# Руководство пользователя JDK

## Руководство по использованию компилятора Java

### Базовая работа с javac

javac — это компилятор Java, отвечающий за компиляцию исходного кода Java в файлы байт-кода. Базовая работа очень проста: javac HelloWorld.java. Это создаст файл Hello.class, который затем можно запустить на JVM.

В реальных проектах разработки обычно необходимо указать больше параметров компиляции. Кчасто используемые параметры включают: -d указывает каталог вывода компиляции, например javac -d out HelloWorld.java сохранит скомпилированные результаты в каталоге out; -classpath (или -cp) указывает путь поиска классов для ссылки на внешние классы во время компиляции; -source указывает версию Java исходного кода; -target указывает версию Java генерируемого байт-кода.

Для больших проектов обычно используются инструменты сборки, такие как Maven или Gradle, для управления процессом компиляции. Эти инструменты автоматически обрабатывают зависимости и параметры компиляции.

### Компиляция нескольких файлов

Если ваш проект содержит несколько исходных файлов Java, вы можете скомпилировать весь каталог сразу. Например: javac -d out src/*.java скомпилирует все файлы Java в каталоге src.

Более сложные структуры проекта могут потребовать использования опции -sourcepath для указания пути поиска исходного кода, чтобы компилятор мог найти зависимые классы. Например: javac -d out -sourcepath src src/Main.java.

### Обработка ошибок компиляции

Если во время компиляции возникают ошибки, javac выдаст подробную информацию об ошибках, включая файл, номер строки и конкретную причину. Распространённые ошибки компиляции включают: синтаксические ошибки (отсутствующая точка с запятой, несовпадающие скобки и т.д.), ошибки типов (несовместимые типы, несуществующие методы и т.д.) и ошибки прав доступа (попытка доступа к закрытым членам и т.д.).

Понимание этих сообщений об ошибках имеет решающее значение для быстрого обнаружения и устранения проблем. Компилятор пытается сообщить о как можно большем количестве ошибок; иногда исправление первой ошибки автоматически устраняет последующие ошибки.

## Руководство по запуску Java-программ

### Базовая работа с командой java

Используйте команду java для запуска скомпилированных Java-программ. Базовая синтаксис: java HelloWorld. Обратите внимание, что расширение .class здесь не требуется. JVM автоматически найдёт Hello.class в текущем каталоге и CLASSPATH.

Если класс находится в пакете, используйте полное имя. Например: java com.example.HelloWorld.

### Classpath (CLASSPATH)

Classpath указывает JVM, где найти файлы классов. Его можно указать с помощью опций -classpath или -cp. Например: java -cp out:lib/* com.example.HelloWorld. В Windows разделитель путей — точка с запятой (;); в Linux и macOS — двоеточие (:).

Если classpath не указан, JVM будет использовать текущий каталог и пути, указанные в переменной среды CLASSPATH.

### Опции JVM

JVM предоставляет множество параметров запуска для управления поведением во время выполнения. Кчасто используемые опции включают: -Xms указывает начальный размер кучи, например -Xms256m устанавливает начальную кучу в 256 МБ; -Xmx указывает максимальный размер кучи, например -Xmx2g устанавливает максимальную кучу в 2 ГБ; -XX:+UseG1GC использует сборщик мусора G1.

В JDK 9 и более поздних версиях можно использовать опцию --add-open для открытия внутреннего доступа к модулям, что полезно при запуске библиотек, которым необходим доступ к внутренним API JDK.

## Руководство по использованию инструментов JDK

### Инструмент архива jar

Инструмент jar используется для создания и управления архивными файлами Java (файлы .jar). Файлы .jar на самом деле представляют собой сжатые файлы в формате ZIP, содержащие скомпилированные файлы .class и метаданные.

Базовая команда для создания файла jar: jar cf myapp.jar -C out . Это создаст myapp.jar из содержимого каталога out. Используйте параметр tf для просмотра содержимого файла jar: jar tf myapp.jar. Используйте xf для извлечения: jar xf myapp.jar.

При создании исполняемого файла jar необходимо указать главный класс в файле MANIFEST.MF. Вы можете добавить собственный MANIFEST.MF с помощью параметра m: jar cfm myapp.jar MANIFEST.MF -C out .

### Генерация документации javadoc

Инструмент javadoc генерирует документацию API в формате HTML из комментариев в исходном коде Java. Стандартная документация библиотеки классов Java генерируется таким образом.

Базовая работа: javadoc -d doc src/*.java. Это сгенерирует документацию для всех файлов Java в каталоге src и сохранит её в каталоге doc.

При написании кода можно использовать определённые форматы комментариев для добавления документации к классам, методам и полям. Распространённые теги комментариев включают: @author автор, @param параметр, @return возвращаемое значение, @throws исключение и т.д.

### Отладчик jdb

jdb — это отладчик Java через командную строку, предоставляющий базовые функции отладки. Запуск отладки: jdb MyClass. Установка точек останова: stop in MyClass.method. Просмотр точек останова: clear. Запуск программы: run. Продолжение выполнения: cont. Пошаговое выполнение: step или next. Просмотр переменных: print varname. Просмотр исходного кода: list.

Хотя jdb не такой мощный, как графические отладчики, его можно использовать в средах без IDE или для удалённой отладки на серверах.

## Оптимизация производительности JVM

### Выбор сборщика мусора

JDK предоставляет несколько сборщиков мусора, каждый из которых имеет различные характеристики и сценарии применения:

Serial GC: Однопоточный сборщик, подходит для небольших приложений и клиентских приложений. Активировать с помощью -XX:+UseSerialGC.

Parallel GC (Throughput GC): Многопоточный сборщик, подходит для приложений с высокой пропускной способностью. Активировать с помощью -XX:+UseParallelGC.

CMS GC: Параллельный сборщик с пометкой и очисткой, стремится к низким паузам. Активировать с помощью -XX:+UseConcMarkSweepGC (устарел в JDK 14).

G1 GC: Сборщик мусора с разделением на области, подходит для приложений с большой памятью и стал вариантом по умолчанию в JDK 9 и более поздних версиях. Рекомендуется использовать параметры по умолчанию или -XX:+UseG1GC.

ZGC: Сборщик мусора с низкой задержкой, поддерживает память кучи на уровне терабайт, паузы не превышают 10 мс. Активировать с помощью -XX:+UseZGC (JDK 15+).

Shenandoah: Ещё один сборщик мусора с низкой задержкой, паузы не зависят от размера кучи. Активировать с помощью -XX:+UseShenandoahGC.

### Настройка памяти

Правильная настройка размера кучи памяти имеет решающее значение для производительности приложения. Распространённые параметры конфигурации:

-Xms: Начальный размер кучи, напр., -Xmx2g.
-Xmx: Максимальный размер кучи, напр., -Xmx4g.
-Xmn: Размер молодого поколения, напр., -Xmn1g.
-XX:MetaspaceSize: Начальный размер метапространства.
-XX:MaxMetaspaceSize: Максимальный размер метапространства.

Рекомендуется установить -Xms и -Xmx на одно и то же значение, чтобы избежать накладных расходов на изменение размера кучи во время выполнения.

### Инструменты мониторинга производительности

JDK предоставляет богатый набор инструментов мониторинга и диагностики производительности:

jps: Показывает список процессов Java, базовое использование jps -l.
jstat: Показывает статистику JVM, например статистику сборки мусора: jstat -gcutil pid 1000.
jstack: Выводит информацию о трассировке стека потоков Java, используется для диагностики блокировок: jstack pid.
jmap: Показывает информацию о памяти кучи Java, может генерировать файлы дампа кучи для анализа: jmap -dump:format=b,file=heap.bin pid.
jcmd: Многофункциональный диагностический инструмент, может выполнять различные диагностические команды: jcmd pid GC.heap_info.

JConsole и VisualVM: Графические инструменты мониторинга и диагностики, могут отслеживать память, потоки, использование процессора и т.д.

## Система модулей

### Обзор модулей

Начиная с JDK 9, Java представила систему модулей (Project Jigsaw), которая делит JDK и приложения на модули. Каждый модуль имеет чёткие зависимости и обеспечивает лучшую инкапсуляцию и удобство сопровождения.

Сам JDK разделён на несколько модулей, таких как java.base, java.sql, java.xml. Приложения могут создавать собственные модули, объявляя имя модуля, зависимости и экспортируемые пакеты через файлы module-info.java.

### Параметры, связанные с модулями

JDK предоставляет несколько параметров командной строки, связанных с модулями:

--module-path (или -p): Указывает путь к модулю, аналогично classpath, но для модулей.
--add-modules: Добавляет модули в набор модулей по умолчанию.
--add-exports: Открывает указанные пакеты модулей для доступа других модулей.
--add-opens: Открывает указанные пакеты модулей для доступа через отражение.

Эти параметры очень важны в определённых ситуациях, например, при использовании отражения для доступа к внутренним API JDK или при использовании библиотек, несовместимых с модульностью.

## Примеры конфигурации

### Конфигурация среды разработки

Для повседневных сред разработки рекомендуемая конфигурация включает: достаточный размер кучи (-Xmx2g или выше), использование сборщика мусора G1 (по умолчанию), включение совместного использования данных классов (-Xshare:classes) для ускорения запуска. Если вы используете IDE, вы можете установить эти параметры непосредственно в конфигурации запуска IDE.

### Конфигурация производственной среды

Конфигурация производственной среды должна быть адаптирована к конкретной ситуации приложения. Основные принципы включают: установку размера кучи в соответствии с потребностями приложения в памяти, обычно резервируя 25-30% памяти для операционной системы и других процессов; выбор подходящего сборщика мусора, G1 обычно хороший выбор; для приложений, требующих низкой задержки, рассмотрите возможность использования ZGC или Shenandoah.

Рекомендуется включить ведение журнала GC в производственных средах, записывающее подробную информацию о сборке мусора для анализа проблем и оптимизации производительности. Пример конфигурации журнала GC: -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M.

### Конфигурация контейнера Docker

При запуске Java-приложений в средах контейнеров обратите внимание на осведомлённость JVM о памяти. Версии JDK 10+ по умолчанию осведомлены об ограничениях памяти контейнера, но в более ранних версиях может потребоваться ручная настройка.

Рекомендуемая конфигурация: установите правильное максимальное значение кучи (-XX:MaxRAMPercentage=75.0), убедитесь, что JVM использует не более 75% доступной памяти контейнера; используйте -XX:+UseContainerSupport (включено по умолчанию), чтобы JVM распознавала ограничения ресурсов контейнера.

Для облачных приложений рассмотрите возможность использования GraalVM Native Image для компиляции приложений в собственные исполняемые файлы, что может значительно сократить время запуска и использование памяти.

## Различия в использовании по версиям

### JDK 8

JDK 8 — последняя полностью бесплатная версия Oracle JDK для коммерческого использования. Основные функции включают лямбда-выражения, API стримов и новый API даты/времени. Установка относительно проста, используется традиционный установщик. Обратите внимание, что JAVA_HOME должен указывать на каталог JDK (не JRE). Некоторые команды и инструменты могут вести себя по-разному в новых версиях.

### JDK 11

JDK 11 ознаменовал начало платных функций Java. Основные функции включают вывод типов локальных переменных (var), новый HTTP-клиент и ZGC (экспериментальный). JRE больше не предлагается как отдельная загрузка; доступен только JDK. Команда java может запускать программы с одним исходным файлом напрямую без компиляции.

### JDK 17

JDK 17 — последняя бесплатная версия Oracle JDK по NFTC. Основные функции включают запечатанные классы, текстовые блоки и выражения switch. Установка и использование аналогичны JDK 11. Интерфейс RandomGenerator предоставляет новые возможности генерации случайных чисел. API доступа к внешней памяти вошёл в фазу инкубации.

### JDK 21

JDK 21 представляет важные функции, включая виртуальные потоки (стабильные), сопоставление с образцом для switch (стабильное) и записи (стабильные). Виртуальные потоки требуют других подходов к параллельному программированию. Приложения, использующие виртуальные потоки, могут потребовать корректировки настроек кучи. Оптимизация производительности должна учитывать характеристики виртуальных потоков.

### JDK 25

JDK 25 — последняя версия с новейшими языковыми функциями и улучшениями производительности. Функции могут всё ещё находиться в режиме предварительного просмотра или быть экспериментальными. Не рекомендуется для производственных сред, требующих долгосрочной поддержки. Конфигурация и использование аналогичны JDK 21, но с большим количеством новых функций, которые могут потребовать дополнительного обучения.
