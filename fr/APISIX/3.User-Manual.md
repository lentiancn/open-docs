# APISIX User Manual

This document introduces the core features and usage of APISIX.

## Basic Concepts

### Route

Route is the core concept in APISIX, used to match client requests and forward them to upstream services.

```bash
# Create a route
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "my-route",
  "uri": "/api/v1/*",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "backend1.example.com:8080": 1,
      "backend2.example.com:8080": 1
    }
  }
}'
```

### Upstream

Upstream is a set of nodes providing the same service. APISIX supports multiple load balancing strategies:

```bash
# Create upstream
curl -i "http://127.0.0.1:9180/apisix/admin/upstreams" -X PUT -d '
{
  "id": "my-upstream",
  "type": "roundrobin",
  "nodes": {
    "host1:8080": 1,
    "host2:8080": 1
  }
}'
```

### Plugin

APISIX provides a rich plugin system that executes various functions during request processing.

## Route Matching Rules

### 1. Exact Match

```json
{
  "uri": "/api/users"
}
```

### 2. Parameter Match

```json
{
  "uri": "/api/users/:id"
}
```

### 3. Regex Match

```json
{
  "uri": "/api/*",
  "vars": [["uri", "==", "/api/users"]]
}
```

### 4. Conditional Match

```json
{
  "uri": "/api/*",
  "vars": [
    ["http_x_api_version", "==", "v2"],
    ["arg_page", ">", 1]
  ]
}
```

## Load Balancing Strategies

### 1. Round Robin

```json
{
  "type": "roundrobin"
}
```

### 2. Hash

```json
{
  "type": "hash",
  "key": "remote_addr"
}
```

### 3. Consistent Hash

```json
{
  "type": "chash",
  "key": "remote_addr"
}
```

## Common Plugins

### 1. Rate Limiting (limit-req)

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rate-limit-route",
  "uri": "/api/*",
  "plugins": {
    "limit-req": {
      "rate": 100,
      "burst": 50,
      "key": "remote_addr"
    }
  },
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "backend:8080": 1
    }
  }
}'
```

### 2. Key Authentication (key-auth)

```bash
  },
  "upstream": {
    "type": "roundrobin",

  }

```

### 3. JWT Authentication (jwt-auth)


## Service Discovery

APISIX supports multiple service discovery mechanisms:

### 1. Consul

```yaml
discovery:
  consul:
    servers:
      - "http://127.0.0.1:8500"
```

### 2. Nacos

```yaml
discovery:
  nacos:
    host:
      - "http://127.0.0.1:8848"
    prefix: /nacos/v1
```

### 3. Eureka

```yaml
discovery:
  eureka:
    host:
      - "http://127.0.0.1:8761"
    port: 8761
```

## Health Checks

APISIX supports both active and passive health checks:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/upstreams" -X PUT -d '
{
  "id": "health-check-upstream",
  "type": "roundrobin",
  "nodes": {
    "backend1:8080": 1,
    "backend2:8080": 1
  },
  "checks": {
    "active": {
      "type": "http",
      "http_path": "/health",
      "interval": 5,
      "timeout": 2
    },
    "passive": {
      "type": "http",
      "healthy": {
        "http_statuses": [200, 201, 202, 203, 204],
        "successes": 3
      },
      "unhealthy": {
        "http_statuses": [500, 502, 503, 504],
        "http_failures": 3,
        "timeouts": 3
      }
    }
  }
}'
```

## Best Practices

1. **Use Upstream**: Separate upstream configuration for easier management and reuse
2. **Use Plugins Reasonably**: Enable plugins as needed to avoid unnecessary performance overhead
3. **Configure Health Checks**: Ensure high availability of services
4. **Use Service Discovery**: Automatically sense service changes in dynamic environments
5. **Monitor and Log**: Integrate Prometheus and logging systems for problem diagnosis
