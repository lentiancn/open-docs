# Hadoop Häufig gestellte Fragen

## HDFS-bezogene Probleme

### Was tun bei unzureichendem HDFS-Speicherplatz?

Unzureichender HDFS-Speicherplatz ist ein häufiges Problem, das mehrere Ursachen haben kann. Überprüfen Sie zunächst die DataNode-Festplattennutzung:

```bash
hdfs dfsadmin -report
```

Wenn der Speicherplatz tatsächlich unzureichend ist, können folgende Maßnahmen ergriffen werden: Löschen Sie nicht mehr benötigte historische Daten; Verwenden Sie hdfs balancer zum Ausbalancieren der Datenverteilung und zum Freigeben einiger Knoten; Fügen Sie neue DataNode-Knoten hinzu, um die Cluster-Speicherkapazität zu erweitern; Reduzieren Sie den Replikationsfaktor, um den Speicheroverhead zu verringern (Datensicherheit abwägen).

```bash
# Dateien löschen
hdfs dfs -rm -r /user/olddata

# Daten ausbalancieren
hdfs balancer -threshold 10
```

### Wie man mit NameNode-Speicherproblemen umgeht

NameNode speichert alle Metadaten im Speicher; bei zu vielen Dateien kann der Speicher knapp werden. Dies kann durch folgende Maßnahmen gemildert werden: Erhöhen Sie die NameNode-Speicherkonfiguration in hadoop-env.sh durch Festlegen des -Xmx-Parameters in HADOOP_NAMENODE_OPTS; Aktivieren Sie die NameNode-Snapshot-Funktion, um regelmäßig schreibgeschützte Kopien wichtiger Daten zu erstellen; Upgrade auf Hadoop 2.x oder höher, das über eine bessere Metadatenverwaltungsoptimierung verfügt.

```bash
# In hadoop-env.sh festlegen
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### Was tun, wenn DataNode nicht startet

DataNode-Startfehler können mehrere Ursachen haben. Überprüfen Sie zunächst die DataNode-Protokolle:

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

Häufige Ursachen und Lösungen: Unzureichender Festplattenspeicher – Festplatten reinigen oder neue hinzufügen; Berechtigungsprobleme – Stellen Sie sicher, dass der Hadoop-Benutzer Lese-/Schreibzugriff auf Datenverzeichnisse hat; Inkonsistente Cluster-ID – Wenn Sie einen neuen DataNode hinzufügen, führen Sie hdfs namenode -upgrade auf dem DataNode aus oder löschen Sie das Speicherverzeichnis und formatieren Sie neu.

```bash
# DataNode-Verzeichnisberechtigungen überprüfen
ls -la /usr/local/hadoop/data/

# DataNode-Speicherverzeichnis löschen und neu starten
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### Wie man versehentlich gelöschte Dateien wiederherstellt

HDFS bietet Papierkorbfunktionalität zum Wiederherstellen gelöschter Dateien. Bestätigen Sie zunächst, dass der Papierkorb aktiviert ist:

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

Wenn Sie versehentliches Löschen feststellen, stellen Sie aus dem Papierkorb wieder her:

```bash
# Datei wiederherstellen
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

Wenn der Papierkorb geleert wurde oder die Aufbewahrungsfrist abgelaufen ist, können die Daten nicht wiederhergestellt werden. Daher wird im täglichen Betrieb empfohlen: Papierkorbfunktion aktiviert lassen (standardmäßig aktiviert); Wichtige Daten sichern; Vor dem Löschen sorgfältig überprüfen.

### Wie man mit beschädigten HDFS-Blöcken umgeht

Datenblöcke können durch Festplattenfehler oder Netzwerkprobleme beschädigt werden. HDFS hat integrierte Datenintegritätserkennung – wenn beschädigte Blöcke erkannt werden, werden automatisch Daten aus anderen Replikaten zur Wiederherstellung repliziert. Manuelle Überprüfung und Reparatur:

```bash
# Dateisystemintegrität überprüfen
hdfs fsck / -files -blocks -locations

# Beschädigte Blockdateien löschen
hdfs fsck / -delete
```

## MapReduce-bezogene Probleme

### Was tun, wenn Jobs langsam ausgeführt werden?

Langsame MapReduce-Jobs können mehrere Ursachen haben. Überprüfen Sie zunächst die Jobverlaufsprotokolle und analysieren Sie den Zeitaufwand für jede Phase:

```bash
# Job-Details anzeigen
yarn logs -applicationId application_xxx -logFiles *

# Im JobHistory Server anzeigen
mapred job -history job_history_file
```

Häufige Optimierungsrichtungen: Erhöhen Sie die Anzahl der Reduce-Aufgaben, um den Parallelitätsgrad zu verbessern; Passen Sie die JVM-Heap-Größe an, um OOM zu vermeiden; Aktivieren Sie Combiner, um das Netzwerkübertragungsvolumen zu reduzieren; Optimieren Sie Datenschiefe durch benutzerdefinierte Partitionierer zum Ausbalancieren der Last; Wählen Sie geeignete Dateiformate (Parquet, ORC usw.).

### Umgang mit Datenschiefe

Datenschiefe bedeutet, dass Daten während der Reduce-Phase ungleichmäßig verteilt sind, sodass einige Reduce-Aufgaben viel länger ausgeführt werden als andere. Lösungen: Verwenden Sie benutzerdefinierte Partitionierer, um Daten basierend auf der Geschäftslogik neu zu verteilen; Fügen Sie zufällige Präfixe hinzu, um Daten zunächst auf mehrere Reduces zu verteilen, und aggregieren Sie dann sekundär; Verwenden Sie Key-Salting – entfernen Sie das Präfix nach der Aggregation.

```java
// Beispiel für benutzerdefinierten Partitionierer
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // Basierend auf Geschäftslogik neu verteilen
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### Fehlerbehebung bei MapReduce-Job-Fehlern

Wenn Jobs fehlschlagen, überprüfen Sie zunächst die Fehlerprotokolle, um die Fehlerursache zu ermitteln:

```bash
# Job-Status anzeigen
yarn application -status application_xxx

# Aufgabenfehlerursachen anzeigen
mapred job -status job_id
```

Häufige Fehlerursachen: Java OutOfMemory (OOM) – erhöhen Sie mapreduce.map.memory.mb und mapreduce.reduce.memory.mb; Eingabedatenformatfehler – überprüfen Sie, ob die Daten dem erwarteten Format entsprechen; Benutzerdefinierter Code-Ausnahmefehler – überprüfen Sie die Codelogik und Ausnahmemeldungen; Netzwerk-Timeout – erhöhen Sie den Parameter mapreduce.task.timeout.

## YARN-bezogene Probleme

### Was tun, wenn ResourceManager nicht startet?

ResourceManager-Startfehler werden normalerweise durch Konfigurationsfehler oder Portkonflikte verursacht. Überprüfen Sie die Protokolle:

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

Häufige Probleme und Lösungen: Port wird verwendet – ändern Sie den ResourceManager-Port in yarn-site.xml; ZooKeeper-Konfigurationsfehler (bei HA) – überprüfen Sie die zkfc-bezogene Konfiguration; Unzureichender Speicher – stellen Sie sicher, dass der Computer über ausreichend physischen Speicher verfügt; Konfigurationsdateifehler – überprüfen Sie sorgfältig yarn-site.xml und capacity-scheduler.xml.

```bash
# Portnutzung überprüfen
netstat -tlnp | grep 8032
```

### Umgang mit geringer Cluster-Ressourcenauslastung

Geringe Cluster-Ressourcenauslastung kann folgende Ursachen haben: Unvernünftige Job-Übermittlungsmuster – Jobs sind zu klein, sodass der Planungsaufwand die Berechnungszeit übersteigt; Ungeeignete Scheduler-Konfiguration – unvernünftige Warteschlangenressourcenzuweisung; Schlechte Datenlokalität – Eingabedaten und Computeknoten nicht am selben Standort.

Optimierungsvorschläge: Kleine Jobs stapelweise übermitteln; Warteschlangenkapazität und Prioritäten vernünftig konfigurieren; Datenlayout optimieren, um Datenlokalität zu verbessern; Geeigneten Scheduler verwenden (Capacity Scheduler oder Fair Scheduler).

### Wie man die Benutzerressourcennutzung einschränkt

Verwenden Sie YARN-Warteschlangen und Zugriffskontrollen, um die Benutzerressourcennutzung einzuschränken:

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- Maximales Vielfaches der Warteschlangenressourcen für einzelnen Benutzer -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- Maximale Anzahl laufender Jobs in Warteschlange -->
</property>
```

## Cluster-Betriebsprobleme

### Wie man Hadoop-Cluster überwacht

Hadoop bietet mehrere Überwachungsmethoden: Web-Benutzeroberfläche (NameNode-Port 9870, ResourceManager-Port 8088); JMX-Überwachung (Java Management Extensions); Externe Überwachungssysteme wie Ganglia, Prometheus; Befehlszeilentools (hdfs dfsadmin, yarn node usw.).

Es wird empfohlen, eine vollständige Überwachungsplattform mit Prometheus + Grafana aufzubauen – sammeln Sie Hadoop-JMX-Metriken und visualisieren Sie diese.

### Wie man Cluster-Upgrades durchführt

Hadoop-Cluster-Upgrades erfordern sorgfältige Vorgänge. Empfohlene Schritte: Zuerst die Kompatibilität der neuen Version in einer Testumgebung überprüfen; NameNode-Metadaten und HDFS-Daten sichern; DataNode-Knoten einzeln aktualisieren, dann NameNode aktualisieren; Rolling Upgrade verwenden, um Ausfallzeiten zu reduzieren.

```bash
# DataNode-Rolling-Upgrade
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Hadoop upgraden
tar -xzf hadoop-new.tar.gz -C /usr/local/
# DataNode starten
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### Wie man NameNode-Hochverfügbarkeit konfiguriert

NameNode HA vermeidet NameNode-Single-Points-of-Failure. Konfigurationsschritte: Konfigurieren Sie zwei NameNodes in hdfs-site.xml; Konfigurieren Sie das gemeinsame Bearbeitungsverzeichnis (normalerweise ZooKeeper oder NFS); Starten Sie den ZooKeeper Failover Controller (ZKFC).

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

Nach der Konfiguration initialisieren Sie ZKFC:

```bash
hdfs zkfc -formatZK
```

### Umgang mit Knoten-Ausmusterung

Die Ausmusterung von Knoten erfordert sorgfältigen Umgang, um Datenverluste zu vermeiden. Verwenden Sie die Decommission-Funktion, um Knoten sicher außer Betrieb zu nehmen:

```bash
# Knoten zur Exclude-Datei in Workers hinzufügen
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# Knotenliste aktualisieren
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

Warten Sie, bis die Datenreplikation abgeschlossen ist – der Knoten wechselt automatisch in den Decommissioned-Status. Bestätigen Sie, dass Daten auf andere Knoten repliziert wurden, bevor Sie den Knoten herunterfahren.

## Entwicklungsbezogene Probleme

### Wie man MapReduce-Programme debuggt

Das Debugging von MapReduce-Programmen ist schwierig. Empfohlene Methoden: Testen Sie zuerst die Logik im lokalen Modus; Verwenden Sie Counter, um wichtige Metriken aufzuzeichnen; Fügen Sie Protokollausgaben hinzu – drucken Sie wichtige Variablenwerte in Map- oder Reduce-Methoden; Verwenden Sie MRUnit für Komponententests.

```java
// Counter verwenden
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### Auswahl von InputFormat und OutputFormat

InputFormat bestimmt, wie Eingabedaten gelesen werden: TextInputFormat (standardmäßig, liest zeilenweise); KeyValueTextInputFormat (teilt nach Tab); SequenceFileInputFormat (Binärformat); DBInputFormat (liest aus Datenbank).

OutputFormat bestimmt, wie Ergebnisse ausgegeben werden: TextOutputFormat (standardmäßig, Textformat); SequenceFileOutputFormat (Binärformat); NullOutputFormat (keine Dateiausgabe); DBOutputFormat (gibt in Datenbank aus).

### Umgang mit dem Problem kleiner Dateien

Große Mengen kleiner Dateien erhöhen den NameNode-Speicherdruck und reduzieren die Verarbeitungseffizienz. Lösungen: Verwenden Sie Hadoop Archives (HAR), um kleine Dateien zu paketieren; Verwenden Sie SequenceFile, um kleine Dateien zusammenzuführen; Verwenden Sie HBase, um große Mengen kleiner Dateien zu speichern; Passen Sie die Blockgröße an, um die Anzahl kleiner Dateien zu reduzieren.
