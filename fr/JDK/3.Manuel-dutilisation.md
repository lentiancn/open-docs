# JDK Manuel d'utilisation

## Guide d'utilisation du compilateur Java

### Utilisation de base de javac

javac est le compilateur Java, responsable de la compilation du code source Java en fichiers bytecode. L'utilisation de base est très simple : javac HelloWorld.java. Cela générera un fichier Hello.class, qui peut ensuite être exécuté sur la JVM.

Dans les projets de développement réels, vous devez généralement spécifier plus d'options de compilation. Les options courantes incluent : -d spécifie le répertoire de sortie de compilation, par exemple javac -d out HelloWorld.java placera les résultats compilés dans le répertoire out ; -classpath (ou -cp) spécifie le chemin de recherche de classes pour référencer des classes externes pendant la compilation ; -source spécifie la version Java du code source ; -target spécifie la version Java du bytecode généré.

Pour les grands projets, on utilise généralement des outils de construction comme Maven ou Gradle pour gérer le processus de compilation. Ces outils gèrent automatiquement les dépendances et les options de compilation.

### Compilation de plusieurs fichiers

Si votre projet contient plusieurs fichiers source Java, vous pouvez compiler tout un répertoire à la fois. Par exemple : javac -d out src/*.java compilera tous les fichiers Java du répertoire src.

Des structures de projet plus complexes peuvent nécessiter l'utilisation de l'option -sourcepath pour spécifier le chemin de recherche du code source, permettant au compilateur de trouver les classes dépendantes. Par exemple : javac -d out -sourcepath src src/Main.java.

### Gestion des erreurs de compilation

Si des erreurs se produisent pendant la compilation, javac émettra des informations d'erreur détaillées, y compris le fichier, le numéro de ligne et la cause spécifique. Les erreurs de compilation courantes incluent : erreurs de syntaxe (point-virgule manquant, parenthèses non appariées, etc.), erreurs de type (types incompatibles, méthodes inexistantes, etc.), et erreurs de permissions d'accès (tentative d'accès à des membres privés, etc.).

Comprendre ces messages d'erreur est crucial pour localiser et résoudre les problèmes rapidement. Le compilateur essaie de signaler autant d'erreurs que possible ; parfois, corriger la première erreur fera disparaître les erreurs suivantes.

## Guide d'exécution de programmes Java

### Utilisation de base de la commande java

Utilisez la commande java pour exécuter des programmes Java compilés. La syntaxe de base est : java HelloWorld. Notez que l'extension .class n'est pas nécessaire ici. La JVM recherchera automatiquement Hello.class dans le répertoire actuel et CLASSPATH.

Si la classe est dans un package, utilisez le nom pleinement qualifié. Par exemple : java com.example.HelloWorld.

### Classpath (CLASSPATH)

Classpath indique à la JVM où trouver les fichiers de classe. Il peut être spécifié en utilisant les options -classpath ou -cp. Par exemple : java -cp out:lib/* com.example.HelloWorld. Sous Windows, le séparateur de chemin est le point-virgule (;) ; sous Linux et macOS, c'est le deux-points (:).

Si classpath n'est pas spécifié, la JVM utilisera le répertoire actuel et les chemins spécifiés dans la variable d'environnement CLASSPATH.

### Options de JVM

La JVM fournit de nombreuses options de démarrage pour contrôler le comportement d'exécution. Les options courantes incluent : -Xms spécifie la taille initiale du tas, comme -Xms256m définit le tas initial à 256 Mo ; -Xmx spécifie la taille maximale du tas, comme -Xmx2g définit le tas maximum à 2 Go ; -XX:+UseG1GC utilise le nettoyeur de mémoire G1.

Sous JDK 9 et ultérieur, vous pouvez utiliser l'option --add-opens pour ouvrir l'accès interne au module, ce qui est utile lors de l'exécution de bibliothèques qui doivent accéder aux API internes de JDK.

## Guide d'utilisation des outils JDK

### Outil d'archive jar

L'outil jar est utilisé pour créer et gérer des fichiers d'archivage Java (fichiers .jar). Les fichiers .jar sont en fait des fichiers compressés au format ZIP contenant des fichiers .class compilés et des métadonnées.

La commande de base pour créer un fichier jar : jar cf myapp.jar -C out . Cela crée myapp.jar à partir du contenu du répertoire out. Utilisez le paramètre tf pour voir le contenu du fichier jar : jar tf myapp.jar. Utilisez xf pour extraire : jar xf myapp.jar.

Lors de la création d'un fichier jar exécutable, vous devez spécifier la classe principale dans le fichier MANIFEST.MF. Vous pouvez ajouter un MANIFEST.MF personnalisé en utilisant le paramètre m : jar cfm myapp.jar MANIFEST.MF -C out .

### Génération de documentation javadoc

L'outil javadoc génère une documentation d'API au format HTML à partir de commentaires dans le code source Java. La documentation standard de la bibliothèque de classes Java est générée de cette manière.

Utilisation de base : javadoc -d doc src/*.java. Cela générera de la documentation pour tous les fichiers Java dans le répertoire src et la placera dans le répertoire doc.

Lors de l'écriture de code, l'utilisation de formats de commentaire spécifiques peut ajouter de la documentation pour les classes, méthodes et champs. Les balises de commentaire courantes incluent : @author auteur, @param paramètre, @return valeur de retour, @throws exception, etc.

### Débogueur jdb

jdb est le débogueur en ligne de commande Java, fournissant des fonctionnalités de débogage de base. Démarrer le débogage : jdb MyClass. Définir des points d'arrêt : stop in MyClass.method. Voir les points d'arrêt : clear. Exécuter le programme : run. Continuer l'exécution : cont. Exécuter pas à pas : step ou next. Voir les variables : print varname. Lister le code source : list.

Bien que jdb ne soit pas aussi puissant que les débogueurs graphiques, il peut être utilisé dans des environnements sans IDE ou pour le débogage distant sur les serveurs.

## Optimisation des performances JVM

### Sélection du nettoyeur de mémoire

JDK fournit plusieurs nettoyeurs de mémoire, chacun ayant différentes caractéristiques et scénarios d'application :

Serial GC : Nettoyeur mono-thread, adapté aux petites applications et applications clientes. Activer avec -XX:+UseSerialGC.

Parallel GC (Throughput GC) : Nettoyeur multi-thread, adapté aux applications cherchant un haut débit. Activer avec -XX:+UseParallelGC.

CMS GC : Nettoyeur concurrent par balayage, cherchant de faibles temps de pause. Activer avec -XX:+UseConcMarkSweepGC (Obsolète dans JDK 14).

G1 GC : Nettoyeur de mémoire partitionné, adapté aux applications à grande mémoire, et est devenu l'option par défaut dans JDK 9 et ultérieur. Il est recommandé d'utiliser les paramètres par défaut ou -XX:+UseG1GC.

ZGC : Nettoyeur à faible latence, prend en charge la mémoire de tas au niveau To, temps de pause ne dépassant pas 10 ms. Activer avec -XX:+UseZGC (JDK 15+).

Shenandoah : Un autre nettoyeur à faible latence, temps de pause indépendants de la taille du tas. Activer avec -XX:+UseShenandoahGC.

### Configuration de la mémoire

Configurer correctement la taille de la mémoire du tas est essentiel pour les performances de l'application. Options de configuration courantes :

-Xms : Taille initiale du tas, ex., -Xms2g.
-Xmx : Taille maximale du tas, ex., -Xmx4g.
-Xmn : Taille de la jeune génération, ex., -Xmn1g.
-XX:MetaspaceSize : Taille initiale du métaspace.
-XX:MaxMetaspaceSize : Taille maximale du métaspace.

Il est recommandé de définir -Xms et -Xmx à la même valeur pour éviter la surcharge de l'ajustement du tas à l'exécution.

### Outils de surveillance des performances

JDK fournit des outils riches de surveillance et de diagnostic des performances :

jps : Affiche la liste des processus Java, utilisation de base jps -l.
jstat : Affiche les statistiques JVM, ex., statistiques de collecte des ordures : jstat -gcutil pid 1000.
jstack : Imprime les informations de trace de pile des threads, utilisé pour diagnostiquer les blocages : jstack pid.
jmap : Affiche les informations de mémoire du tas Java, peut générer des fichiers de vidage de tas pour analyse : jmap -dump:format=b,file=heap.bin pid.
jcmd : Outil de diagnostic multifonction, peut exécuter diverses commandes de diagnostic : jcmd pid GC.heap_info.

JConsole et VisualVM : Outils de surveillance et de diagnostic graphiques, peuvent surveiller la mémoire, les threads, l'utilisation du CPU, etc.

## Système de modules

### Vue d'ensemble des modules

À partir de JDK 9, Java a introduit le système de modules (Project Jigsaw), divisant JDK et applications en modules. Chaque module a des dépendances claires, offrant une meilleure encapsulation et maintenabilité.

JDK lui-même est divisé en plusieurs modules comme java.base, java.sql, java.xml. Les applications peuvent créer leurs propres modules, déclarant le nom du module, les dépendances et les export via des fichiers module-info.java.

### Options liées aux modules

JDK fournit plusieurs options de ligne de commande liées aux modules :

--module-path (ou -p) : Spécifie le chemin du module, similaire à classpath mais pour les modules.
--add-modules : Ajoute des modules à l'ensemble de modules par défaut.
--add-exports : Ouvre les packages de modules spécifiés pour que d'autres modules y accèdent.
--add-opens : Ouvre les packages de modules spécifiés pour l'accès par réflexion.

Ces options sont très importantes dans certaines situations, comme l'utilisation de la réflexion pour accéder aux API internes de JDK ou l'utilisation de bibliothèques non compatibles avec la modularisation.

## Exemples de configuration courants

### Configuration de l'environnement de développement

Pour les environnements de développement quotidiens, la configuration recommandée comprend : suffisamment de mémoire de tas (-Xmx2g ou plus), utiliser le nettoyeur de mémoire G1 (par défaut), activer le partage des données de classe (-Xshare:classes) pour accélérer le démarrage. Si vous utilisez l'IDE, vous pouvez définir ces options directement dans la configuration d'exécution de l'IDE.

### Configuration de l'environnement de production

La configuration de l'environnement de production doit être ajustée en fonction de la situation spécifique de l'application. Les principes de base incluent : définir la taille du tas selon les besoins en mémoire de l'application, généralement en réservant 25-30% de mémoire pour le système d'exploitation et d'autres processus ; choisir un nettoyeur de mémoire approprié, G1 est généralement un bon choix ; pour les applications cherchant une faible latence, envisagez d'utiliser ZGC ou Shenandoah.

Il est recommandé d'activer la journalisation GC dans les environnements de production, enregistrant des informations détaillées de collecte des ordures pour l'analyse des problèmes et l'optimisation des performances. Exemple de configuration de journalisation GC : -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M.

### Configuration de conteneur Docker

Lors de l'exécution d'applications Java dans des environnements de conteneurs, faites attention à la conscience de la mémoire de la JVM. Les versions de JDK 10+ sont conscientes des limites de mémoire de conteneur par défaut, mais peuvent nécessiter une configuration manuelle dans les versions antérieures.

Configuration recommandée : définir la valeur maximale du tas correcte (-XX:MaxRAMPercentage=75.0), assurant que la JVM n'utilise pas plus de 75% de la mémoire disponible du conteneur ; utiliser -XX:+UseContainerSupport (activé par défaut) pour permettre à la JVM d'être consciente des limites des ressources du conteneur.

Pour les applications cloud natives, envisagez d'utiliser GraalVM Native Image pour compiler des applications en exécutables natifs, ce qui peut réduire considérablement le temps de démarrage et l'utilisation de la mémoire.

## Différences d'utilisation par version

### JDK 8

JDK 8 est la dernière version d'Oracle JDK entièrement gratuite pour utilisation commerciale. Les fonctionnalités principales incluent les expressions lambda, l'API streams et la nouvelle API date/heure. L'installation est relativement simple, utilisant l'installeur traditionnel. Notez que JAVA_HOME doit pointer vers le répertoire JDK (pas JRE). Certaines commandes et outils peuvent se comporter différemment dans les versions plus récentes.

### JDK 11

JDK 11 a marqué le début des fonctionnalités payantes de Java. Les fonctionnalités principales incluent l'inférence de type de variable locale (var), le nouveau client HTTP API et ZGC (expérimental). Le JRE n'est plus fourni en téléchargement séparé ; seul JDK est disponible. La commande java peut exécuter des programmes source à fichier unique directement sans compilation.

### JDK 17

JDK 17 est la dernière version Oracle JDK gratuite sous NFTC. Les fonctionnalités principales incluent les classes scellées, les blocs de texte et les expressions switch. L'installation et l'utilisation sont similaires à JDK 11. L'interface RandomGenerator fournit de nouvelles capacités de génération de nombres aléatoires. L'API d'accès à la mémoire étrangère est entrée en phase d'incubation.

### JDK 21

JDK 21 introduit des fonctionnalités importantes, notamment les threads virtuels (stables), la correspondance de motifs pour switch (stable) et les records (stables). Les threads virtuels nécessitent différentes approches de programmation concurrente. Les applications utilisant des threads virtuels peuvent nécessiter des ajustements de configuration de tas. L'optimisation des performances doit considérer les caractéristiques des threads virtuels.

### JDK 25

JDK 25 est la dernière version avec les dernières fonctionnalités de langage et les améliorations de performances. Les fonctionnalités peuvent encore être en état de prévisualisation ou expérimental. Non recommandé pour les environnements de production nécessitant un support à long terme. La configuration et l'utilisation sont similaires à JDK 21, mais avec plus de nouvelles fonctionnalités qui peuvent nécessiter un apprentissage supplémentaire.
