# JDK 사용자 매뉴얼

## Java 컴파일러 사용 가이드

### javac 기본 사용법

javac는 Java 소스 코드를 바이트코드 파일로 컴파일하는 Java 컴파일러입니다. 기본 사용법은 매우 간단합니다: javac HelloWorld.java. 이렇게 하면 Hello.class 파일이 생성되고 JVM에서 실행할 수 있습니다.

실제 개발 프로젝트에서는 일반적으로 더 많은 컴파일 옵션을 지정해야 합니다. 일반적인 옵션에는 다음이 포함됩니다: -d는 컴파일 출력 디렉토리를 지정합니다(예: javac -d out HelloWorld.java는 컴파일된 결과를 out 디렉토리에 저장). -classpath(또는 -cp)는 컴파일 시 외부 클래스를 참조하기 위한 클래스 검색 경로를 지정합니다. -source는 소스 코드의 Java 버전을 지정합니다. -target은 생성되는 바이트코드의 Java 버전을 지정합니다.

대규모 프로젝트에서는 일반적으로 Maven 또는 Gradle과 같은 빌드 도구를 사용하여 컴파일 프로세스를 관리합니다. 이러한 도구는 종속성과 컴파일 옵션을 자동으로 처리합니다.

### 여러 파일 컴pilizing

프로젝트에 여러 Java 소스 파일이 있는 경우 한 번에 전체 디렉토리를 컴파일할 수 있습니다. 예: javac -d out src/*.java는 src 디렉토리의 모든 Java 파일을 컴파일합니다.

더 복잡한 프로젝트 구조에서는 컴파일러가 종속 클래스를 찾을 수 있도록 소스 코드 검색 경로를 지정하기 위해 -sourcepath 옵션을 사용해야 할 수 있습니다. 예: javac -d out -sourcepath src src/Main.java.

### 컴파일 오류 처리

컴파일 중 오류가 발생하면 javac는 파일, 줄 번호 및 특정 원인을 포함한 상세한 오류 정보를 출력합니다. 일반적인 컴파일 오류에는 다음이 포함됩니다: 구문 오류(세미콜론 누락, 일치하지 않는 괄호 등), 타입 오류(호환되지 않는 타입, 존재하지 않는 메서드 등), 액세스 권한 오류(비공개 멤버에 대한 액세스 시도 등).

이러한 오류 메시지를 이해하는 것은 문제를 신속하게 찾고 수정하는 데 중요합니다. 컴파일러는 최대한 많은 오류를 보고하려고 합니다. 때때로 첫 번째 오류를 수정하면 후속 오류가 자동으로 해결됩니다.

## Java 프로그램 실행 가이드

### java 명령 기본 사용법

컴파일된 Java 프로그램을 실행하려면 java 명령을 사용합니다. 기본 구문: java HelloWorld. 여기서 .class 확장자는 필요하지 않습니다. JVM은 현재 디렉토리와 CLASSPATH에서 Hello.class를 자동으로 찾습니다.

클래스가 패키지에 있는 경우 정규화된 이름을 사용합니다. 예: java com.example.HelloWorld.

### Classpath (CLASSPATH)

Classpath는 JVM에게 클래스 파일의 위치를 알려줍니다. -classpath 또는 -cp 옵션을 사용하여 지정할 수 있습니다. 예: java -cp out:lib/* com.example.HelloWorld. Windows에서는 경로 구분 기호가 세미콜론(;)이고 Linux 및 macOS에서는 콜론(:)입니다.

Classpath가 지정되지 않으면 JVM은 현재 디렉토리와 CLASSPATH 환경 변수에 지정된 경로를 사용합니다.

### JVM 옵션

JVM은 런타임 동작을 제어하기 위한 numerous 시작 옵션을 제공합니다. 일반적인 옵션에는 다음이 포함됩니다: -Xms는 초기 힙 크기를 지정합니다(예: -Xms256m은 초기 힙을 256MB로 설정). -Xmx는 최대 힙 크기를 지정합니다(예: -Xmx2g는 최대 힙을 2GB로 설정). -XX:+UseG1GC는 G1 가비지 컬렉터를 사용합니다.

JDK 9 이상에서는 --add-open 옵션을 사용하여 모듈에 대한 내부 액세스를 열 수 있습니다. 이는 JDK 내부 API에 액세스해야 하는 라이브러리를 실행할 때 유용합니다.

## JDK 도구 사용 가이드

### jar 아카이브 도구

jar 도구는 Java 아카이브(.jar 파일) 생성 및 관리에 사용됩니다. .jar 파일은 기본적으로 컴파일된 .class 파일과 메타데이터를 포함하는 ZIP 형식의 압축 파일입니다.

jar 파일을 생성하는 기본 명령: jar cf myapp.jar -C out . 이렇게 하면 out 디렉토리의 내용으로 myapp.jar가 생성됩니다. jar tf myapp.jar를 사용하여 jar 파일 내용을 보고, jar xf myapp.jar를 사용하여 추출합니다.

실행 가능한 jar 파일을 생성할 때 MANIFEST.MF 파일에서 기본 클래스를 지정해야 합니다. m 매개변수를 사용하여 사용자 정의 MANIFEST.MF를 추가할 수 있습니다: jar cfm myapp.jar MANIFEST.MF -C out .

### javadoc 문서 생성

javadoc 도구는 Java 소스 코드의 주석에서 HTML 형식의 API 문서를 생성합니다. 표준 Java 클래스 라이브러리 문서는 이러한 방식으로 생성됩니다.

기본 사용법: javadoc -d doc src/*.java. 이렇게 하면 src 디렉토리의 모든 Java 파일에 대한 문서가 생성되어 doc 디렉토리에 저장됩니다.

코드 작성 시 특정 주석 형식을 사용하여 클래스, 메서드 및 필드에 대한 문서를 추가할 수 있습니다. 일반적인 주석 태그에는 @author(저자), @param(매개변수), @return(반환값), @throws(예외) 등이 있습니다.

### jdb 디버거

jdb는 기본 디버깅 기능을 제공하는 Java 명령줄 디버거입니다. 디버깅 시작: jdb MyClass. 중단점 설정: stop in MyClass.method. 중단점 보기: clear. 프로그램 실행: run. 실행 계속: cont. 단계별 실행: step 또는 next. 변수 보기: print varname. 소스 코드 보기: list.

jdb가 그래픽 디버거만큼 강력하지는 않지만, IDE가 없는 환경 또는 서버에서 원격 디버깅에 사용할 수 있습니다.

## JVM 성능 최적화

### 가비지 컬렉터 선택

JDK는 각각異なる 특성 및 애플리케이션 시나리오를 가진 여러 가비지 컬렉터를 제공합니다:

Serial GC: 단일 스레드 컬렉터로, 작은 애플리케이션 및 클라이언트 애플리케이션에 적합합니다. -XX:+UseSerialGC로 활성화합니다.

Parallel GC (Throughput GC): 다중 스레드 컬렉터로, 높은 처리량을 추구하는 애플리케이션에 적합합니다. -XX:+UseParallelGC로 활성화합니다.

CMS GC: 동시 마크 앤 스위프 컬렉터로, 낮은 일시 정지 시간을 추구합니다. -XX:+UseConcMarkSweepGC로 활성화합니다(JDK 14에서 폐기됨).

G1 GC: 분할 가비지 컬렉터로, 대용량 메모리 애플리케이션에 적합하며 JDK 9 이상에서 기본 옵션이 되었습니다. 기본 설정 사용하거나 -XX:+UseG1GC를 사용하는 것이 좋습니다.

ZGC: 저지연 가비지 컬렉터로, 테라바이트 수준의 힙 메모리를 지원하며 일시 정지가 10ms를 초과하지 않습니다. -XX:+UseZGC로 활성화합니다(JDK 15+).

Shenandoah: 또 다른 저지연 가비지 컬렉터로, 힙 크기에 관계없이 일시 정지가 독립적입니다. -XX:+UseShenandoahGC로 활성화합니다.

### 메모리 구성

힙 메모리 크기를 올바르게 구성하는 것은 애플리케이션 성능에至关重要합니다. 일반적인 구성 옵션:

-Xms: 초기 힙 크기(예: -Xms2g).
-Xmx: 최대 힙 크기(예: -Xmx4g).
-Xmn: Young 세대 크기(예: -Xmn1g).
-XX:MetaspaceSize: 초기 메타스페이스 크기.
-XX:MaxMetaspaceSize: 최대 메타스페이스 크기.

런타임 시 힙 크기 조정 오버헤드를 피하기 위해 -Xms와 -Xmx를 동일한 값으로 설정하는 것이 좋습니다.

### 성능 모니터링 도구

JDK는 풍부한 성능 모니터링 및 진단 도구를 제공합니다:

jps: Java 프로세스 목록 표시, 기본 사용 jps -l.
jstat: JVM 통계 표시(예: 가비지 컬렉션 통계: jstat -gcutil pid 1000).
jstack: 스레드의 스택 추적 정보 인쇄, 정지 문제 진단에 사용: jstack pid.
jmap: Java 힙 메모리 정보 표시, 분석을 위한 힙 덤프 파일 생성 가능: jmap -dump:format=b,file=heap.bin pid.
jcmd: 다기능 진단 도구, 다양한 진단 명령 실행 가능: jcmd pid GC.heap_info.

JConsole 및 VisualVM: 그래픽 모니터링 및 진단 도구로, 메모리, 스레드, CPU 사용량 등을 모니터링할 수 있습니다.

## 모듈 시스템

### 모듈 개요

JDK 9부터 Java는 시스템 모듈(Project Jigsaw)을 도입하여 JDK와 애플리케이션을 모듈로 분할했습니다. 각 모듈은 명확한 종속성을 가지며 더 나은 캡슐화와 유지 관리성을 제공합니다.

JDK 자체는 java.base, java.sql, java.xml와 같은 여러 모듈로 분할됩니다. 애플리케이션은 자체 모듈을 생성할 수 있으며, module-info.java 파일을 통해 모듈 이름, 종속성 및 내보내기 패키지를 선언합니다.

### 모듈 관련 옵션

JDK는 모듈과 관련된 여러 명령줄 옵션을 제공합니다:

--module-path(또는 -p): 모듈 경로를 지정하며, classpath와 유사하지만 모듈용입니다.
--add-modules: 기본 모듈 세트에 모듈을 추가합니다.
--add-exports: 지정된 모듈 패키지를 다른 모듈이 액세스할 수 있도록 합니다.
--add-opens: 지정된 모듈 패키지를 리플렉션 액세스를 위해 엽니다.

이러한 옵션은 JDK 내부 API에 리플렉션을 사용하여 액세스하거나 모듈화와 호환되지 않는 라이브러리를 사용할 때 특정 상황에서 매우 중요합니다.

## 일반적인 구성 예

### 개발 환경 구성

일상적인 개발 환경의 권장 구성은 충분한 힙 메모리(-Xmx2g 이상), G1 가비지 컬렉터 사용(기본값), 클래스 데이터 공유 활성화(-Xshare:classes)로 시작 시간을 단축합니다. IDE를 사용하는 경우 IDE 실행 구성에서 이러한 옵션을 직접 설정할 수 있습니다.

### 프로덕션 환경 구성

프로덕션 환경 구성은 애플리케이션의 특정 상황에 맞게 조정해야 합니다. 기본 원칙: 애플리케이션 메모리 요구 사항에 따라 힙 크기 설정, 일반적으로 운영 체제 및 다른 프로세스를 위해 25-30% 메모리 예약; 적절한 가비지 컬렉터 선택, G1이 일반적으로 좋은 선택; 저지연이 필요한 애플리케이션의 경우 ZGC 또는 Shenandoah 사용 고려.

프로덕션 환경에서 GC 로깅을 활성화하여 문제 분석 및 성능 최적화를 위한 상세한 가비지 컬렉션 정보를 기록하는 것이 좋습니다. GC 로깅 구성 예: -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M.

### Docker 컨테이너 구성

컨테이너 환경에서 Java 애플리케이션을 실행할 때 JVM의 메모리 인식에 주의해야 합니다. JDK 10+ 버전은 기본적으로 컨테이너 메모리 제한을 인식하지만 이전 버전에서는 수동 구성이 필요할 수 있습니다.

권장 구성: 올바른 최대 힙 값 설정(-XX:MaxRAMPercentage=75.0), JVM이 사용 가능한 컨테이너 메모리의 75% 이상을 사용하지 않도록 보장; -XX:+UseContainerSupport(기본적으로 활성화됨)를 사용하여 JVM이 컨테이너 리소스 제한을 인식하도록 합니다.

클라우드 네이티브 애플리케이션의 경우 GraalVM Native Image를 사용하여 애플리케이션을 네이티브 실행 파일로 컴파일하는 것을 고려하세요. 이렇게 하면 시작 시간과 메모리 사용량을 크게 줄일 수 있습니다.

## 버전별 사용 차이점

### JDK 8

JDK 8은 상업적 사용을 위한 마지막 무료 Oracle JDK 버전입니다. 주요 기능에는 람다 표현식, 스트림 API 및 새로운 날짜/시간 API가 포함됩니다. 설치는 비교적 간단하며 기존 설치 프로그램을 사용합니다. JAVA_HOME이 JDK 디렉토리(JRE 아님)를 가리키야 합니다. 일부 명령어와 도구는 최신 버전에서 다르게 동작할 수 있습니다.

### JDK 11

JDK 11은 Java 유료 기능의 시작을 나타냅니다. 주요 기능에는 로컬 변수 타입 추론(var), 새로운 HTTP 클라이언트 및 ZGC(실험적)가 포함됩니다. JRE는 더 이상 별도의 다운로드로 제공되지 않습니다; JDK만 사용 가능합니다. java 명령은 컴파일 없이 단일 소스 프로그램 파일을 직접 실행할 수 있습니다.

### JDK 17

JDK 17은 NFTC에 따른 마지막 무료 Oracle JDK 버전입니다. 주요 기능에는 sealed 클래스, 텍스트 블록 및 switch 표현식이 포함됩니다. 설치 및 사용은 JDK 11과 유사합니다. RandomGenerator 인터페이스는 새로운 난수 생성 기능을 제공합니다. 외부 메모리 액세스 API가 인큐베이션 단계에 진입했습니다.

### JDK 21

JDK 21은 가상 스레드(안정), switch용 패턴 매칭(안정) 및 레코드(안정)를 포함한 중요한 기능을 도입합니다. 가상 스레드는 다른 병렬 프로그래밍 접근 방식을 필요로 합니다. 가상 스레드를 사용하는 애플리케이션은 힙 설정을 조정해야 할 수 있습니다. 성능 최적화는 가상 스레드의 특성을 고려해야 합니다.

### JDK 25

JDK 25는 최신 언어 기능과 성능 개선이 포함된 최신 버전입니다. 기능은 아직 미리보기 또는 실험 단계에 있을 수 있습니다. 장기 지원이 필요한 프로덕션 환경에는 권장되지 않습니다. 구성 및 사용은 JDK 21과 유사하지만, 추가 학습이 필요할 수 있는 더 많은 새로운 기능이 있습니다.
