# FAQ GitLab

## Problèmes d'Installation

### Que faire si l'installation de GitLab échoue ?

L'échec de l'installation est généralement causé par quelques raisons communes. Premièrement, vérifiez si le système répond aux exigences minimales, y compris la version du système d'exploitation, la mémoire et l'espace disque. Si l'installation utilisant le script auto-hébergé échoue, essayez d'installer manuellement les paquets de dépendances, puis d'installer étape par étape.

Consultez les journaux d'installation pour obtenir des informations détaillées sur les erreurs :

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# Voir les journaux
sudo tail -f /var/log/gitlab/install.log
```

Les erreurs courantes incluent : conflits de ports (ports 80, 443, 22 occupés), mémoire insuffisante, problèmes de certificats SSL, etc. S'il y a un conflit de ports, soit arrêtez le service occupant le port, soit modifiez la configuration de GitLab pour utiliser d'autres ports.

### Comment modifier l'adresse d'accès GitLab ?

Si vous devez modifier l'adresse d'accès GitLab après l'installation, modifiez le paramètre external_url dans /etc/gitlab/gitlab.rb, puis reconfigurez :

```bash
sudo nano /etc/gitlab/gitlab.rb
# Modifier external_url
sudo gitlab-ctl reconfigure
```

Après modification, asegurez-vous que la résolution DNS ou le fichier hosts local est correctement configuré pour que la nouvelle adresse soit accessible normalement. Si vous passez de HTTP à HTTPS, vous devez également configurer les certificats SSL.

## Problèmes d'Utilisation

### Que faire si la connexion par clé SSH échoue ?

L'échec de connexion par clé SSH peut être causé par plusieurs raisons. Premièrement, confirmez que la clé publique a été correctement ajoutée aux clés SSH du compte GitLab :

```bash
# Voir la clé publique locale
cat ~/.ssh/id_ed25519.pub

# Tester la connexion
ssh -T git@gitlab.example.com
```

Si le test de connexion échoue, vérifiez la configuration du client SSH :

```bash
# Modifier la config SSH
nano ~/.ssh/config

# Ajouter ce qui suit
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

Asegurez-vous que l'URL du dépôt distant utilise le format SSH (git@gitlab.example.com:group/project.git), pas le format HTTPS.

### Que faire si une Merge Request ne peut pas être fusionnée ?

Les Merge Requests peuvent ne pas pouvoir être fusionnées pour les raisons suivantes : des conflits existent, les tests CI/CD ont échoué, les règles de branche protégée ne sont pas satisfaites, des approbations manquantes, etc. La solution est de vérifier point par point :

- Conflits de fusion : Cliquez sur "Mettre à jour la branche" sur la page Merge Request, fusionnez la branche cible dans la branche source, résolvez les conflits, puis repoussez
- Échec CI : Consultez le journal du travail échoué, résolvez le problème, puis réexécutez le pipeline
- Branche protégée : Contactez l'administrateur du projet pour modifier les paramètres des branches protégées, ou demandez à un membre avec les permissions de fusionner

### Comment diagnostiquer les échecs d'exécution de pipeline CI/CD ?

Lorsqu'un pipeline échoue, consultez d'abord le journal du travail échoué sur la page CI/CD → Pipelines. Le journal affichera les informations d'erreur spécifiques et les raisons de l'échec. Les problèmes courants incluent : échec du téléchargement des dépendances, erreurs de commande de script, échecs de cas de test, etc.

Si le problème est difficile à localiser, vous pouvez ajouter des étapes de débogage dans la configuration du pipeline, comme la sortie des variables d'environnement, la vérification de l'état des fichiers avant et après l'installation, etc. Vous pouvez également utiliser la fonction de débogage interactif pour démarrer un shell interactif dans les travaux de pipeline pour le diagnostic.

## Problèmes de Performance

### L'accès à GitLab est très lent, que faire ?

L'accès lent peut être causé par des ressources serveur insuffisantes, une latence réseau ou une configuration inappropriée. Vérifiez d'abord la charge du serveur :

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

Si l'utilisation du CPU ou de la mémoire est très élevée, vous devrez peut-être mettre à niveau la configuration du serveur ou optimiser les paramètres GitLab. Vous pouvez également vérifier les performances de la base de données :

```bash
sudo gitlab-rake gitlab:git:fsck  # Vérifier l'intégrité du dépôt Git
sudo gitlab-rake db:index_cache_daily  # Optimiser le cache de la base de données
```

Pour les problèmes réseau, envisagez d'utiliser un CDN pour accélérer la distribution des ressources statiques, ou déployez GitLab à des emplacements réseau plus proches des utilisateurs.

### Comment optimiser les performances de GitLab ?

L'optimisation des performances de GitLab peut être abordée sous plusieurs angles. L'optimisation de la base de données est importante. La maintenance régulière peut améliorer considérablement les performances :

```bash
# Nettoyer la base de données
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# Nettoyer les fichiers inutilisés
sudo gitlab-rake gitlab:cleanup:builds
```

Pour les grands dépôts, vous pouvez activer les pools d'objets Git pour réduire l'utilisation du disque, ou configurer Git LFS pour stocker les fichiers volumineux. Configurer raisonnablement le cache CI/CD peut accélérer les constructions.

## Problèmes de Sécurité

### Comment configurer les certificats SSL ?

GitLab prend en charge la configuration automatique SSL avec Let's Encrypt, ou vous pouvez utiliser des certificats personnalisés. Modifiez d'abord le fichier de configuration :

```ruby
# Configuration automatique Let's Encrypt
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# Certificat personnalisé
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

Exécutez sudo gitlab-ctl reconfigure pour appliquer la configuration. Si vous utilisez des certificats auto-signés, les clients doivent approuver le certificat pour accéder via HTTPS.

### Comment sauvegarder et restaurer GitLab ?

Les sauvegardes régulières sont importantes pour prévenir la perte de données. GitLab fournit un outil de sauvegarde intégré :

```bash
# Créer une sauvegarde
sudo gitlab-backup create

# Emplacement des fichiers de sauvegarde
ls -la /var/opt/gitlab/backups/
```

Les fichiers de sauvegarde incluent les bases de données, les fichiers téléchargés et les dépôts Git. Pour restaurer : arrêtez le service GitLab, copiez le fichier de sauvegarde à l'emplacement par défaut, puis exécutez la commande de restauration :

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

Il est recommandé de synchroniser automatiquement les fichiers de sauvegarde vers un stockage distant, comme les services de stockage d'objets ou un autre serveur.

## Problèmes de Mise à Niveau

### Que faire si des erreurs apparaissent après la mise à niveau ?

Après une mise à niveau, des erreurs de compatibilité ou des anomalies fonctionnelles peuvent se produire. Vérifiez d'abord l'état de GitLab :

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

Consultez les journaux de mise à niveau pour comprendre les erreurs spécifiques :

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

Les problèmes courants incluent : échec de migration de base de données, erreurs de syntaxe de fichier de configuration, dépendances manquantes, etc. S'il s'agit d'un problème de migration de base de données, vous devrez peut-être corriger manuellement :

```bash
sudo gitlab-rake db:migrate
```

Il est recommandé de sauvegarder les données avant la mise à niveau et de lire attentivement la documentation officielle de mise à niveau pour comprendre les changements importants entre les versions.

### Comment mettre à niveau en toute sécurité entre différentes versions ?

La mise à niveau de GitLab doit suivre les limites de saut de version, en ne sautant pas plus d'une version majeure à la fois. Par exemple, passez de 15.x à 16.x, pas directement de 15.x à 17.x. Passez d'abord à la version intermédiaire, confirmez le fonctionnement normal, puis continuez la mise à niveau.

Avant la mise à niveau, effectuez les étapes suivantes : sauvegarder les données, vérifier les problèmes de compatibilité, lire les notes de version. Maintenez l'énergie et le réseau stables pendant le processus de mise à niveau, et vérifiez que toutes les fonctionnalités sont normales après la mise à niveau.
