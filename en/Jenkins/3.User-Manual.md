# User Manual

## Pipeline Basics

Pipeline is a core feature of Jenkins that automates the entire software development process from building, testing, to deployment.

### What is Pipeline

Pipeline is a suite of Jenkins plugins that enables implementing and integrating continuous delivery pipelines into Jenkins. Simply put, Pipeline automates the entire process of getting software from version control to users.

Pipeline definitions are written in a text file (Jenkinsfile) that can be committed to the project's source control repository. This implements "Pipeline as Code" philosophy.

### Benefits of Pipeline

- **Code-based**: Pipelines are versioned like code, enabling review and iteration
- **Durable**: Pipelines survive both planned and unplanned restarts of Jenkins
- **Pausable**: Pipelines can stop and wait for human input or approval
- **Versatile**: Supports complex real-world CD requirements including parallel execution and loops
- **Extensible**: Plugins can extend Pipeline DSL and integrate with other tools

### Pipeline Concepts

#### Node

A node is a machine that is part of the Jenkins environment and is capable of executing a Pipeline. Nodes can be the master or agent machines.

#### Stage

A stage defines a conceptually distinct subset of tasks performed through the entire Pipeline, such as "Build", "Test", and "Deploy" stages. Stages are used to visualize Pipeline execution status.

#### Step

A step is a fundamental action in Jenkins Pipeline that tells Jenkins what to do at a particular point in time. For example, execute a shell command using the `sh` step.

### Pipeline Syntax

#### Declarative Pipeline

Declarative Pipeline is newer syntax with richer features, easier to write and read:

```groovy
pipeline {
    agent any
    
    stages {
        stage('Build') {
            steps {
                echo 'Building application...'
            }
        }
        stage('Test') {
            steps {
                echo 'Running tests...'
            }
        }
        stage('Deploy') {
            steps {
                echo 'Deploying application...'
            }
        }
    }
}
```

#### Scripted Pipeline

Scripted Pipeline uses Groovy syntax, more flexible but requires more programming experience:

```groovy
node {
    stage('Build') {
        echo 'Building application...'
    }
    stage('Test') {
        echo 'Running tests...'
    }
    stage('Deploy') {
        echo 'Deploying application...'
    }
}
```

## Creating Your First Pipeline

### Using Blue Ocean

Blue Ocean is Jenkins' modern interface that makes creating pipelines more intuitive:

1. Install Blue Ocean plugin (if not pre-installed)
2. Click "Open Blue Ocean"
3. Click "Create Pipeline"
4. Select code repository location (GitHub, GitLab, etc.)
5. Select repository and branch
6. Blue Ocean will auto-detect Jenkinsfile or help create one

### Using Jenkinsfile

Create a Jenkinsfile in your project, and Jenkins will automatically recognize it:

1. Create Jenkinsfile in project root
2. Define pipeline stages
3. Commit to version control
4. Create pipeline job in Jenkins and link to repository

## Credentials Management

Jenkins needs to access various external resources like code repositories and deployment targets, which require credentials management.

### Adding Credentials

1. Go to "Manage Jenkins" > "Manage Credentials"
2. Click "Add Credentials"
3. Choose credential type (Username with Password, SSH Username with Private Key, Secret File, etc.)
4. Fill in credential information
5. Set ID and description for the credential

### Using Credentials in Pipeline

Use credentials with the withCredentials step:

```groovy
pipeline {
    agent any
    stages {
        stage('Use Credentials') {
            steps {
                withCredentials([usernamePassword(credentialsId: 'my-cred', usernameVariable: 'USER', passwordVariable: 'PASS')]) {
                    echo "Username: $USER"
                }
            }
        }
    }
}
```

## Build Triggers

Automated builds require triggers. Jenkins supports multiple trigger methods.

### Polling Source Code Management

Regularly check if source code has changed:

```groovy
pipeline {
    agent any
    triggers {
        pollSCM('H/5 * * * *')  // Check every 5 minutes
    }
    stages {
        stage('Build') {
            steps {
                echo 'Code change detected, starting build'
            }
        }
    }
}
```

### Webhook Trigger

Trigger builds via repository webhooks for better efficiency:

1. Install corresponding plugin (e.g., GitHub Hook Trigger)
2. Configure repository webhook pointing to Jenkins
3. Build triggers automatically on code commit

### Scheduled Builds

Execute automatically on a schedule:

```groovy
pipeline {
    agent any
    triggers {
        cron('H 2 * * *')  // Run at 2 AM daily
    }
    stages {
        stage('Scheduled Build') {
            steps {
                echo 'Scheduled build running'
            }
        }
    }
}
```

## Using Agents and Containers

### Configuring Agents

Jenkins supports distributed builds, distributing build tasks across multiple machines:

1. Go to "Manage Jenkins" > "Manage Nodes"
2. Add agent node and configure connection information
3. Specify agent in pipeline

### Using Docker in Pipeline

Declarative Pipeline can execute builds in Docker containers:

```groovy
pipeline {
    agent {
        docker {
            image 'maven:3.8-openjdk-8'
            args '-v /root/.m2:/root/.m2'
        }
    }
    stages {
        stage('Build') {
            steps {
                sh 'mvn --version'
            }
        }
    }
}
```

## Test Reporting

Jenkins can integrate with various test frameworks to generate visual test reports.

### JUnit Test Reports

```groovy
pipeline {
    agent any
    stages {
        stage('Test') {
            steps {
                sh 'mvn test'
            }
        }
    }
    post {
        always {
            junit 'target/surefire-reports/*.xml'
        }
    }
}
```

### Other Test Reports

Jenkins supports various test report plugins:

- TestNG
- Cobertura (code coverage)
- JaCoCo (code coverage)
- ESLint (frontend code analysis)

## Deployment Practices

### Deploying to Different Environments

Typical multi-stage deployment pipeline:

```groovy
pipeline {
    agent any
    stages {
        stage('Build') {
            steps {
                sh 'mvn clean package'
            }
        }
        stage('Deploy to Test') {
            when {
                branch 'develop'
            }
            steps {
                sh './deploy.sh test'
            }
        }
        stage('Deploy to Production') {
            when {
                branch 'main'
            }
            steps {
                input message: 'Confirm deployment to production?', ok: 'Deploy'
                sh './deploy.sh prod'
            }
        }
    }
}
```

### Blue-Green Deployment

Implement blue-green deployment with Jenkins for zero-downtime updates:

1. Maintain two identical environments (blue and green)
2. Switch traffic to new version
3. Rollback quickly if issues arise

## Best Practices

### Jenkinsfile Best Practices

- Keep Jenkinsfile in version control
- Use declarative pipeline syntax
- Keep pipelines simple and clear
- Divide stages reasonably
- Add necessary timeout and retry mechanisms

### Security Recommendations

- Regularly update Jenkins and plugins
- Limit anonymous user permissions
- Use credentials for sensitive information
- Configure appropriate access control
- Enable security audit logging

### Performance Optimization

- Configure build nodes properly
- Use incremental builds to reduce time
- Configure build caching
- Regularly clean old build records
- Use parallel stages to speed up builds

## Troubleshooting

### Viewing Build Logs

When a build fails, first check the build logs:

1. Go to job page
2. Click failed build number
3. Check console output

### Common Problem Solutions

- Build stuck: Check for pending input or resource deadlocks
- Credentials invalid: Check if credentials expired or revoked
- Plugin conflicts: Try disabling recently installed plugins
- Out of memory: Adjust JVM heap parameters
