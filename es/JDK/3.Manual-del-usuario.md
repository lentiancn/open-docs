# JDK Manual de Usuario

## Guía de uso del compilador Java

### Uso básico de javac

javac es el compilador de Java, responsable de compilar código fuente Java en archivos de bytecode. El uso básico es muy simple: javac HelloWorld.java. Esto generará un archivo HelloWorld.class, que luego puede ejecutarse en JVM.

En proyectos de desarrollo reales, generalmente necesita especificar más opciones de compilación. Las opciones comunes incluyen: -d especifica el directorio de salida de compilación, por ejemplo, javac -d out HelloWorld.java pondrá los resultados compilados en el directorio out; -classpath (o -cp) especifica la ruta de búsqueda de clases para referenciar clases externas durante la compilación; -source especifica la versión Java del código fuente; -target especifica la versión Java del bytecode generado.

Para proyectos grandes, generalmente se usan herramientas de construcción como Maven o Gradle para gestionar el proceso de compilación, estas herramientas manejan automáticamente las dependencias y opciones de compilación.

### Compilación de múltiples archivos

Si su proyecto contiene múltiples archivos fuente Java, puede compilar todo un directorio a la vez. Por ejemplo: javac -d out src/*.java compilará todos los archivos Java en el directorio src.

Las estructuras de proyectos más complejas pueden requerir el uso de la opción -sourcepath para especificar la ruta de búsqueda del código fuente, permitiendo que el compilador encuentre clases dependientes. Por ejemplo: javac -d out -sourcepath src src/Main.java.

### Manejo de errores de compilación

Si ocurren errores durante la compilación, javac emitirá información detallada del error, incluyendo el archivo, número de línea y causa específica. Los errores de compilación comunes incluyen: errores de sintaxis (punto y coma faltante, paréntesis desbalanceados, etc.), errores de tipo (tipos incompatibles, métodos no existentes, etc.), y errores de permisos de acceso (intentar acceder a miembros privados, etc.).

Entender estos mensajes de error es crucial para localizar y resolver problemas rápidamente. El compilador intenta reportar tantos errores como sea posible; a veces, corregir el primer error hará que los errores subsiguientes desaparezcan.

## Guía para ejecutar programas Java

### Uso básico del comando java

Use el comando java para ejecutar programas Java compilados. La sintaxis básica es: java HelloWorld. Note que no necesita la extensión .class aquí. JVM buscará automáticamente Hello.class en el directorio actual y CLASSPATH.

Si la clase está en un paquete, use el nombre completamente calificado. Por ejemplo: java com.example.HelloWorld.

### Classpath (CLASSPATH)

Classpath le dice a JVM dónde encontrar archivos de clase. Puede especificarse usando las opciones -classpath o -cp. Por ejemplo: java -cp out:lib/* com.example.HelloWorld. En Windows, el separador de ruta es punto y coma (;); en Linux y macOS, es dos puntos (:).

Si no se especifica classpath, JVM usará el directorio actual y las rutas especificadas en la variable de entorno CLASSPATH.

### Opciones de JVM

JVM proporciona numerosas opciones de inicio para controlar el comportamiento en tiempo de ejecución. Las opciones comunes incluyen: -Xms especifica el tamaño inicial del heap, como -Xms256m establece el heap inicial en 256MB; -Xmx especifica el tamaño máximo del heap, como -Xmx2g establece el heap máximo en 2GB; -XX:+UseG1GC usa el recolector de basura G1.

En JDK 9 y posteriores, puede usar la opción --add-opens para abrir el acceso interno del módulo, lo cual es útil al ejecutar bibliotecas que necesitan acceder a APIs internas de JDK.

## Guía de herramientas de JDK

### Herramienta de archivos jar

La herramienta jar se usa para crear y administrar archivos de almacenamiento Java (archivos .jar). Los archivos .jar son en realidad archivos comprimidos en formato ZIP que contienen archivos .class compilados y algunos metadatos.

El comando básico para crear un archivo jar: jar cf myapp.jar -C out . Esto crea myapp.jar del contenido del directorio out. Use el parámetro tf para ver el contenido del archivo jar: jar tf myapp.jar. Use xf para extraer: jar xf myapp.jar.

Al crear un archivo jar ejecutable, necesita especificar la clase principal en el archivo MANIFEST.MF. Puede agregar un MANIFEST.MF personalizado usando el parámetro m: jar cfm myapp.jar MANIFEST.MF -C out .

### Generación de documentación javadoc

La herramienta javadoc genera documentación de API en formato HTML a partir de comentarios en el código fuente Java. La documentación estándar de la biblioteca de clases Java se genera de esta manera.

Uso básico: javadoc -d doc src/*.java. Esto generará documentación para todos los archivos Java en el directorio src y la colocará en el directorio doc.

Al escribir código, usar formatos de comentarios específicos puede agregar documentación para clases, métodos y campos. Las etiquetas de comentario comunes incluyen: @author autor, @param parámetro, @return valor de retorno, @throws excepción, etc.

### Depurador jdb

jdb es el depurador de línea de comandos de Java, proporcionando funcionalidad básica de depuración. Iniciar depuración: jdb MyClass. Establecer puntos de interrupción: stop in MyClass.method. Ver puntos de interrupción: clear. Ejecutar programa: run. Continuar ejecución: cont. Ejecutar paso a paso: step o next. Ver variables: print varname. Listar código fuente: list.

Aunque jdb no es tan potente como los depuradores gráficos, puede usarse en entornos sin IDE o para depuración remota en servidores.

## Ajuste de rendimiento de JVM

### Selección del recolector de basura

JDK proporciona múltiples recolectores de basura, cada uno con diferentes características y escenarios de aplicación:

Serial GC: Recolector de hilo único, adecuado para aplicaciones pequeñas y aplicaciones de cliente. Habilitar con -XX:+UseSerialGC.

Parallel GC (Throughput GC): Recolector de múltiples hilos, adecuado para aplicaciones que buscan alto rendimiento. Habilitar con -XX:+UseParallelGC.

CMS GC: Recolector de marca y barrido concurrente, buscando bajos tiempos de pausa. Habilitar con -XX:+UseConcMarkSweepGC (Obsoleto en JDK 14).

G1 GC: Recolector de basura por particiones, adecuado para aplicaciones de gran memoria, y se convirtió en la opción predeterminada en JDK 9 y posteriores. Se recomienda usar la configuración predeterminada o -XX:+UseG1GC.

ZGC: Recolector de baja latencia, soporta memoria de heap a nivel de TB, tiempos de pausa que no exceden 10ms. Habilitar con -XX:+UseZGC (JDK 15+).

Shenandoah: Otro recolector de baja latencia, tiempos de pausa independientes del tamaño del heap. Habilitar con -XX:+UseShenandoahGC.

### Configuración de memoria

Configurar apropiadamente el tamaño de la memoria heap es esencial para el rendimiento de la aplicación. Opciones de configuración comunes:

-Xms: Tamaño inicial del heap, ej., -Xms2g.
-Xmx: Tamaño máximo del heap, ej., -Xmx4g.
-Xmn: Tamaño de la generación joven, ej., -Xmn1g.
-XX:MetaspaceSize: Tamaño inicial de Metaspace.
-XX:MaxMetaspaceSize: Tamaño máximo de Metaspace.

Se recomienda establecer -Xms y -Xmx en el mismo valor para evitar la sobrecarga del ajuste del heap en tiempo de ejecución.

### Herramientas de monitoreo de rendimiento

JDK proporciona herramientas ricas de monitoreo y diagnóstico de rendimiento:

jps: Muestra lista de procesos Java, uso básico jps -l.
jstat: Muestra estadísticas de JVM, ej., estadísticas de recolección de basura: jstat -gcutil pid 1000.
jstack: Imprime información de seguimiento de pila de hilos, usado para diagnosticar bloqueos: jstack pid.
jmap: Muestra información de memoria heap, ej., generar volcado de heap: jmap -dump:format=b,file=heap.bin pid.
jcmd: Herramienta de diagnóstico multifunción, puede ejecutar varios comandos de diagnóstico: jcmd pid GC.heap_info.

JConsole y VisualVM: Herramientas gráficas de monitoreo y diagnóstico, pueden monitorear memoria, hilos, uso de CPU, etc.

## Sistema de módulos

### Descripción general de módulos

A partir de JDK 9, Java introdujo el sistema de módulos (Project Jigsaw), dividiendo JDK y aplicaciones en módulos. Cada módulo tiene dependencias claras, proporcionando mejor encapsulamiento y mantenibilidad.

JDK mismo se divide en múltiples módulos como java.base, java.sql, java.xml. Las aplicaciones pueden crear sus propios módulos, declarando nombre de módulo, dependencias y exportaciones a través de archivos module-info.java.

### Opciones relacionadas con módulos

JDK proporciona múltiples opciones de línea de comandos relacionadas con módulos:

--module-path (o -p): Especifica la ruta del módulo, similar a classpath pero para módulos.
--add-modules: Agrega módulos al conjunto de módulos predeterminado.
--add-exports: Abre paquetes de módulos especificados para que otros módulos accedan.
--add-opens: Abre paquetes de módulos especificados para acceso por reflexión.

Estas opciones son muy importantes en ciertas situaciones, como usar reflexión para acceder a APIs internas de JDK o al usar bibliotecas no compatibles con modularización.

## Ejemplos de configuración común

### Configuración de entorno de desarrollo

Para entornos de desarrollo diario, la configuración recomendada incluye: suficiente memoria heap (-Xmx2g o mayor), usar recolector de basura G1 (predeterminado), habilitar uso de datos de clase compartidos (-Xshare:classes) para acelerar el inicio. Si usa IDE, puede establecer estas opciones directamente en la configuración de ejecución del IDE.

### Configuración de entorno de producción

La configuración del entorno de producción necesita ajustarse según la situación específica de la aplicación. Los principios básicos incluyen: establecer el tamaño del heap según las necesidades de memoria de la aplicación, generalmente reservando 25-30% de memoria para el sistema operativo y otros procesos; elegir un recolector de basura apropiado, G1 generalmente es una buena opción; para aplicaciones que buscan baja latencia, considere usar ZGC o Shenandoah.

Se recomienda habilitar el registro de GC en entornos de producción, grabando información detallada de recolección de basura para análisis de problemas y ajuste de rendimiento. Ejemplo de configuración de registro de GC: -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M.

### Configuración de contenedor Docker

Al ejecutar aplicaciones Java en entornos de contenedores, preste atención a la aware de memoria de JVM. Las versiones de JDK 10+ son automaticamente aware de los límites de memoria del contenedor, pero puede requerir configuración manual en versiones anteriores.

Configuración recomendada: establecer el valor máximo de heap correcto (-XX:MaxRAMPercentage=75.0), asegurando que JVM no use más del 75% de la memoria disponible del contenedor; usar -XX:+UseContainerSupport (habilitado por defecto) para permitir que JVM sea aware de los límites de recursos del contenedor.

Para aplicaciones nativas en la nube, considere usar GraalVM Native Image para compilar aplicaciones en ejecutables nativos, lo cual puede reducir significativamente el tiempo de inicio y el uso de memoria.

## Diferencias de uso por versión

### JDK 8

JDK 8 es la última versión de Oracle JDK completamente gratuita para uso comercial. Las características principales incluyen expresiones lambda, API de streams y nueva API de fecha/hora. La instalación es relativamente simple, usando el instalador tradicional. Note que JAVA_HOME debe apuntar al directorio JDK (no JRE). Algunos comandos y herramientas pueden comportarse de manera diferente en versiones más nuevas.

### JDK 11

JDK 11 marcó el comienzo de las características pagadas de Java. Las características principales incluyen inferencia de tipo de variable local (var), nuevo cliente HTTP API y ZGC (experimental). El JRE ya no se proporciona como descarga separada; solo JDK está disponible. El comando java puede ejecutar programas de código fuente de un solo archivo directamente sin compilación.

### JDK 17

JDK 17 es la última versión de Oracle JDK gratuita bajo NFTC. Las características principales incluyen clases selladas, bloques de texto y expresiones switch. La instalación y uso son similares a JDK 11. La interfaz RandomGenerator proporciona nuevas capacidades de generación de números aleatorios. La API de acceso a memoria extranjera entró en fase de incubación.

### JDK 21

JDK 21 introduce características importantes incluyendo Hilos Virtuales (estable), coincidencia de patrones para switch (estable) y registros (estable). Los hilos virtuales requieren diferentes enfoques de programación concurrente. Las aplicaciones que usan hilos virtuales pueden necesitar ajustes de configuración de heap. El ajuste de rendimiento debe considerar las características de los hilos virtuales.

### JDK 25

JDK 25 es la versión más reciente con las últimas características de lenguaje y mejoras de rendimiento. Las características aún pueden estar en estado de vista previa o experimental. No recomendada para entornos de producción que requieren soporte a largo plazo. La configuración y uso son similares a JDK 21, pero con más funciones nuevas que pueden requerir aprendizaje adicional.
