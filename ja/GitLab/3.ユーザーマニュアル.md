# GitLab ユーザーマニュアル

## アカウント管理

### 登録とログイン

GitLabを初めて使用するには、ログインページで「登録」をクリックしてアカウントを作成します。登録時にはユーザー名、メールアドレス、パスワードが必要です。ユーザー名の選択は慎重にしてください。作成後は変更できません。登録完了後、検証メールを受け取りメール内のリンクをクリックしてメールの確認を完了します。

ログイン後、個人設定でプロフィールを完成させることができます。アバター、姓名、简介などを含め、これらの情報はプロジェクトメンバーリストやコミット履歴に表示されます。コマンドラインでコードをプッシュするためにSSH鍵をバインドすることもできます。

### SSH鍵の設定

SSH鍵を使用すると、ユーザー名とパスワードを入力せずにGitLabレポジトリと安全に認証できます。まずSSH鍵ペアを生成します：

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

生成された公開鍵は~/.ssh/id_ed25519.pubファイルにあります。GitLabにログインし、ユーザー設定ページで「SSH鍵」タブをクリックし、公開鍵の内容をテキストボックスに貼り付けて保存します。鍵を追加すると、SSHでレポジトリをクローンできます：

```bash
git clone git@gitlab.example.com:username/project.git
```

## プロジェクト操作

### プロジェクトの作成

ログイン後、トップナビゲーションバーの「新規プロジェクト」ボタンをクリックして新しいプロジェクトを作成します。空のプロジェクトを作成するか、テンプレートから作成するか、既存のプロジェクトをインポートするかを選択できます。作成時にプロジェクト名、説明、可視性レベル（公開、内部、私有）、デフォルトブランチ名を設定する必要があります。

プロジェクト名は簡潔で明確に、英数字とハイフンを使用して命名してください。私有プロジェクトはプロジェクトメンバーのみアクセス可能、内部プロジェクトは企業内のログイン済みすべてのユーザーに公開、公開プロジェクトはすべての人に表示されます。作成後、プロジェクト設定ページでこれらのオプションを変更できます。

### ブランチ管理

GitLabはデフォルトでmainをメインブランチとして使用します（以前はmaster）。プロジェクトページで「ブランチ」をクリックすると、すべてのブランチを表示・管理できます。新規ブランチの作成は：ブランチページで「新規ブランチ」をクリックするか、Gitコマンドを使用します：

```bash
git checkout -b feature/new-feature
```

保護ブランチはチームコラボレーションの重要な機能です。プロジェクト設定 → リポジトリ → 保護ブランチに移動し、どのユーザーが特定のブランチにプッシュまたはマージできるかを設定できます。mainブランチを保護し、特定の役割またはメンテナー만がマージできるようにすることを推奨します。

### マージリクエスト

マージリクエストはGitLabにおけるコードコラボレーションのコア機能です。機能開発を完了してブランチをプッシュした後、プロジェクトページで「マージリクエスト」→「新規マージリクエスト」をクリックし、ソースブランチとターゲットブランチを選択して、タイトルと説明を入力して作成します。

マージリクエストページには、ソースブランチとターゲットブランチの差分が表示されます。これにはファイルの変更とコミット履歴が含まれます。チームメンバーはここでコードレビューを行い、行ごとにコメントを追加できます。レビューが完了し、マージ条件（CIテストの合格、承認の取得など）が満たされていれば、ブランチをマージできます。

## CI/CDパイプライン

### .gitlab-ci.yml設定

GitLab CI/CDはプロジェクトルートにある.gitlab-ci.ymlファイルを使用してパイプラインを定義します。これは各ビルドステージとジョブを記述するYAML設定ファイルです。簡単な例：

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

この設定は3つのステージを定義しています：buildはプロジェクトのビルド、testはテストの実行、deployはアプリケーションのデプロイです。artifactsはジョブ間でビルドアーティファクトを渡すために使用され、onlyはデプロイジョブをmainブランチでのみ実行するように指定しています。

### GitLab Runner

GitLab RunnerはCI/CDタスクを実行するエージェントです。GitLabは共有Runnerを提供していますが、独自のプライベートRunnerを登録することもできます。サーバーにGitLab Runnerをインストールします：

```bash
# インストール
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Runnerを登録
sudo gitlab-runner register
```

登録にはGitLabインスタンスURLとRegistration Tokenが必要です。これらはプロジェクト設定 → CI/CD → Runnersにあります。Runnerを並行実行に設定して、ビルド速度を向上させることができます。

### パイプライン監視

コードをプッシュするたびに、パイプラインが自動的にトリガーされます。プロジェクトのCI/CD → パイプラインページで、すべてのパイプライン実行状態を表示できます。特定のパイプラин 클릭하면、各ジョブの実行状況を確認できます。これにはログ出力、ビルド時間、失敗理由が含まれます。

GitLabにはパイプラインエディタもあり、CI/CD設定を作成・編集するための視覚的な方法で、手動でYAMLファイルを記述する必要がありません。エディタは構文チェックとリアルタイムプレビューを提供し、設定エラーを削減します。

## 問題追跡

### 問題の作成と管理

問題システムはGitLabプロジェクト管理の基礎です。プロジェクトページで「問題」→「新規問題」をクリックして新しい問題を作成します。タイトルと説明を入力し、特定のメンバーに割り当てたり、ラベル、マイルストーン、期限日を設定したりできます。

問題はラベルまたはマイルストーンで管理されます。一般的なラベルにはbug、feature、enhancement等等があります。ボードビューを使用して問題を視覚化し、問題を列（例：未着手、進行中、完了）ごとにグループ化できます。

### 問題とコードのリンク

コミットメッセージやマージリクエストの説明でキーワード和问题番号を使用すると、コードと問題が自動的にリンクされます。例えば：

- Fixes #123 - 問題を修正し、マージ後に自動的に問題をクローズ
- Closes #456 - 問題をクローズ
- See #789 - 問題を参照

このリンクメカニズムは、チームが作業進捗を追跡し、コードレビュー時に問題の背景と目的を理解するのに役立ちます。

## Wikiとドキュメント

すべてのGitLabプロジェクトには、プロジェクトドキュメントを作成するための組み込みWiki機能があります。WikiページはMarkdown形式で記述され、コード強調表示、表、画像をサポートします。Wikiコンテンツはコードレポジトリと同様にバージョンコントロールをサポートしており、過去のバージョンを表示したりロールバックしたりできます。

プロジェクトに以下のドキュメントを維持することを推奨します：README（プロジェクト紹介）、CONTRIBUTING（コントリビューションガイドライン）、デプロイガイド、APIドキュメント。これらのドキュメントは、オンボーディングの難易度を下げ、社区へのコントリビューションを促進します。
