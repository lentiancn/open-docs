# JDK Preguntas Frecuentes

## Preguntas básicas

### ¿Qué es JDK?

JDK (Java Development Kit) es el kit de desarrollo de software para el lenguaje de programación Java. Es el paquete de software necesario para desarrollar aplicaciones Java. JDK incluye el compilador de Java (javac), la máquina virtual de Java (JVM), bibliotecas de clases centrales de Java y varias herramientas de desarrollo. Sin JDK, no puede desarrollar ni ejecutar programas Java.

La diferencia entre JDK y JRE es: JRE (Java Runtime Environment) solo contiene el entorno necesario para ejecutar programas Java, sin herramientas de desarrollo. Si solo necesita ejecutar programas Java desarrollados por otros, instale JRE; si necesita desarrollar programas Java usted mismo, debe instalar JDK.

### ¿Qué es JVM?

JVM (Java Virtual Machine) es el núcleo de la ejecución de programas Java. JVM es responsable de interpretar o compilar bytecode de Java (.class archivos) para ejecución de código nativo. La existencia de JVM permite que los programas de Java logren la capacidad multiplataforma de "Escribe una vez, ejecuta en cualquier lugar".

JVM no solo ejecuta programas Java, sino que también maneja la gestión de memoria (recolección de basura), gestión de subprocesos, gestión de seguridad, etc. Entender cómo funciona JVM es esencial para escribir aplicaciones Java de alto rendimiento.

### ¿Cuál es la diferencia entre JDK y JRE?

JDK (Java Development Kit) es el kit de desarrollo completo, que contiene JRE y herramientas de desarrollo. JRE (Java Runtime Environment) es el entorno de ejecución de Java, que contiene solo los componentes necesarios para ejecutar programas Java.

En términos simples: si solo quiere ejecutar programas Java hechos por otros, instalar JRE es suficiente; si quiere desarrollar programas Java usted mismo, debe instalar JDK. JDK contiene JRE completo, por lo que después de instalar JDK no necesita instalar JRE por separado.

### ¿Cuál es la mejor versión de JDK?

No existe una "mejor" versión absoluta; la elección depende de las necesidades específicas. Para proyectos nuevos, se recomienda JDK 21 (la última versión LTS) o JDK 25 (la versión más reciente). Para proyectos empresariales que requieren mantenimiento a largo plazo, se recomienda usar versiones LTS como JDK 21 o JDK 17.

JDK 21 es la última versión de soporte a largo plazo, que recibirá al menos 8 años de soporte. JDK 17 todavía se usa ampliamente y es gratuita. JDK 8, aunque antiguo, todavía se usa en muchos sistemas heredados.

## Preguntas de instalación

### ¿Cómo verificar si JDK está instalado correctamente?

Ejecute java -version en la línea de comando. Si se muestra información de la versión, JDK está instalado correctamente. Ejecute javac -version para verificar si el compilador está disponible. Ejecute echo %JAVA_HOME% (Windows) o echo $JAVA_HOME (Linux/macOS) para verificar si las variables de entorno están configuradas correctamente.

Si estos comandos muestran errores en la línea de comando, primero confirme si JDK está instalado correctamente, luego verifique si las variables de entorno están configuradas correctamente.

### ¿Qué es JAVA_HOME?

JAVA_HOME es una variable de entorno que apunta al directorio de instalación de JDK. Muchas herramientas Java y scripts de compilación (como Maven, Gradle) dependen de JAVA_HOME para localizar JDK. Configurar correctamente JAVA_HOME asegura que estas herramientas puedan funcionar correctamente.

En Windows, JAVA_HOME típicamente se establece en rutas como C:\Program Files\Java\jdk-21. En Linux/macOS, típicamente se establece en rutas como /usr/lib/jvm/java-21-openjdk.

### ¿Cómo instalar múltiples versiones de JDK en Windows?

Se pueden instalar múltiples versiones de JDK en la misma máquina. Después de instalar múltiples versiones, cambie la versión predeterminada modificando el orden de los directorios bin de JDK en la variable de entorno PATH, o modificando la variable de entorno JAVA_HOME para apuntar a diferentes directorios de instalación de JDK.

También puede usar herramientas de terceros para administrar múltiples versiones de JDK, como la versión de Windows de SDKMAN! o JetBrains Toolbox. Estas herramientas facilitan el cambio entre versiones.

### ¿Cuál es la diferencia entre Oracle JDK y OpenJDK?

Oracle JDK y OpenJDK son casi idénticos a nivel de código, con las principales diferencias siendo los acuerdos de licencia y soporte técnico. Oracle JDK proporciona soporte comercial y funciones adicionales de nivel empresarial, pero algunas versiones pueden requerir pago para uso comercial. OpenJDK es completamente de código abierto y gratuito, mantenido por la comunidad.

Para la mayoría de los casos de uso, OpenJDK es perfectamente adecuado. Muchas organizaciones eligen usar OpenJDK para evitar problemas de licencia, obteniendo al mismo tiempo un rendimiento y funcionalidad casi idénticos a Oracle JDK.

## Preguntas de uso

### ¿Qué es classpath?

Classpath es la ruta que JVM usa para buscar archivos de clase. Cuando ejecuta java MyClass, JVM buscará MyClass.class en los directorios y archivos JAR especificados por classpath.

Al compilar y ejecutar programas Java, classpath se puede especificar usando las opciones -classpath o -cp. Si no se especifica, JVM usa el directorio actual como classpath.

### ¿Qué es bytecode?

Bytecode es una forma intermedia de código compilado por el compilador Java a partir del código fuente. Bytecode no es código nativo y no puede ejecutarse directamente en la CPU; solo puede ser interpretado por JVM o compilado JIT para ejecución.

La existencia de bytecode da a los programas Java la capacidad multiplataforma. El mismo bytecode puede ejecutarse en cualquier sistema con JVM instalado, ya sea Windows, Linux o macOS.

### ¿Qué es un archivo JAR?

JAR (Java Archive) es un formato para empacar múltiples archivos de clase Java en un solo archivo. Los archivos JAR son en realidad archivos comprimidos en formato ZIP que contienen archivos .class compilados y algunos metadatos.

La herramienta jar se puede usar para crear, ver y extraer archivos JAR. Los archivos JAR ejecutables contienen un archivo MANIFEST.MF que especifica la clase principal, y se pueden ejecutar directamente con java -jar.

### ¿Qué es el sistema de módulos?

El sistema de módulos Java (Project Jigsaw) se introdujo en JDK 9, dividiendo JDK y aplicaciones en módulos. Los módulos proporcionan mejor encapsulamiento y declaran claramente las dependencias, ayudando a construir aplicaciones más robustas y mantenibles.

Los módulos se declaran a través de archivos module-info.java, definiendo el nombre del módulo, dependencias de otros módulos, paquetes exportados, etc. El sistema de módulos ayuda a resolver conflictos de classpath y proporciona mejor gestión de versiones.

## Preguntas de rendimiento

### ¿Cómo optimizar el rendimiento de JVM?

La optimización del rendimiento de JVM se basa principalmente en: configurar apropiadamente el tamaño de la memoria heap, evitando configuraciones demasiado grandes o pequeñas; elegir el recolector de basura apropiado, G1 es una buena opción para la mayoría de los casos; para aplicaciones que buscan baja latencia, considere usar ZGC o Shenandoah; habilitar uso de datos de clase compartidos (-Xshare:classes) para acelerar el inicio.

Use herramientas de monitoreo de rendimiento (como VisualVM, JConsole) para analizar el rendimiento de la aplicación, encontrar cuellos de botella y realizar optimización dirigida.

### ¿Qué es la recolección de basura?

La Recolección de Basura (GC) es el mecanismo por el cual JVM gestiona automáticamente la memoria. Durante la ejecución del programa, se crean objetos que consumen memoria. Cuando los objetos ya no son referenciados, el recolector de basura identifica y recupera automáticamente la memoria ocupada por estos objetos.

La recolección de basura de Java simplifica enormemente el trabajo de gestión de memoria; los programadores no necesitan liberar memoria explícitamente. Sin embargo, entender los principios de la recolección de basura ayuda a escribir código más eficiente, evitando fugas de memoria y GC frecuente.

### ¿Qué es una fuga de memoria?

Una fuga de memoria se refiere a la memoria que no se puede recuperar durante la ejecución del programa, causando que la memoria disponible disminuya gradualmente. En Java, las fugas de memoria generalmente son causadas por colecciones (como Map, List estáticas) que continuamente agregan objetos sin limpiarlos oportunamente.

Para investigar fugas de memoria, puede usar jmap para generar archivos de volcado de heap, luego usar herramientas como MAT (Memory Analyzer Tool) para analizar y encontrar los objetos que causan la fuga.

### ¿Qué es el compilador JIT?

El compilador JIT (Just-In-Time) es parte de JVM, responsable de compilar bytecode caliente en código nativo para mejorar el rendimiento de ejecución. Durante el tiempo de ejecución, JVM analiza la ejecución del código; para código ejecutado frecuentemente (puntos calientes), el compilador JIT lo compila en código de máquina nativo, mejorando significativamente el rendimiento.

La compilación JIT ocurre en tiempo de ejecución sin pasos de compilación adicionales. El compilador C1 compila rápido pero con menos optimización, el compilador C2 compila lento pero con más optimización. La compilación por niveles combina las ventajas de ambos.

## Preguntas de versiones

### ¿Qué versión de Java debo usar?

Para proyectos nuevos, se recomienda la última versión LTS JDK 21. Proporciona buen rendimiento, características ricas y soporte a largo plazo. Para proyectos que necesitan compatibilidad con Java 8/11, pueden usar esas versiones.

No se recomienda usar versiones que han dejado de recibir soporte, como JDK 8 (a menos que tenga suscripción pagada). Actualizar a nuevas versiones de JDK a tiempo proporciona actualizaciones de seguridad y mejoras de rendimiento.

### ¿Qué significa LTS en los números de versión de Java?

LTS (Long-Term Support) son versiones de soporte a largo plazo. Las versiones LTS reciben al menos 8 años de soporte técnico y actualizaciones de parches de Oracle. Las versiones LTS actuales incluyen JDK 8, JDK 11, JDK 17 y JDK 21.

Las versiones no LTS (como JDK 25) tienen ciclos de soporte más cortos, generalmente solo 6 meses. Estas versiones son adecuadas para desarrolladores que desean las últimas características, pero se recomiendan versiones LTS para entornos de producción.

### ¿Puedo seguir usando Java 17?

Java 17 es una versión LTS madura y todavía se puede usar normalmente. Oracle cambió la licencia de JDK 17 en octubre de 2024. Sin embargo, según la declaración de Oracle, JDK 17 todavía se puede usar gratuitamente para desarrollo, uso personal y ejecutar productos con licencia de Oracle.

Muchas empresas todavía usan JDK 17 porque es una versión muy madura y estable con muchos recursos de soporte comunitario. Si el proyecto ya está usando JDK 17, puede seguir usarlo; si comienza un nuevo proyecto, JDK 21 es más recomendado.

## Preguntas de desarrollo

### ¿Cómo compilar archivos Java?

Use el comando javac para compilar archivos Java. Uso básico: javac MyClass.java. Después de la compilación, se genera un archivo de bytecode MyClass.class.

Para proyectos, generalmente se usan herramientas de construcción como Maven o Gradle para gestionar el proceso de compilación. Estas herramientas manejan automáticamente dependencias y opciones de compilación.

### ¿Cómo ejecutar programas Java?

Use el comando java para ejecutar programas Java compilados. Uso básico: java MyClass. Note que no necesita la extensión .class.

Si el programa está empacado en un archivo JAR, use java -jar MyApp.jar para ejecutar.

### ¿Cómo configurar classpath?

Especifique classpath en tiempo de ejecución usando -classpath o -cp: java -cp lib/*:out MyClass. En Windows, el separador es punto y coma (;); en Linux/macOS, es dos puntos (:).

También puede configurar a través de la variable de entorno CLASSPATH, pero especificar en línea de comandos es más flexible.

### ¿Qué son Maven y Gradle?

Maven y Gradle son herramientas de construcción para proyectos Java. Pueden descargar automáticamente dependencias, compilar código, ejecutar pruebas, empaquetar para publicación, simplificando enormemente el proceso de construcción de proyectos.

Maven usa XML (pom.xml) para definir la configuración del proyecto, mientras que Gradle usa scripts Groovy o Kotlin (build.gradle) para definir la lógica de construcción. Ambas son herramientas de construcción ampliamente utilizadas en proyectos Java.
