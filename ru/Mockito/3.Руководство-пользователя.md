# Руководство пользователя Mockito

## Создание макетных объектов

Создание макетных объектов — это первый шаг в использовании Mockito для тестирования. Существует несколько способов создания макетных объектов, и разработчики могут выбрать подходящий метод в зависимости от своих потребностей.

### Использование метода mock()

Самый базовый способ — использовать статический метод mock() из класса Mockito для создания макетных объектов. Этот метод принимает параметр Class и возвращает экземпляр этого класса в виде макета.

```java
import static org.mockito.Mockito.*;

List<String> mockedList = mock(List.class);
Map<String, Integer> mockedMap = mock(Map.class);
```

Начиная с Mockito 4.10.0, вы можете опустить параметр Class, и компилятор автоматически выведет тип.

```java
List<String> mockedList = mock();
```

### Использование аннотации @Mock

Аннотация @Mock предоставляет более чистый стиль объявления, что особенно полезно, когда в тестовом классе нужно создать несколько макетных объектов. Использование аннотации требует предварительной инициализации фреймворка Mockito.

```java
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @Before
    public void setup() {
        MockitoAnnotations.initMocks(this);
    }
}
```

Использование MockitoJUnitRunner может автоматически обрабатывать инициализацию.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

Использование аннотации @ExtendWith из JUnit 5 также достигает автоматической инициализации.

```java
@ExtendWith(MockitoExtension.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

## Определение поведения методов

Определение поведения методов также известно как заглушка. Оно используется для указания, какие значения должны возвращаться или какие исключения должны выбрасываться при вызове методов макетных объектов.

### Базовая заглушка возвращаемых значений

Используйте цепочку синтаксиса when().thenReturn() для определения возвращаемых значений.

```java
when(mockedList.size()).thenReturn(100);
when(mockedList.get(0)).thenReturn("first");
when(mockedList.get(1)).thenReturn("second");
```

Вы можете определить несколько последовательных возвращаемых значений, и каждый последующий вызов будет возвращать разные значения.

```java
when(mockedList.get(anyInt()))
    .thenReturn("first")
    .thenReturn("second")
    .thenThrow(new RuntimeException("Error"));

System.out.println(mockedList.get(0)); // выводит: first
System.out.println(mockedList.get(0)); // выводит: second
System.out.println(mockedList.get(0)); // выбрасывает исключение
```

### Определение выброса исключений

Используйте thenThrow() для определения исключений, которые должны выбрасываться при вызове методов.

```java
when(mockedList.get(0)).thenThrow(new IndexOutOfBoundsException("Index 0"));
```

Для методов с возвращаемым типом void используйте doThrow().

```java
doThrow(new RuntimeException("Error")).when(mockedList).clear();
```

### Использование Answer для пользовательских ответов

Если предопределенных возвращаемых значений недостаточно, вы можете реализовать интерфейс Answer для динамического вычисления возвращаемых значений.

```java
when(mockedList.get(anyInt())).thenAnswer(invocation -> {
    int index = invocation.getArgument(0);
    return "Value at index " + index;
});
```

## Проверка взаимодействий

Проверка используется для подтверждения того, что тестируемый код правильно взаимодействует с макетными объектами, как ожидалось.

### Базовая проверка

Используйте verify() для проверки, был ли вызван метод.

```java
mockedList.add("one");
verify(mockedList).add("one");
```

### Проверка количества вызовов

Вы можете проверить, сколько раз был вызван метод.

```java
mockedList.add("one");
mockedList.add("one");
mockedList.add("one");

verify(mockedList, times(3)).add("one");       // ровно 3 раза
verify(mockedList, atLeast(2)).add("one");     // минимум 2 раза
verify(mockedList, atMost(3)).add("one");      // максимум 3 раза
verify(mockedList, never()).add("two");        // никогда не вызывался
verify(mockedList, atLeastOnce()).add("one");  // хотя бы один раз
```

### Проверка порядка вызовов

Проверщик InOrder может проверять, были ли вызовы методов выполнены в определённом порядке.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
inOrder.verify(mockedList).add("third");
```

### Проверка аргументов вызовов

Для гибкой проверки аргументов вызовов можно использовать matchеры аргументов.

```java
verify(mockedList).add(anyString());
verify(mockedList).get(anyInt());
verify(mockedList).contains(argThat(s -> s.length() > 5));
```

## Matchеры аргументов

Mockito предоставляет богатый набор matchеров аргументов для гибкого сопоставления параметров методов.

### Встроенные matchеры

```java
anyInt()          // любой целочисленный
anyString()       // любая строка
any()             // любой объект
anyList()         // любой список
anyMap()          // любая карта
isNull()          // значение null
isNotNull()       // значение не null
eq(value)         // точное совпадение
startsWith()      // начинается с указанной строки
endsWith()        // заканчивается указанной строкой
contains()        // содержит указанную строку
```

### Пользовательские matchеры

Пользовательские matchеры аргументов можно создать с помощью метода argThat().

```java
when(mockedList.add(argThat(s -> s != null && s.length() > 3)))
    .thenReturn(true);
```

## Использование Spy

В отличие от mock-объектов, spy-объекты сохраняют大部分 поведения реального объекта, позволяя при этом заглушать конкретные методы.

### Создание Spy

```java
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);
```

Или используйте аннотацию @Spy.

```java
@Spy
private List<String> list = new ArrayList<>();
```

### Определение поведения Spy

В отличие от mock-объектов, нужно быть осторожным при использовании when() со spy, поскольку это может повлиять на реальные вызовы методов. Рекомендуется использовать синтаксис doReturn().when().

```java
doReturn(100).when(spyList).size();
```

Spy вызывают реальные методы, если они явно не заглушены.

```java
spyList.add("element");  // реальный метод будет вызван
verify(spyList).add("element");  // проверка пройдена
```

## Использование @InjectMocks

Аннотация @InjectMocks используется для автоматического создания тестируемого объекта и внедрения полей с аннотациями @Mock или @Spy в него.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @InjectMocks
    private UserService userService;
    
    @Test
    public void testCreateUser() {
        User user = new User("testUser");
        when(userRepository.save(any(User.class))).thenReturn(user);
        
        User result = userService.createUser("testUser");
        
        assertEquals("testUser", result.getUsername());
        verify(userRepository).save(any(User.class));
    }
}
```

## Использование ArgumentCaptor

ArgumentCaptor используется для захвата аргументов, переданных во время вызовов методов, для более детальных проверок.

```java
@Mock
private List<String> mockedList;

@Test
public void testCaptureArgument() {
    mockedList.add("test");
    
    ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
    verify(mockedList).add(captor.capture());
    
    assertEquals("test", captor.getValue());
}
```

Когда метод вызывается несколько раз, вы можете захватить все аргументы.

```java
mockedList.add("first");
mockedList.add("second");
mockedList.add("third");

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList, times(3)).add(captor.capture());

assertEquals(3, captor.getAllValues().size());
assertEquals("second", captor.getAllValues().get(1));
```

## BDD-стиль тестирования

Mockito поддерживает тестирование в стиле разработки, управляемой поведением (BDD), используя синтаксис given().willReturn().

```java
import static org.mockito.BDDMockito.*;

@Test
public void testBddStyle() {
    // Given
    given(mockedList.get(0)).willReturn("first");
    
    // When
    String result = mockedList.get(0);
    
    // Then
    assertEquals("first", result);
}
```

## Распространённые шаблоны использования

### Тестирование void-методов

```java
@Test
public void testVoidMethod() {
    doNothing().when(mockedList).clear();
    mockedList.clear();
    verify(mockedList).clear();
}
```

### Мокирование цепочек методов

```java
@Test
public void testMethodChaining() {
    when(mockedList.iterator()).thenReturn(Collections.emptyIterator());
    
    Iterator<String> iterator = mockedList.iterator();
    assertFalse(iterator.hasNext());
}
```

### Мокирование обработки исключений

```java
@Test(expected = RuntimeException.class)
public void testException() {
    when(mockedList.get(0)).thenThrow(new RuntimeException("Error"));
    mockedList.get(0);
}
```

## Лучшие практики

При использовании Mockito следуйте лучшим практикам для обеспечения качества и поддерживаемости тестового кода. Не мокируйте типы, которыми вы владеете или объекты-значения, поскольку это нарушает достоверность тестов. Мокируйте только необходимые зависимости для поддержания достоверности тестов. Каждый тестовый метод должен фокусироваться на тестировании одной функции. Правильно организуйте тестовый код, используя @Before для общей настройки. Проверка должна быть умеренной, фокусируясь на важных взаимодействиях, а не на проверке каждого вызова метода.
