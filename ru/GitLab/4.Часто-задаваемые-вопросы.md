# FAQ GitLab

## Проблемы установки

### Что делать, если установка GitLab не удалась?

Сбой установки обычно вызывается несколькими распространёнными причинами. Сначала проверьте, соответствует ли система минимальным требованиям, включая версию операционной системы, память и дисковое пространство. Если установка с помощью скрипта самостоятельного размещения не удалась, попробуйте вручную установить пакеты зависимостей, а затем устанавливать пошагово.

Просмотрите журналы установки для получения подробной информации об ошибках:

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# Просмотр журналов
sudo tail -f /var/log/gitlab/install.log
```

Распространённые ошибки включают: конфликты портов (порты 80, 443, 22 заняты), недостаточно памяти, проблемы с SSL-сертификатами и другие. При конфликте портов либо остановите службу, занимающую порт, либо измените конфигурацию GitLab для использования других портов.

### Как изменить адрес доступа к GitLab?

Если нужно изменить адрес доступа к GitLab после установки, отредактируйте параметр external_url в файле /etc/gitlab/gitlab.rb, затем перенастройте:

```bash
sudo nano /etc/gitlab/gitlab.rb
# Изменить external_url
sudo gitlab-ctl reconfigure
```

После изменения убедитесь, что разрешение DNS или локальный файл hosts правильно настроены, чтобы новый адрес был доступен. При изменении с HTTP на HTTPS также необходимо настроить SSL-сертификаты.

## Проблемы использования

### Что делать, если подключение по SSH-ключу не удаётся?

Сбой подключения по SSH-ключу может быть вызван несколькими причинами. Сначала подтвердите, что открытый ключ был правильно добавлен в SSH-ключи учётной записи GitLab:

```bash
# Просмотр локального открытого ключа
cat ~/.ssh/id_ed25519.pub

# Тест подключения
ssh -T git@gitlab.example.com
```

Если тест подключения не удаётся, проверьте конфигурацию SSH-клиента:

```bash
# Редактирование конфигурации SSH
nano ~/.ssh/config

# Добавить следующее
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

Убедитесь, что URL удалённого репозитория использует формат SSH (git@gitlab.example.com:group/project.git), а не формат HTTPS.

### Что делать, если запрос на слияние нельзя слить?

Запросы на слияние могут не сливаться по следующим причинам: существуют конфликты, тесты CI/CD не пройдены, правила защищённых веток не выполнены, отсутствуют утверждения и другие. Решение — проверять по пунктам:

- Конфликты слияния: Нажмите «Обновить ветку» на странице запроса на слияние, влейте целевую ветку в исходную, разрешите конфликты и повторно отправьте
- Сбой CI: Просмотрите журнал невыполненной задачи, устраните проблему и повторно запустите конвейер
- Защищённая ветка: Свяжитесь с администратором проекта для изменения настроек защищённых веток или попросите участника с правами выполнить слияние

### Как устранить сбои выполнения конвейеров CI/CD?

Когда конвейер не выполняется, сначала просмотрите журнал невыполненной задачи на странице CI/CD → Конвейеры. Журнал покажет конкретную информацию об ошибках и причины сбоя. Распространённые проблемы включают: сбой загрузки зависимостей, ошибки команд скрипта, неудачные тестовые случаи и другие.

Если проблему сложно определить, можно добавить шаги отладки в конфигурацию конвейера, например вывод переменных окружения, проверку состояния файлов до и после установки и другие. Также можно использовать функцию интерактивной отладки для запуска интерактивной оболочки в задачах конвейера для устранения неполадок.

## Проблемы производительности

### Доступ к GitLab очень медленный, что делать?

Медленный доступ может быть вызван недостаточными ресурсами сервера, задержкой сети или неправильной конфигурацией. Сначала проверьте нагрузку на сервер:

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

Если использование ЦП или памяти очень высокое, может потребоваться обновить конфигурацию сервера или оптимизировать настройки GitLab. Также можно проверить производительность базы данных:

```bash
sudo gitlab-rake gitlab:git:fsck  # Проверка целостности репозитория Git
sudo gitlab-rake db:index_cache_daily  # Оптимизация кэша базы данных
```

При проблемах с сетью рассмотрите возможность использования CDN для ускорения доставки статических ресурсов или развёртывания GitLab в местах, ближе расположенных к пользователям.

### Как оптимизировать производительность GitLab?

Оптимизация производительности GitLab может быть достигнута несколькими способами. Оптимизация базы данных важна. Регулярное обслуживание может значительно улучшить производительность:

```bash
# Очистка базы данных
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# Очистка неиспользуемых файлов
sudo gitlab-rake gitlab:cleanup:builds
```

Для больших репозиториев можно включить пулы объектов Git для уменьшения использования диска или настроить Git LFS для хранения больших файлов. Правильная настройка кэширования CI/CD может ускорить сборки.

## Проблемы безопасности

### Как настроить SSL-сертификаты?

GitLab поддерживает автоматическую настройку SSL с Let's Encrypt или можно использовать собственные сертификаты. Сначала отредактируйте файл конфигурации:

```ruby
# Автоматическая настройка Let's Encrypt
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# Пользовательский сертификат
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

Выполните sudo gitlab-ctl reconfigure для применения конфигурации. При использовании самоподписанных сертификатов клиенты должны доверять сертификату для доступа по HTTPS.

### Как создавать резервные копии и восстанавливать GitLab?

Регулярное резервное копирование важно для предотвращения потери данных. GitLab предоставляет встроенный инструмент резервного копирования:

```bash
# Создание резервной копии
sudo gitlab-backup create

# Расположение файлов резервных копий
ls -la /var/opt/gitlab/backups/
```

Файлы резервных копий содержат базы данных, загруженные файлы и репозитории Git. Для восстановления: остановите службу GitLab, скопируйте файл резервной копии в расположение по умолчанию, затем выполните команду восстановления:

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

Рекомендуется автоматически синхронизировать файлы резервных копий с удалённым хранилищем, например службами объёмного хранения или другим сервером.

## Проблемы обновления

### Что делать, если после обновления появляются ошибки?

После обновления могут возникнуть ошибки совместимости или функциональные аномалии. Сначала проверьте состояние GitLab:

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

Просмотрите журналы обновления для понимания конкретных ошибок:

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

Распространённые проблемы включают: сбой миграции базы данных, ошибки синтаксиса файла конфигурации, отсутствующие зависимости и другие. При проблеме с миграцией базы данных может потребоваться ручное исправление:

```bash
sudo gitlab-rake db:migrate
```

Рекомендуется создавать резервную копию данных перед обновлением и внимательно читать официальную документацию по обновлению для понимания важных изменений между версиями.

### Как безопасно обновляться между разными версиями?

Обновление GitLab должно следовать ограничениям на пропуск версий, не пропуская более одной основной версии за раз. Например, обновляйтесь с 15.x на 16.x, а не напрямую с 15.x на 17.x. Сначала обновитесь до промежуточной версии, подтвердите нормальную работу, затем продолжайте обновление.

Перед обновлением выполните следующие шаги: создайте резервную копию данных, проверьте проблемы совместимости, прочитайте примечания к выпуску. Поддерживайте стабильное питание и сеть во время процесса обновления и проверьте, что все функции работают нормально после обновления.
