# Foire Aux Questions Hadoop

## Problèmes Liés à HDFS

### Que Faire Lorsque l'Espace de Stockage HDFS Est Insuffisant

L'espace de stockage HDFS insuffisant est un problème courant qui peut avoir plusieurs causes. Vérifiez d'abord l'utilisation du disque des DataNodes :

```bash
hdfs dfsadmin -report
```

Si l'espace est effectivement insuffisant, prenez les mesures suivantes : supprimez les données historiques qui ne sont plus nécessaires ; utilisez hdfs balancer pour équilibrer la distribution des données et libérer certains nœuds ; ajoutez de nouveaux nœuds DataNode pour étendre la capacité de stockage du cluster ; réduisez le facteur de réplication pour diminuer la surcharge de stockage (équilibrer avec la sécurité des données).

```bash
# Supprimer des fichiers
hdfs dfs -rm -r /user/olddata

# Équilibrer les données
hdfs balancer -threshold 10
```

### Comment Gérer la Mémoire Insuffisante du NameNode

NameNode stocke toutes les métadonnées en mémoire ; s'il y a trop de fichiers, la mémoire peut devenir insuffisante. Atténuez en : augmentant la configuration de mémoire du NameNode dans hadoop-env.sh en définissant le paramètre -Xmx dans HADOOP_NAMENODE_OPTS ; activant la fonctionnalité de snapshots NameNode pour créer régulièrement des copies en lecture seule des données importantes ; passant à Hadoop 2.x ou supérieur, qui a une meilleure optimisation de gestion des métadonnées.

```bash
# Définir dans hadoop-env.sh
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### Que Faire Quand le DataNode Ne Démarre Pas

L'échec du démarrage du DataNode peut avoir plusieurs causes. Vérifiez d'abord les journaux du DataNode :

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

Causes et solutions courantes : espace disque insuffisant — nettoyez les disques ou ajoutez-en de nouveaux ; problèmes de permissions — assurez-vous que l'utilisateur Hadoop a un accès lecture/écriture aux répertoires de données ; ID de cluster incohérent — si vous ajoutez un nouveau DataNode, exécutez hdfs namenode -upgrade sur le DataNode ou supprimez le répertoire de stockage et reformatez.

```bash
# Vérifier les permissions du répertoire DataNode
ls -la /usr/local/hadoop/data/

# Supprimer le répertoire de stockage DataNode et redémarrer
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### Comment Récupérer des Fichiers Supprimés Accidentellement

HDFS fournit une fonctionnalité de corbeille pour récupérer les fichiers supprimés. Confirmez d'abord que la corbeille est activée :

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

Si vous constatez une suppression accidentelle, restaurez depuis la corbeille :

```bash
# Restaurer le fichier
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

Si la corbeille a été vidée ou que la période de conservation est passée, les données ne peuvent pas être récupérées. Par conséquent, dans les opérations quotidiennes : gardez la corbeille activée (activée par défaut) ; sauvegardez les données importantes ; confirmez soigneusement avant de supprimer.

### Comment Gérer les Blocs HDFS Endommagés

Les blocs de données peuvent être endommagés en raison de pannes de disque ou de problèmes réseau. HDFS a des mécanismes intégrés de détection d'intégrité des données : lorsque des blocs endommagés sont détectés, il réplique automatiquement les données à partir d'autres réplicas pour la récupération. Vérification et réparation manuelle :

```bash
# Vérifier l'intégrité du système de fichiers
hdfs fsck / -files -blocks -locations

# Supprimer les fichiers de blocs endommagés
hdfs fsck / -delete
```

## Problèmes Liés à MapReduce

### Que Faire Quand les Travaux Lent

Les travaux MapReduce lents peuvent être causés par plusieurs facteurs. Vérifiez d'abord les journaux d'historique des travaux et analysez le temps passé dans chaque phase :

```bash
# Voir les détails du travail
yarn logs -applicationId application_xxx -logFiles *

# Voir dans JobHistory Server
mapred job -history job_history_file
```

Orientations d'optimisation courantes : augmentez le nombre de tâches Reduce pour améliorer le parallélisme ; ajustez la taille du tas JVM pour éviter OOM ; activez Combiner pour réduire le volume de transmission réseau ; optimisez l'asymétrie des données en utilisant des partitionneurs personnalisés pour équilibrer la charge ; choosez des formats de fichiers appropriés (Parquet, ORC, etc.).

### Comment Gérer l'Asymétrie des Données

L'asymétrie des données signifie que les données sont inégalement distribuées pendant la phase Reduce, provoquant l'exécution de certaines tâches Reduce beaucoup plus longtemps que d'autres. Les solutions incluent : utilisez des partitionneurs personnalisés pour redistribuer les données selon la logique métier ; ajoutez un préfixe aléatoire pour laisser les données se disperser d'abord vers plusieurs Reduces, puis agrégez secondairement ; utilisez le salting de clé — supprimez le préfixe après l'agrégation.

```java
// Exemple de partitionneur personnalisé
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // Redistribuer selon la logique métier
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### Comment Dépanner les Échecs de Travaux MapReduce

Lorsque les travaux échouent, vérifiez d'abord les journaux d'erreurs pour déterminer la cause de l'échec :

```bash
# Voir l'état du travail
yarn application -status application_xxx

# Voir les raisons de l'échec des tâches
mapred job -status job_id
```

Causes courantes d'échec : Java OutOfMemory (OOM) — augmentez mapreduce.map.memory.mb et mapreduce.reduce.memory.mb ; erreur de format des données d'entrée — vérifiez si les données correspondent au format attendu ; exception de code personnalisé — vérifiez la logique du code et les messages d'exception ; timeout réseau — augmentez le paramètre mapreduce.task.timeout.

## Problèmes Liés à YARN

### Que Faire Quand le ResourceManager Ne Démarre Pas

L'échec du démarrage du ResourceManager est généralement causé par des erreurs de configuration ou des conflits de ports. Vérifiez les journaux :

```bash
cat $HARNESS_HOME/logs/yarn-resourcemanager-*.log
```

Problèmes et solutions courants : port déjà utilisé — modifiez le port du ResourceManager dans yarn-site.xml ; erreur de configuration ZooKeeper (si HA) — vérifiez la configuration liée à zkfc ; mémoire insuffisante — assurezvous que la machine a suffisamment de mémoire physique ; erreur de fichier de configuration — vérifiez soigneusement yarn-site.xml et capacity-scheduler.xml.

```bash
# Vérifier l'utilisation du port
netstat -tlnp | grep 8032
```

### Comment Gérer la Faible Utilisation des Ressources du Cluster

La faible utilisation des ressources du cluster peut être causée par : modèles de soumission de travaux déraisonnables — travaux trop petits faisant que la surcharge d'ordonnancement dépasse le temps de calcul ; configuration inappropriée de l'ordonnanceur — allocation des ressources de file d'attente déraisonnable ; mauvaise localité des données — données d'entrée et nœuds de calcul pas au même endroit.

Suggestions d'optimisation : soumettez les petits travaux en lots ; configurez raisonnablement la capacité et les priorités des files d'attente ; optimisez la disposition des données pour améliorer la localité des données ; utilisez l'ordonnanceur approprié (Capacity Scheduler ou Fair Scheduler).

### Comment Limiter l'Utilisation des Ressources par les Utilisateurs

Utilisez les files d'attente YARN et le contrôle d'accès pour limiter l'utilisation des ressources par les utilisateurs :

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- Multiple maximum des ressources de file d'attente pour un seul utilisateur -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- Nombre maximum de travaux en cours dans la file d'attente -->
</property>
```

## Problèmes d'Opération du Cluster

### Comment Surveiller le Cluster Hadoop

Hadoop fournit plusieurs méthodes de surveillance : Interface Web (NameNode port 9870, ResourceManager port 8088) ; Surveillance JMX (Java Management Extensions) ; Systèmes de surveillance externes comme Ganglia, Prometheus ; Outils de ligne de commande (hdfs dfsadmin, yarn node, etc.).

Il est recommandé d'utiliser Prometheus + Grafana pour construire une plateforme de surveillance complète — collectez les métriques JMX d'Hadoop et visualisez-les.

### Comment Effectuer une Mise à Niveau du Cluster

La mise à niveau du cluster Hadoop nécessite une manipulation prudente. Étapes recommandées : vérifiez d'abord la compatibilité de la nouvelle version dans un environnement de test ; sauvegardez les métadonnées NameNode et les données HDFS ; mettez à niveau les nœuds DataNode un par un, puis mettez à niveau NameNode ; utilisez la mise à niveau progressive pour réduire le temps d'indisponibilité.

```bash
# Mise à niveau progressive DataNode
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Mettre à niveau Hadoop
tar -xzf hadoop-new.tar.gz -C /usr/local/
# Démarrer DataNode
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### Comment Configurer la Haute Disponibilité du NameNode

NameNode HA évite le point de défaillance unique du NameNode. Étapes de configuration : configurez deux NameNodes dans hdfs-site.xml ; configurez le répertoire d'éditions partagé (utilisant généralement ZooKeeper ou NFS) ; démarrez ZooKeeper Failover Controller (ZKFC).

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

Après la configuration, initialisez ZKFC :

```bash
hdfs zkfc -formatZK
```

### Comment Gérer la Désaffectation des Nœuds

La désaffectation des nœuds nécessite une manipulation prudente pour assurer aucune perte de données. Utilisez la fonctionnalité de decommission pour mettre hors service les nœuds en toute sécurité :

```bash
# Ajouter le nœud à exclure dans le fichier workers
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# Actualiser la liste des nœuds
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

Attendez que la réplication des données soit terminée — le nœud entrera automatiquement en état decommissioned. Confirmez que les données ont été répliquées vers d'autres nœuds avant d'arrêter le nœud.

## Problèmes de Développement

### Comment Déboguer les Programmes MapReduce

Le débogage des programmes MapReduce est difficile. Méthodes recommandées : testez d'abord la logique en mode local ; utilisez des Compteurs pour enregistrer les métriques importantes ; ajoutez une sortie de journal — imprimez les valeurs des variables clés dans les méthodes Map ou Reduce ; utilisez MRUnit pour les tests unitaires.

```java
// Utiliser un Compteur
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### Comment Choisir InputFormat et OutputFormat

InputFormat détermine comment lire les données d'entrée : TextInputFormat (par défaut, lit par ligne) ; KeyValueTextInputFormat (divise par tabulation) ; SequenceFileInputFormat (format binaire) ; DBInputFormat (lit depuis une base de données).

OutputFormat détermine comment émettre les résultats : TextOutputFormat (par défaut, format texte) ; SequenceFileOutputFormat (format binaire) ; NullOutputFormat (pas de sortie de fichier) ; DBOutputFormat (émet vers une base de données).

### Comment Gérer le Problème des Petits Fichiers

Les grandes quantités de petits fichiers augmentent la pression mémoire du NameNode et réduisent l'efficacité du traitement. Solutions : utilisez Hadoop Archives (HAR) pour emballer les petits fichiers ; utilisez SequenceFile pour fusionner les petits fichiers ; utilisez HBase pour stocker de grandes quantités de petits fichiers ; ajustez la taille des blocs pour réduire le nombre de petits fichiers.
