# Руководство пользователя Kubernetes

## Основные концепции

### Pod
Pod — наименьшая развёртываемая единица в Kubernetes, представляющая один или несколько контейнеров, работающих в кластере. Обычно Pod содержит один контейнер, но также может содержать несколько тесно связанных контейнеров.

### Deployment
Deployments используются для управления приложениями без состояния, предоставляя возможности декларативного обновления для простого развёртывания, масштабирования и обновления.

### Service
Services предоставляют стабильные сетевые конечные точки для набора подов, сохраняя стабильность даже при изменении IP-адресов подов.

### Namespace
Namespaces делят ресурсы кластера на несколько логических разделов, подходят для нескольких команд или проектов, использующих один кластер.

## Общие команды

### Управление кластером

```bash
# Просмотреть информацию о кластере
kubectl cluster-info

# Просмотреть узлы
kubectl get nodes

# Просмотреть все поды
kubectl get pods --all-namespaces

# Просмотреть использование ресурсов кластера
kubectl top nodes
kubectl top pods
```

### Развёртывание приложений

```bash
# Создать Deployment
kubectl create deployment nginx --image=nginx

# Просмотреть Deployments
kubectl get deployments

# Масштабировать
kubectl scale deployment nginx --replicas=3

# Обновить образ
kubectl set image deployment/nginx nginx=nginx:1.21

# Откатить
kubectl rollout undo deployment/nginx

# Просмотреть статус развёртывания
kubectl rollout status deployment/nginx
```

### Управление сервисами

```bash
# Открыть сервис
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Просмотреть сервисы
kubectl get services

# Удалить сервис
kubectl delete service <service-name>
```

### Отладка

```bash
# Просмотреть логи пода
kubectl logs <pod-name>

# Войти в контейнер
kubectl exec -it <pod-name> -- /bin/bash

# Описать детали ресурса
kubectl describe pod <pod-name>

# Перенаправление портов для отладки
kubectl port-forward <pod-name> 8080:80
```

## Манифесты ресурсов

Kubernetes поддерживает декларативное управление ресурсами с помощью YAML-файлов. Примеры распространённых определений ресурсов:

### Пример Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        ports:
        - containerPort: 8080
        resources:
          limits:
            memory: "128Mi"
            cpu: "500m"
          requests:
            memory: "64Mi"
            cpu: "250m"
```

### Пример Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
  - protocol: TCP
    port: 80
    targetPort: 8080
  type: ClusterIP
```

### Пример ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
data:
  database_url: "postgres://db:5432/myapp"
  cache_enabled: "true"
```

### Пример Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
stringData:
  username: admin
  password: changeme
```

## Управление жизненным циклом приложений

### Развёртывание приложений

```bash
# Развернуть с помощью YAML-файла
kubectl apply -f deployment.yaml

# Просмотреть статус развёртывания
kubectl get pods -l app=my-app
```

### Обновление приложений

```bash
# Обновить версию образа
kubectl set image deployment/my-app my-app=my-app:v2

# Просмотреть прогресс обновления
kubectl rollout status deployment/my-app
```

### Откат

```bash
# Откатить к предыдущей версии
kubectl rollout undo deployment/my-app

# Откатить к конкретной ревизии
kubectl rollout undo deployment/my-app --to-revision=2

# Просмотреть историю ревизий
kubectl rollout history deployment/my-app
```

### Масштабирование

```bash
# Ручное масштабирование
kubectl scale deployment my-app --replicas=5

# Автоматическое масштабирование (HPA)
kubectl autoscale deployment my-app --min=2 --max=10 --cpu-percent=80
```

## Управление хранилищем

### Создание PersistentVolume

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  hostPath:
    path: /data/my-app
```

### Создание PersistentVolumeClaim

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

### Использование PVC в Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: my-app
    image: my-app
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

## Управление конфигурацией

### Использование ConfigMap

```bash
# Создать ConfigMap из файла
kubectl create configmap my-config --from-file=config.properties

# Создать из файла переменных окружения
kubectl create configmap my-config --from-env-file=env.properties

# Использовать в Pod
kubectl apply -f pod-with-configmap.yaml
```

### Использование Secret

```bash
# Создать Secret из файла
kubectl create secret generic my-secret --from-file=credentials.json

# Использовать в Pod
kubectl apply -f pod-with-secret.yaml
```

## Квоты и ограничения ресурсов

### ResourceQuota

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: my-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
```

### LimitRange

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: my-limits
spec:
  limits:
  - max:
      cpu: "2"
      memory: 1Gi
    min:
      cpu: 100m
      memory: 128Mi
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 200m
      memory: 256Mi
    type: Container
```

## Мониторинг и журналы

### Мониторинг

```bash
# Просмотреть использование ресурсов с помощью kubectl top
kubectl top nodes
kubectl top pods

# Просмотреть подробный статус
kubectl describe node <node-name>
kubectl describe pod <pod-name>
```

### Журналы

```bash
# Просмотреть логи пода
kubectl logs my-app-pod

# Просматривать логи в реальном времени
kubectl logs -f my-app-pod

# Просмотреть логи предыдущего экземпляра контейнера
kubectl logs --previous my-app-pod
```

## Рекомендации

1. Использовать декларативную конфигурацию для управления инфраструктурой
2. Устанавливать соответствующие ограничения и запросы ресурсов
3. Организовывать ресурсы с помощью меток
4. Регулярно очищать неиспользуемые ресурсы
5. Использовать namespaces для изоляции различных сред
6. Настраивать проверки жизнеспособности
7. Использовать RBAC для контроля разрешений
8. Регулярно создавать резервные копии данных etcd
