# GitLab FAQ

## Installationsprobleme

### Was tun, wenn die GitLab-Installation fehlschlägt?

Installationsfehler werden normalerweise durch einige häufige Ursachen verursacht. Überprüfen Sie zunächst, ob das System die Mindestanforderungen erfüllt, einschließlich Betriebssystemversion, Speicher und Festplattenplatz. Wenn die Installation mit dem Self-Hosted-Skript fehlschlägt, versuchen Sie, Abhängigkeitspakete manuell zu installieren und dann Schritt für Schritt zu installieren.

Zeigen Sie Installationsprotokolle an, um detaillierte Fehlerinformationen zu erhalten:

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# Protokolle anzeigen
sudo tail -f /var/log/gitlab/install.log
```

Häufige Fehler sind: Port-Konflikte (Ports 80, 443, 22 werden verwendet), unzureichender Speicher, SSL-Zertifikatsprobleme usw. Wenn ein Port-Konflikt vorliegt, stoppen Sie entweder den Dienst, der den Port verwendet, oder ändern Sie die GitLab-Konfiguration, um andere Ports zu verwenden.

### Wie ändere ich die GitLab-Zugriffsadresse?

Wenn Sie die GitLab-Zugriffsadresse nach der Installation ändern müssen, bearbeiten Sie den Parameter external_url in /etc/gitlab/gitlab.rb und konfigurieren Sie dann neu:

```bash
sudo nano /etc/gitlab/gitlab.rb
# external_url ändern
sudo gitlab-ctl reconfigure
```

Nach der Änderung stellen Sie sicher, dass die DNS-Auflösung oder die lokale Hosts-Datei korrekt konfiguriert ist, damit die neue Adresse normal erreicht werden kann. Wenn Sie von HTTP zu HTTPS wechseln, müssen Sie auch SSL-Zertifikate konfigurieren.

## Nutzungsprobleme

### Was tun, wenn die SSH-Schlüssel-Verbindung fehlschlägt?

SSH-Schlüssel-Verbindungsfehler können durch mehrere Gründe verursacht werden. Bestätigen Sie zunächst, dass der öffentliche Schlüssel korrekt zu den SSH-Schlüsseln des GitLab-Kontos hinzugefügt wurde:

```bash
# Lokalen öffentlichen Schlüssel anzeigen
cat ~/.ssh/id_ed25519.pub

# Verbindung testen
ssh -T git@gitlab.example.com
```

Wenn der Verbindungstest fehlschlägt, überprüfen Sie die SSH-Client-Konfiguration:

```bash
# SSH-Konfiguration bearbeiten
nano ~/.ssh/config

# Folgendes hinzufügen
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

Stellen Sie sicher, dass die Remote-Repository-URL das SSH-Format verwendet (git@gitlab.example.com:group/project.git), nicht das HTTPS-Format.

### Was tun, wenn ein Merge Request nicht gemergt werden kann?

Merge Requests können aus folgenden Gründen nicht gemergt werden: Konflikte vorhanden, CI/CD-Tests fehlgeschlagen, geschützte Branch-Regeln nicht erfüllt, Genehmigungen fehlen usw. Die Lösung ist, Punkt für Punkt zu überprüfen:

- Merge-Konflikte: Klicken Sie auf "Branch aktualisieren" auf der Merge Request-Seite, mergen Sie den Zielbranch in den Quellbranch, lösen Sie die Konflikte und pushen Sie erneut
- CI-Fehler: Zeigen Sie das Protokoll der fehlgeschlagenen Aufgabe an, beheben Sie das Problem und führen Sie die Pipeline erneut aus
- Geschützter Branch: Wenden Sie sich an den Projektadministrator, um die Einstellungen für geschützte Branches zu ändern, oder lassen Sie ein Mitglied mit Berechtigungen mergen

### Wie behebe ich fehlgeschlagene CI/CD-Pipeline-Ausführungen?

Wenn eine Pipeline fehlschlägt, zeigen Sie zunächst das Protokoll der fehlgeschlagenen Aufgabe auf der Seite CI/CD → Pipelines an. Das Protokoll zeigt spezifische Fehlerinformationen und Fehlerursachen. Häufige Probleme sind: Abhängigkeits-Download fehlgeschlagen, Skriptbefehlsfehler, Testfallfehler usw.

Wenn das Problem schwer zu lokalisieren ist, können Sie Debugging-Schritte zur Pipeline-Konfiguration hinzufügen, z.B. Umgebungsvariablen ausgeben, Dateizustände vor und nach der Installation überprüfen usw. Sie können auch die interaktive Debugging-Funktion verwenden, um eine interaktive Shell in Pipeline-Aufgaben zur Fehlerbehebung zu starten.

## Leistungsprobleme

### GitLab-Zugriff ist sehr langsam, was tun?

Langsamer Zugriff kann durch unzureichende Serverressourcen, Netzwerklatenz oder unsachgemäße Konfiguration verursacht werden. Überprüfen Sie zunächst die Serverlast:

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

Wenn die CPU- oder Speicherauslastung sehr hoch ist, müssen Sie möglicherweise die Serverkonfiguration upgraden oder GitLab-Einstellungen optimieren. Sie können auch die Datenbankleistung überprüfen:

```bash
sudo gitlab-rake gitlab:git:fsck  # Git-Repository-Integrität prüfen
sudo gitlab-rake db:index_cache_daily  # Datenbank-Cache optimieren
```

Bei Netzwerkproblemen sollten Sie die Verwendung eines CDN zur Beschleunigung der Bereitstellung statischer Ressourcen in Betracht ziehen oder GitLab an netzwerknäheren Standorten bereitstellen.

### Wie optimiere ich die GitLab-Leistung?

Die Optimierung der GitLab-Leistung kann auf mehrere Arten angegangen werden. Die Datenbankoptimierung ist wichtig. Regelmäßige Wartung kann die Leistung erheblich verbessern:

```bash
# Datenbank bereinigen
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# Unbenutzte Dateien bereinigen
sudo gitlab-rake gitlab:cleanup:builds
```

Für große Repositorys können Sie Git-Objektpools aktivieren, um die Festplattennutzung zu reduzieren, oder Git LFS zum Speichern großer Dateien konfigurieren. Das vernünftige Einrichten von CI/CD-Caching kann die Build-Geschwindigkeit verbessern.

## Sicherheitsprobleme

### Wie konfiguriere ich SSL-Zertifikate?

GitLab unterstützt die automatische SSL-Konfiguration mit Let's Encrypt, oder Sie können benutzerdefinierte Zertifikate verwenden. Bearbeiten Sie zunächst die Konfigurationsdatei:

```ruby
# Let's Encrypt-Autokonfiguration
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# Benutzerdefiniertes Zertifikat
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

Führen Sie sudo gitlab-ctl reconfigure aus, um die Konfiguration anzuwenden. Wenn Sie selbstsignierte Zertifikate verwenden, müssen Clients das Zertifikat vertrauen, um über HTTPS zuzugreifen.

### Wie sichere und stelle ich GitLab wieder her?

Regelmäßige Backups sind wichtig, um Datenverlust zu verhindern. GitLab bietet ein integriertes Backup-Tool:

```bash
# Backup erstellen
sudo gitlab-backup create

# Backup-Dateistandort
ls -la /var/opt/gitlab/backups/
```

Backup-Dateien enthalten Datenbanken, hochgeladene Dateien und Git-Repositorys. Zur Wiederherstellung: GitLab-Dienst stoppen, Backup-Datei an den Standardort kopieren, dann Wiederherstellungsbefehl ausführen:

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

Es wird empfohlen, Backup-Dateien automatisch mit Remote-Speicher zu synchronisieren, z.B. mit Object-Storage-Diensten oder einem anderen Server.

## Upgrade-Probleme

### Was tun, wenn nach dem Upgrade Fehler auftreten?

Nach einem Upgrade können Kompatibilitätsfehler oder Funktionsanomalien auftreten. Überprüfen Sie zunächst den GitLab-Status:

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

Zeigen Sie Upgrade-Protokolle an, um spezifische Fehler zu verstehen:

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

Häufige Probleme sind: Datenbank-Migrationsfehler, Konfigurationsdatei-Syntaxfehler, fehlende Abhängigkeiten usw. Wenn es sich um ein Datenbank-Migrationsproblem handelt, müssen Sie möglicherweise manuell beheben:

```bash
sudo gitlab-rake db:migrate
```

Es wird empfohlen, vor dem Upgrade ein Backup zu erstellen und die offizielle Upgrade-Dokumentation sorgfältig zu lesen, um wichtige Änderungen zwischen Versionen zu verstehen.

### Wie upgrade ich sicher zwischen verschiedenen Versionen?

Das Upgrade von GitLab sollte Versionssprunggrenzen folgen, wobei nicht mehr als eine Hauptversion auf einmal übersprungen werden sollte. Upgrade Sie beispielsweise von 15.x auf 16.x, nicht direkt von 15.x auf 17.x. Upgrade Sie zuerst auf die Zwischenversion, bestätigen Sie den normalen Betrieb und fahren Sie dann mit dem Upgrade fort.

Führen Sie vor dem Upgrade folgende Schritte durch: Datenbackup, Kompatibilitätsprobleme prüfen, Versionshinweise lesen. Halten Sie Strom und Netzwerk während des Upgrade-Vorgangs stabil und überprüfen Sie, ob alle Funktionen nach dem Upgrade normal funktionieren.
