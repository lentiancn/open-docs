# Часто задаваемые вопросы о Kubernetes

## Основные концепции

### Что такое Kubernetes?

Kubernetes (K8s) — это открытое программное обеспечение для автоматизации развёртывания, масштабирования и управления контейнеризированными приложениями. Kubernetes группирует контейнеры в логические единицы для удобного управления и обнаружения.

### В чём разница между Pod и контейнером?

Pod — наименьшая развёртываемая единица в Kubernetes. Pod может содержать один или несколько контейнеров. Рекомендуется запускать только один контейнер на Pod. Контейнеры в одном Podе совместно используют сетевые и хранилищные ресурсы.

### В чём разница между Deployment и StatefulSet?

Deployment используется для управления приложениями без состояния, поддерживает декларативные обновления, масштабирование и откаты. StatefulSet используется для управления приложениями с состоянием, предоставляет стабильные сетевые идентификаторы и постоянное хранилище, подходит для баз данных и других приложений, требующих постоянства.

### Что такое Namespace?

Namespace делит ресурсы кластера на несколько логических разделов, помогая изолировать ресурсы между различными командами или проектами. Распространённые пространства имён по умолчанию включают default, kube-system и kube-public.

## Установка и настройка

### В чём разница между Minikube и Kind?

Minikube создаёт полноценный однодноузловой кластер Kubernetes, подходит для обучения и разработки. Kind (Kubernetes in Docker) использует контейнеры Docker для имитации узлов Kubernetes, более легковесный, подходит для быстрого тестирования.

### Как выбрать среду выполнения контейнеров?

Распространённые среды выполнения контейнеров включают Docker, containerd и CRI-O. containerd — это основная среда выполнения контейнеров от Docker, в настоящее время широко рекомендуется.

### Что делать, если kubeadm init завершается ошибкой?

Распространённые причины:
- Порты уже используются: проверьте порты 6443, 10250, 10255
- Swap не отключён: выполните `swapoff -a`
- Правила брандмауэра: проверьте сетевое сообщение между узлами
- SELinux/AppArmor: можно установить в разрешающий режим или отключить

### Как настроить кластер с несколькими плоскостями управления?

Использование kubeadm для создания высокодоступных кластеров:
1. Подготовьте несколько серверных узлов
2. Используйте kubeadm init для создания первого узла плоскости управления
3. Сгенерируйте сертификаты и скопируйте на другие узлы
4. Используйте kubeadm join для добавления дополнительных плоскостей управления

## Сеть и хранилище

### Что делать, если поды не могут общаться друг с другом?

Распространённые причины:
- Сетевой плагин установлен неправильно
- Брандмауэр блокирует связь в CIDR сети подов
- Ошибки конфигурации плагина CNI
- Проверьте, работает ли kube-proxy правильно

### Как настроить постоянное хранилище для подов?

Необходимо создать PersistentVolume (PV) и PersistentVolumeClaim (PVC). PV может быть локальным хранилищем, сетевым хранилищем (NFS, iSCSI) или облачным хранилищем (AWS EBS, GCE PD, Azure Disk).

### Какие типы сервисов существуют?

- ClusterIP: виртуальный IP, доступный внутри кластера
- NodePort: открывает порт на каждом узле
- LoadBalancer: использует балансировщик нагрузки облачного провайдера
- ExternalName: сопоставляет сервис с внешним доменным именем

## Управление ресурсами

### Как установить ограничения ресурсов контейнера?

Добавьте поле resources в spec Pod или Deployment:
```yaml
resources:
  limits:
    memory: "512Mi"
    cpu: "500m"
  requests:
    memory: "256Mi"
    cpu: "250m"
```

### Что такое ResourceQuota?

ResourceQuota используется для ограничения общего количества ресурсов в namespace, предотвращая чрезмерное потребление ресурсов одной командой или проектом.

### Что такое LimitRange?

LimitRange используется для установки ограничений ресурсов по умолчанию и значений запросов для подов и контейнеров в namespace.

## Развёртывание и эксплуатация

### Как реализовать непрерывное обновление приложений?

При использовании Deployment Kubernetes автоматически выполняет непрерывное обновление:
```bash
kubectl set image deployment/my-app my-app=my-app:v2
kubectl rollout status deployment/my-app
```

### Как откатить развёртывание?

```bash
# Откатить к предыдущей версии
kubectl rollout undo deployment/my-app

# Откатить к конкретной ревизии
kubectl rollout undo deployment/my-app --to-revision=2
```

### Что такое HPA?

Horizontal Pod Autoscaler (HPA) автоматически регулирует количество реплик подов на основе метрик, таких как процессор и память. Требует Metrics Server.

### Как настроить проверки здоровья?

Настройте пробы готовности и жизнеспособности:
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

## Безопасность

### Что такое RBAC?

Управление доступом на основе ролей (RBAC) используется для управления разрешениями в кластерах Kubernetes. Можно создавать Role и ClusterRole и связывать их с пользователями или группами.

### Как управлять конфиденциальной информацией?

Используйте Secret для хранения конфиденциальной информации, такой как пароли, ключи и сертификаты. Можно создать с помощью командной строки kubectl или определить через YAML-файлы.

### Как защитить кластер?

- Включить аутентификацию RBAC
- Использовать сетевые политики для ограничения связи между подами
- Регулярно обновлять версии Kubernetes
- Шифровать данные etcd
- Использовать TLS для связи
- Ограничить права доступа kubelet

## Устранение неполадок

### Pod остаётся в состоянии Pending?

Возможные причины:
- Недостаточно ресурсов (процессор или память)
- Селектор узла или taint не совпадают
- PVC не связан правильно
- Используйте `kubectl describe pod <pod-name>` для получения подробностей

### Pod остаётся в состоянии ImagePullBackOff?

Распространённые причины:
- Неправильное имя образа
- Тег образа не существует
- Проблемы с аутентификацией в приватном реестре
- Проблемы с сетью, препятствующие загрузке образа

### Pod остаётся в состоянии CrashLoopBackOff?

- Проверьте логи контейнера: kubectl logs --previous <pod-name>
- Проверьте, запускается ли приложение правильно
- Проверьте правильность конфигурации ресурсов
- Проверьте доступность зависимых сервисов

### Как отлаживать сетевые проблемы?

- Проверьте, работает ли kube-proxy правильно
- Проверьте состояние плагина CNI
- Используйте kubectl exec для входа в контейнер и проверки сетевого подключения
- Просмотрите правила iptables
- Используйте инструменты диагностики сети (curl, ping, traceroute)

## Оптимизация производительности

### Как улучшить производительность кластера?

- Настройте соответствующие запросы и ограничения ресурсов
- Используйте анти-аффинность для распределения подов
- Включите кэширование API для уменьшения запросов
- Используйте локальное хранилище для уменьшения задержки сети
- Оптимизируйте размеры образов для уменьшения времени загрузки

### Как обрабатывать крупномасштабные кластеры?

- Используйте Federation для управления несколькими кластерами
- Настройте подходящее оборудование для etcd
- Включите кэширование API
- Используйте балансировщики нагрузки для распределения запросов
