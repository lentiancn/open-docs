# GitLab Manual del Usuario

## Gestión de Cuentas

### Registro e Inicio de Sesión

Para usar GitLab por primera vez, haga clic en "Registrarse" en la página de inicio de sesión para crear una cuenta. El registro requiere proporcionar un nombre de usuario, dirección de correo electrónico y contraseña. La elección del nombre de usuario debe ser cuidadosa, ya que no se puede cambiar después de la creación. Después del registro, recibirá un correo de verificación. Haga clic en el enlace del correo para completar la verificación del correo electrónico.

Después de iniciar sesión, puede mejorar su perfil en configuración personal, incluyendo avatar, nombre, nombre completo y bio. Se recomienda completar la información, ya que esto se mostrará en las listas de miembros del proyecto y registros de confirmaciones. También puede vincular claves SSH para empujar código a través de la línea de comandos.

### Configuración de Clave SSH

Las claves SSH permiten autenticarse de forma segura con repositorios GitLab sin ingresar nombre de usuario y contraseña cada vez. Primero, genere un par de claves SSH:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

La clave pública generada se encuentra en ~/.ssh/id_ed25519.pub. Inicie sesión en GitLab, vaya a configuración de usuario, haga clic en la pestaña "Claves SSH", pegue el contenido de la clave pública en el cuadro de texto y guarde. Después de agregar la clave, puede clonar repositorios usando SSH:

```bash
git clone git@gitlab.example.com:username/project.git
```

## Operaciones de Proyecto

### Creando un Proyecto

Después de iniciar sesión, haga clic en el botón "Nuevo Proyecto" en la barra de navegación superior para crear un nuevo proyecto. Puede elegir crear un proyecto en blanco, crear desde una plantilla o importar un proyecto existente. Al crear, necesita establecer el nombre del proyecto, descripción, nivel de visibilidad (Público, Interno, Privado) y nombre de rama predeterminado.

Los nombres de proyecto deben ser concisos y claros, usando inglés y guiones para el nombres. Los proyectos privados solo son accesibles para miembros del proyecto. Los proyectos internos son visibles para todos los usuarios conectados dentro de la empresa. Los proyectos públicos son visibles para todos. Después de la creación, puede modificar estas opciones en la página de configuración del proyecto.

### Gestión de Ramas

GitLab usa main como la rama principal predeterminada (las versiones anteriores usaban master). Haga clic en "Ramas" en la página del proyecto para ver y gestionar todas las ramas. Cree una nueva rama haciendo: clic en "Nueva Rama" en la página de ramas, o usando comandos Git:

```bash
git checkout -b feature/new-feature
```

Las ramas protegidas son una función importante para la colaboración en equipo. Vaya a Configuración del Proyecto → Repositorio → Ramas Protegidas para establecer qué usuarios pueden empujar o fusionar a ramas específicas. Se recomienda proteger la rama main, permitiendo solo a roles específicos o mantenedores fusionar código.

### Merge Requests

Los Merge Requests son la función central para la colaboración de código en GitLab. Después de completar el desarrollo de una función y empujar la rama, haga clic en "Merge Requests" → "Nuevo Merge Request" en la página del proyecto. Seleccione la rama fuente y la rama objetivo, complete el título y la descripción, luego cree.

La página de Merge Request muestra las diferencias entre las ramas fuente y objetivo, incluyendo cambios de archivos e historial de confirmaciones. Los miembros del equipo pueden realizar revisión de código aquí, agregando comentarios línea por línea. Después de que la revisión esté completa, si se cumplen las condiciones de fusión (como pasar pruebas CI y recibir aprobaciones), la rama puede ser fusionada.

## Pipelines CI/CD

### Configuración .gitlab-ci.yml

GitLab CI/CD usa el archivo .gitlab-ci.yml en la raíz del proyecto para definir pipelines. Este es un archivo de configuración YAML que describe cada etapa de construcción y trabajo. Aquí hay un ejemplo simple:

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

Esta configuración define tres etapas: build para construir el proyecto, test para ejecutar pruebas, y deploy para desplegar la aplicación. Los artifacts se usan para pasar artefactos de construcción entre trabajos, y only especifica ejecutar trabajos de despliegue solo en la rama main.

### GitLab Runner

GitLab Runner es el agente que ejecuta trabajos CI/CD. GitLab proporciona Runners compartidos, o puede registrar sus propios Runners privados. Instale GitLab Runner en su servidor:

```bash
# Instalar
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Registrar Runner
sudo gitlab-runner register
```

El registro requiere proporcionar la URL de la instancia de GitLab y el token de registro, que pueden encontrarse en Configuración del Proyecto → CI/CD → Runners. Los Runners pueden configurarse para ejecutar trabajos en paralelo para mejorar la velocidad de construcción.

### Monitoreo de Pipelines

Después de cada empujón de código, el pipeline se activa automáticamente. Puede ver todos los estados de ejecución de pipelines en la página del Proyecto → CI/CD → Pipelines. Haga clic en un pipeline específico para ver el estado de ejecución de cada trabajo, incluyendo salida de registros, tiempo de construcción y razones de falla.

GitLab también proporciona un Editor de Pipeline que crea y edita configuraciones CI/CD visualmente, eliminando la necesidad de escribir archivos YAML manualmente. El editor proporciona verificación de sintaxis y vista previa en tiempo real, reduciendo errores de configuración.

## Seguimiento de Problemas

### Creando y Gestionando Problemas

El sistema de problemas es la base de la gestión de proyectos de GitLab. En la página del proyecto, haga clic en "Problemas" → "Nuevo Problema" para crear un nuevo problema. Necesita completar el título y la descripción, y puede elegir asignar a miembros específicos, establecer etiquetas, hitos y fechas límite.

El estado de los problemas se gestiona a través de etiquetas o hitos. Las etiquetas comunes incluyen bug, feature, enhancement, etc. Puede usar la vista de tablero para visualizar el estado de los problemas, agrupando problemas por columnas (p.ej., Por Hacer, En Progreso, Hecho).

### Vinculando Problemas al Código

Usando palabras clave más números de problema en mensajes de confirmación o descripciones de Merge Request vincula automáticamente código a problemas. Por ejemplo:

- Fixes #123 - Arregla el problema 123, cerrará automáticamente el problema después de fusionar
- Closes #456 - Cierra el problema 456
- See #789 - Referencia el problema 789

Este mecanismo de vinculación ayuda a los equipos a seguir el progreso del trabajo y entender el fondo y propósito del problema durante la revisión de código.

## Wikis y Documentación

Cada proyecto de GitLab incluye funcionalidad de wiki incorporada para crear documentación del proyecto. Las páginas de wiki se escriben en formato Markdown, soportando resaltado de código, tablas e imágenes. El contenido del wiki soporta control de versiones al igual que los repositorios de código, permitiéndole ver versiones históricas y revertir.

Se recomienda mantener la siguiente documentación para proyectos: README para introducción del proyecto, CONTRIBUTING para guía de contribuciones, guías de despliegue y documentación de API. Estos documentos pueden reducir significativamente la dificultad de incorporación y promover contribuciones de la comunidad.
