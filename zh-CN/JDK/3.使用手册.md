# JDK 使用手册

## Java 编译器使用指南

### javac 基本用法

javac 是 Java 的编译器，负责将 Java 源代码编译成字节码文件。基本的使用方法非常简单：javac HelloWorld.java。这会生成 HelloWorld.class 文件，然后在 JVM 上运行。

在实际的开发项目中，通常需要指定更多的编译选项。常用的选项包括：-d 指定编译输出目录，例如 javac -d out HelloWorld.java 会将编译结果放到 out 目录中；-classpath（或 -cp）指定类搜索路径，用于编译时引用外部类；-source 指定源代码的 Java 版本；-target 指定生成的字节码的 Java 版本。

对于大型项目，通常会使用构建工具如 Maven 或 Gradle 来管理编译过程，这些工具会自动处理依赖和编译选项。

### 编译多个文件

如果项目包含多个 Java 源文件，可以一次性编译整个目录。例如：javac -d out src/*.java 会编译 src 目录下的所有 Java 文件。

更复杂的项目结构可能需要使用 -sourcepath 选项指定源代码的搜索路径，让编译器能够找到依赖的类。例如：javac -d out -sourcepath src src/Main.java。

### 编译错误处理

编译过程中如果出现错误，javac 会输出详细的错误信息，包括错误所在的文件、行号和具体原因。常见的编译错误包括：语法错误（缺少分号、括号不匹配等）、类型错误（类型不兼容、方法不存在等）、访问权限错误（尝试访问 private 成员等）。

理解这些错误信息对于快速定位和解决问题非常重要。编译器会尽可能多地报告错误，有时修复第一个错误后，后续的错误可能就自动消失了。

## Java 程序运行指南

### java 命令基本用法

使用 java 命令运行编译后的 Java 程序。基本语法是：java HelloWorld。注意这里不需要 .class 扩展名。JVM 会自动在当前目录和 CLASSPATH 中查找 HelloWorld.class 文件。

如果类在包中，需要使用完全限定名。例如：java com.example.HelloWorld。

### 类路径（CLASSPATH）

类路径告诉 JVM 在哪里查找类文件。可以通过 -classpath 或 -cp 选项指定。例如：java -cp out:lib/* com.example.HelloWorld。在 Windows 上，路径分隔符是分号（;），在 Linux 和 macOS 上是冒号（:）。

如果未指定类路径，JVM 会使用当前目录以及 CLASSPATH 环境变量中指定的路径。

### JVM 选项

JVM 提供了大量的启动选项，用于控制运行时的行为。常用选项包括：-Xms 指定堆的初始大小，如 -Xms256m 设置初始堆为 256MB；-Xmx 指定堆的最大大小，如 -Xmx2g 设置最大堆为 2GB；-XX:+UseG1GC 使用 G1 垃圾回收器。

在 JDK 9 及以后，可以使用 --add-opens 选项来开放模块的内部访问，这在运行需要访问 JDK 内部 API 的库时很有用。

## JDK 工具使用指南

### jar 归档工具

jar 工具用于创建和管理 Java 归档文件（.jar 文件）。.jar 文件实际上就是 ZIP 格式的压缩包，里面包含了编译后的 .class 文件和一些元数据。

创建 jar 文件的基本命令是：jar cf myapp.jar -C out .。这会从 out 目录的内容创建 myapp.jar。使用 tf 参数可以查看 jar 文件的内容：jar tf myapp.jar。使用 xf 可以解压：jar xf myapp.jar。

在创建可执行的 jar 文件时，需要在 MANIFEST.MF 文件中指定主类。可以通过 m 参数添加自定义的 MANIFEST.MF：jar cfm myapp.jar MANIFEST.MF -C out .。

### javadoc 文档生成

javadoc 工具根据 Java 源代码中的注释生成 HTML 格式的 API 文档。标准的 Java 类库文档就是这样生成的。

基本用法是：javadoc -d doc src/*.java。这会为 src 目录下所有 Java 文件生成文档，放到 doc 目录中。

在编写代码时，使用特定的注释格式可以为类、方法、字段添加文档。常用的注释标签包括：@author 作者、@param 参数、@return 返回值、@throws 异常等。

### jdb 调试器

jdb 是 Java 的命令行调试器，提供基本的调试功能。启动调试：jdb MyClass。设置断点：stop in MyClass.method。查看断点：clear。运行程序：run。继续执行：cont。单步执行：step 或 next。查看变量：print varname。列出源代码：list。

虽然 jdb 功能不如图形化调试器强大，但它可以在没有 IDE 的环境中使用，或者在服务器上进行远程调试。

## JVM 性能调优

### 垃圾回收器选择

JDK 提供了多种垃圾回收器，每种都有不同的特点和适用场景：

Serial GC：单线程收集器，适合小型应用和客户端应用。使用 -XX:+UseSerialGC 启用。

Parallel GC（Throughput GC）：多线程收集器，适合追求高吞吐量的应用。使用 -XX:+UseParallelGC 启用。

CMS GC：并发标记清除收集器，追求低停顿时间。使用 -XX:+UseConcMarkSweepGC 启用（JDK 14 已废弃）。

G1 GC：分区式垃圾回收器，适合大内存应用，在 JDK 9 及以后成为默认选项。推荐使用默认设置或 -XX:+UseG1GC。

ZGC：低延迟垃圾回收器，支持 TB 级堆内存，停顿时间不超过 10ms。使用 -XX:+UseZGC 启用（JDK 15+）。

Shenandoah：另一个低延迟垃圾回收器，停顿时间与堆大小无关。使用 -XX:+UseShenandoahGC 启用。

### 内存配置

合理配置堆内存大小对应用性能至关重要。常用配置选项：

-Xms：堆初始大小，如 -Xms2g。

-Xmx：堆最大大小，如 -Xmx4g。

-Xmn：年轻代大小，如 -Xmn1g。

-XX:MetaspaceSize：元空间初始大小。

-XX:MaxMetaspaceSize：元空间最大大小。

建议将 -Xms 和 -Xmx 设置为相同的值，避免运行时调整堆大小带来的开销。

### 性能监控工具

JDK 提供了丰富的性能监控和诊断工具：

jps：显示 Java 进程列表，基本用法是 jps -l。

jstat：显示 JVM 统计信息，如垃圾回收统计：jstat -gcutil pid 1000。

jstack：打印线程堆栈信息，用于诊断死锁等问题：jstack pid。

jmap：显示堆内存信息，如生成堆转储：jmap -dump:format=b,file=heap.bin pid。

jcmd：多功能诊断工具，可以执行各种诊断命令：jcmd pid GC.heap_info。

JConsole 和 VisualVM：图形化的监控和诊断工具，可以监控内存、线程、CPU 使用等情况。

## 模块系统

### 模块化概述

从 JDK 9 开始，Java 引入了模块系统（Project Jigsaw），将 JDK 本身和应用程序划分为模块。每个模块都有明确的依赖关系，提供了更好的封装和可维护性。

JDK 本身被划分为多个模块，如 java.base、java.sql、java.xml 等。应用程序可以创建自己的模块，通过 module-info.java 文件声明模块名、依赖和其他模块的导出。

### 模块相关选项

JDK 提供了多个与模块相关的命令行选项：

--module-path（或 -p）：指定模块路径，类似于 classpath 但用于模块。

--add-modules：添加模块到默认的模块集合中。

--add-exports：开放指定模块的包供其他模块访问。

--add-opens：开放指定模块的包供反射访问。

这些选项在某些情况下非常重要，例如在使用反射访问 JDK 内部 API 时，或者在使用不兼容模块化的库时。

## 常用配置示例

### 开发环境配置

对于日常开发环境，推荐的配置包括：足够的堆内存（-Xmx2g 或更高），使用 G1 垃圾回收器（默认），启用类数据共享（-Xshare:classes）以加快启动速度。如果使用 IDE，可以直接在 IDE 的运行配置中设置这些选项。

### 生产环境配置

生产环境的配置需要根据应用的具体情况调整。基本原则包括：根据应用内存需求设置堆大小，通常建议预留 25-30% 的内存给操作系统和其他进程；选择合适的垃圾回收器，G1 通常是较好的选择；对于追求低延迟的应用，可以考虑 ZGC 或 Shenandoah。

建议在生产环境中启用 GC 日志，记录详细的垃圾回收信息，便于问题分析和性能调优。GC 日志配置示例：-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M。

### Docker 容器配置

在容器环境中运行 Java 应用需要注意 JVM 的内存感知。JDK 10+ 版本默认会感知容器内存限制，但在早期版本中可能需要手动配置。

推荐配置：设置正确的堆最大值（-XX:MaxRAMPercentage=75.0），确保 JVM 不会使用超过容器可用内存的 75%；使用 -XX:+UseContainerSupport（默认启用）让 JVM 感知容器资源限制。

对于云原生应用，可以考虑使用 GraalVM Native Image 将应用编译为原生可执行文件，可以显著减少启动时间和内存占用。
