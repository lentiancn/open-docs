# Questions fréquentes sur Node.js (FAQ)

## Questions générales

### Qu'est-ce que Node.js ?
Node.js est un environnement d'exécution JavaScript open-source et multiplateforme qui exécute du code JavaScript en dehors d'un navigateur web. Il permet aux développeurs d'utiliser JavaScript pour écrire des outils en ligne de commande et pour le scripting côté serveur, générant ainsi du contenu web dynamique avant que la page ne soit envoyée au navigateur de l'utilisateur.

### Pourquoi Node.js a-t-il été créé ?
Node.js a été créé par Ryan Dahl en 2009 pour répondre aux limitations des serveurs web traditionnels qui utilisaient un modèle basé sur les threads pour gérer les connexions concurrentes. Node.js a introduit un modèle d'E/S orienté événements et non bloquant qui le rend léger et efficace pour construire des applications réseau évolutives.

### Node.js est-il monothread ?
Les applications Node.js s'exécutent sur un seul thread pour l'exécution JavaScript, mais elles peuvent gérer plusieurs opérations concurrentes grâce aux E/S asynchrones et à la boucle d'événements. De plus, Node.js fournit des API comme `worker_threads` et `child_process` pour utiliser plusieurs cœurs de CPU lorsque nécessaire.

### Qu'est-ce que npm ?
npm (Node Package Manager) est le gestionnaire de packages par défaut pour Node.js et le plus grand registre de logiciels au monde. Il permet aux développeurs de partager et réutiliser des packages de code, de gérer les dépendances et d'exécuter des scripts définis dans les fichiers package.json.

## Installation et configuration

### Quelle version dois-je installer : LTS ou Current ?
- **LTS (Support à Long Terme)** : Recommandé pour les applications de production. Offre stabilité, mises à jour de sécurité et support à long terme (30 mois au total).
- **Current** : Contient les dernières fonctionnalités mais a un cycle de support plus court. Meilleur pour le développement, les tests de nouvelles fonctionnalités ou l'apprentissage.

### Comment changer entre les versions de Node.js ?
Utilisez un gestionnaire de versions Node :
- **nvm** (macOS/Linux) : `nvm install 18 && nvm use 18`
- **nvm-windows** (Windows) : Commandes similaires à nvm basé sur Unix
- **fnm** (Fast Node Manager) : Alternative multiplateforme avec de meilleures performances

### Pourquoi obtiens-je des erreurs de permission lors de l'installation de packages globaux ?
Cela se produit généralement lorsque npm tente d'écrire dans des répertoires système qui nécessitent des permissions élevées. Solutions :
1. Configurez npm pour utiliser un répertoire différent : `npm config set prefix ~/.npm-global`
2. Ajoutez `~/.npm-global/bin` à votre PATH
3. Utilisez un gestionnaire de versions Node qui gère correctement les permissions

### Comment vérifier mon installation Node.js ?
Exécutez ces commandes dans votre terminal :
```bash
node --version    # Affiche la version de Node.js
npm --version     # Affiche la version de npm
node -e "console.log('Ça fonctionne !')"  # Teste la fonctionnalité de base
```

## Questions de développement

### Quelle est la différence entre CommonJS et ES Modules ?
- **CommonJS** (`require()`/`module.exports`) : Le système de modules original de Node.js, synchrone, largement pris en charge
- **ES Modules** (`import`/`export`) : Standard JavaScript moderne, asynchrone, nécessite l'extension `.mjs` ou `"type": "module"` dans package.json

Exemple CommonJS :
```javascript
// math.js
exports.add = (a, b) => a + b;

// app.js
const { add } = require('./math');
```

Exemple ES Modules :
```javascript
// math.mjs
export const add = (a, b) => a + b;

// app.mjs
import { add } from './math.mjs';
```

### Comment gérer les opérations asynchrones ?
Trois approches principales :

1. **Callbacks** (traditionnel mais peut conduire à l'"enfer des callbacks") :
```javascript
fs.readFile('file.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log(data);
});
```

2. **Promesses** (meilleure gestion des erreurs et chaînage) :
```javascript
fs.promises.readFile('file.txt', 'utf8')
  .then(data => console.log(data))
  .catch(err => console.error(err));
```

3. **Async/Await** (le plus lisible, ressemble à du code synchrone) :
```javascript
async function readFile() {
  try {
    const data = await fs.promises.readFile('file.txt', 'utf8');
    console.log(data);
  } catch (err) {
    console.error(err);
  }
}
```

### Qu'est-ce que la boucle d'événements et comment fonctionne-t-elle ?
La boucle d'événements est le mécanisme de Node.js pour gérer les opérations asynchrones sans bloquer le thread principal. Elle vérifie continuellement les callbacks en attente et les exécute lorsque leurs opérations associées sont terminées.

Phases de la boucle d'événements :
1. **Timers** : Exécute les callbacks setTimeout() et setInterval()
2. **Callbacks en attente** : Exécute les callbacks d'E/S reportés à la prochaine itération de la boucle
3. **Idle, préparation** : Usage interne uniquement
4. **Poll** : Récupère de nouveaux événements d'E/S ; exécute les callbacks liés à l'E/S
5. **Check** : Exécute les callbacks setImmediate()
6. **Callbacks de fermeture** : Exécute les callbacks d'événements de fermeture (par exemple socket.on('close'))

### Comment déboguer les applications Node.js ?
Plusieurs options de débogage :

1. **Inspecteur intégré** : `node --inspect app.js`
2. **DevTools Chrome** : Se connecter à `chrome://inspect`

5. **Outils de débogage dédiés** : Comme Node.js Inspector Manager (NiM)

## Performance et mise à l'échelle

### Comment mettre à l'échelle les applications Node.js ?
Plusieurs stratégies :

1. **Clustering** : Utilisez le module intégré `cluster` pour créer des processus enfants qui partagent les ports du serveur
2. **Équilibrage de charge** : Utilisez des proxys inversés comme NGINX pour distribuer le trafic entre plusieurs instances Node.js
3. **Microservices** : Divisez les grandes applications en services plus petits et indépendants
4. **Mise en cache** : Implémentez des couches de mise en cache (Redis, Memcached) pour réduire la charge de la base de données
5. **Optimisation de la base de données** : Utilisez le regroupement de connexions et l'optimisation des requêtes

### Qu'est-ce qui cause les fuites de mémoire dans Node.js ?
Causes courantes :
- **Variables globales** : Accumulation de données dans la portée globale
- **Écouteurs d'événements non fermés** : Ne pas supprimer les écouteurs d'événements lorsque les objets sont détruits
- **Fermetures** : Maintenir des références à de gros objets plus longtemps que nécessaire
- **Timers** : Ne pas effacer les intervalles et les timeouts
- **Tampons** : Ne pas gérer correctement les grandes données binaires

Outils de détection :
- `process.memoryUsage()` pour la surveillance de base
- Instantanés de tas avec DevTools Chrome
- Outils de profilage de mémoire comme Clinic.js

### Comment gérer les tâches intensives en CPU ?
Puisque Node.js est monothread pour l'exécution JavaScript, les tâches intensives en CPU peuvent bloquer la boucle d'événements. Solutions :

1. **Threads de travail** : Utilisez le module `worker_threads` pour le traitement parallèle
```javascript
const { Worker } = require('worker_threads');
const worker = new Worker('./cpu-intensive-task.js');
```

2. **Processus enfants** : Lancez des processus séparés en utilisant `child_process`
```javascript
const { spawn } = require('child_process');
const child = spawn('python', ['script.py']);
```

3. **Déléguer à des services externes** : Utilisez des files d'attente de messages (RabbitMQ, Kafka) pour traiter les tâches lourdes de manière asynchrone
4. **Optimiser les algorithmes** : Profilez et optimisez le code réel qui cause des problèmes de performance

## Sécurité

### Comment sécuriser mon application Node.js ?
Pratiques de sécurité clés :

1. **Validation des entrées** : Validez et nettoyez toujours les entrées utilisateur
2. **Gestion des dépendances** : Mettez à jour régulièrement les dépendances et exécutez `npm audit`
3. **Variables d'environnement** : Stockez les secrets dans les variables d'environnement, jamais dans le code
4. **Limitation de débit** : Prévenez les attaques par force brute avec un middleware de limitation de débit
5. **Helmet.js** : Utilisez un middleware d'en-têtes de sécurité
6. **CORS** : Configurez correctement le partage de ressources cross-origin
7. **HTTPS** : Utilisez toujours HTTPS en production
8. **Authentification** : Implémentez une authentification et une autorisation appropriées

### Comment prévenir les vulnérabilités courantes ?
- **Injection SQL** : Utilisez des requêtes paramétrées ou des ORM
- **XSS (Cross-Site Scripting)** : Échappez les entrées utilisateur avant le rendu
- **CSRF (Cross-Site Request Forgery)** : Utilisez des jetons CSRF
- **Pollution de prototype** : Évitez les fonctions de fusion d'objets non sécurisées
- **Traversée de chemin** : Validez les chemins de fichiers et utilisez `path.resolve()` avec prudence

## Déploiement et production

### Quel gestionnaire de processus dois-je utiliser ?
Options populaires :

1. **PM2** : Le plus populaire, riche en fonctionnalités, prend en charge le clustering et la surveillance
2. **Forever** : Gestionnaire de processus simple, bon pour les cas d'utilisation de base
3. **Systemd** : Gestionnaire de services natif Linux, s'intègre bien avec le système d'exploitation
4. **Docker** : Approche de conteneurisation, excellente pour les microservices

### Comment gérer la configuration spécifique à l'environnement ?
Utilisez des variables d'environnement et des fichiers de configuration :

```javascript
// config.js
const config = {
  development: {
    port: 3000,
    dbUrl: 'mongodb://localhost:27017/myapp'
  },
  production: {
    port: process.env.PORT || 8080,
    dbUrl: process.env.DATABASE_URL
  }
};

module.exports = config[process.env.NODE_ENV || 'development'];
```

### Quels outils de surveillance dois-je utiliser ?
- **Surveillance des performances d'application (APM)** : New Relic, Datadog, AppSignal
- **Journalisation** : Winston, Bunyan, avec journalisation centralisée (pile ELK, Splunk)
- **Vérifications de santé** : Points de terminaison personnalisés qui vérifient la santé de l'application
- **Métriques** : Prometheus avec Grafana pour la visualisation
- **Suivi des erreurs** : Sentry, Rollbar pour la surveillance des exceptions

### Comment implémenter un arrêt gracieux ?
Gérez les signaux de processus pour nettoyer les ressources :

```javascript
const server = http.createServer(app);

const shutdown = () => {
  server.close(() => {
    console.log('Serveur fermé');
    process.exit(0);
  });
  
  // Arrêt forcé après 10 secondes
  setTimeout(() => {
    console.error('Impossible de fermer les connexions à temps, arrêt forcé');
    process.exit(1);
  }, 10000);
};

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## Écosystème et communauté

### Quels sont les frameworks Node.js les plus populaires ?
- **Express.js** : Minimaliste, flexible, le plus largement utilisé
- **Fastify** : Hautes performances, faible surcharge
- **Koa** : Moderne, utilise async/await, créé par les créateurs d'Express

- **Hapi** : Centré sur la configuration, orienté entreprise
