# Kubernetes Benutzerhandbuch

## Grundkonzepte

### Pod
Pod ist die kleinste bereitstellbare Einheit in Kubernetes und repräsentiert einen oder mehrere Container, die im Cluster ausgeführt werden. Normalerweise enthält ein Pod einen einzelnen Container, aber es kann auch mehrere eng gekoppelte Container enthalten.

### Deployment
Deployments werden zur Verwaltung von zustandslosen Anwendungen verwendet und bieten deklarative Aktualisierungsfunktionen für einfache Bereitstellung, Skalierung und Updates.

### Service
Services bieten stabile Netzwerkendpunkte für eine Gruppe von Pods und behalten ihre Stabilität bei, auch wenn sich die IP-Adressen der Pods ändern.

### Namespace
Namespaces teilen Cluster-Ressourcen in mehrere logische Partitionen auf, geeignet für mehrere Teams oder Projekte, die dasselbe Cluster teilen.

## Häufige Befehle

### Cluster-Verwaltung

```bash
# Cluster-Informationen anzeigen
kubectl cluster-info

# Knoten anzeigen
kubectl get nodes

# Alle Pods anzeigen
kubectl get pods --all-namespaces

# Cluster-Ressourcennutzung anzeigen
kubectl top nodes
kubectl top pods
```

### Anwendungsbereitstellung

```bash
# Deployment erstellen
kubectl create deployment nginx --image=nginx

# Deployments anzeigen
kubectl get deployments

# Skalieren
kubectl scale deployment nginx --replicas=3

# Image aktualisieren
kubectl set image deployment/nginx nginx=nginx:1.21

# Rollback
kubectl rollout undo deployment/nginx

# Rollout-Status anzeigen
kubectl rollout status deployment/nginx
```

### Service-Verwaltung

```bash
# Service exponieren
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Services anzeigen
kubectl get services

# Service löschen
kubectl delete service <service-name>
```

### Fehlerbehebung

```bash
# Pod-Logs anzeigen
kubectl logs <pod-name>

# In Container wechseln
kubectl exec -it <pod-name> -- /bin/bash

# Ressourcendetails beschreiben
kubectl describe pod <pod-name>

# Port-Weiterleitung zur Fehlerbehebung
kubectl port-forward <pod-name> 8080:80
```

## Ressourcen-Manifeste

Kubernetes unterstützt deklarative Ressourcenverwaltung mit YAML-Dateien. Beispiele für häufige Ressourcendefinitionen:

### Deployment-Beispiel

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        ports:
        - containerPort: 8080
        resources:
          limits:
            memory: "128Mi"
            cpu: "500m"
          requests:
            memory: "64Mi"
            cpu: "250m"
```

### Service-Beispiel

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
  - protocol: TCP
    port: 80
    targetPort: 8080
  type: ClusterIP
```

### ConfigMap-Beispiel

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
data:
  database_url: "postgres://db:5432/myapp"
  cache_enabled: "true"
```

### Secret-Beispiel

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
stringData:
  username: admin
  password: changeme
```

## Anwendungslebenszyklusverwaltung

### Anwendungen bereitstellen

```bash
# Mit YAML-Datei bereitstellen
kubectl apply -f deployment.yaml

# Bereitstellungsstatus anzeigen
kubectl get pods -l app=my-app
```

### Anwendungen aktualisieren

```bash
# Image-Version aktualisieren
kubectl set image deployment/my-app my-app=my-app:v2

# Aktualisierungsfortschritt anzeigen
kubectl rollout status deployment/my-app
```

### Rollback

```bash
# Auf vorherige Version zurücksetzen
kubectl rollout undo deployment/my-app

# Auf bestimmte Revision zurücksetzen
kubectl rollout undo deployment/my-app --to-revision=2

# Revisionsverlauf anzeigen
kubectl rollout history deployment/my-app
```

### Skalierung

```bash
# Manuelle Skalierung
kubectl scale deployment my-app --replicas=5

# Automatische Skalierung (HPA)
kubectl autoscale deployment my-app --min=2 --max=10 --cpu-percent=80
```

## Speicherverwaltung

### PersistentVolume erstellen

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  hostPath:
    path: /data/my-app
```

### PersistentVolumeClaim erstellen

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

### PVC in Pod verwenden

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: my-app
    image: my-app
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

## Konfigurationsverwaltung

### ConfigMap verwenden

```bash
# ConfigMap aus Datei erstellen
kubectl create configmap my-config --from-file=config.properties

# Aus Umgebungsdatei erstellen
kubectl create configmap my-config --from-env-file=env.properties

# In Pod verwenden
kubectl apply -f pod-with-configmap.yaml
```

### Secret verwenden

```bash
# Secret aus Datei erstellen
kubectl create secret generic my-secret --from-file=credentials.json

# In Pod verwenden
kubectl apply -f pod-with-secret.yaml
```

## Ressourcenquoten und -limits

### ResourceQuota

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: my-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
```

### LimitRange

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: my-limits
spec:
  limits:
  - max:
      cpu: "2"
      memory: 1Gi
    min:
      cpu: 100m
      memory: 128Mi
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 200m
      memory: 256Mi
    type: Container
```

## Überwachung und Protokollierung

### Überwachung

```bash
# Ressourcennutzung mit kubectl top anzeigen
kubectl top nodes
kubectl top pods

# Detaillierten Status anzeigen
kubectl describe node <node-name>
kubectl describe pod <pod-name>
```

### Protokollierung

```bash
# Pod-Logs anzeigen
kubectl logs my-app-pod

# Logs in Echtzeit anzeigen
kubectl logs -f my-app-Pod

# Logs der vorherigen Container-Instanz anzeigen
kubectl logs --previous my-app-pod
```

## Best Practices

1. Verwende deklarative Konfiguration zur Infrastrukturverwaltung
2. Setze angemessene Ressourcenlimits und -anforderungen
3. Organisiere Ressourcen mit Labels
4. Bereinige regelmäßig ungenutzte Ressourcen
5. Verwende Namespaces zur Isolierung verschiedener Umgebungen
6. Konfiguriere Health-Check-Sonden
7. Verwende RBAC zur Zugriffskontrolle
8. Sichere etcd-Daten regelmäßig
