# JDK Benutzerhandbuch

## Java-Compiler-Verwendungsanleitung

### Grundlegende javac-Verwendung

javac ist der Java-Compiler und dient zum Kompilieren von Java-Quellcode in Bytecode-Dateien. Die grundlegende Verwendung ist sehr einfach: javac HelloWorld.java. Dies erstellt eine HelloWorld.class-Datei, die dann auf der JVM ausgeführt werden kann.

In tatsächlichen Entwicklungsprojekten müssen normalerweise mehr Kompilierungsoptionen angegeben werden. Zu den gängigen Optionen gehören: -d gibt das Kompilierungsausgabeverzeichnis an, z.B. javac -d out HelloWorld.java speichert die kompilierten Ergebnisse im out-Verzeichnis; -classpath (oder -cp) gibt den Klassensuchpfad an, um externe Klassen während der Kompilierung zu referenzieren; -source gibt die Java-Version des Quellcodes an; -target gibt die Java-Version des generierten Bytecodes an.

Für große Projekte werden normalerweise Build-Tools wie Maven oder Gradle verwendet, um den Kompilierungsprozess zu verwalten. Diese Tools behandeln automatisch Abhängigkeiten und Kompilierungsoptionen.

### Mehrere Dateien kompilieren

Wenn Ihr Projekt mehrere Java-Quelldateien enthält, können Sie ein ganzes Verzeichnis auf einmal kompilieren. Beispiel: javac -d out src/*.java kompiliert alle Java-Dateien im src-Verzeichnis.

Komplexere Projektstrukturen erfordern möglicherweise die Verwendung der -sourcepath-Option, um den Quellcode-Suchpfad anzugeben, damit der Compiler abhängige Klassen finden kann. Beispiel: javac -d out -sourcepath src src/Main.java.

### Kompilierungsfehler behandeln

Wenn während der Kompilierung Fehler auftreten, gibt javac detaillierte Fehlerinformationen aus, einschließlich der Datei, der Zeilennummer und der spezifischen Ursache. Zu den häufigsten Kompilierungsfehlern gehören: Syntaxfehler (fehlende Semikolon, nicht übereinstimmende Klammern usw.), Typfehler (inkompatible Typen, nicht existierende Methoden usw.) und Zugriffsrechtefehler (Zugriffsversuch auf private Mitglieder usw.).

Das Verstehen dieser Fehlermeldungen ist entscheidend, um Probleme schnell zu lokalisieren und zu beheben. Der Compiler versucht, so viele Fehler wie möglich zu melden; manchmal behebt das Beheben des ersten Fehlers die nachfolgenden Fehler automatisch.

## Java-Programm-Ausführungsanleitung

### Grundlegende java-Befehlsverwendung

Verwenden Sie den java-Befehl, um kompilierte Java-Programme auszuführen. Die grundlegende Syntax lautet: java HelloWorld. Beachten Sie, dass hier die .class-Erweiterung nicht erforderlich ist. Die JVM sucht automatisch nach Hello.class im aktuellen Verzeichnis und CLASSPATH.

Wenn sich die Klasse in einem Paket befindet, verwenden Sie den vollständig qualifizierten Namen. Beispiel: java com.example.HelloWorld.

### Classpath (CLASSPATH)

Classpath teilt der JVM mit, wo Klassendateien zu finden sind. Es kann mit den Optionen -classpath oder -cp angegeben werden. Beispiel: java -cp out:lib/* com.example.HelloWorld. Unter Windows ist das Pfadtrennzeichen ein Semikolon (;), unter Linux und macOS ein Doppelpunkt (:).

Wenn kein Classpath angegeben ist, verwendet die JVM das aktuelle Verzeichnis und die in der CLASSPATH-Umgebungsvariable angegebenen Pfade.

### JVM-Optionen

Die JVM bietet zahlreiche Startoptionen zur Steuerung des Laufzeitverhaltens. Zu den gängigen Optionen gehören: -Xms gibt die anfängliche Heap-Größe an, z.B. -Xms256m setzt den anfänglichen Heap auf 256MB; -Xmx gibt die maximale Heap-Größe an, z.B. -Xmx2g setzt den maximalen Heap auf 2GB; -XX:+UseG1GC verwendet den G1-Garbage-Collector.

Unter JDK 9 und höher können Sie die Option --add-open verwenden, um den internen Zugriff auf Module zu öffnen, was nützlich ist, wenn Bibliotheken ausgeführt werden, die auf JDK-interne APIs zugreifen müssen.

## JDK-Werkzeugverwendungsanleitung

### jar-Archivwerkzeug

Das jar-Werkzeug wird zum Erstellen und Verwalten von Java-Archivdateien (.jar-Dateien) verwendet. .jar-Dateien sind eigentlich ZIP-formatierte komprimierte Pakete, die kompilierte .class-Dateien und Metadaten enthalten.

Der grundlegende Befehl zum Erstellen einer jar-Datei lautet: jar cf myapp.jar -C out . Dies erstellt myapp.jar aus dem Inhalt des out-Verzeichnisses. Verwenden Sie den tf-Parameter, um den Inhalt der jar-Datei anzuzeigen: jar tf myapp.jar. Verwenden Sie xf zum Entpacken: jar xf myapp.jar.

Beim Erstellen einer ausführbaren jar-Datei müssen Sie die Hauptklasse in der MANIFEST.MF-Datei angeben. Sie können eine benutzerdefinierte MANIFEST.MF mit dem m-Parameter hinzufügen: jar cfm myapp.jar MANIFEST.MF -C out .

### javadoc-Dokumentationsgenerierung

Das javadoc-Werkzeug generiert API-Dokumentation im HTML-Format aus Kommentaren im Java-Quellcode. Die Standard-Java-Bibliotheksdokumentation wird auf diese Weise generiert.

Grundlegende Verwendung: javadoc -d doc src/*.java. Dies generiert Dokumentation für alle Java-Dateien im src-Verzeichnis und speichert sie im doc-Verzeichnis.

Beim Schreiben von Code können Sie spezifische Kommentarformate verwenden, um Dokumentation für Klassen, Methoden und Felder hinzuzufügen. Zu den gängigen Kommentar-Tags gehören: @author Autor, @param Parameter, @return Rückgabewert, @throws Ausnahme usw.

### jdb-Debugger

jdb ist der Java-Befehlszeilen-Debugger und bietet grundlegende Debugging-Funktionen. Debugging starten: jdb MyClass. Haltepunkte setzen: stop in MyClass.method. Haltepunkte anzeigen: clear. Programm ausführen: run. Ausführung fortsetzen: cont. Schrittweise ausführen: step oder next. Variablen anzeigen: print varname. Quellcode auflisten: list.

Obwohl jdb nicht so leistungsstark wie grafische Debugger ist, kann es in Umgebungen ohne IDE oder für Remote-Debugging auf Servern verwendet werden.

## JVM-Leistungsoptimierung

### Garbage-Collector-Auswahl

JDK bietet mehrere Garbage-Collector, die jeweils unterschiedliche Eigenschaften und Anwendungszenarien haben:

Serial GC: Einkthread-Sammler, geeignet für kleine Anwendungen und Client-Anwendungen. Aktivieren mit -XX:+UseSerialGC.

Parallel GC (Throughput GC): Multithread-Sammler, geeignet für Anwendungen, die hohen Durchsatz anstreben. Aktivieren mit -XX:+UseParallelGC.

CMS GC: Gleichzeitiger Mark-and-Sweep-Sammler, strebt niedrige Pausenzeiten an. Aktivieren mit -XX:+UseConcMarkSweepGC (in JDK 14 veraltet).

G1 GC: Partitionierender Garbage-Collector, geeignet für Anwendungen mit großem Speicher, in JDK 9 und höher zur Standardoption geworden. Verwenden Sie die Standardeinstellung oder -XX:+UseG1GC.

ZGC: Niedriglatenz-Garbage-Collector, unterstützt Heap-Speicher auf TB-Ebene, Pausenzeiten überschreiten nicht 10ms. Aktivieren mit -XX:+UseZGC (JDK 15+).

Shenandoah: Ein weiterer Niedriglatenz-Garbage-Collector, Pausenzeiten unabhängig von der Heap-Größe. Aktivieren mit -XX:+UseShenandoahGC.

### Speicherkonfiguration

Die angemessene Konfiguration der Heap-Speichergröße ist für die Anwendungsleistung entscheidend. Gängige Konfigurationsoptionen:

-Xms: Anfängliche Heap-Größe, z.B. -Xms2g.
-Xmx: Maximale Heap-Größe, z.B. -Xmx4g.
-Xmn: Größe der Young Generation, z.B. -Xmn1g.
-XX:MetaspaceSize: Anfängliche Metaspace-Größe.
-XX:MaxMetaspaceSize: Maximale Metaspace-Größe.

Es wird empfohlen, -Xms und -Xmx auf denselben Wert zu setzen, um den Overhead durch Heap-Größenanpassungen zur Laufzeit zu vermeiden.

### Leistungsüberwachungswerkzeuge

JDK bietet reichhaltige Leistungsüberwachungs- und Diagnosewerkzeuge:

jps: Zeigt Liste der Java-Prozesse, grundlegende Verwendung jps -l.
jstat: Zeigt JVM-Statistiken, z.B. Garbage-Collection-Statistiken: jstat -gcutil pid 1000.
jstack: Druckt Stack-Trace-Informationen von Threads, verwendet zur Diagnose von Deadlocks: jstack pid.
jmap: Zeigt Heap-Speicherinformationen, kann Heap-Dump-Dateien zur Analyse generieren: jmap -dump:format=b,file=heap.bin pid.
jcmd: Multifunktions-Diagnosewerkzeug, kann verschiedene Diagnosebefehle ausführen: jcmd pid GC.heap_info.

JConsole und VisualVM: Grafische Überwachungs- und Diagnosewerkzeuge, können Speicher, Threads, CPU-Auslastung usw. überwachen.

## Modulsystem

### Modulübersicht

Ab JDK 9 führte Java das Modulsystem (Project Jigsaw) ein, das JDK und Anwendungen in Module unterteilt. Jedes Modul hat klare Abhängigkeiten und bietet bessere Kapselung und Wartbarkeit.

JDK selbst ist in mehrere Module wie java.base, java.sql, java.xml unterteilt. Anwendungen können eigene Module erstellen, die über module-info.java-Dateien Modulnamen, Abhängigkeiten und Exportpakete deklarieren.

### Modulbezogene Optionen

JDK bietet mehrere Befehlszeilenoptionen im Zusammenhang mit Modulen:

--module-path (oder -p): Gibt den Modulpfad an, ähnlich wie Classpath, aber für Module.
--add-modules: Fügt Module zum Standard-Modulset hinzu.
--add-exports: Öffnet angegebene Modulpakete für den Zugriff durch andere Module.
--add-opens: Öffnet angegebene Modulpakete für Reflexionszugriff.

Diese Optionen sind in bestimmten Situationen sehr wichtig, z.B. bei der Verwendung von Reflexion für den Zugriff auf JDK-interne APIs oder bei der Verwendung von Bibliotheken, die nicht mit Modularisierung kompatibel sind.

## Konfigurationsbeispiele

### Entwicklungsumgebung

Für tägliche Entwicklungsumgebungen wird empfohlene Konfiguration verwendet: ausreichend Heap-Speicher (-Xmx2g oder höher), G1 Garbage-Collector (Standard), Klassendatenfreigabe aktivieren (-Xshare:classes), um den Start zu beschleunigen. Wenn Sie eine IDE verwenden, können Sie diese Optionen direkt in der IDE-Ausführungskonfiguration festlegen.

### Produktionsumgebung

Die Produktionsumgebungskonfiguration muss an die spezifische Situation der Anwendung angepasst werden. Grundprinzipien: Heap-Größe gemäß den Anwendungsanforderungen festlegen, normalerweise 25-30% Speicher für das Betriebssystem und andere Prozesse reservieren; geeigneten Garbage-Collector auswählen, G1 ist normalerweise eine gute Wahl; für Anwendungen, die niedrige Latenz anstreben, sollten Sie ZGC oder Shenandoah in Betracht ziehen.

Es wird empfohlen, in Produktionsumgebungen GC-Logging zu aktivieren, das detaillierte Garbage-Collection-Informationen aufzeichnet, um Problemanalyse und Leistungsoptimierung zu erleichtern. GC-Logging-Konfigurationsbeispiel: -Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M.

### Docker-Container-Konfiguration

Beim Ausführen von Java-Anwendungen in Containerumgebungen ist auf das Speicherbewusstsein der JVM zu achten. JDK 10+ Versionen sind standardmäßig bewusst für Container-Speichergrenzen, aber in früheren Versionen kann eine manuelle Konfiguration erforderlich sein.

Empfohlene Konfiguration: Richtigen maximalen Heap-Wert festlegen (-XX:MaxRAMPercentage=75.0), sicherstellen, dass die JVM nicht mehr als 75% des verfügbaren Container-Speichers verwendet; -XX:+UseContainerSupport (standardmäßig aktiviert) verwenden, damit die JVM Container-Ressourcengrenzen erkennt.

Für Cloud-native Anwendungen sollten Sie die Verwendung von GraalVM Native Image in Betracht ziehen, das Anwendungen in native ausführbare Dateien kompiliert und den Start und die Speichernutzung erheblich reduzieren kann.

## Versionsspezifische Nutzungsunterschiede

### JDK 8

JDK 8 ist die letzte vollständig kostenlose Oracle JDK-Version für kommerzielle Nutzung. Hauptfunktionen umfassen Lambda-Ausdrücke, Streams-API und neue Datums-/Zeit-API. Die Installation ist relativ einfach und verwendet den traditionellen Installer. Beachten Sie, dass JAVA_HOME auf das JDK-Verzeichnis (nicht JRE) zeigen sollte. Einige Befehle und Tools verhalten sich in neueren Versionen möglicherweise anders.

### JDK 11

JDK 11 markierte den Beginn der kostenpflichtigen Java-Funktionen. Hauptfunktionen umfassen lokale Variablentyp-Inferenz (var), neuen HTTP-Client und ZGC (experimentell). Das JRE wird nicht mehr als separater Download angeboten; nur JDK ist verfügbar. Der java-Befehl kann Einzell quellcode-Programme direkt ohne Kompilierung ausführen.

### JDK 17

JDK 17 ist die letzte vollständig kostenlose Oracle JDK-Version unter NFTC. Hauptfunktionen umfassen versiegelte Klassen, Textblöcke und switch-Ausdrücke. Installation und Verwendung ähneln JDK 11. Die RandomGenerator-Schnittstelle bietet neue Funktionen zur Zufallszahlengenerierung. Die Foreign Memory Access API trat in die Inkubationsphase ein.

### JDK 21

JDK 21 führt wichtige Funktionen ein, darunter virtuelle Threads (stabil), Pattern Matching für switch (stabil) und Records (stabil). Virtuelle Threads erfordern andere Ansätze zur nebenläufigen Programmierung. Anwendungen, die virtuelle Threads verwenden, müssen möglicherweise die Heap-Einstellungen anpassen. Die Leistungsoptimierung sollte die Eigenschaften virtueller Threads berücksichtigen.

### JDK 25

JDK 25 ist die neueste Version mit neuesten Sprachfunktionen und Leistungsverbesserungen. Funktionen können sich noch im Vorschaustadium oder experimentell befinden. Nicht für Produktionsumgebungen empfohlen, die langfristigen Support erfordern. Konfiguration und Verwendung ähneln JDK 21, aber mit mehr neuen Funktionen, die möglicherweise zusätzliches Lernen erfordern.
