# Hadoop 常見問題

## HDFS 相關問題

### HDFS 儲存空間不足怎麼辦

HDFS 儲存空間不足是一個常見問題，可能由多種原因導致。首先檢查 DataNode 的磁碟使用情況：

```bash
hdfs dfsadmin -report
```

如果確實空間不足，可以採取以下措施：刪除不再需要的歷史資料；使用 hdfs balancer 均衡資料分布，釋放部分節點；增加新的 DataNode 節點擴展叢集儲存容量；調整副本數，減少儲存開銷（需權衡資料安全性）。

```bash
# 刪除檔案
hdfs dfs -rm -r /user/olddata

# 均衡資料
hdfs balancer -threshold 10
```

### NameNode 記憶體不足如何處理

NameNode 將所有中繼資料儲存在記憶體中，如果檔案數量過多，可能導致記憶體不足。可以透過以下方式緩解：增加 NameNode 的記憶體配置，在 hadoop-env.sh 中設定 HADOOP_NAMENODE_OPTS 的 -Xmx 參數；啟用 NameNode 快照功能，定期建立唯讀副本以備份重要資料；升級到 Hadoop 2.x 或更高版本，它們對中繼資料管理有更好的優化。

```bash
# 在 hadoop-env.sh 中設定
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### DataNode 啟動失敗怎麼辦

DataNode 啟動失敗可能由多種原因導致。首先檢查 DataNode 日誌：

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

常見原因和解決方案包括：磁碟空間不足，需要清理磁碟或新增磁碟；權限問題，確保 Hadoop 使用者對資料目錄有讀寫權限；叢集 ID 不一致，如果是新加入的 DataNode，需要在 DataNode 上執行 hdfs namenode -upgrade 或刪除 storage 目錄後重新格式化。

```bash
# 檢查 DataNode 目錄權限
ls -la /usr/local/hadoop/data/

# 刪除 DataNode 的 storage 目錄並重新啟動
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### 如何恢復誤刪的檔案

HDFS 提供了回收筒功能，可以在刪除檔案後恢復。首先確認回收筒是否啟用：

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

如果發現誤刪，可以從回收筒恢復：

```bash
# 恢復檔案
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

如果回收筒已被清空或過了保留時間，資料將無法恢復。因此，日常操作中建議：啟用回收筒功能（預設啟用）；重要資料做好備份；刪除前仔細確認。

### HDFS 塊損壞如何處理

資料塊可能因磁碟故障或網路問題而損壞。HDFS 有內建的資料完整性檢測機制，發現損壞塊時會自動從其他副本複製資料進行恢復。手動檢查和修復：

```bash
# 檢查檔案系統完整性
hdfs fsck / -files -blocks -locations

# 刪除損壞的塊檔案
hdfs fsck / -delete
```

## MapReduce 相關問題

### 作業執行緩慢怎麼辦

MapReduce 作業執行緩慢可能由多種因素導致。首先查看作業歷史日誌，分析各個階段的耗時：

```bash
# 查看作業詳情
yarn logs -applicationId application_xxx -logFiles *

# 在 JobHistory Server 查看
mapred job -history job_history_file
```

常見優化方向包括：增加 Reduce 任務數量以提高並行度；調整 JVM 堆記憶體大小避免 OOM；啟用 Combiner 減少網路傳輸資料量；優化資料傾斜，使用自定義分區器平衡負載；選擇合適的檔案格式（Parquet、ORC 等）。

### 如何處理資料傾斜

資料傾斜是指資料在 Reduce 階段分布不均，導致部分 Reduce 任務執行時間遠超其他任務。解決方案包括：使用自定義分區器，根據業務特點重新分配資料；添加隨機前綴使資料先分散到多個 Reduce，再二次聚合；對 Key 進行加鹽處理，聚合後再去除前綴。

```java
// 自定義分區器範例
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // 根據業務邏輯重新分配
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### MapReduce 作業失敗如何排查

作業失敗時，首先查看錯誤日誌確定失敗原因：

```bash
# 查看作業狀態
yarn application -status application_xxx

# 查看任務失敗原因
mapred job -status job_id
```

常見失敗原因包括：Java 記憶體溢出（OOM），增加 mapreduce.map.memory.mb 和 mapreduce.reduce.memory.mb；輸入資料格式錯誤，檢查資料是否符合預期格式；自定義程式碼異常，檢查程式碼邏輯和異常資訊；網路超時，增加 mapreduce.task.timeout 參數。

## YARN 相關問題

### ResourceManager 無法啟動怎麼辦

ResourceManager 啟動失敗通常由設定錯誤或連接埠衝突導致。檢查日誌：

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

常見問題和解決方案：連接埠被佔用，修改 yarn-site.xml 中的 ResourceManager 連接埠；ZooKeeper 設定錯誤（如果使用 HA），檢查 zkfc 相關設定；記憶體不足，確保機器有足夠的實體記憶體；設定檔案錯誤，仔細檢查 yarn-site.xml 和 capacity-scheduler.xml。

```bash
# 查看連接埠佔用
netstat -tlnp | grep 8032
```

### 叢集資源利用率低怎麼辦

叢集資源利用率低可能由以下原因導致：任務提交模式不合理，作業過小導致調度開銷大於計算時間；調度器配置不當，佇列資源分配不合理；資料本地性差，輸入資料與計算節點不在同一位置。

優化建議：批量提交小作業；合理配置佇列容量和優先級；優化資料布局，提高資料本地性；使用合適的調度器（Capacity Scheduler 或 Fair Scheduler）。

### 如何限制使用者資源使用

可以透過 YARN 佇列和存取控制限制使用者資源使用：

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- 單使用者最大可用佇列資源的倍數 -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- 佇列最大執行作業數 -->
</property>
```

## 叢集運維問題

### 如何監控 Hadoop 叢集

Hadoop 提供了多種監控方式：Web UI（NameNode 9870 連接埠、ResourceManager 8088 連接埠）；JMX（Java Management Extensions）監控；Ganglia、Prometheus 等外部監控系統；命令列工具（hdfs dfsadmin、yarn node 等）。

推薦使用 Prometheus + Grafana 搭建完整的監控平台，收集 Hadoop 的 JMX 指標並可視化展示。

### 如何進行叢集升級

Hadoop 叢集升級需要謹慎操作，建議按以下步驟進行：在測試環境先驗證新版本的相容性；備份 NameNode 的中繼資料和 HDFS 資料；逐一升級 DataNode 節點，最後升級 NameNode；使用 rolling upgrade 進行滾動升級，減少停機時間。

```bash
# 滾動升級 DataNode
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# 升級 Hadoop
tar -xzf hadoop-new.tar.gz -C /usr/local/
# 啟動 DataNode
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### NameNode 高可用如何配置

NameNode HA 可以避免 NameNode 單點故障。配置步驟包括：在 hdfs-site.xml 中配置兩個 NameNode；配置共用編輯目錄（通常使用 ZooKeeper 或 NFS）；啟動 ZooKeeper Failover Controller (ZKFC)。

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

配置完成後，使用以下命令初始化 ZKFC：

```bash
hdfs zkfc -formatZK
```

### 如何處理節點下線

節點下線需要謹慎處理，確保資料不丟失。使用 decommission 功能讓節點安全下線：

```bash
# 在 workers 檔案中添加要下線的節點到 exclude 檔案
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# 重新整理節點列表
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

等待資料複製完成後，節點會自動進入 decommissioned 狀態，確認資料已複製到其他節點後再關閉節點。

## 開發相關問題

### 如何調試 MapReduce 程式

MapReduce 程式調試比較困難，推薦以下方法：在本地模式下先測試邏輯；使用 Counter 記錄關鍵指標；添加日誌輸出，在 Map 或 Reduce 方法中列印關鍵變數值；使用 MRUnit 進行單元測試。

```java
// 使用 Counter 計數
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### 如何選擇 InputFormat 和 OutputFormat

InputFormat 決定如何讀取輸入資料：TextInputFormat（預設，按行讀取）；KeyValueTextInputFormat（按定位點分割）；SequenceFileInputFormat（二進制格式）；DBInputFormat（從資料庫讀取）。

OutputFormat 決定如何輸出結果：TextOutputFormat（預設，文字格式）；SequenceFileOutputFormat（二進制格式）；NullOutputFormat（不輸出檔案）；DBOutputFormat（輸出到資料庫）。

### 如何處理小檔案問題

大量小檔案會導致 NameNode 記憶體壓力增大和處理效率降低。解決方案包括：使用 Hadoop Archives (HAR) 將小檔案打包；使用 SequenceFile 合併小檔案；使用 HBase 儲存大量小檔案；調整塊大小，減少小檔案數量。
