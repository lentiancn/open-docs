# Manuel d'utilisation de Mockito

## Creation d'objets Mock

La creation d'objets mock est la premiere etape pour utiliser Mockito dans les tests. Il existe plusieurs facons de creer des objets mock, et les developpeurs peuvent choisir la methode appropriee selon leurs besoins.

### Utilisation de la methode mock()

La facon la plus simple est d'utiliser la methode statique mock() de la classe Mockito pour creer des objets mock. Cette methode accepte un parametre Class et retourne une instance mock de cette classe.

```java
import static org.mockito.Mockito.*;

List<String> mockedList = mock(List.class);
Map<String, Integer> mockedMap = mock(Map.class);
```

A partir de Mockito 4.10.0, vous pouvez omettre le parametre Class et le compilateur inferera automatiquement le type.

```java
List<String> mockedList = mock();
```

### Utilisation de l'annotation @Mock

L'annotation @Mock offre un style de declaration plus propre, particulierement utile lorsque plusieurs objets mock doivent etre creates dans une classe de test. L'utilisation de l'annotation necessite d'abord l'initialisation du framework Mockito.

```java
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @Before
    public void setup() {
        MockitoAnnotations.initMocks(this);
    }
}
```

L'utilisation de MockitoJUnitRunner peut gerer l'initialisation automatiquement.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

L'utilisation de l'annotation @ExtendWith de JUnit 5 realise egalement l'initialisation automatique.

```java
@ExtendWith(MockitoExtension.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

## Definition du comportement des methodes

Definir le comportement des methodes est aussi appele stubbing. Il est utilise pour definir quelles valeurs doivent etre retournees ou quelles exceptions doivent etre lancees lorsque des methodes sont appelees sur des objets mock.

### Configuration basique des valeurs de retour

Utilisez la syntaxe chainee when().thenReturn() pour definir les valeurs de retour.

```java
when(mockedList.size()).thenReturn(100);
when(mockedList.get(0)).thenReturn("first");
when(mockedList.get(1)).thenReturn("second");
```

Vous pouvez definir plusieurs valeurs de retour consecutives, et chaque appel subsequent retournera des valeurs differentes.

```java
when(mockedList.get(anyInt()))
    .thenReturn("first")
    .thenReturn("second")
    .thenThrow(new RuntimeException("Error"));

System.out.println(mockedList.get(0)); // affiche: first
System.out.println(mockedList.get(0)); // affiche: second
System.out.println(mockedList.get(0)); // lance une exception
```

### Definition du lancement d'exceptions

Utilisez thenThrow() pour definir les exceptions qui doivent etre lancees lorsque des methodes sont appelees.

```java
when(mockedList.get(0)).thenThrow(new IndexOutOfBoundsException("Index 0"));
```

Pour les methodes avec un type de retour void, utilisez doThrow().

```java
doThrow(new RuntimeException("Error")).when(mockedList).clear();
```

### Utilisation de Answer pour les reponses personnalisees

Si les valeurs de retour predefinies sont insuffisantes, vous pouvez implementer l'interface Answer pour calculer dynamiquement les valeurs de retour.

```java
when(mockedList.get(anyInt())).thenAnswer(invocation -> {
    int index = invocation.getArgument(0);
    return "Value at index " + index;
});
```

## Verification des interactions

La verification est utilisee pour confirmer que le code teste interagit correctement avec les objets mock comme attendu.

### Verification basique

Utilisez verify() pour verifier si une methode a ete appelee.

```java
mockedList.add("one");
verify(mockedList).add("one");
```

### Verification du nombre d'appels

Vous pouvez verifier combien de fois une methode a ete appelee.

```java
mockedList.add("one");
mockedList.add("one");
mockedList.add("one");

verify(mockedList, times(3)).add("one");       // exactement 3 fois
verify(mockedList, atLeast(2)).add("one");     // au moins 2 fois
verify(mockedList, atMost(3)).add("one");     // au plus 3 fois
verify(mockedList, never()).add("two");        // jamais appele
verify(mockedList, atLeastOnce()).add("one"); // au moins une fois
```

### Verification de l'ordre des appels

Le verificateur InOrder peut verifier si les appels de methodes ont ete executes dans un ordre specifique.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
inOrder.verify(mockedList).add("third");
```

### Verification des arguments d'appel

Des matchers d'arguments peuvent etre utilises pour verifier flexiblement les arguments d'appel.

```java
verify(mockedList).add(anyString());
verify(mockedList).get(anyInt());
verify(mockedList).contains(argThat(s -> s.length() > 5));
```

## Matchers d'arguments

Mockito fournit des matchers darguments riches pour le matching flexible des parametres de methodes.

### Matchers integres

```java
anyInt()          // n'importe quel entier
anyString()       // n'importe quelle chaine
any()             // n'importe quel objet
anyList()         // n'importe quelle liste
anyMap()          // n'importe quelle map
isNull()          // valeur null
isNotNull()       // valeur non null
eq(value)         // correspondance exacte
startsWith()      // commence par la chaine specifiee
endsWith()        // finit par la chaine specifiee
contains()        // contient la chaine specifiee
```

### Matchers personnalises

Des matchers d'arguments personnalises peuvent etre crees en utilisant la methode argThat().

```java
when(mockedList.add(argThat(s -> s != null && s.length() > 3)))
    .thenReturn(true);
```

## Utilisation de Spy

Contrairement aux mocks, les spies conservent la majorite du comportement de l'objet reel tout en permettant a des methodes specifiques d'etre stubbees.

### Creation d'un Spy

```java
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);
```

Ou utilisez l'annotation @Spy.

```java
@Spy
private List<String> list = new ArrayList<>();
```

### Definition du comportement du Spy

Contrairement aux mocks, vous devez etre prudent lorsque vous utilisez when() avec des spies car cela peut affecter les appels de methodes reelles. Il est recommande d'utiliser la syntaxe doReturn().when().

```java
doReturn(100).when(spyList).size();
```

Les Spy appellent les methodes reales sauf si elles sont explicitement stubbees.

```java
spyList.add("element");  // la methode reelle sera appelee
verify(spyList).add("element");  // la verification reussit
```

## Utilisation de @InjectMocks

L'annotation @InjectMocks est utilisee pour creer automatiquement l'objet teste et y injecter les champs annotes avec @Mock ou @Spy.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @InjectMocks
    private UserService userService;
    
    @Test
    public void testCreateUser() {
        User user = new User("testUser");
        when(userRepository.save(any(User.class))).thenReturn(user);
        
        User result = userService.createUser("testUser");
        
        assertEquals("testUser", result.getUsername());
        verify(userRepository).save(any(User.class));
    }
}
```

## Utilisation d'ArgumentCaptor

ArgumentCaptor est utilise pour capturer les arguments passes lors des appels de methodes pour des assertions plus detaillees.

```java
@Mock
private List<String> mockedList;

@Test
public void testCaptureArgument() {
    mockedList.add("test");
    
    ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
    verify(mockedList).add(captor.capture());
    
    assertEquals("test", captor.getValue());
}
```

Lorsqu'une methode est appelee plusieurs fois, vous pouvez capturer les arguments de tous les appels.

```java
mockedList.add("first");
mockedList.add("second");
mockedList.add("third");

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList, times(3)).add(captor.capture());

assertEquals(3, captor.getAllValues().size());
assertEquals("second", captor.getAllValues().get(1));
```

## Tests style BDD

Mockito supporte les tests style Development Guide par le Comportement (BDD) en utilisant la syntaxe given().willReturn().

```java
import static org.mockito.BDDMockito.*;

@Test
public void testBddStyle() {
    // Given
    given(mockedList.get(0)).willReturn("first");
    
    // When
    String result = mockedList.get(0);
    
    // Then
    assertEquals("first", result);
}
```

## Modeles d'utilisation courants

### Test des methodes void

```java
@Test
public void testVoidMethod() {
    doNothing().when(mockedList).clear();
    mockedList.clear();
    verify(mockedList).clear();
}
```

### Simulation de chaines de methodes

```java
@Test
public void testMethodChaining() {
    when(mockedList.iterator()).thenReturn(Collections.emptyIterator());
    
    Iterator<String> iterator = mockedList.iterator();
    assertFalse(iterator.hasNext());
}
```

### Simulation de la gestion des exceptions

```java
@Test(expected = RuntimeException.class)
public void testException() {
    when(mockedList.get(0)).thenThrow(new RuntimeException("Error"));
    mockedList.get(0);
}
```

## Bonnes pratiques

Lors de l'utilisation de Mockito, suivez les bonnes pratiques pour assurer la qualite et la maintenabilite du code de test. Ne simulez pas les types que vous possedez ou les objets de valeur, car cela rompt l'authenticite des tests. Ne simulez que les dependances necessaires pour maintenir l'authenticite du test. Chaque methode de test doit se concentrer sur le test d'une seule fonctionnalite. Organisez correctement le code de test en utilisant @Before pour la configuration commune. La verification doit etre moderee, en se concentrant sur les interactions importantes plutot que de verifier chaque appel de methode.
