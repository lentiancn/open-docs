# Mockito よくある質問

## 基本的な質問

### Mockito と PowerMock の違い

Mockitoは通常のオブジェクトのモック、主に使用しますが、PowerMockは静的メソッド、コンストラクタ、私有メソッドなどをモックできます。ただし、PowerMockはバイトコード操作技術を使用してテスト分離を破り、テストをより統合テストに近い单元テスト而不是にします。公式の推奨は最初にMockitoを使用し、Mockitoが本当に处理できないシナリオ（静的メソッドのモックなど）がある場合にのみPowerMockを検討することです。Mockito 2.x以降では、mockito-inlineを通じて静的メソッドのモックもサポートされています。

### Spy と Mock の使い分け

Spyは大部分の実動作が必要で、少数のメソッドのみをモックしたい場合に使用します。Spyは実オブジェクトの機能の大部分を保持しながら、特定のメソッドのみをスタブできます。Mockは実オブジェクトを完全に置き換えるため、実際のメソッドロジックは実行されません。オブジェクト動作を完全に制御する必要がある場合はMockを使用します。実際にはSpyの使用シナリオは比較的少ないです。なぜならほとんどの場合、実テストオブジェクトまたは完全にMockすれば要件を満たすことができるからです。

### Mockito は final クラスやメソッドに使用できるか

Mockito 2.1.0以降では、final クラスやメソッドをモックするようにMockitoを設定できます。mockito-coreではなく、mockito-inline依存関係をプロジェクトに追加する必要があります。mockito-inlineはサブクラスの動的サブクラス化メカニズムを有効にし、final クラスやメソッドを処理できます。この機能はMockitoの適用範囲を大幅に強化し、ほぼすべてのJavaクラスをモック 가능にします。

### モックオブジェクトのメソッドが null を返すのはなぜですか

デフォルトでは、モックオブジェクトのすべてのメソッドはnullまたは対応する型のデフォルト値を返します（intは0を返し、booleanはfalseを返し、コレクション型は空のコレクションを返しますなど）。これはMockitoのデフォルトAnswer動作です。特定の戻り値が必要な場合は、when().thenReturn()を使用して明示的に設定する必要があります。

## 使用に関する質問

### @Mock アノテーションが機能しない場合はどうしますか

@Mockアノテーションは、フレームワークを初期化しないと機能しません。テストでMockitoAnnotations.initMocks(this)、@RunWith(MockitoJUnitRunner.class)、または@ExtendWith(MockitoExtension.class)（JUnit 5）のいずれかを使用していることを確認してください。正しく初期化されると、@Mockアノテーションが付けられたフィールドはテストメソッドの前に自動的にモックオブジェクトを作成します。

### void メソッドをモックするには

void戻り型のメソッドには、doThrow()、doAnswer()、またはdoNothing()を使用して動作を定義します。

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### メソッド呼び出し引数を検証するには

引数マッチャーを使用して検証するか、ArgumentCaptorを使用して詳細なアサーションのために引数をキャプチャできます。

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### イテレータをモックするには

イテレータのモックには、hasNext()とnext()メソッドの調整に注意する必要があります。

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### メソッドオーバーロード怎么处理

クラスにオーバーロードされたメソッドがある場合、Mockitoはパラメータの型と数に基づいて異なるメソッドを区別します。when()のパラメータ型がモックしたいメソッドの署名と完全に一致していることを確認してください。

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // オーバーロードされたメソッド
```

## 上級者向け質問

### when().thenReturn() が動作しないのはなぜですか

一般的な原因には、同じメソッドに複数回動作を設定する（最後の1つのみが有効）；Spyで реальメソッドを呼び出す；メソッドの戻り型がvoid（doReturn().when()を使用すべき）；引数マッチャーの使用法が正しくない导致匹配失败。動作設定の構文が正しいことを確認してください。

### 抽象クラスをテストするには

抽象クラスをテストする場合、Mockitoを使用して抽象クラスの具象メソッドをモックするか、Spyを使用して一部の実際動作を保持できます。もう1つの方法は、テスト用の具体的なサブクラスを作成してテストを実行することです。

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### ジェネリックメソッドをモックするには

ジェネリックメソッドをモックする場合、型消去による制限に注意してください。any()またはanyObject()を使用してジェネリックパラメータをマッチングします。

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### 呼び出し順序を検証するには

InOrderを使用して、メソッド呼び出しが特定の順序で実行されたことを確認します。

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### オブジェクトを部分的にモックするには

Spyを使用して部分的なモックを行うか、withSettings()を使用してMockの動作をカスタマイズします。

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## パフォーマンスの質問

### テストの実行が遅い場合はどうしますか

不必要な реальメソッド呼び出しを使用していないことを確認してください。各テストで新しいMockオブジェクトを作成していないか確認してください。Mockitoを大量に使用するプロジェクトでは、@RunWith(MockitoJUnitRunner.Silent.class)を使用してログ出力を削減することを検討してください。

### モックオブジェクトのメモリ使用量を減らすには

Mockitoのモックオブジェクト自体は大容量のメモリを消費しません。ただし、大量のモックオブジェクトを作成するとパフォーマンスに影響する場合、@Mockアノテーションと@Scopeを使用してモックオブジェクトを再利用することを検討するか、静的Mockファクトリメソッドを使用してください。

## デバッグの質問

### モックオブジェクトの呼び出しログを表示するには

Mockitoのログ機能を使用するか、リスナーを追加して呼び出しを追跡できます。

```java
Mockito.framework().addListener(new VerificationListener());
```

またはMaven/Gradleのテスト出力を使用して詳細なMock呼び出し情報を表示します。

### テストが失敗した場合のデバッグ方法

Mockitoは詳細なエラーメッセージを提供します。verify失敗時に表示される予期された呼び出しと実際の呼び出しの違いを確認します。Print Mockito Logs機能を使用してモックオブジェクトの動作を理解します。テストケースを段階的に簡素化して問題を特定します。
