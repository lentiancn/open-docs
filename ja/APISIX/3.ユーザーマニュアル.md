# ユーザーマニュアル

このドキュメントでは、Apache APISIX のコア機能と使用方法について詳しく説明します。ルーティング設定、アップストリーム管理、負荷分散、レート制限、認証などの主要機能をカバーしています。

## 基本概念

### Route（ルート）

ルートはアップストリームターゲットへのアクセス経路です。APISIXでは、ルートはまず事前定義されたルールでクライアントリクエストを一致させ соответствующиеプラグインを読み込んで実行してから、特定の上流サービスにリクエストを転送します。

最もシンプルなルートは、マッチングパスとアップストリームアドレスの2つの情報だけで構成されます。

### Upstream（アップストリーム）

アップストリームは、同じ機能を持つノードの集合であり、バーチャルホストの抽象化です。アップストリームは事前設定されたルールに基づいて、複数のサービスノードに対して負荷分散を行うことができます。

### Consumer（コンシューマ）

コンシューマは、APIを使用するアプリケーションや開発者です。APISIXでは、コンシューマにはグローバルに一意の名前が必要で、認証プラグインにバインドする必要があります。

### Plugin（プラグイン）

APISIXは豊富なプラグインシステムを提供しており、ゲートウェイ機能を拡張できます。プラグインはルート、コンシューマ、またはグローバルレベルで設定できます。

## ルーティングの設定

管理APIでルートを簡単に作成できます。以下は、httpbin.orgにリクエストを転送するルートを作成する例です：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

作成に成功すると `HTTP/1.1 201 Created` が返されます。

### ルートの確認

```bash
curl "http://127.0.0.1:9080/ip"
```

正常に動作している場合、以下のような内容が返されます：

```json
{
  "origin": "183.94.122.205"
}
```

### ルートのマッチルール

APISIXは複数のルートマッチ方式をサポートしています：

**URI マッチ：**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**Host マッチ：**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**リクエストメソッドマッチ：**

```json
{
  "methods": ["GET", "POST"]
}
```

**ヘッダーマッチ：**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## 負荷分散

APISIXは、ラウンドロбин、加重ラウンドロビン、最少接続など、複数の負荷分散アルゴリズムをサポートしています。

### 重み付き負荷分散を設定

複数のアップストリームサービスを持つルートを作成します：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**パラメータ説明：**

- `type`：負荷分散アルゴリズムのタイプ。`roundrobin`（ラウンドロビン）、`weighted_roundrobin`（加重ラウンドロビン）、`least_conn`（最少接続）をサポート
- `nodes`：アップストリームサービスノード。フォーマットは `{"host:port": weight}`
- `pass_host`：`node` に設定すると、リクエストヘッダーをアップストリームに転送
- `scheme`：アップストリームプロトコル。`http` と `https` をサポート

### 負荷分散の確認

100個のリクエストを生成して分散効果をテストします：

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

リクエストはほとんど均等に2つのアップストリームに分配されます。

## レート制限

APISIXは柔軟なレート制限機能を提供しており、`limit-count` プラグインで実装されています。

### レート制限を有効化

既存のルートにレート制限プラグインを追加します：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**パラメータ説明：**

- `count`：時間枠内で許可される最大リクエスト数
- `time_window`：時間枠のサイズ（秒）
- `rejected_code`：制限を超えたときに返すHTTPステータスコード

上記の設定では、10秒あたり最大2リクエストまでの受信トラフィックに制限されます。

### レート制限の確認

100個のリクエストを生成してレート制限効果をテストします：

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

結果：2件のリクエストが成功（200）、98件のリクエストが拒否（503）されます。

### レート制限を無効化

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### 高度なレート制限設定

**変数の使用：**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**Redis クラスタの使用：**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## 認証

APISIXは、Key Auth、Basic Auth、JWT、OAuth2 など、複数の認証方式をサポートしています。

### Key Authentication（APIキー認証）

#### コンシューマを作成

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "secret-key"
    }
  }
}'
```

> ⚠️ 本番環境では複雑なキーを使用してください。

#### ルートでAPIキー認証を有効化

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### 認証の確認

**キーを含まないリクエスト：**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

返答：`HTTP/1.1 401 Unauthorized`

**誤ったキーを含むリクエスト：**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: wrong-key'
```

返答：`HTTP/1.1 401 Unauthorized`

**正しいキーを含むリクエスト：**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: secret-key'
```

返答：`HTTP/1.1 200 OK`

### その他の認証方式

**JWT Authentication：**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "user-key",
      "secret": "my-secret-key"
    }
  }
}
```

**Basic Authentication：**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**HMAC Authentication：**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "my-secret-key"
    }
  }
}
```

## サービスガバナンス

### グレーディングリリース

APISIXは重みベースのグレーディングリリースとカナリアリリースをサポートしています：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### リクエスト書き出し

`proxy-rewrite` プラグインを使用してリクエストを書き出します：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### レスポンスキャッシュ

`proxy-cache` プラグインを使用してアップストリームレスポンスをキャッシュします：

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## プラグイン設定

### プラグイン実行順序

`plugins` 設定の `phase` でプラグイン実行フェーズを指定できます：

- `rewrite`：リクエスト書き出しフェーズ
- `access`：アクセス制御フェーズ
- `before_proxy`：プロキシ前フェーズ
- `header_filter`：レスポンスヘッダーフィルター段階
- `body_filter`：レスポンスボディフィルター段階
- `log`：ログ記録フェーズ

### プラグインを無効化

プラグインの `_meta.disable` を `true` に設定します：

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/route-id" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## モニタリングと可観測性

### Prometheus との連携

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

メトリクスにアクセス：

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### ログ記録

APISIXは複数のログ出力をサポートしています：

**HTTP ログ：**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

**エラーログ：**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

## ベストプラクティス

1. **意味のあるIDを使用する**：ルート、アップストリームなどのリソースには説明的なIDを使用しましょう
2. **適切なタイムアウトを設定**：アップストリームの応答時間に基づいて合理的なタイムアウトパラメータを設定しましょう
3. **ヘルスチェックを有効化**：重要なサービスには能動的・受動的なヘルスチェックを設定しましょう
4. **適切なレート制限を設定**：ビジネスニーズとサーバー能力に基づいてレート制限ルールを設定しましょう
5. **認証を有効化**：本番環境では必ず認証を有効にしましょう
6. **TLS を使用**：外部に公開するインターフェースには HTTPS を使用しましょう
7. **主要指標をモニタリング**：Prometheus モニタリングを設定し、アラートを設定しましょう
