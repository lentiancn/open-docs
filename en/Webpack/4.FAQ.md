# Webpack Frequently Asked Questions

## Installation Related Questions

### Q: Should I install Webpack globally or locally?
**A**: Always install Webpack locally as a development dependency. Global installation can cause version conflicts between projects and makes it harder to manage different Webpack versions across projects.

### Q: What's the difference between webpack and webpack-cli?
**A**: 
- **webpack**: The core bundling engine
- **webpack-cli**: The command-line interface that provides the CLI commands

Both are required for basic usage. Install them together: `npm install --save-dev webpack webpack-cli`

### Q: How do I choose between Webpack v4 and v5?
**A**: 
- **New projects**: Use Webpack v5 for better performance, persistent caching, and modern features
- **Existing projects**: If already using v4, you can continue using it or migrate to v5
- **Enterprise environments**: v4 provides stable long-term support, while v5 offers cutting-edge features

## Configuration Related Questions

### Q: Do I need a configuration file?
**A**: Since Webpack 4, you don't need a configuration file for basic usage. Webpack will use default settings and look for `./src/index.js` as the entry point. However, most real-world projects will need custom configuration.

### Q: What's the difference between CommonJS and ES Modules in Webpack config?
**A**: 
- **CommonJS** (Webpack v4 style): Uses `module.exports` and `require()`
- **ES Modules** (Webpack v5 style): Uses `export default` and `import`

Webpack v5 supports both, but ES Modules are recommended for new projects.

### Q: How do I handle multiple entry points?
**A**: Specify an object in the entry configuration:

```javascript
export default {
  entry: {
    app: './src/app.js',
    admin: './src/admin.js'
  }
};
```

This will create separate bundles for each entry point.

## Performance Related Questions

### Q: Why is my build so slow?
**A**: Common causes and solutions:
- **Large node_modules**: Use `SplitChunksPlugin` to split vendor code
- **No caching**: Enable persistent caching in Webpack v5
- **Many loaders**: Optimize loader configurations and use include/exclude rules
- **Source maps**: Use faster source map types in development (`eval` instead of `source-map`)

### Q: How do I enable caching for production builds?
**A**: Use content hashes in output filenames:

```javascript
output: {
  filename: '[name].[contenthash].js',
  chunkFilename: '[name].[contenthash].js'
}
```

This ensures that files only change when their content changes, allowing browsers to cache them effectively.

### Q: What is persistent caching in Webpack v5?
**A**: Persistent caching stores build information on disk between runs, significantly speeding up subsequent builds. Enable it with:

```javascript
cache: {
  type: 'filesystem'
}
```

## Development Related Questions

### Q: How do I set up hot module replacement (HMR)?
**A**: Use webpack-dev-server with HMR enabled:

```javascript
// webpack.config.js
export default {
  devServer: {
    hot: true
  }
};
```

And add HMR code to your application:

```javascript
if (module.hot) {
  module.hot.accept('./module', () => {
    // Handle updates
  });
}
```

### Q: How do I debug my bundled code?
**A**: Enable source maps in your configuration:

```javascript
export default {
  devtool: 'source-map' // for production
  // or
  devtool: 'eval-source-map' // for development (faster)
};
```

This allows you to debug your original source code in the browser's developer tools.

### Q: How do I handle CSS and images?
**A**: Use appropriate loaders or Webpack v5's built-in asset modules:

**Webpack v5 (recommended):**
```javascript
module: {
  rules: [
    {
      test: /\.css$/,
      use: ['style-loader', 'css-loader']
    },
    {
      test: /\.(png|svg|jpg|jpeg|gif)$/i,
      type: 'asset/resource'
    }
  ]
}
```

**Webpack v4:**
```javascript
module: {
  rules: [
    {
      test: /\.css$/,
      use: ['style-loader', 'css-loader']
    },
    {
      test: /\.(png|svg|jpg|jpeg|gif)$/i,
      use: ['file-loader']
    }
  ]
}
```

## Migration Questions

### Q: How do I migrate from Webpack v4 to v5?
**A**: Follow these steps:
1. Update dependencies: `npm install webpack@latest webpack-cli@latest`
2. Review the official migration guide for breaking changes
3. Replace file-loader/url-loader with asset modules
4. Update plugin configurations for Webpack v5 compatibility
5. Test thoroughly to ensure everything works correctly

### Q: What happened to file-loader and url-loader in Webpack v5?
**A**: Webpack v5 introduced built-in asset modules that replace file-loader and url-loader:

- **asset/resource**: Replaces file-loader (emits separate files)
- **asset/inline**: Replaces url-loader (inlines as data URLs)
- **asset**: Automatically chooses between resource and inline based on file size

### Q: Are my existing plugins compatible with Webpack v5?
**A**: Most popular plugins have been updated for Webpack v5 compatibility. Check the plugin's documentation or npm page for Webpack v5 support. If a plugin isn't compatible, look for alternatives or check if there are beta versions available.

## Advanced Usage Questions

### Q: What is Module Federation?
**A**: Module Federation is a Webpack v5 feature that enables sharing code between separate applications at runtime. It's particularly useful for micro-frontend architectures where different teams own different parts of the application.

### Q: How do I optimize bundle size?
**A**: Several strategies:
- **Code splitting**: Split code into smaller chunks that load on demand
- **Tree shaking**: Remove unused code (ensure you're using ES6 modules)
- **Minification**: Enable minification in production mode
- **Compression**: Use compression plugins like CompressionWebpackPlugin
- **Analyze bundles**: Use webpack-bundle-analyzer to identify large dependencies

### Q: How do I handle environment variables?
**A**: Use the DefinePlugin to inject environment variables:

```javascript
import webpack from 'webpack';

export default {
  plugins: [
    new webpack.DefinePlugin({
      'process.env.NODE_ENV': JSON.stringify('production'),
      'process.env.API_URL': JSON.stringify('https://api.example.com')
    })
  ]
};
```

These variables will be replaced at build time with their actual values.
