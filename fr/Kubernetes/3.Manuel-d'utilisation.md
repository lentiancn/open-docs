# Manuel d'utilisation de Kubernetes

## Concepts de Base

### Pod
Pod est la plus petite unité déployable dans Kubernetes, représentant un ou plusieurs conteneurs s'exécutant dans le cluster. Typiquement un Pod contient un seul conteneur, mais peut aussi contenir plusieurs conteneurs fortement couplés.

### Deployment
Les Deployments sont utilisés pour gérer les applications sans état, fournissant des capacités de mise à jour déclarative pour un déploiement, une mise à l'échelle et des mises à jour facilités.

### Service
Les Services fournissent des points de terminaison réseau stables pour un ensemble de Pods, maintenant la stabilité même lorsque les adresses IP des Pods changent.

### Namespace
Les Namespaces divisent les ressources du cluster en plusieurs partitions logiques, adaptées à plusieurs équipes ou projets partageant le même cluster.

## Commandes Courantes

### Gestion du Cluster

```bash
# Voir les informations du cluster
kubectl cluster-info

# Voir les nœuds
kubectl get nodes

# Voir tous les pods
kubectl get pods --all-namespaces

# Voir l'utilisation des ressources du cluster
kubectl top nodes
kubectl top pods
```

### Déploiement d'Applications

```bash
# Créer un Deployment
kubectl create deployment nginx --image=nginx

# Voir les Deployments
kubectl get deployments

# Mettre à l'échelle
kubectl scale deployment nginx --replicas=3

# Mettre à jour l'image
kubectl set image deployment/nginx nginx=nginx:1.21

# Restaurer
kubectl rollout undo deployment/nginx

# Voir le statut du déploiement
kubectl rollout status deployment/nginx
```

### Gestion des Services

```bash
# Exposer un Service
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# Voir les Services
kubectl get services

# Supprimer un Service
kubectl delete service <service-name>
```

### Débogage

```bash
# Voir les logs du Pod
kubectl logs <pod-name>

# Exécuter dans le conteneur
kubectl exec -it <pod-name> -- /bin/bash

# Décrire les détails de la ressource
kubectl describe pod <pod-name>

# Transfert de port pour le débogage
kubectl port-forward <pod-name> 8080:80
```

## Manifestes de Ressources

Kubernetes prend en charge la gestion déclarative des ressources via des fichiers YAML. Exemples de définitions de ressources courantes :

### Exemple de Deployment

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
  labels:
    app: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: my-app
        image: my-app:latest
        ports:
        - containerPort: 8080
        resources:
          limits:
            memory: "128Mi"
            cpu: "500m"
          requests:
            memory: "64Mi"
            cpu: "250m"
```

### Exemple de Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app-service
spec:
  selector:
    app: my-app
  ports:
  - protocol: TCP
    port: 80
    targetPort: 8080
  type: ClusterIP
```

### Exemple de ConfigMap

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-config
data:
  database_url: "postgres://db:5432/myapp"
  cache_enabled: "true"
```

### Exemple de Secret

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
type: Opaque
stringData:
  username: admin
  password: changeme
```

## Gestion du Cycle de Vie des Applications

### Déployer des Applications

```bash
# Déployer avec un fichier YAML
kubectl apply -f deployment.yaml

# Voir le statut du déploiement
kubectl get pods -l app=my-app
```

### Mettre à jour les Applications

```bash
# Mettre à jour la version de l'image
kubectl set image deployment/my-app my-app=my-app:v2

# Voir la progression de la mise à jour
kubectl rollout status deployment/my-app
```

### Restauration

```bash
# Restaurer à la version précédente
kubectl rollout undo deployment/my-app

# Restaurer à une révision spécifique
kubectl rollout undo deployment/my-app --to-revision=2

# Voir l'historique des révisions
kubectl rollout history deployment/my-app
```

### Mise à l'échelle

```bash
# Mise à l'échelle manuelle
kubectl scale deployment my-app --replicas=5

# Mise à l'échelle automatique (HPA)
kubectl autoscale deployment my-app --min=2 --max=10 --cpu-percent=80
```

## Gestion du Stockage

### Créer un PersistentVolume

```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: my-pv
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  storageClassName: standard
  hostPath:
    path: /data/my-app
```

### Créer un PersistentVolumeClaim

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

### Utiliser un PVC dans un Pod

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
spec:
  containers:
  - name: my-app
    image: my-app
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: my-pvc
```

## Gestion de la Configuration

### Utiliser ConfigMap

```bash
# Créer ConfigMap depuis un fichier
kubectl create configmap my-config --from-file=config.properties

# Créer depuis un fichier de variables d'environnement
kubectl create configmap my-config --from-env-file=env.properties

# Utiliser dans un Pod
kubectl apply -f pod-with-configmap.yaml
```

### Utiliser Secret

```bash
# Créer Secret depuis un fichier
kubectl create secret generic my-secret --from-file=credentials.json

# Utiliser dans un Pod
kubectl apply -f pod-with-secret.yaml
```

## Quotas et Limites de Ressources

### ResourceQuota

```yaml
apiVersion: v1
kind: ResourceQuota
metadata:
  name: my-quota
spec:
  hard:
    requests.cpu: "4"
    requests.memory: 8Gi
    limits.cpu: "8"
    limits.memory: 16Gi
    pods: "20"
```

### LimitRange

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: my-limits
spec:
  limits:
  - max:
      cpu: "2"
      memory: 1Gi
    min:
      cpu: 100m
      memory: 128Mi
    default:
      cpu: 500m
      memory: 512Mi
    defaultRequest:
      cpu: 200m
      memory: 256Mi
    type: Container
```

## Surveillance et Journaux

### Surveillance

```bash
# Voir l'utilisation des ressources avec kubectl top
kubectl top nodes
kubectl top pods

# Voir le statut détaillé
kubectl describe node <node-name>
kubectl describe pod <pod-name>
```

### Journaux

```bash
# Voir les logs du Pod
kubectl logs my-app-pod

# Voir les logs en temps réel
kubectl logs -f my-app-pod

# Voir les logs de l'instance de conteneur précédente
kubectl logs --previous my-app-pod
```

## Meilleures Pratiques

1. Utiliser la configuration déclarative pour gérer l'infrastructure
2. Définir des limites et demandes de ressources appropriées
3. Organiser les ressources avec des labels
4. Nettoyer régulièrement les ressources non utilisées
5. Utiliser les namespaces pour isoler les différents environnements
6. Configurer les sondes de vérification de santé
7. Utiliser RBAC pour contrôler les permissions
8. Sauvegarder régulièrement les données etcd
