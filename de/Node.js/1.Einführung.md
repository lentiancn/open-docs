# Einführung in Node.js

Node.js® ist eine kostenlose, Open-Source, plattformübergreifende JavaScript-Laufzeitumgebung, die Entwicklern ermöglicht, Server, Webanwendungen, Kommandozeilentools und Skripte zu erstellen.

## Hauptmerkmale

Als asynchrone, ereignisgesteuerte JavaScript-Laufzeitumgebung ist Node.js darauf ausgelegt, skalierbare Netzwerkanwendungen zu erstellen. Im klassischen "Hello World"-Beispiel können viele Verbindungen gleichzeitig verarbeitet werden. Bei jeder Verbindung wird der Callback ausgelöst, aber wenn keine Arbeit zu erledigen ist, wechselt Node.js in den Ruhezustand.

```javascript
const { createServer } = require('node:http');

const hostname = '127.0.0.1';
const port = 3000;

const server = createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello World');
});

server.listen(port, hostname, () => {
  console.log(`Server running at http://${hostname}:${port}/`);
});
```

## Design-Philosophie

Dies steht im Gegensatz zum heute üblicheren Nebenläufigkeitsmodell, bei dem Betriebssystem-Threads verwendet werden. Thread-basierte Vernetzung ist relativ ineffizient und sehr schwer zu verwenden. Außerdem müssen sich Node.js-Benutzer keine Sorgen um Deadlocks im Prozess machen, da es keine Sperren gibt.

Fast keine Funktion in Node.js führt direkt E/A-Operationen durch, daher blockiert der Prozess nie, außer wenn synchrone Methoden der Node.js-Standardbibliothek verwendet werden. Da nichts blockiert, ist die Entwicklung skalierbarer Systeme in Node.js sehr sinnvoll.

## Ereignisgesteuerte Architektur

Node.js ähnelt in seinem Design Systemen wie Ruby's Event Machine und Python's Twisted und wurde von ihnen beeinflusst. Node.js treibt das Ereignismodell noch einen Schritt weiter. Es stellt eine Ereignisschleife als Laufzeitkonstrukt dar, anstatt sie als Bibliothek bereitzustellen.

In anderen Systemen gibt es immer einen blockierenden Aufruf, um die Ereignisschleife zu starten. Typischerweise wird das Verhalten zu Beginn eines Skripts durch Callbacks definiert, und am Ende wird ein Server durch einen blockierenden Aufruf wie `EventMachine::run()` gestartet.

In Node.js gibt es keinen solchen Aufruf zum Starten der Ereignisschleife. Node.js tritt einfach nach der Ausführung des Eingabeskripts in die Ereignisschleife ein. Node.js verlässt die Ereignisschleife, wenn keine weiteren Callbacks ausgeführt werden müssen. Dieses Verhalten ähnelt dem Browser-JavaScript – die Ereignisschleife ist für den Benutzer verborgen.

## HTTP-Unterstützung

HTTP ist in Node.js ein vollwertiger Bestandteil und wurde mit Blick auf Streaming und geringe Latenz entwickelt. Dies macht Node.js besonders gut geeignet als Grundlage für eine Web-Bibliothek oder ein Framework.

## Unterstützung für Mehrkern-Prozessoren

Die Tatsache, dass Node.js ohne Threads entwickelt wurde, bedeutet nicht, dass Sie nicht von mehreren Kernen in Ihrer Umgebung profitieren können. Kindprozesse können mithilfe unserer `child_process.fork()`-API erzeugt werden und sind so konzipiert, dass die Kommunikation mit ihnen einfach ist.

Auf derselben Schnittstelle basiert das `cluster`-Modul, das es Ihnen ermöglicht, Sockets zwischen Prozessen zu teilen, um Load Balancing über Ihre Kerne zu ermöglichen.
