# Руководство пользователя GitLab

## Управление учётными записями

### Регистрация и вход

Чтобы использовать GitLab впервые, нажмите «Зарегистрироваться» на странице входа, чтобы создать учётную запись. При регистрации необходимо указать имя пользователя, адрес электронной почты и пароль. Выбор имени пользователя следует делать осторожно, так как его нельзя изменить после создания. После регистрации вы получите письмо для подтверждения. Нажмите на ссылку в письме, чтобы завершить проверку электронной почты.

После входа можно улучшить профиль в личных настройках, включая аватар, имя, полное имя и краткую информацию. Рекомендуется заполнить полную информацию, так как она будет отображаться в списках участников проекта и журналах коммитов. Также можно привязать SSH-ключи для отправки кода через командную строку.

### Настройка SSH-ключей

SSH-ключи позволяют безопасно аутентифицироваться в репозиториях GitLab без ввода имени пользователя и пароля каждый раз. Сначала сгенерируйте пару SSH-ключей:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Сгенерированный открытый ключ находится в файле ~/.ssh/id_ed25519.pub. Войдите в GitLab, перейдите в настройки пользователя, нажмите на вкладку «SSH-ключи», вставьте содержимое открытого ключа в текстовое поле и сохраните. После добавления ключа можно клонировать репозитории через SSH:

```bash
git clone git@gitlab.example.com:username/project.git
```

## Операции с проектами

### Создание проекта

После входа нажмите кнопку «Новый проект» в верхней навигационной панели, чтобы создать новый проект. Можно создать пустой проект, создать из шаблона или импортировать существующий проект. При создании необходимо указать название проекта, описание, уровень видимости (Публичный, Внутренний, Частный) и имя ветки по умолчанию.

Названия проектов должны быть краткими и понятными, использовать английский язык и дефисы для именования. Частные проекты доступны только участникам проекта. Внутренние проекты видны всем авторизованным пользователям в организации. Публичные проекты видны всем. После создания можно изменить эти параметры на странице настроек проекта.

### Управление ветками

GitLab использует main как основную ветку по умолчанию (в более ранних версиях использовался master). Нажмите «Ветки» на странице проекта, чтобы просмотреть и управлять всеми ветками. Создать новую ветку можно: нажав «Новая ветка» на странице веток или используя команды Git:

```bash
git checkout -b feature/new-feature
```

Защищённые ветки — важная функция для командной работы. Перейдите в Настройки проекта → Репозиторий → Защищённые ветки, чтобы указать, какие пользователи могут отправлять или сливать в определённые ветки. Рекомендуется защитить ветку main, разрешая сливать только определённым ролям или сопровождающим.

### Запросы на слияние

Запросы на слияние — это основная функция совместной работы над кодом в GitLab. После завершения разработки функции и отправки ветки нажмите «Запросы на слияние» → «Новый запрос на слияние» на странице проекта. Выберите исходную и целевую ветки, заполните название и описание, затем создайте.

На странице запроса на слияние показаны различия между исходной и целевой ветками, включая изменения файлов и историю коммитов. Участники команды могут здесь выполнять проверку кода, добавляя построчные комментарии. После завершения проверки, если условия слияния выполнены (например, пройдены тесты CI и получены утверждения), ветку можно слить.

## Конвейеры CI/CD

### Настройка .gitlab-ci.yml

GitLab CI/CD использует файл .gitlab-ci.yml в корне проекта для определения конвейеров. Это файл конфигурации YAML, описывающий каждый этап сборки и задачу. Вот простой пример:

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

Эта конфигурация определяет три этапа: build — сборка проекта, test — выполнение тестов, deploy — развёртывание приложения. Артефакты используются для передачи сборок между задачами, а only указывает выполнять задачи развёртывания только в ветке main.

### GitLab Runner

GitLab Runner — это агент, выполняющий задачи CI/CD. GitLab предоставляет общие раннеры, или можно зарегистрировать собственные частные раннеры. Установите GitLab Runner на своём сервере:

```bash
# Установка
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Регистрация раннера
sudo gitlab-runner register
```

Для регистрации требуется указать URL-адрес экземпляра GitLab и токен регистрации, которые можно найти в Настройки проекта → CI/CD → Раннеры. Раннеры можно настроить для параллельного выполнения задач, чтобы повысить скорость сборки.

### Мониторинг конвейеров

После каждой отправки кода конвейер запускается автоматически. Вы можете просмотреть все статусы выполнения конвейеров на странице Проект → CI/CD → Конвейеры. Нажмите на конкретный конвейер, чтобы увидеть статус выполнения каждой задачи, включая вывод журналов, время сборки и причины сбоя.

GitLab также предоставляет Редактор конвейеров, который визуально создаёт и редактирует конвейеры CI/CD, устраняя необходимость ручного написания файлов YAML. Редактор проверяет синтаксис и показывает предпросмотр в реальном времени, уменьшая ошибки конфигурации.

## Отслеживание проблем

### Создание и управление проблемами

Система проблем — основа управления проектами в GitLab. На странице проекта нажмите «Проблемы» → «Новая проблема», чтобы создать новую проблему. Необходимо заполнить название и описание, можно назначить конкретных участников, установить метки, вехи и сроки выполнения.

Статус проблем управляется через метки или вехи. Распространённые метки включают bug, feature, enhancement и другие. Можно использовать вид доски для визуализации статуса проблем, группируя их по столбцам (например: К выполнению, В работе, Готово).

### Связывание проблем с кодом

Использование ключевых слов и номеров проблем в сообщениях коммитов или описаниях запросов на слияние автоматически связывает код с проблемами. Например:

- Fixes #123 — Исправляет проблему 123, автоматически закроет её после слияния
- Closes #456 — Закрывает проблему 456
- See #789 — Ссылается на проблему 789

Этот механизм связи помогает командам отслеживать прогресс работы и понимать контекст и цель проблемы при проверке кода.

## Вики и документация

Каждый проект GitLab имеет встроенную функцию вики для создания проектной документации. Страницы вики пишутся в формате Markdown с поддержкой подсветки синтаксиса, таблиц и изображений. Содержимое вики поддерживает контроль версий, как и репозитории кода, позволяя просматривать исторические версии и откатываться.

Рекомендуется поддерживать следующую документацию для проектов: README (введение в проект), CONTRIBUTING (руководство по участию), руководства по развёртыванию и документацию API. Эта документация может значительно снизить сложность адаптации и способствовать участию сообщества.
