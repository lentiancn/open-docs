# Hadoop よくある質問

## HDFS関連の問題

### HDFSストレージ容量不足の対処方法

HDFSストレージ容量不足は一般的な問題で、複数の原因で発生する可能性があります。まずDataNodeのディスク使用状況を確認します：

```bash
hdfs dfsadmin -report
```

実際に容量不足の場合は、以下の対策を講じることができます：不要になった過去のデータを削除する、hdfs balancerを使用してデータ分布をバランスさせて一部のノードを解放する、新しいDataNodeノードを追加してクラスターストレージ容量を拡張する、レプリカ数を調整してストレージオーバーヘッドを削減する（データセキュリティとのトレードオフ要考虑）。

```bash
# ファイルを削除
hdfs dfs -rm -r /user/olddata

# データをバランス
hdfs balancer -threshold 10
```

### NameNodeメモリ不足の処理方法

NameNodeはすべてのメタデータをメモリに保存するため、ファイル数が多すぎるとメモリ不足が発生する可能性があります。以下の方法で緩和できます：hadoop-env.shでNameNodeメモリ設定 увеличить -XmxパラメータをHADOOP_NAMENODE_OPTSで設定する、NameNodeスナップショット機能を有効にして重要なデータの読み取り専用コピーを定期的に作成する、Hadoop 2.x以上にアップグレードする（メタデータ管理が最適化されている）。

```bash
# hadoop-env.shで設定
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### DataNode起動失敗の対処方法

DataNode起動失敗は複数の原因で発生する可能性があります。まずDataNodeログを確認します：

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

一般的な原因と解決策：ディスク容量不足—ディスクをクリーンするか新しいディスクを追加する、権限の問題—Hadoopユーザーがデータディレクトリへの読み書き権限を持っていることを確認する、クラスターID不一致—新しいDataNodeを追加する場合は、DataNodeでhdfs namenode -upgradeを実行するかストレージディレクトリを削除して再フォーマットする。

```bash
# DataNodeディレクトリ権限を確認
ls -la /usr/local/hadoop/data/

# DataNodeストレージディレクトリを削除して再起動
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### 誤って削除したファイルの回復方法

HDFSはゴミ箱機能を提供しており、ファイルを削除した後でも回復できます。まずゴミ箱が有効になっていることを確認します：

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

誤削除が発覚した場合は、ゴミ箱から回復できます：

```bash
# ファイルを回復
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

ゴミ箱が既に空になっているか、保持期間を過ぎている場合は、データを回復できません。したがって、日常操作では以下を推奨します：ゴミ箱機能を有効にする（デフォルトで有効）、重要なデータのバックアップを取る、削除前に慎重に確認する。

### HDFSブロック破損の処理方法

データブロックはディスク障害やネットワーク問題で破損する可能性があります。HDFSには内置のデータ整合性検出メカニズムがあり、破損したブロックが見つかった場合、他のレプリカからデータを自動的に複製して回復します。手動での確認と修復：

```bash
# ファイルシステム整合性を確認
hdfs fsck / -files -blocks -locations

# 破損したブロックファイルを削除
hdfs fsck / -delete
```

## MapReduce関連の問題

### ジョブの実行が遅い場合の対処方法

MapReduceジョブの実行が遅い原因是複数考えられます。まずジョブ履歴ログを確認し、各フェーズでの所要時間を分析します：

```bash
# ジョブ詳細を確認
yarn logs -applicationId application_xxx -logFiles *

# JobHistory Serverで確認
mapred job -history job_history_file
```

一般的な最適化方向：Reduceタスク数を 增加させて並列度を向上させる、JVMヒープサイズを調整してOOMを回避する、Combinerを有効にしてネットワーク転送量を削減する、カスタムパーティショナーを使用してデータスキューを最適化し負荷をバランスさせる、適切なファイルフォーマット（Parquet、ORCなど）を選択する。

### データスキューの処理方法

データスキューとは、Reduceフェーズでデータが不均一に分布し、一部のReduceタスクの実行時間が他のタスク的比で著しく長い状態を指します。解決策：カスタムパーティショナーを使用してビジネスロジックに基づいてデータを再配分する、ランダムプレフィックスを追加してデータを最初に複数のReduceに分散させ、その後二次的に集約する、Keyにソルト処理を行い、集約後にプレフィックスを削除する。

```java
// カスタムパーティショナーの例
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // ビジネスロジックに基づいて再配分
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### MapReduceジョブ失敗のトラブルシューティング

ジョブが失敗した場合は、まずエラーログを確認して失敗原因を特定します：

```bash
# ジョブ状態を確認
yarn application -status application_xxx

# タスク失敗理由を確認
mapred job -status job_id
```

一般的な失敗原因：Java OutOfMemory（OOM）—mapreduce.map.memory.mbとmapreduce.reduce.memory.mb увеличить、入力データフォーマットエラー—データが予期されるフォーマット是否符合するか確認、カスタムコード例外—コードロジックと例外メッセージを確認、ネットワークタイムアウト—mapreduce.task.timeoutパラメータ увеличить。

## YARN関連の問題

### ResourceManager起動失敗の対処方法

ResourceManager起動失敗は通常、設定エラーまたはポート衝突によって発生します。ログを確認します：

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

一般的な問題と解決策：ポート在使用中—yarn-site.xmlでResourceManagerポートを変更する、ZooKeeper設定エラー（HAを使用する場合）—zkfc関連設定を確認、メモリ不足—マシンに十分な物理メモリがあるか確認、設定ファイルエラー—yarn-site.xmlとcapacity-scheduler.xmlを注意深く確認。

```bash
# ポート使用状況を確認
netstat -tlnp | grep 8032
```

### クラスターリソース利用率低い場合の対処方法

クラスターリソース利用率が低い原因是以下が考えられます：ジョブ提交パターンが不合理—ジョブが小さすぎてスケジューリングオーバーヘッドが計算時間を超える、スケジューラー設定が不適切—キューリソース分配が不合理、データローカリティが悪い—入力データとコンピューティングノードが同じ位置にない。

最適化提案：小規模ジョブをバッチで提交する、キュー容量と優先度を合理的に構成する、データレイアウトを最適化してデータローカリティを向上させる、適切なスケジューラー（Capacity SchedulerまたはFair Scheduler）を使用する。

### ユーザーリソース使用を制限する方法

YARNキューとアクセス制御を使用してユーザーリソース使用を制限できます：

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- 単一ユーザーの最大キュー資源倍数 -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- キューで実行中の最大ジョブ数 -->
</property>
```

## クラスター運用問題

### Hadoopクラスターを監視する方法

Hadoopは複数の監視方法を提供しています：Web UI（NameNode 9870ポート、ResourceManager 8088ポート）、JMX（Java Management Extensions）監視、Ganglia、Prometheusなどの外部監視システム、コマンドラインツール（hdfs dfsadmin、yarn nodeなど）。

Prometheus + Grafanaを使用して完全な監視プラットフォームを構築することをお勧めします—HadoopのJMXメトリクスを収集して可視化します。

### クラスターアップグレードの実行方法

Hadoopクラスターアップグレードは慎重な操作が必要です。推奨される手順：テスト環境でまず新バージョンの互換性を確認する、NameNodeメタデータとHDFSデータをバックアップする、DataNodeノードを1つずつアップグレードし、最後にNameNodeをアップグレードする、rolling upgradeを使用して停止時間を削減する。

```bash
# DataNodeをローリングアップグレード
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Hadoopをアップグレード
tar -xzf hadoop-new.tar.gz -C /usr/local/
# DataNodeを起動
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### NameNode高可用性の設定方法

NameNode HAはNameNodeの単一障害点を回避できます。設定手順：hdfs-site.xmlで2つのNameNodeを構成する、共有編集ディレクトリ（通常はZooKeeperまたはNFSを使用）を構成する、ZooKeeper Failover Controller（ZKFC）を起動する。

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

設定完了後、以下のコマンドでZKFCを初期化します：

```bash
hdfs zkfc -formatZK
```

### ノード退役の処理方法

ノード退役はデータ損失がないように慎重に処理する必要があります。decommission機能を使用してノードを安全に退役させます：

```bash
# workersファイルのexcludeファイルに退役するノードを追加
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# ノードリストを更新
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

データ複製が完了するのを待ちます—ノードは自動的にdecommissioned状態になります。データが他のノードに複製されたことを確認してからノードをシャットダウンします。

## 開発関連の問題

### MapReduceプログラムをデバッグする方法

MapReduceプログラムのデバッグは困難です。推奨される方法：ローカルモードでまずロジックをテストする、Counterを使用して重要な指標を記録する、ログ出力を追加する—MapまたはReduceメソッドで重要な変数値を出力する、MRUnitを使用してユニットテストを行う。

```java
// Counterを使用
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### InputFormatとOutputFormatの選択方法

InputFormatは入力データをどのように読み取るかを決定します：TextInputFormat（デフォルト、行ごとに読み取り）、KeyValueTextInputFormat（タブで分割）、SequenceFileInputFormat（バイナリフォーマット）、DBInputFormat（データベースから読み取り）。

OutputFormatは結果の出力をどのように行うかを決定します：TextOutputFormat（デフォルト、テキストフォーマット）、SequenceFileOutputFormat（バイナリフォーマット）、NullOutputFormat（ファイル出力なし）、DBOutputFormat（データベースに出力）。

### 小ファイル問題の処理方法

大量の小さなファイルはNameNodeのメモリ圧力を増大させ、处理効率を低下させます。解決策：Hadoop Archives（HAR）を使用して小さなファイルをパッケージ化する、SequenceFileを使用して小さなファイルをマージする、HBaseを使用して大量の小さなファイルを保存する、ブロックサイズを調整して小さなファイルの数を減らす。
