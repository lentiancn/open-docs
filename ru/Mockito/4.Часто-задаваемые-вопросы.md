# Часто задаваемые вопросы о Mockito

## Базовые вопросы

### В чём разница между Mockito и PowerMock

Mockito в основном используется для мокирования обычных объектов, в то время как PowerMock может мокировать статические методы, конструкторы, приватные методы и многое другое. Однако PowerMock использует методы манипуляции байт-кодом, которые нарушают изоляцию тестов, делая тесты ближе к интеграционным тестам, а не к юнит-тестам. Официальная рекомендация — сначала предпочитать Mockito, и рассматривать PowerMock только тогда, когда Mockito действительно не может справиться с определёнными сценариями, такими как мокирование статических методов. Начиная с Mockito 2.x, мокирование статических методов также поддерживается через mockito-inline.

### Когда следует использовать Spy вместо Mock

Используйте Spy, когда вам нужнобольшинство реального поведения, но вы хотите замокировать только несколько методов. Spy сохраняет большую часть функциональности реального объекта, позволяя при этом заглушать конкретные методы. Mock полностью заменяет реальный объект, поэтому никакая логика реальных методов не выполняется. Используйте Mock, когда вам нужен полный контроль над поведением объекта. На практике Spy используется относительно редко, поскольку в большинстве случаев использование реальных тестовых объектов или полных моков удовлетворяет требованиям.

### Можно ли использовать Mockito для финальных классов и методов

Начиная с Mockito 2.1.0, вы можете настроить Mockito для мокирования финальных классов и методов. Вам нужно добавить зависимость mockito-inline вместо mockito-core. mockito-inline включает механизм динамического создания подклассов для обработки финальных классов и методов. Эта функция значительно улучшает применимость Mockito, делая почти любой Java-класс мокируемым.

### Почему методы макетных объектов возвращают null

По умолчанию все методы макетных объектов возвращают null или значения по умолчанию для своих типов (int возвращает 0, boolean возвращает false, типы коллекций возвращают пустые коллекции и так далее). Это поведение Answer по умолчанию в Mockito. Если вам нужны конкретные возвращаемые значения, вы должны явно установить их с помощью when().thenReturn().

## Вопросы использования

### Аннотация @Mock не работает, что делать

Аннотация @Mock требует инициализации фреймворка Mockito, чтобы она вступила в силу. Убедитесь, что вы используете один из вариантов: MockitoAnnotations.initMocks(this), @RunWith(MockitoJUnitRunner.class) или @ExtendWith(MockitoExtension.class) (для JUnit 5) в ваших тестах. При правильной инициализации поля с аннотацией @Mock будут автоматически создавать макетные объекты перед выполнением тестовых методов.

### Как мокировать void-методы

Для методов с возвращаемым типом void используйте doThrow(), doAnswer() или doNothing() для определения поведения.

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### Как проверить аргументы вызовов методов

Вы можете использовать matchеры аргументов для проверки или использовать ArgumentCaptor для захвата аргументов для детальных проверок.

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### Как мокировать итераторы

Мокирование итераторов требует внимания к координации методов hasNext() и next().

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### Как обрабатывать перегрузку методов

Когда класс имеет перегруженные методы, Mockito различает разные методы на основе типов и количества параметров. Убедитесь, что типы параметров в when() точно соответствуют сигнатуре метода, который вы хотите замокировать.

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // перегруженный метод
```

## Продвинутые вопросы

### Почему when().thenReturn() не работает

К наиболее распространённым причинам относятся: установка поведения на один и тот же метод несколько раз (действует только последнее); использование Spy и вызов реального метод; метод возвращает void (следует использовать doReturn().when()); неправильное использование matchера аргументов, вызывающее сбой сопоставления. Проверьте свой код, чтобы убедиться в правильности синтаксиса установки поведения.

### Как тестировать абстрактные классы

При тестировании абстрактных классов вы можете либо замокировать конкретные методы абстрактного класса с помощью Mockito, либо использовать Spy для сохранения некоторого реального поведения. Другой подход — создать конкретный тестовый подкласс для запуска тестов.

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### Как мокировать обобщённые методы

При мокировании обобщённых методов учитывайте ограничения, вызванные стиранием типов. Используйте any() или anyObject() для сопоставления обобщённых параметров.

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### Как проверить порядок вызовов

Используйте InOrder для проверки того, что вызовы методов выполняются в определённом порядке.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### Как частично мокировать объекты

Используйте Spy для частичного мокирования или используйте withSettings() для настройки поведения Mock.

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## Вопросы производительности

### Тесты работают медленно, что делать

Убедитесь, что вы не используете ненужные вызовы реальных методов. Проверьте, создаёте ли вы новые Mock-объекты в каждом тесте. Для проектов с большим использованием Mockito рассмотрите использование @RunWith(MockitoJUnitRunner.Silent.class) для сокращения вывода журнала.

### Как уменьшить использование памяти Mock-объектами

Сами по себе макетные объекты Mockito потребляют немного памяти. Однако если создание большого количества макетных объектов всё ещё влияет на производительность, рассмотрите использование аннотации @Mock с областью видимости @Scope для повторного использования макетных объектов или используйте статические фабричные методы для создания моков.

## Вопросы отладки

### Как просмотреть журналы вызовов макетных объектов

Вы можете использовать функцию логирования Mockito или добавить слушателей для отслеживания вызовов.

```java
Mockito.framework().addListener(new VerificationListener());
```

Или используйте вывод тестов Maven/Gradle для просмотра подробной информации о вызовах моков.

### Как отлаживать, когда тесты падают

Mockito предоставляет подробные сообщения об ошибках. Проверьте различия между ожидаемыми и фактическими вызовами, показанными при сбоях проверки. Используйте функцию Print Mockito Logs для понимания поведения макетных объектов. Упрощайте тестовые случаи шаг за шагом для обнаружения проблем.
