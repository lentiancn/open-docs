# Mockito 常見問題

## 基礎問題

### Mockito 和 PowerMock 有什麼區別

Mockito 主要用於普通物件的模擬，而 PowerMock 可以模擬靜態方法、建構子、私有方法等。然而，PowerMock 透過位元組碼操作技術破壞測試的隔離性，使得測試更接近整合測試而非單元測試。官方建議優先使用 Mockito，只有在確實需要模擬靜態方法等 Mockito 無法做到的場景時才考慮 PowerMock。從 Mockito 2.x 開始，透過 mockito-inline 也支援模擬靜態方法。

### 什麼時候應該使用 Spy 而不是 Mock

使用 Spy 的場景是當需要大部分真實行為，只對少數方法進行模擬時。Spy 會保留真實物件的大部分功能，只對明確指定的方法進行模擬。而 Mock 則完全替換了真實物件，所有方法呼叫都不會執行真實邏輯。如果需要完全控制物件行為，則使用 Mock。在實際開發中，Spy 使用場景相對較少，因為大多數情況下使用真實的測試物件或完全 Mock 就能滿足需求。

### Mockito 能否用於 final 類或方法

從 Mockito 2.1.0 版本開始，透過設定可以模擬 final 類和方法。需要在專案中新增 mockito-inline 依賴，而不是使用 mockito-core。mockito-inline 會啟用子類的動態代理機制，從而能夠處理 final 類和方法。這個功能大大增強了 Mockito 的適用範圍，使得幾乎所有 Java 類都可以被模擬。

### 為什麼 Mock 物件的方法回傳 null

預設情況下，Mock 物件的所有方法都會回傳 null 或對應類型的預設值（int 回傳 0，boolean 回傳 false，集合類型回傳空集合等）。這是 Mockito 的預設 Answer 行為。如果需要特定 的回傳值，需要使用 when().thenReturn() 進行明確的設定。

## 使用問題

### @Mock 注解不生效怎麼辦

@Mock 注解需要先初始化 Mockito 框架才能生效。確保在測試中使用了 MockitoAnnotations.initMocks(this)、@RunWith(MockitoJUnitRunner.class) 或 @ExtendWith(MockitoExtension.class)（JUnit 5）中的一種方式來初始化。如果初始化正確，@Mock 注解的欄位會在測試方法執行前自動建立模擬物件。

### 如何模擬 void 方法

對於回傳類型為 void 的方法，使用 doThrow()、doAnswer() 或 doNothing() 來設定行為。

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### 如何驗證方法的呼叫參數

可以使用參數匹配器進行驗證，或者使用 ArgumentCaptor 擷取參數進行詳細斷言。

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### 如何模擬迭代器

迭代器的模擬需要注意 hasNext() 和 next() 方法的配合。

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### 如何處理方法重載

如果類別中有方法重載，Mockito 會根據參數類型和數量來區分不同的方法。確保 when() 中的參數類型與要模擬的方法簽章完全匹配。

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any");  // 多載方法
```

## 高級問題

### 為什麼 when().thenReturn() 不工作

常見原因包括：對同一個方法多次設定行為（只有最後一次生效）；使用 Spy 時呼叫了真實方法；方法回傳類型為 void（應該使用 doReturn().when()）；參數匹配器使用不當導致匹配失敗。檢查程式碼確保行為設定的語法正確。

### 如何測試抽象類別

測試抽象類別時，可以使用 Mockito 模擬抽象類別的具體方法，或者使用 Spy 來保留部分真實行為。另一種方式是建立一個具體的測試子類別來執行測試。

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### 如何模擬泛型方法

模擬泛型方法時，需要注意類型擦除帶來的限制。使用 any() 或 anyObject() 來匹配泛型參數。

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### 如何驗證呼叫順序

使用 InOrder 來驗證方法呼叫按照特定順序執行。

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### 如何部分模擬物件

使用 Spy 進行部分模擬，或者使用 withSettings() 自定義 Mock 的行為。

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## 效能問題

### 測試執行很慢怎麼辦

確保沒有使用不必要的 real method calls。檢查是否在每個測試中都建立新的 Mock 物件。對於大量使用 Mockito 的專案，可以考慮使用 @RunWith(MockitoJUnitRunner.Silent.class) 來減少日誌輸出。

### 如何減少 Mock 物件的記憶體佔用

Mockito 的 Mock 物件本身不會佔用太多記憶體。但如果建立大量 Mock 物件仍然影響效能，可以考慮使用 @Mock 注解配合 @Scope 來復用 Mock 物件，或者使用靜態的 Mock 工廠方法。

## 除錯問題

### 如何查看 Mock 物件的呼叫日誌

可以使用 Mockito 的日誌功能或透過新增監聽器來追蹤呼叫。

```java
Mockito.framework().addListener(new VerificationListener());
```

或者使用 Maven/Gradle 的測試輸出查看詳細的 Mock 呼叫資訊。

### 測試失敗時如何除錯

Mockito 提供了詳細的錯誤訊息。檢查 verify 失敗時顯示的預期呼叫與實際呼叫的差異。使用 Print Mockito Logs 功能可以幫助了解 Mock 物件的具體行為。逐步簡化測試用例，定位問題所在。
