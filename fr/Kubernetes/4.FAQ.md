# Foire Aux Questions sur Kubernetes

## Concepts de Base

### Qu'est-ce que Kubernetes ?

Kubernetes (K8s) est un système open-source permettant d'automatiser le déploiement, la mise à l'échelle et la gestion des applications conteneurisées. Il regroupe les conteneurs en unités logiques pour faciliter la gestion et la découverte.

### Quelle est la différence entre Pod et Conteneur ?

Pod est la plus petite unité déployable dans Kubernetes. Un Pod peut contenir un ou plusieurs conteneurs. Il est recommandé de n'exécuter qu'un seul conteneur par Pod. Les conteneurs au sein du même Pod partagent les ressources réseau et stockage.

### Quelle est la différence entre Deployment et StatefulSet ?

Deployment est utilisé pour gérer les applications sans état, supportant les mises à jour déclaratives, la mise à l'échelle et les restaurations. StatefulSet est utilisé pour gérer les applications avec état, fournissant des identités réseau stables et un stockage persistant, adapté aux bases de données et autres applications nécessitant la persistance.

### Qu'est-ce qu'un Namespace ?

Namespace divise les ressources du cluster en plusieurs partitions logiques, permettant d'isoler les ressources entre différentes équipes ou projets. Les namespaces par défaut courants incluent default, kube-system et kube-public.

## Installation et Configuration

### Quelle est la différence entre Minikube et Kind ?

Minikube crée un cluster Kubernetes complet à nœud unique, adapté à l'apprentissage et au développement. Kind (Kubernetes in Docker) utilise des conteneurs Docker pour simuler des nœuds Kubernetes, plus léger, adapté aux tests rapides.

### Comment choisir un runtime de conteneurs ?

Les runtimes de conteneurs courants incluent Docker, containerd et CRI-O. containerd est le runtime de conteneurs central de Docker et est actuellement largement recommandé.

### Que faire si kubeadm init échoue ?

Raisons courantes :
- Ports déjà utilisés : vérifier les ports 6443, 10250, 10255
- Swap non désactivé : exécuter `swapoff -a`
- Règles de pare-feu : assurer la communication réseau entre les nœuds
- SELinux/AppArmor : peut être défini en mode permissif ou désactivé

### Comment configurer un cluster multi-plans de contrôle ?

Utilisation de kubeadm pour créer des clusters haute disponibilité :
1. Préparer plusieurs nœuds serveur
2. Utiliser kubeadm init pour créer le premier nœud du plan de contrôle
3. Générer des certificats et les copier sur d'autres nœuds
4. Utiliser kubeadm join pour ajouter des plans de contrôle supplémentaires

## Réseau et Stockage

### Que faire si les Pods ne peuvent pas communiquer entre eux ?

Raisons courantes :
- Plugin réseau pas installé correctement
- Pare-feu bloquant la communication CIDR réseau Pod
- Erreurs de configuration du plugin CNI
- Vérifier si kube-proxy s'exécute correctement

### Comment configurer un stockage persistant pour les Pods ?

Vous devez créer un PersistentVolume (PV) et un PersistentVolumeClaim (PVC). PV peut être un stockage local, un stockage réseau (NFS, iSCSI) ou un stockage cloud (AWS EBS, GCE PD, Azure Disk).

### Quels sont les types de Service ?

- ClusterIP : IP virtuelle accessible au sein du cluster
- NodePort : Ouvre un port sur chaque nœud
- LoadBalancer : Utilise l'équilibreur de charge du fournisseur cloud
- ExternalName : Mappe le service à un nom de domaine externe

## Gestion des Ressources

### Comment définir des limites de ressources pour les conteneurs ?

Ajouter le champ resources dans le spec du Pod ou du Deployment :
```yaml
resources:
  limits:
    memory: "512Mi"
    cpu: "500m"
  requests:
    memory: "256Mi"
    cpu: "250m"
```

### Qu'est-ce que ResourceQuota ?

ResourceQuota est utilisé pour limiter le total des ressources dans un namespace, empêchant une seule équipe ou projet de consommer trop de ressources.

### Qu'est-ce que LimitRange ?

LimitRange est utilisé pour définir les limites de ressources par défaut et les valeurs de demande pour les Pods et conteneurs dans un namespace.

## Déploiement et Opérations

### Comment réaliser des mises à jour continues des applications ?

Lors de l'utilisation de Deployment, Kubernetes effectue automatiquement des mises à jour continues :
```bash
kubectl set image deployment/my-app my-app=my-app:v2
kubectl rollout status deployment/my-app
```

### Comment restaurer un déploiement ?

```bash
# Restaurer à la version précédente
kubectl rollout undo deployment/my-app

# Restaurer à une révision spécifique
kubectl rollout undo deployment/my-app --to-revision=2
```

### Qu'est-ce que HPA ?

Horizontal Pod Autoscaler (HPA) ajuste automatiquement le nombre de répliques de Pod en fonction de métriques comme le CPU et la mémoire. Nécessite Metrics Server.

### Comment configurer les vérifications de santé ?

Configurer les sondes de vivantsité et de préparation :
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

## Sécurité

### Qu'est-ce que RBAC ?

Le contrôle d'accès basé sur les rôles (RBAC) est utilisé pour gérer les permissions dans les clusters Kubernetes. Vous pouvez créer Role et ClusterRole, et les lier à des utilisateurs ou groupes.

### Comment gérer les informations sensibles ?

Utilisez Secret pour stocker des informations sensibles comme les mots de passe, clés et certificats. Peut être créé via la commande kubectl ou défini via des fichiers YAML.

### Comment protéger votre cluster ?

- Activer l'authentification RBAC
- Utiliser les politiques de réseau pour restreindre la communication entre Pods
- Mettre à jour régulièrement les versions Kubernetes
- Chiffrer les données etcd
- Utiliser TLS pour la communication
- Restreindre les permissions d'accès kubelet

## Dépannage

### Le Pod reste à l'état Pending ?

Causes possibles :
- Ressources insuffisantes (CPU ou mémoire)
- Sélecteur de nœud ou taint non correspondants
- PVC non lié correctement
- Utiliser `kubectl describe pod <pod-name>` pour les détails

### Le Pod reste à l'état ImagePullBackOff ?

Raisons courantes :
- Nom d'image incorrect
- Tag d'image inexistant
- Problèmes d'authentification de registre privé
- Problèmes réseau empêchant le téléchargement de l'image

### Le Pod reste à l'état CrashLoopBackOff ?

- Vérifier les logs du conteneur : kubectl logs --previous <pod-name>
- Vérifier que l'application démarre correctement
- Vérifier que la configuration des ressources est correcte
- Vérifier que les services dépendants sont disponibles

### Comment déboguer les problèmes réseau ?

- Vérifier que kube-proxy s'exécute correctement
- Vérifier l'état du plugin CNI
- Utiliser kubectl exec pour entrer dans le conteneur et tester la connectivité réseau
- Voir les règles iptables
- Utiliser les outils de diagnostic réseau (curl, ping, traceroute)

## Optimisation des Performances

### Comment améliorer les performances du cluster ?

- Configurer les demandes et limites de ressources appropriées
- Utiliser l'anti-affinité pour distribuer les Pods
- Activer le cache API pour réduire les requêtes
- Utiliser le stockage local pour réduire la latence réseau
- Optimiser les tailles d'image pour réduire le temps de téléchargement

### Comment gérer les clusters à grande échelle ?

- Utiliser la Fédération pour gérer plusieurs clusters
- Configurer le matériel approprié pour etcd
- Activer le cache API
- Utiliser des équilibreurs de charge pour distribuer les requêtes
