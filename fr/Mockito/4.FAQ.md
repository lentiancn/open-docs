# Foire aux questions sur Mockito

## Questions de base

### Quelle est la difference entre Mockito et PowerMock

Mockito est principalement utilise pour simuler des objets reguliers, tandis que PowerMock peut simuler des methodes statiques, des constructeurs, des methodes privees, et plus encore. Cependant, PowerMock utilise des techniques de manipulation de bytecode qui brisent l'isolation des tests, rendant les tests plus proches de tests d'integration que de tests unitaires. La recommandation officielle est de preferer Mockito d'abord, et de considerer PowerMock seulement lorsque Mockito ne peut vraiment pas gerer certains scenarii comme la simulation de methodes statiques. Depuis Mockito 2.x, la simulation de methodes statiques est egalement prise en charge via mockito-inline.

### Quand dois-je utiliser Spy au lieu de Mock

Utilisez Spy lorsque vous avez besoin de la majeure partie du comportement reel mais souhaitez ne simuler que quelques methodes. Spy conserve la majeure partie de la fonctionnalite de l'objet reel tout en permettant a des methodes specifiques d'etre stubbees. Mock remplace completement l'objet reel, donc aucune logique de methode reelle n'est executee. Utilisez Mock lorsque vous avez besoin d'un controle total sur le comportement de l'objet. Dans la pratique, Spy est relativement peu utilise car dans la plupart des cas, l'utilisation d'objets de test reels ou de mocks complets repond aux exigences.

### Mockito peut-il etre utilise pour les classes et methodes finales

Depuis Mockito 2.1.0, vous pouvez configurer Mockito pour simuler des classes et methodes finales. Vous devez ajouter la dependance mockito-inline au lieu de mockito-core dans votre projet. mockito-inline active le mecanisme de sous-classes dynamiques pour gerer les classes et methodes finales. Cette fonction amelior considerablement l'applicabilite de Mockito, rendant presque toute classe Java simulable.

### Pourquoi les methodes des objets mock retournent null

Par defaut, toutes les methodes des objets mock retournent null ou des valeurs par defaut pour leurs types (int retourne 0, boolean retourne false, les types de collection retournent des collections vides, etc.). C'est le comportement de Answer par defaut de Mockito. Si vous avez besoin de valeurs de retour specifiques, vous devez les definir explicitement en utilisant when().thenReturn().

## Questions d'utilisation

### L'annotation @Mock ne fonctionne pas, que dois-je faire

L'annotation @Mock necessite l'initialisation du framework Mockito pour prendre effet. Assurez-vous d'utiliser l'un de MockitoAnnotations.initMocks(this), @RunWith(MockitoJUnitRunner.class), ou @ExtendWith(MockitoExtension.class) (pour JUnit 5) dans vos tests. Si l'initialisation est correcte, les champs annotes avec @Mock creeront automatiquement des objets mock avant que les methodes de test ne soient executees.

### Comment simuler des methodes void

Pour les methodes avec un type de retour void, utilisez doThrow(), doAnswer(), ou doNothing() pour definir le comportement.

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### Comment verifier les arguments d'appel de methode

Vous pouvez utiliser des matchers d'arguments pour la verification, ou utiliser ArgumentCaptor pour capturer les arguments pour des assertions detaillees.

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### Comment simuler des iterateurs

La simulation d'iterateurs necessite une attention a la coordination entre les methodes hasNext() et next().

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### Comment gerer la surcharge de methodes

Lorsqu'une classe a des methodes surchargees, Mockito distingue les differentes methodes en fonction des types et nombres de parametres. Assurez-vous que les types de parametres dans when() correspondent exactement a la signature de la methode que vous souhaitez simuler.

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // methode surchargee
```

## Questions avancees

### Pourquoi when().thenReturn() ne fonctionne pas

Les raisons courantes incluent: definir le comportement sur la meme methode plusieurs fois (seul le dernier prend effet); utiliser Spy et appeler la methode reelle; la methode retourne void (devrait utiliser doReturn().when()); le matcher d'arguments utilise incorrectement causant un echec de correspondance. Verifiez votre code pour vous assurer que la syntaxe pour definir le comportement est correcte.

### Comment tester des classes abstraites

Lors du test de classes abstraites, vous pouvez utiliser Mockito pour simuler des methodes concretes de la classe abstraite, ou utiliser Spy pour conserver une partie du comportement reel. Une autre approche consiste a creer une sous-classe de test concrete pour executer des tests.

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### Comment simuler des methodes generiques

Lors de la simulation de methodes generiques, soyez conscient des limitations causees par l'effacement de type. Utilisez any() ou anyObject() pour faire correspondre les parametres generiques.

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### Comment verifier l'ordre des appels

Utilisez InOrder pour verifier que les appels de methodes ont ete executes dans un ordre specifique.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### Comment simuler des objets partiellement

Utilisez Spy pour une simulation partielle, ou utilisez withSettings() pour personnalizer le comportement de Mock.

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## Questions de performance

### Les tests s'executent lentement, que dois-je faire

Assurez-vous de ne pas utiliser d'appels de methodes reelles inutiles. Verifiez si vous creez de nouveaux objets Mock dans chaque test. Pour les projets utilisant beaucoup Mockito, envisagez d'utiliser @RunWith(MockitoJUnitRunner.Silent.class) pour reduire la sortie des logs.

### Comment reduire l'utilisation de la memoire des objets mock

Les objets mock de Mockito eux-memes ne consomment pas beaucoup de memoire. Cependant, si la creation d'un grand nombre d'objets mock affecte toujours les performances, envisagez d'utiliser l'annotation @Mock avec @Scope pour reutiliser les objets mock, ou utilisez des methodes d'usine de mock statiques.

## Questions de debogage

### Comment voir les journaux d'appels des objets mock

Vous pouvez utiliser la fonction de journalisation de Mockito ou ajouter des ecouteurs pour suivre les appels.

```java
Mockito.framework().addListener(new VerificationListener());
```

Ou utilisez la sortie de test Maven/Gradle pour voir les informations detaillees des appels mock.

### Comment deboguer cuando les tests echouent

Mockito fournit des messages d'erreur detailles. Verifiez les differences entre les appels attendus et reels affiches dans les echecs de verify. Utilisez la fonction Imprimer les logs Mockito pour comprendre le comportement de l'objet mock. Simplifiez les cas de test pas a pas pour localiser les problemes.
