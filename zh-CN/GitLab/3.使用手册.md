# GitLab 使用手册

## 账户管理

### 注册与登录

首次使用 GitLab，需要在登录页面点击“注册”创建账户。注册时需要提供用户名、电子邮箱和密码。用户名的选择需谨慎，创建后无法更改。完成注册后，会收到一封验证邮件，点击邮件中的链接完成邮箱验证。

登录后可以在个人设置中完善个人资料，包括头像、姓名、全名和-bio 等信息。建议填写完整，因为这些信息会在项目成员列表和提交记录中显示。还可以绑定 SSH 密钥，用于通过命令行推送代码。

### SSH 密钥配置

SSH 密钥允许安全地与 GitLab 仓库进行身份验证，无需每次输入用户名和密码。首先需要生成 SSH 密钥对：

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

生成的公钥位于 ~/.ssh/id_ed25519.pub 文件中。登录 GitLab，进入用户设置页面，点击“SSH 密钥”选项卡，将公钥内容粘贴到文本框中并保存。添加密钥后，可以使用 SSH 方式克隆仓库：

```bash
git clone git@gitlab.example.com:username/project.git
```

## 项目操作

### 创建项目

登录后，点击顶部导航栏的“新建项目”按钮创建新项目。可以选择创建空白项目、从模板创建或导入已有项目。创建时需要设置项目名称、描述、可见性级别（公开、内部、私有）以及默认分支名称。

项目名称应简洁明了，使用英文和短横线命名。私有项目仅项目成员可以访问，内部项目对企业内所有登录用户可见，公开项目对所有人可见。创建后可以进入项目设置页面修改这些选项。

### 分支管理

GitLab 默认使用 main 作为主分支（早期版本为 master）。在项目页面点击“分支”可以查看和管理所有分支。创建新分支可以通过以下方式：在分支页面点击“新建分支”，或者使用 Git 命令：

```bash
git checkout -b feature/new-feature
```

保护分支是团队协作的重要功能。进入项目设置 → 仓库 → 分支保护，可以设置哪些用户可以推送或合并到特定分支。推荐保护 main 分支，只允许特定角色或维护者合并代码。

### Merge Request

Merge Request（合并请求）是 GitLab 代码协作的核心功能。完成功能开发并推送分支后，在项目页面点击“合并请求”→“新建合并请求”，选择源分支和目标分支，填写标题和描述后创建。

合并请求页面显示了源分支与目标分支的差异，包括文件变更和提交历史。团队成员可以在这里进行代码审查，逐行添加评论。审查完成后，如果满足合并条件（如通过 CI 测试、获得审批），则可以合并分支。

## CI/CD 流水线

### .gitlab-ci.yml 配置

GitLab CI/CD 使用项目根目录下的 .gitlab-ci.yml 文件定义流水线。这是一个 YAML 格式的配置文件，描述了各个构建阶段和作业。以下是一个简单的示例：

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

这个配置定义了三个阶段：build 构建项目，test 运行测试，deploy 部署应用。artifacts 用于在作业之间传递构建产物，only 指定只在 main 分支上运行部署作业。

### GitLab Runner

GitLab Runner 是执行 CI/CD 作业的代理程序。GitLab 官方提供共享 Runner，也可以自行注册专有 Runner。在服务器上安装 GitLab Runner：

```bash
# 安装
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# 注册 Runner
sudo gitlab-runner register
```

注册时需要提供 GitLab 实例 URL 和注册令牌，这些信息可以在项目设置 → CI/CD → Runners 中找到。Runner 可以配置为并行执行作业，提高构建速度。

### 流水线监控

每次推送代码后，流水线会自动触发。在项目的 CI/CD → 流水线页面可以查看所有流水线运行状态。点击具体的流水线可以看到各个作业的执行情况，包括日志输出、构建时间和失败原因。

GitLab 还提供流水线编辑器，采用可视化方式创建和编辑 CI/CD 配置，无需手动编写 YAML 文件。编辑器提供语法检查和实时预览功能，降低了配置错误的概率。

## 问题跟踪

### 创建和管理问题

问题系统是 GitLab 项目管理的基础。在项目页面点击“问题”→“新建问题”创建新问题。需要填写标题和描述，可以选择指派给特定成员、设置标签、里程碑和截止日期。

问题状态通过标签或里程碑来管理，常用标签包括 bug、feature、enhancement 等。可以通过看板视图可视化问题状态，将问题按列分组（如待办、处理中、已完成）。

### 问题与代码关联

在提交信息或合并请求描述中使用关键词加问题编号，可以自动关联代码与问题。例如：

- Fixes #123 - 修复问题 123，合并后会自动关闭问题
- Closes #456 - 关闭问题 456
- See #789 - 引用问题 789

这种关联机制帮助团队追踪工作进度，在代码审查时了解问题的背景和目的。

## 维基与文档

每个 GitLab 项目都有内置的维基功能，用于创建项目文档。维基页面使用 Markdown 格式编写，支持代码高亮、表格和图片。维基内容与代码仓库一样支持版本控制，可以查看历史版本和回滚。

建议为项目维护以下文档：README 项目简介、CONTRIBUTING 贡献指南、部署指南和 API 文档。这些文档可以显著降低项目上手难度，促进社区贡献。
