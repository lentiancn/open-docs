# Manual del Usuario

Este manual describe en detalle las funciones principales y cómo usar Apache APISIX, incluyendo configuración de rutas, gestión de upstream, balance de carga, limitación de velocidad, autenticación y otras funciones clave.

## Conceptos Básicos

### Route (Ruta)

Una ruta es el camino para acceder a un destino upstream. En APISIX, una ruta primero hace coincidir las solicitudes de los clientes con reglas predefinidas, luego carga y ejecuta los plugins correspondientes, y finalmente reenvía la solicitud a un servicio upstream específico.

La ruta más simple solo necesita la ruta de coincidencia y la dirección del upstream.

### Upstream (Servidor de Origen)

Un upstream es un conjunto de nodos que proporcionan la misma funcionalidad, es una abstracción del host virtual. El upstream puede realizar balance de carga entre múltiples nodos de servicio utilizando reglas preconfiguradas.

### Consumer (Consumidor)

Un consumidor es la aplicación o desarrollador que usa la API. En APISIX, un consumidor necesita un nombre globalmente único y debe vincularse a un plugin de autenticación.

### Plugin

APISIX proporciona un extenso sistema de plugins para ampliar las funcionalidades del gateway. Los plugins pueden configurarse a nivel de ruta, consumidor o de forma global.

## Configurar Rutas

Crear rutas es fácil a través de la API de administración. El siguiente ejemplo crea una ruta que reenvía las solicitudes a httpbin.org:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

Si se crea correctamente, devolverá `HTTP/1.1 201 Created`.

### Verificar la Ruta

```bash
curl "http://127.0.0.1:9080/ip"
```

Debería devolver algo similar a:

```json
{
  "origin": "183.94.122.205"
}
```

### Reglas de Coincidencia de Rutas

APISIX soporta múltiples formas de coincidir rutas:

**Coincidencia por URI:**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**Coincidencia por Host:**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**Coincidencia por Método de Solicitud:**

```json
{
  "methods": ["GET", "POST"]
}
```

**Coincidencia por Encabezados:**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## Balance de Carga

APISIX soporta múltiples algoritmos de balance de carga, incluyendo roundrobin (轮询), weighted roundrobin (加权轮询), least connections (最小连接数), entre otros.

### Configurar Balance de Carga Ponderado

Crear una ruta con múltiples servicios upstream:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**Parámetros:**

- `type`: Tipo de algoritmo de balance de carga. Soporta `roundrobin`, `weighted_roundrobin`, `least_conn`
- `nodes`: Nodos del servicio upstream, en formato `{"host:puerto": peso}`
- `pass_host`: Cuando se establece en `node`, los encabezados de solicitud se pasan al upstream
- `scheme`: Protocolo del upstream, soporta `http` y `https`

### Verificar el Balance de Carga

Genera 100 solicitudes para probar el equilibrio:

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

Las solicitudes se distribuirán casi uniformemente entre los dos servicios upstream.

## Limitación de Velocidad

APISIX ofrece funciones flexibles de limitación de velocidad a través del plugin `limit-count`.

### Activar la Limitación de Velocidad

Añadir el plugin de limitación a una ruta existente:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**Parámetros:**

- `count`: El número máximo de solicitudes permitidas dentro de la ventana de tiempo
- `time_window`: Tamaño de la ventana de tiempo en segundos
- `rejected_code`: Código de estado HTTP a devolver cuando se supere el límite

La configuración anterior limita el tráfico entrante a un máximo de 2 solicitudes cada 10 segundos.

### Verificar la Limitación de Velocidad

Genera 100 solicitudes para probar el efecto:

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

Resultado: 2 solicitudes exitosas (200), 98 solicitudes rechazadas (503).

### Desactivar la Limitación de Velocidad

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### Configuración Avanzada

**Usar variables:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**Usar Redis Cluster:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## Autenticación

APISIX soporta múltiples métodos de autenticación, incluyendo Key Auth, Basic Auth, JWT, OAuth2, entre otros.

### Autenticación con Clave (Key Auth)

#### Crear un Consumidor

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "clave-secreta"
    }
  }
}'
```

> ⚠️ En producción, usa claves más complejas.

#### Activar la Autenticación con Clave en la Ruta

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### Probar la Autenticación

**Sin clave:**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

Devuelve: `HTTP/1.1 401 Unauthorized`

**Con clave incorrecta:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: clave-incorrecta'
```

Devuelve: `HTTP/1.1 401 Unauthorized`

**Con clave correcta:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: clave-secreta'
```

Devuelve: `HTTP/1.1 200 OK`

### Otros Métodos de Autenticación

**Autenticación JWT:**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "clave-usuario",
      "secret": "mi-clave-secreta"
    }
  }
}
```

**Autenticación Básica:**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**Autenticación HMAC:**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "mi-clave-secreta"
    }
  }
}
```

## Gobierno de Servicios

### Despliegue Progresivo (Canary)

APISIX soporta despliegues progresivos basados en peso:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### Reescritura de Solicitudes

Usar el plugin `proxy-rewrite` para reescribir solicitudes:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### Caché de Respuestas

Usar el plugin `proxy-cache` para almacenar en caché las respuestas del upstream:

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## Configuración de Plugins

### Orden de Ejecución de Plugins

Puedes especificar la fase de ejecución del plugin en la configuración de `plugins`:

- `rewrite`: Fase de reescritura de solicitud
- `access`: Fase de control de acceso
- `before_proxy`: Fase previa al proxy
- `header_filter`: Fase de filtrado de encabezados de respuesta
- `body_filter`: Fase de filtrado del cuerpo de respuesta
- `log`: Fase de registro de logs

### Desactivar un Plugin

Establecer `_meta.disable` del plugin en `true`:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/id-ruta" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## Monitoreo y Observabilidad

### Integrar Prometheus

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

Acceder a las métricas:

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### Registro de Logs

APISIX soporta múltiples salidas de logs:

**Logs HTTP:**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://servidor-log:8080/logs"
    }
  }
}
```

**Logs de Errores:**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://servidor-log:8080/logs"
    }
  }
}
```

## Mejores Prácticas

1. **Usa IDs significativos**: Utiliza IDs descriptivos para rutas, upstream y otros recursos
2. **Configura tiempos de espera adecuados**: Ajusta los parámetros de tiempo de espera según el tiempo de respuesta del upstream
3. **Activa verificaciones de salud**: Configura verificaciones de salud activas y pasivas para servicios críticos
4. **Configura limitación de velocidad apropiada**: Establece reglas de limitación según las necesidades del negocio y la capacidad del servidor
5. **Activa la autenticación**: En producción, la autenticación es obligatoria
6. **Usa TLS**: Las interfaces expuestas públicamente deben usar HTTPS
7. **Monitorea métricas clave**: Configura Prometheus y establece alertas
