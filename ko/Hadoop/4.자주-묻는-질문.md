# Hadoop 자주 묻는 질문

## HDFS 관련 문제

### HDFS 스토리지 공간이 부족할 때怎么做

HDFS 스토리지 공간 부족은 여러 원인으로 발생할 수 있는 일반적인 문제입니다. 먼저 DataNode 디스크 사용량을 확인하세요:

```bash
hdfs dfsadmin -report
```

공간이 실제로 부족하면 다음 조치를 취할 수 있습니다: 더 이상 필요 없는 과거 데이터를 삭제; hdfs balancer를 사용하여 데이터 분포를 균형 조정하고 일부 노드 확보; 새 DataNode 노드를 추가하여 클러스터 스토리지 용량 확장; 복제 계수를 조정하여 스토리지 오버헤드 줄이기 (데이터 보안과의 균형 고려).

```bash
# 파일 삭제
hdfs dfs -rm -r /user/olddata

# 데이터 균형 조정
hdfs balancer -threshold 10
```

### NameNode 메모리 부족 처리 방법

NameNode는 모든 메타데이터를 메모리에 저장합니다. 파일이 너무 많으면 메모리가 부족해질 수 있습니다. 다음으로 완화: hadoop-env.sh에서 NameNode 메모리 구성 증가 (-Xmx 매개변수 설정); NameNode 스냅샷 기능을 활성화하여 중요한 데이터의 읽기 전용 복사본을 정기적으로 생성; 메타데이터 관리가 더 최적화된 Hadoop 2.x 이상으로 업그레이드.

```bash
# hadoop-env.sh에서 설정
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### DataNode가 시작되지 않을 때怎么做

DataNode 시작 실패는 여러 원인으로 발생할 수 있습니다. 먼저 DataNode 로그를 확인하세요:

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

일반적인 원인 및 해결책: 디스크 공간 부족 - 디스크 정리 또는 새 디스크 추가; 권한 문제 - Hadoop 사용자가 데이터 디렉토리에 읽기/쓰기 액세스 권한이 있는지 확인; 클러스터 ID 불일치 - 새 DataNode를 추가하는 경우 DataNode에서 hdfs namenode -upgrade 실행하거나 스토리지 디렉토리 삭제 후 재포맷.

```bash
# DataNode 디렉토리 권한 확인
ls -la /usr/local/hadoop/data/

# DataNode 스토리지 디렉토리 삭제 및 재시작
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### 실수로 삭제한 파일 복구 방법

HDFS는 삭제한 파일을 복구할 수 있는 휴지통 기능을 제공합니다. 먼저 휴지통이 활성화되어 있는지 확인하세요:

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

실수로 삭제한 것을 발견하면 휴지통에서 복구하세요:

```bash
# 파일 복구
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

휴지통이 비워졌거나 보존 기간이 지났으면 데이터를 복구할 수 없습니다. 따라서 일상적인 작업에서: 휴지통 기능 활성화 유지 (기본적으로 활성화); 중요한 데이터 백업; 삭제 전 주의 깊게 확인.

### 손상된 HDFS 블록 처리 방법

데이터 블록은 디스크 오류 또는 네트워크 문제로 인해 손상될 수 있습니다. HDFS에는 데이터 무결성 감지 메커니즘이 내장되어 있습니다: 손상된 블록이 발견되면 복구하기 위해 다른 복제본에서 자동으로 데이터를 복제합니다. 수동 확인 및 복구:

```bash
# 파일 시스템 무결성 확인
hdfs fsck / -files -blocks -locations

# 손상된 블록 파일 삭제
hdfs fsck / -delete
```

## MapReduce 관련 문제

### 작업이 느리게 실행될 때怎么做

느린 MapReduce 작업은 여러 요인으로 발생할 수 있습니다. 먼저 작업 기록 로그를 확인하고 각 단계에서 소요된 시간을 분석하세요:

```bash
# 작업 세부 정보 보기
yarn logs -applicationId application_xxx -logFiles *

# JobHistory Server에서 보기
mapred job -history job_history_file
```

일반적인 최적화 방향: 병렬도를 높이려면 Reduce 작업 수 증가; JVM 힙 크기 조정하여 OOM 피하기; Combiner 활성화하여 네트워크 전송 볼륨 줄이기; 사용자 정의 파티셔너를 사용하여 데이터 기울기 최적화 및 부하 균형 조정; 적절한 파일 형식 선택 (Parquet, ORC 등).

### 데이터 기울기 처리 방법

데이터 기울기는 Reduce 단계에서 데이터가 고르게 분산되지 않아 일부 Reduce 작업이 다른 작업보다 훨씬 오래 실행되는 것을 의미합니다. 해결책: 사용자 정의 파티셔너를 사용하여 비즈니스 로직에 따라 데이터 재분배; 무작위 접두사를 추가하여 데이터를 먼저 여러 Reduce에 분산시킨 다음 2차 집계; 키 소팅 사용 - 집계 후 접두사 제거.

```java
// 사용자 정의 파티셔너 예제
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // 비즈니스 로직에 따라 재분배
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### MapReduce 작업 실패 문제 해결

작업이 실패하면 먼저 오류 로그를 확인하여 실패 원인을 파악하세요:

```bash
# 작업 상태 보기
yarn application -status application_xxx

# 작업 실패 이유 보기
mapred job -status job_id
```

일반적인 실패 원인: Java OutOfMemory (OOM) - mapreduce.map.memory.mb와 mapreduce.reduce.memory.mb 증가; 입력 데이터 형식 오류 - 데이터가 예상 형식과 일치하는지 확인; 사용자 정의 코드 예외 - 코드 로직 및 예외 메시지 확인; 네트워크 타임아웃 - mapreduce.task.timeout 매개변수 증가.

## YARN 관련 문제

### ResourceManager가 시작되지 않을 때怎么做

ResourceManager 시작 실패는 일반적으로 구성 오류 또는 포트 충돌로 인해 발생합니다. 로그를 확인하세요:

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

일반적인 문제 및 해결책: 포트 사용 중 - yarn-site.xml에서 ResourceManager 포트 변경; ZooKeeper 구성 오류 (HA 사용 시) - zkfc 관련 구성 확인; 메모리 부족 - 시스템에 충분한 물리적 메모리가 있는지 확인; 구성 파일 오류 - yarn-site.xml과 capacity-scheduler.xml을 주의 깊게 확인.

```bash
# 포트 사용 확인
netstat -tlnp | grep 8032
```

### 클러스터 리소스 활용도가 낮을 때怎么做

클러스터 리소스 활용도가 낮을 수 있는 원인: 작업 제출 패턴이 합리적이지 않음 - 작업이 너무 작아 스케줄링 오버헤드가 계산 시간을 초과함; 스케줄러 구성 불량 - 대기열 리소스 할당 합리적이지 않음; 데이터 지역성 불량 - 입력 데이터와 컴퓨팅 노드가 같은 위치에 있지 않음.

최적화 제안: 작은 작업을 배치로 제출; 대기열 용량 및 우선순위 합리적으로 구성; 데이터 레이아웃 최적화하여 데이터 지역성 향상; 적절한 스케줄러 사용 (Capacity Scheduler 또는 Fair Scheduler).

### 사용자 리소스 사용 제한 방법

YARN 대기열 및 액세스 제어를 사용하여 사용자 리소스 사용을 제한할 수 있습니다:

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- 단일 사용자의 최대 대기열 리소스 배수 -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- 대기열의 최대 실행 작업 수 -->
</property>
```

## 클러스터 운영 문제

### Hadoop 클러스터 모니터링 방법

Hadoop은 여러 모니터링 방법을 제공합니다: Web UI (NameNode 포트 9870, ResourceManager 포트 8088); JMX (Java Management Extensions) 모니터링; Ganglia, Prometheus 등의 외부 모니터링 시스템; 명령줄 도구 (hdfs dfsadmin, yarn node 등).

Prometheus + Grafana를 사용하여 전체 모니터링 플랫폼을 구축하는 것이 좋습니다 - Hadoop의 JMX 메트릭을 수집하고 시각화합니다.

### 클러스터 업그레이드 수행 방법

Hadoop 클러스터 업그레이드는 신중하게 작업해야 합니다. 권장 단계: 테스트 환경에서 먼저 새 버전 호환성 확인; NameNode 메타데이터 및 HDFS 데이터 백업; DataNode 노드를 하나씩 업그레이드한 다음 NameNode 업그레이드; 가동 중지 시간을 줄이기 위해 롤링 업그레이드 사용.

```bash
# DataNode 롤링 업그레이드
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Hadoop 업그레이드
tar -xzf hadoop-new.tar.gz -C /usr/local/
# DataNode 시작
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### NameNode 고가용성 구성 방법

NameNode HA는 NameNode 단일 장애 지점을 방지할 수 있습니다. 구성 단계: hdfs-site.xml에서 두 개의 NameNode 구성; 공유 편집 디렉토리 구성 (일반적으로 ZooKeeper 또는 NFS 사용); ZooKeeper Failover Controller (ZKFC) 시작.

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

구성 후 다음 명령으로 ZKFC를 초기화하세요:

```bash
hdfs zkfc -formatZK
```

### 노드 폐기 처리 방법

노드 폐기는 데이터 손실이 없도록 신중하게 처리해야 합니다. decommission 기능을 사용하여 노드를 안전하게 폐기하세요:

```bash
# workers 파일의 제외 파일에 노드 추가
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# 노드 목록 새로 고침
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

데이터 복제가 완료될 때까지 기다리세요 - 노드는 자동으로 decommissioned 상태가 됩니다. 노드를 종료하기 전에 데이터가 다른 노드에 복제되었는지 확인하세요.

## 개발 관련 문제

### MapReduce 프로그램 디버깅 방법

MapReduce 프로그램 디버깅은 어렵습니다. 권장 방법: 로컬 모드에서 먼저 로직 테스트; 카운터를 사용하여 중요한 메트릭 기록; 로그 출력 추가 - Map 또는 Reduce 메서드에서 중요한 변수 값 인쇄; MRUnit을 사용하여 단위 테스트 수행.

```java
// 카운터 사용
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### InputFormat 및 OutputFormat 선택 방법

InputFormat은 입력 데이터를 읽는 방법을 결정합니다: TextInputFormat (기본값, 줄별 읽기); KeyValueTextInputFormat (탭으로 분할); SequenceFileInputFormat (바이너리 형식); DBInputFormat (데이터베이스에서 읽기).

OutputFormat은 결과를 출력하는 방법을 결정합니다: TextOutputFormat (기본값, 텍스트 형식); SequenceFileOutputFormat (바이너리 형식); NullOutputFormat (파일 출력 없음); DBOutputFormat (데이터베이스로 출력).

### 작은 파일 문제 처리 방법

대량의 작은 파일은 NameNode 메모리 압력을 증가시키고 처리 효율을 저하시킵니다. 해결책: Hadoop Archives (HAR)를 사용하여 작은 파일 패키징; SequenceFile을 사용하여 작은 파일 병합; HBase를 사용하여 대량의 작은 파일 저장; 블록 크기를 조정하여 작은 파일 수 감소.
