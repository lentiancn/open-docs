# Introduction à Node.js

Node.js® est un environnement d'exécution JavaScript gratuit, open-source et multiplateforme qui permet aux développeurs de créer des serveurs, des applications web, des outils en ligne de commande et des scripts.

## Fonctionnalités principales

En tant qu'environnement d'exécution JavaScript asynchrone et orienté événements, Node.js est conçu pour construire des applications réseau évolutives. Dans l'exemple classique "hello world", de nombreuses connexions peuvent être gérées simultanément. À chaque connexion, le callback est déclenché, mais s'il n'y a aucun travail à effectuer, Node.js passe en mode veille.

```javascript
const { createServer } = require('node:http');

const hostname = '127.0.0.1';
const port = 3000;

const server = createServer((req, res) => {
  res.statusCode = 200;
  res.setHeader('Content-Type', 'text/plain');
  res.end('Hello World');
});

server.listen(port, hostname, () => {
  console.log(`Server running at http://${hostname}:${port}/`);
});
```

## Philosophie de conception

Ceci contraste avec le modèle de concurrence plus courant aujourd'hui, qui utilise des threads du système d'exploitation. La programmation réseau basée sur les threads est relativement inefficace et très difficile à utiliser. De plus, les utilisateurs de Node.js sont libérés des soucis de blocage du processus, car il n'y a pas de verrous.

Presque aucune fonction dans Node.js n'effectue directement des opérations d'E/S, donc le processus ne se bloque jamais sauf lorsqu'on utilise des méthodes synchrones de la bibliothèque standard de Node.js. Puisque rien ne bloque, il est très raisonnable de développer des systèmes évolutifs en Node.js.

## Architecture orientée événements

Node.js est similaire dans sa conception à des systèmes comme Event Machine de Ruby et Twisted de Python, et a été influencé par eux. Node.js pousse le modèle événementiel un peu plus loin. Il présente une boucle d'événements comme une construction d'exécution plutôt que comme une bibliothèque.

Dans d'autres systèmes, il y a toujours un appel bloquant pour démarrer la boucle d'événements. Typiquement, le comportement est défini par des callbacks au début d'un script, et à la fin, un serveur est démarré par un appel bloquant comme `EventMachine::run()`.

Dans Node.js, il n'y a pas un tel appel pour démarrer la boucle d'événements. Node.js entre simplement dans la boucle d'événements après avoir exécuté le script d'entrée. Node.js quitte la boucle d'événements lorsqu'il n'y a plus de callbacks à exécuter. Ce comportement est similaire à celui du JavaScript dans les navigateurs – la boucle d'événements est cachée à l'utilisateur.

## Prise en charge HTTP

HTTP est un citoyen de première classe dans Node.js, conçu avec le streaming et la faible latence à l'esprit. Cela rend Node.js particulièrement adapté comme base pour une bibliothèque ou un framework web.

## Prise en charge des processeurs multi-cœurs

Le fait que Node.js soit conçu sans threads ne signifie pas que vous ne pouvez pas tirer parti de plusieurs cœurs dans votre environnement. Des processus enfants peuvent être créés en utilisant notre API `child_process.fork()`, et sont conçus pour être faciles à communiquer avec.

Construit sur cette même interface se trouve le module `cluster`, qui vous permet de partager des sockets entre processus pour permettre l'équilibrage de charge sur vos cœurs.
