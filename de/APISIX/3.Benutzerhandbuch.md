# Benutzerhandbuch

Dieses Handbuch erläutert die Kernfunktionen und die Verwendung von Apache APISIX, einschließlich Routing, Upstream-Management, Lastverteilung, Ratenbegrenzung und Authentifizierung.

## Grundkonzepte

### Route (Routing)

Eine Route ist der Pfad, über den ein Client auf ein Upstream-Ziel zugreift. In APISIX wird eine Client-Anfrage zunächst anhand vordefinierter Regeln zugeordnet, dann werden die entsprechenden Plugins geladen und ausgeführt, und schließlich wird die Anfrage an einen bestimmten Upstream-Dienst weitergeleitet.

Die einfachste Route besteht nur aus einem Matching-Pfad und einer Upstream-Adresse.

### Upstream

Ein Upstream ist eine Gruppe von Nodes mit derselben Funktionalität – eine Abstraktion eines virtuellen Hosts. Der Upstream kann über vordefinierte Regeln den Lastenausgleich zwischen mehreren Service-Nodes durchführen.

### Consumer

Ein Consumer ist eine Anwendung oder ein Entwickler, die/der die API nutzt. In APISIX benötigt ein Consumer einen global eindeutigen Namen und ist an ein Authentifizierungs-Plugin gebunden.

### Plugin

APISIX bietet ein umfangreiches Plugin-System zur Erweiterung der Gateway-Funktionalität. Plugins können auf Route-, Consumer- oder globaler Ebene konfiguriert werden.

## Routen konfigurieren

Routen können einfach über die Admin-API erstellt werden. Das folgende Beispiel erstellt eine Route, die Anfragen an httpbin.org weiterleitet:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

Bei erfolgreicher Erstellung wird `HTTP/1.1 201 Created` zurückgegeben.

### Route überprüfen

```bash
curl "http://127.0.0.1:9080/ip"
```

Normalerweise erhalten Sie eine Antwort wie:

```json
{
  "origin": "183.94.122.205"
}
```

### Routing-Matching-Regeln

APISIX unterstützt verschiedene Matching-Methoden:

**URI-Matching:**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**Host-Matching:**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**Request-Methoden-Matching:**

```json
{
  "methods": ["GET", "POST"]
}
```

**Header-Matching:**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## Lastverteilung

APISIX unterstützt verschiedene Lastverteilungsalgorithmen, darunter Round-Robin, Weighted Round-Robin und Least Connections.

### Gewichtete Lastverteilung konfigurieren

Erstellen Sie eine Route mit mehreren Upstream-Diensten:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**Parameter-Erklärung:**

- `type`: Lastverteilungsalgorithmus – `roundrobin`, `weighted_roundrobin` oder `least_conn`
- `nodes`: Upstream-Service-Nodes im Format `{"host:port": gewicht}`
- `pass_host`: Bei `node` werden Request-Headers an den Upstream weitergeleitet
- `scheme`: Upstream-Protokoll, `http` oder `https`

### Lastverteilung überprüfen

Erzeugen Sie 100 Anfragen, um die Verteilung zu testen:

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

Die Anfragen werden nahezu gleichmäßig auf beide Upstreams verteilt.

## Ratenbegrenzung

APISIX bietet flexible Ratenbegrenzung durch das `limit-count`-Plugin.

### Ratenbegrenzung aktivieren

Fügen Sie das Ratenbegrenzungs-Plugin zu einer bestehenden Route hinzu:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**Parameter-Erklärung:**

- `count`: Maximale Anzahl von Anfragen im Zeitfenster
- `time_window`: Zeitfenstergröße in Sekunden
- `rejected_code`: HTTP-Statuscode bei Überschreitung des Limits

Diese Konfiguration begrenzt den eingehenden Verkehr auf maximal 2 Anfragen pro 10 Sekunden.

### Ratenbegrenzung überprüfen

Erzeugen Sie 100 Anfragen, um die Begrenzung zu testen:

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

Ergebnis: 2 Anfragen erfolgreich (200), 98 Anfragen abgelehnt (503).

### Ratenbegrenzung deaktivieren

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### Erweiterte Ratenbegrenzung

**Mit Variablen:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**Mit Redis-Cluster:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## Authentifizierung

APISIX unterstützt verschiedene Authentifizierungsmethoden: Key Auth, Basic Auth, JWT und OAuth2.

### Key Authentication (Schlüssel-Authentifizierung)

#### Consumer erstellen

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "secret-key"
    }
  }
}'
```

> ⚠️ In Produktionsumgebungen sollten Sie einen sicheren Schlüssel verwenden.

#### Schlüssel-Authentifizierung für Route aktivieren

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### Authentifizierung überprüfen

**Anfrage ohne Schlüssel:**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

Antwort: `HTTP/1.1 401 Unauthorized`

**Anfrage mit falschem Schlüssel:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: wrong-key'
```

Antwort: `HTTP/1.1 401 Unauthorized`

**Anfrage mit korrektem Schlüssel:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: secret-key'
```

Antwort: `HTTP/1.1 200 OK`

### Weitere Authentifizierungsmethoden

**JWT Authentication:**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "user-key",
      "secret": "my-secret-key"
    }
  }
}
```

**Basic Authentication:**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**HMAC Authentication:**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "my-secret-key"
    }
  }
}
```

## Service-Governance

### Canary Releases

APISIX unterstützt Canary Releases basierend auf Gewichtung:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### Request-Umschreibung

Verwenden Sie das `proxy-rewrite`-Plugin, um Anfragen umzuschreiben:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### Response-Caching

Verwenden Sie das `proxy-cache`-Plugin, um Upstream-Responses zu cachen:

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## Plugin-Konfiguration

### Plugin-Ausführungsreihenfolge

Die Ausführungsphase eines Plugins kann über die `phase`-Konfiguration festgelegt werden:

- `rewrite`: Request-Umschreibungsphase
- `access`: Zugriffskontrollphase
- `before_proxy`: Phase vor dem Proxying
- `header_filter`: Response-Header-Filterphase
- `body_filter`: Response-Body-Filterphase
- `log`: Logging-Phase

### Plugin deaktivieren

Setzen Sie `_meta.disable` auf `true`:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/route-id" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## Monitoring und Observability

### Prometheus integrieren

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

Metriken abrufen:

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### Logging

APISIX unterstützt verschiedene Log-Ausgaben:

**HTTP-Logging:**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

**Fehler-Logging:**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

## Best Practices

1. **Aussagekräftige IDs verwenden**: Nutzen Sie beschreibende IDs für Routen, Upstreams und andere Ressourcen
2. **Timeouts vernünftig setzen**: Passen Sie Timeout-Parameter an die Antwortzeiten der Upstreams an
3. **Health Checks aktivieren**: Konfigurieren Sie aktive und passive Health Checks für kritische Dienste
4. **Ratenbegrenzung angemessen konfigurieren**: Passen Sie Regeln an Ihre Geschäftsanforderungen und Serverkapazitäten an
5. **Authentifizierung aktivieren**: In Produktionsumgebungen ist Authentifizierung Pflicht
6. **TLS verwenden**: Exponierte Schnittstellen sollten HTTPS nutzen
7. **Wichtige Metriken überwachen**: Konfigurieren Sie Prometheus-Monitoring mit Alarmierungen
