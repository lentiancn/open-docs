# 사용 매뉴얼

이 매뉴얼은 Apache APISIX의 핵심 기능과 사용법에 대한 상세한 정보를 제공합니다. 라우트 구성, 업스트림 관리, 로드 밸런싱, 속도 제한, 인증 등을 포함합니다.

## 기본 개념

### 라우트

라우트는 업스트림 서비스에 접근하는 경로입니다. APISIX에서 라우트는 먼저 사전 정의된 규칙을 사용하여 클라이언트 요청을 일치시킨 다음, 해당 플러그인을 로드하고 실행하며, 마지막으로 요청을 특정 업스트림 서비스로 전달합니다.

가장 간단한 라우트는 일치하는 경로와 업스트림 주소로만 구성됩니다.

### 업스트림

업스트림은 동일한 기능의 노드 모음으로, 가상 호스트의 추상화입니다. 업스트림은 사전 정의된 규칙을 사용하여 여러 서비스 노드에서 로드 밸런싱을 수행할 수 있습니다.

### 컨슈머

컨슈머는 API를 사용하는 애플리케이션 또는 개발자입니다. APISIX에서 컨슈머는 전역적으로 고유한 이름이 필요하며 인증 플러그인에 바인딩되어야 합니다.

### 플러그인

APISIX는 게이트웨이 기능을 확장하기 위해丰富的플러그인 시스템을 제공합니다. 플러그인은 라우트, 컨슈머 또는 전역 수준에서 구성할 수 있습니다.

## 라우트 구성

Admin API를 통해 쉽게 라우트를 생성할 수 있습니다. 다음 예제는 httpbin.org로 요청을 전달하는 라우트를 생성합니다:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

성공적으로 생성되면 `HTTP/1.1 201 Created`가 반환됩니다.

### 라우트 확인

```bash
curl "http://127.0.0.1:9080/ip"
```

반환값:

```json
{
  "origin": "183.94.122.205"
}
```

### 라우트 일치 규칙

APISIX는 다양한 라우트 일치 방법을 지원합니다:

**URI 일치:**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**호스트 일치:**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**메서드 일치:**

```json
{
  "methods": ["GET", "POST"]
}
```

**헤더 일치:**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## 로드 밸런싱

APISIX는 roundrobin, 가중 라운드로빈, 최소 연결을 포함한 다양한 로드 밸런싱 알고리즘을 지원합니다.

### 가중 로드 밸런싱 구성

여러 업스트림 서비스를 가진 라우트 생성:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**매개변수:**

- `type`: 로드 밸런싱 알고리즘 - `roundrobin`, `weighted_roundrobin`, 또는 `least_conn`
- `nodes`: 형식 `{"host:port": weight}`의 업스트림 서비스 노드
- `pass_host`: 요청 헤더를 업스트림에 전달하려면 `node`로 설정
- `scheme`: 업스트림 프로토콜, `http` 또는 `https`

### 로드 밸런싱 확인

밸런싱을 테스트하기 위해 100개의 요청을 생성:

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

요청은 두 업스트림 서비스之间에 거의 균등하게 분배됩니다.

## 속도 제한

APISIX는 `limit-count` 플러그인을 통해 유연한 속도 제한을 제공합니다.

### 속도 제한 활성화

기존 라우트에 limit-count 플러그인 추가:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**매개변수:**

- `count`: 시간 창에서 허용되는 최대 요청 수
- `time_window`: 시간 창 크기 (초)
- `rejected_code`: 제한을 초과할 때 반환되는 HTTP 상태 코드

이 구성은 10초당 최대 2개의 요청으로 수신 트래픽을 제한합니다.

### 속도 제한 확인

테스트하기 위해 100개의 요청을 생성:

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

결과: 2개 요청 성공 (200), 98개 요청 거부 (503).

### 속도 제한 비활성화

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### 고급 속도 제한

**변수 사용:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**Redis 클러스터 사용:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## 인증

APISIX는 키 인증, 기본 인증, JWT, OAuth2 등을 포함한 다양한 인증 방법을 지원합니다.

### 키 인증

#### 컨슈머 생성

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "secret-key"
    }
  }
}'
```

> ⚠️ 프로덕션에서는 복잡한 키를 사용하세요.

#### 라우트에 키 인증 활성화

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### 인증 확인

**키 없이 요청:**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

반환: `HTTP/1.1 401 Unauthorized`

**잘못된 키로 요청:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: wrong-key'
```

반환: `HTTP/1.1 401 Unauthorized`

**올바른 키로 요청:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: secret-key'
```

반환: `HTTP/1.1 200 OK`

### 기타 인증 방법

**JWT 인증:**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "user-key",
      "secret": "my-secret-key"
    }
  }
}
```

**기본 인증:**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**HMAC 인증:**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "my-secret-key"
    }
  }
}
```

## 서비스 거버넌스

### Gray 배포

APISIX는 가중치 기반 Gray 배포 및 카나리 배포를 지원합니다:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### 요청 재작성

`proxy-rewrite` 플러그인을 사용하여 요청을 재작성합니다:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### 응답 캐싱

`proxy-cache` 플러그인을 사용하여 업스트림 응답을 캐싱합니다:

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## 플러그인 구성

### 플러그인 실행 순서

플러그인 구성에서 `phase`를 사용하여 플러그인 실행 단계를 지정합니다:

- `rewrite`: 요청 재작성 단계
- `access`: 접근 제어 단계
- `before_proxy`: 사전 프록시 단계
- `header_filter`: 응답 헤더 필터 단계
- `body_filter`: 응답 본문 필터 단계
- `log`: 로깅 단계

### 플러그인 비활성화

`_meta.disable`를 `true`로 설정합니다:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/route-id" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## 모니터링 및 관측 가능성

### Prometheus 통합

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

메트릭 접근:

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### 로깅

APISIX는 다양한 로그 출력을 지원합니다:

**HTTP 로거:**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

**오류 로거:**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

## 모범 사례

1. **의미 있는 ID 사용**: 라우트, 업스트림 등에 설명적인 ID를 사용하세요
2. **적절한 시간 초과 설정**: 업스트림 응답 시간에 따라 합리적인 시간 초과를 구성하세요
3. **상태 검사 활성화**: 중요한 서비스에 능동적 및 수동적 상태 검사를 구성하세요
4. **적절한 속도 제한 구성**: 비즈니스 요구와 서버 용량에 따라 속도 제한을 설정하세요
5. **인증 활성화**: 프로덕션에서는 반드시 인증을 활성화해야 합니다
6. **TLS 사용**: 외부에 노출되는 인터페이스에는 HTTPS를 사용하세요
7. **주요 메트릭 모니터링**: Prometheus 모니터링을 구성하고 알림을 설정하세요
