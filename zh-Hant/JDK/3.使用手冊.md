# JDK 使用手冊

## Java 編譯器使用指南

### javac 基本用法

javac 是 Java 的編譯器，負責將 Java 原始碼編譯成位元組碼檔案。基本的使用方法非常簡單：javac HelloWorld.java。這會生成 HelloWorld.class 檔案，然後在 JVM 上運行。

在實際的開發專案中，通常需要指定更多的編譯選項。常用的選項包括：-d 指定編譯輸出目錄，例如 javac -d out HelloWorld.java 會將編譯結果放到 out 目錄中；-classpath（或 -cp）指定類別搜尋路徑，用於編譯時參考外部類別；-source 指定原始碼的 Java 版本；-target 指定生成的位元組碼的 Java 版本。

對於大型專案，通常會使用建置工具如 Maven 或 Gradle 來管理編譯過程，這些工具會自動處理依賴和編譯選項。

### 編譯多個檔案

如果專案包含多個 Java 原始檔案，可以一次性編譯整個目錄。例如：javac -d out src/*.java 會編譯 src 目錄下的所有 Java 檔案。

更複雜的專案結構可能需要使用 -sourcepath 選項指定原始碼的搜尋路徑，讓編譯器能夠找到依賴的類別。例如：javac -d out -sourcepath src src/Main.java。

### 編譯錯誤處理

編譯過程中如果出現錯誤，javac 會輸出詳細的錯誤資訊，包括錯誤所在的檔案、行號和具體原因。常見的編譯錯誤包括：語法錯誤（缺少分號、括號不相符等）、類型錯誤（類型不相容、方法不存在等）、訪問權限錯誤（嘗試存取 private 成員等）。

理解這些錯誤資訊對於快速定位和解決問題非常重要。編譯器會盡可能多地報告錯誤，有時修復第一個錯誤後，後續的錯誤可能就自動消失了。

## Java 程式運行指南

### java 命令基本用法

使用 java 命令運行編譯後的 Java 程式。基本語法是：java HelloWorld。注意這裡不需要 .class 副檔名。JVM 會自動在當前目錄和 CLASSPATH 中尋找 HelloWorld.class 檔案。

如果類別在套件中，需要使用完整限定名。例如：java com.example.HelloWorld。

### 類別路徑（CLASSPATH）

類別路徑告訴 JVM 在哪裡尋找類別檔案。可以通過 -classpath 或 -cp 選項指定。例如：java -cp out:lib/* com.example.HelloWorld。在 Windows 上，路徑分隔符是分號（;），在 Linux 和 macOS 上是冒號（:）。

如果未指定類別路徑，JVM 會使用當前目錄以及 CLASSPATH 環境變數中指定的路徑。

### JVM 選項

JVM 提供了大量的啟動選項，用於控制運行時的行為。常用選項包括：-Xms 指定堆的初始大小，如 -Xms256m 設定初始堆為 256MB；-Xmx 指定堆的最大大小，如 -Xmx2g 設定最大堆為 2GB；-XX:+UseG1GC 使用 G1 垃圾回收器。

在 JDK 9 及之後，可以使用 --add-opens 選項來開放模組的內部訪問，這在運行需要訪問 JDK 內部 API 的庫時很有用。

## JDK 工具使用指南

### jar 歸檔工具

jar 工具用於建立和管理 Java 歸檔檔案（.jar 檔案）。.jar 檔案實際上就是 ZIP 格式的壓縮包，裡面包含了編譯後的 .class 檔案和一些中繼資料。

建立 jar 檔案的基本命令是：jar cf myapp.jar -C out .。這會從 out 目錄的內容建立 myapp.jar。使用 tf 參數可以查看 jar 檔案的內容：jar tf myapp.jar。使用 xf 可以解壓：jar xf myapp.jar。

在建立可執行的 jar 檔案時，需要在 MANIFEST.MF 檔案中指定主類。可以通過 m 參數新增自定義的 MANIFEST.MF：jar cfm myapp.jar MANIFEST.MF -C out .

### javadoc 文檔生成

javadoc 工具根據 Java 原始碼中的註解生成 HTML 格式的 API 文檔。標準的 Java 類別庫文檔就是這樣生成的。

基本用法是：javadoc -d doc src/*.java。這會為 src 目錄下所有 Java 檔案生成文檔，放到 doc 目錄中。

在編寫程式碼時，使用特定的註解格式可以為類別、方法、欄位新增文檔。常用的註解標籤包括：@author 作者、@param 參數、@return 回傳值、@throws 例外等。

### jdb 偵錯器

jdb 是 Java 的命令列偵錯器，提供基本的偵錯功能。啟動偵錯：jdb MyClass。設定中斷點：stop in MyClass.method。查看中斷點：clear。執行程式：run。繼續執行：cont。單步執行：step 或 next。查看變數：print varname。列出原始碼：list。

雖然 jdb 功能不如圖形化偵錯器強大，但它可以在沒有 IDE 的環境中使用，或者在伺服器上進行遠端偵錯。

## JVM 效能調優

### 垃圾回收器選擇

JDK 提供了多種垃圾回收器，每種都有不同的特點和適用場景：

Serial GC：單執行緒收集器，適合小型應用和用戶端應用。使用 -XX:+UseSerialGC 啟用。

Parallel GC（Throughput GC）：多執行緒收集器，適合追求高吞吐量的應用。使用 -XX:+UseParallelGC 啟用。

CMS GC：並發標記清除收集器，追求低停頓時間。使用 -XX:+UseConcMarkSweepGC 啟用（JDK 14 已廢棄）。

G1 GC：分割式垃圾回收器，適合大記憶體應用，在 JDK 9 及之後成為預設選項。建議使用預設設定或 -XX:+UseG1GC。

ZGC：低延遲垃圾回收器，支援 TB 級堆記憶體，停頓時間不超過 10ms。使用 -XX:+UseZGC 啟用（JDK 15+）。

Shenandoah：另一個低延遲垃圾回收器，停頓時間與堆大小無關。使用 -XX:+UseShenandoahGC 啟用。

### 記憶體配置

合理配置堆記憶體大小對應用效能至關重要。常用配置選項：

-Xms：堆初始大小，如 -Xms2g。

-Xmx：堆最大大小，如 -Xmx4g。

-Xmn：年輕代大小，如 -Xmn1g。

-XX:MetaspaceSize：元空間初始大小。

-XX:MaxMetaspaceSize：元空間最大大小。

建議將 -Xms 和 -Xmx 設定為相同的值，避免執行期調整堆大小帶來的開銷。

### 效能監控工具

JDK 提供了豐富的效能監控和診斷工具：

jps：顯示 Java 程式列表，基本用法是 jps -l。

jstat：顯示 JVM 統計資訊，如垃圾回收統計：jstat -gcutil pid 1000。

jstack：列印執行緒堆疊資訊，用於診斷死鎖等問題：jstack pid。

jmap：顯示堆記憶體資訊，如生成堆轉儲：jmap -dump:format=b,file=heap.bin pid。

jcmd：多功能診斷工具，可以執行各種診斷命令：jcmd pid GC.heap_info。

JConsole 和 VisualVM：圖形化的監控和診斷工具，可以監控記憶體、執行緒、CPU 使用等情況。

## 模組系統

### 模組化概述

從 JDK 9 開始，Java 引入了模組系統（Project Jigsaw），將 JDK 本身和應用程式劃分為模組。每個模組都有明確的依賴關係，提供了更好的封裝和可維護性。

JDK 本身被劃分為多個模組，如 java.base、java.sql、java.xml 等。應用程式可以建立自己的模組，透過 module-info.java 檔案聲明模組名、依賴和其他模組的匯出。

### 模組相關選項

JDK 提供了多個與模組相關的命令列選項：

--module-path（或 -p）：指定模組路徑，類似於 classpath 但用於模組。

--add-modules：新增模組到預設的模組集合中。

--add-exports：開放指定模組的套件供其他模組訪問。

--add-opens：開放指定模組的套件供反射訪問。

這些選項在某些情況下非常重要，例如在使用反射訪問 JDK 內部 API 時，或者在使用不相容模組化的庫時。

## 常用配置示例

### 開發環境配置

對於日常開發環境，建議的配置包括：足夠的堆記憶體（-Xmx2g 或更高)，使用 G1 垃圾回收器（預設)，啟用類別資料共享（-Xshare:classes）以加快啟動速度。如果使用 IDE，可以直接在 IDE 的執行配置中設定這些選項。

### 生產環境配置

生產環境的配置需要根據應用的具體情況調整。基本原則包括：根據應用記憶體需求設定堆大小，通常建議預留 25-30% 的記憶體給作業系統和其他程式；選擇合適的垃圾回收器，G1 通常是較好的選擇；對於追求低延遲的應用，可以考慮 ZGC 或 Shenandoah。

建議在生產環境中啟用 GC 日誌，記錄詳細的垃圾回收資訊，便於問題分析和效能調優。GC 日誌配置示例：-Xlog:gc*:file=gc.log:time,uptime,level,tags:filecount=5,filesize=10M。

### Docker 容器配置

在容器環境中運行 Java 應用需要注意 JVM 的記憶體感知。JDK 10+ 版本預設會感知容器記憶體限制，但在早期版本中可能需要手動配置。

建議配置：設定正確的堆最大值（-XX:MaxRAMPercentage=75.0)，確保 JVM 不會使用超過容器可用記憶體的 75%；使用 -XX:+UseContainerSupport（預設啟用）讓 JVM 感知容器資源限制。

對於雲端原生應用，可以考慮使用 GraalVM Native Image 將應用編譯為原生可執行檔案，可以顯著減少啟動時間和記憶體佔用。

## 版本特定使用差異

### JDK 8

JDK 8 是最後一個免費用於商業使用的 Oracle JDK 版本。引入了 Lambda 表達式、串流 API 和新的日期/時間 API。安裝相對簡單，使用傳統的安裝程式。注意 JAVA_HOME 應指向 JDK 目錄（不是 JRE）。某些命令和工具在新版本中可能會有不同的行為。

### JDK 11

JDK 11 標誌著 Java 付費特性的開始。引入了局部變數類型推斷（var）、新的 HTTP 用戶端 API 和 ZGC（實驗性）。不再作為單獨的下載提供 JRE；只有 JDK 可用。java 命令可以直接運行單一檔案原始碼程式而無需編譯。

### JDK 17

JDK 17 是最後一個在 NFTC 下完全免費的 Oracle JDK 版本。引入了密封類、文字區塊和 switch 表達式。安裝和使用與 JDK 11 類似。RandomGenerator 介面提供了新的隨機數生成能力。外部記憶體訪問 API 進入孵化階段。

### JDK 21

JDK 21 引入了重要特性，包括虛擬線程（穩定）、switch 的模式匹配（穩定）和記錄類（穩定）。虛擬線程需要不同的並發程式設計方法。使用虛擬線程的應用可能需要調整堆設定。效能調優應考慮虛擬線程的特性。

### JDK 25

JDK 25 是最新版本，帶來最新的語言特性和效能改進。特性可能仍處於預覽或實驗狀態。不建議在需要長期支援的生產環境中使用。配置和使用與 JDK 21 類似，但有更多新特性可能需要額外學習。
