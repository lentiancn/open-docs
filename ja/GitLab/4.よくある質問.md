# GitLab よくある質問

## インストール問題

### GitLabのインストールが失敗した場合はどうしますか？

インストール失敗はいくつかの一般的な原因で発生します。まず、オペレーティングシステムのバージョン、メモリ、ディスクスペースなど、システムが最小要件を満たしているかどうかを確認してください。セルフホストスクリプトでのインストールが失敗した場合は、依存パッケージを手動でインストールしてからステップバイステップでインストールしてみてください。

詳細なエラー情報を得るためにインストールログを確認します：

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# ログの確認
sudo tail -f /var/log/gitlab/install.log
```

一般的なエラーには、ポートの競合（ポート80、443、22が使用中）、メモリ不足、SSL証明書問題などが含まれます。ポートの競合がある場合は、そのポートを使用しているサービスを停止するか、GitLabの設定を変更して別のポートを使用する必要があります。

### GitLabのアクセスアドレスを変更するには？

インストール後にGitLabのアクセスアドレスを変更する必要がある場合は、/etc/gitlab/gitlab.rbファイルのexternal_urlパラメータを編集し、再設定します：

```bash
sudo nano /etc/gitlab/gitlab.rb
# external_urlを変更
sudo gitlab-ctl reconfigure
```

変更後、新しいアドレスに正常にアクセスできるように、DNS解決またはローカルhostsファイルが正しく設定されていることを確認してください。HTTPからHTTPSに変更する場合は、SSL証明書の設定も必要です。

## 使用問題

### SSH鍵の接続が失敗した場合はどうしますか？

SSH鍵の接続失敗にはいくつかの理由が考えられます。まず、公開鍵がGitLabアカウントのSSH鍵設定に正しく追加されていることを確認します：

```bash
# ローカルの公開鍵を確認
cat ~/.ssh/id_ed25519.pub

# 接続テスト
ssh -T git@gitlab.example.com
```

接続テストが失敗した場合は、SSHクライアント設定を確認します：

```bash
# SSH設定の編集
nano ~/.ssh/config

# 以下を追加
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

リモートリポジトリURLがSSH形式（git@gitlab.example.com:group/project.git）を使用しており、HTTPS形式でないことを確認してください。

### マージリクエストをマージできない場合はどうしますか？

マージリクエストは次の理由でマージできない場合があります：競合が存在する、CI/CDテストが失敗した、保護ブランチルールを満たしていない、承認が欠けている，等等。解決方法は1つずつ確認することです：

- マージ競合：マージリクエストページで「ブランチを更新」をクリックし、ターゲットブランチをソースブランチにマージし、競合を解決してから再プッシュ
- CI失敗：失敗したジョブのログを確認し、問題を修正してからパイプラインを再実行
- ブランチ保護：プロジェクト管理者に連絡して保護ブランチ設定を変更するか、権限を持つメンバーがマージ

### CI/CDパイプラインの実行失敗をトラブルシューティングする方法は？

パイプラインが失敗した場合は、まずCI/CD → パイプラインページで失敗したジョブのログを確認します。ログには具体的なエラー情報と失敗理由が表示されます。一般的な問題には、依存関係のダウンロード失敗、スクリプトコマンドエラー、テストケースの失敗などが含まれます。

問題が特定しにくい場合は、パイプライン設定にデバッグステップを追加して、環境変数を出力したり、インストール前後のファイル状態を確認したりできます。また、interactive debugging機能を使用して、パイプラインジョブでインタラクティブシェルを起動してトラブルシューティングを行うこともできます。

## パフォーマンス問題

### GitLabへのアクセスが非常に遅い場合はどうしますか？

アクセスが遅い原因是、サーバー資源の不足、ネットワーク遅延、設定の不備などが考えられます。まずサーバーの負荷を確認します：

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

CPUまたはメモリの使用率が高い場合は、サーバー設定のアップグレードまたはGitLab設定の最適化が必要な場合があります。データベースのパフォーマンスも確認できます：

```bash
sudo gitlab-rake gitlab:git:fsck  # Gitリポジトリの整合性をチェック
sudo gitlab-rake db:index_cache_daily  # データベースキャッシュを最適化
```

ネットワーク問題の場合は、CDNを使用して静的リソースを配信하거나、よりユーザーの近いネットワーク位置にGitLabをデプロイすることを検討してください。

### GitLabのパフォーマンスを最適化する方法は？

GitLabのパフォーマンス最適化は複数のアプローチがあります。データベースの最適化は重要で、定期的なメンテナンスはパフォーマンスを大幅に向上させることができます：

```bash
# データベースのクリーンアップ
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# 未使用ファイルのクリーンアップ
sudo gitlab-rake gitlab:cleanup:builds
```

大きなリポジトリの場合は、Gitオブジェクトプールを有効にしてディスク使用量を削減するか、Git LFSを設定して大きなファイルを保存できます。CI/CDキャッシュを適切に設定すると、ビルド速度が向上します。

## セキュリティ問題

### SSL証明書を設定する方法は？

GitLabはLet's Encryptによる自動SSL設定をサポートしていますが、カスタム証明書を使用することもできます。まず設定ファイルを編集します：

```ruby
# Let's Encrypt自動設定
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# カスタム証明書
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

設定後、sudo gitlab-ctl reconfigureを実行して設定を適用します。自己署名証明書をを使用する場合は、HTTPSでアクセスするためにクライアントがその証明書を信頼する必要があります。

### GitLabをバックアップして復元する方法は？

定期的なバックアップはデータ損失を防ぐための重要なMeasuresです。GitLabには組み込みバックアップツールがあります：

```bash
# バックアップの作成
sudo gitlab-backup create

# バックアップファイルの場所
ls -la /var/opt/gitlab/backups/
```

バックアップファイルにはデータベース、アップロードされたファイル、Gitリポジトリが含まれます。復元するには：GitLabサービスを停止し、バックアップファイルをデフォルトの場所にコピーし、復元コマンドを実行します：

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

バックアップファイルをオブジェクトストレージサービスや別のサーバーに自動的に同期することを推奨しています。

## アップグレード問題

### アップグレード後にエラーが発生した場合はどうしますか？

アップグレード後に互換性エラーや機能の異常が発生する可能性があります。まずGitLabの状態を確認します：

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

具体的なエラーを確認するために upgrade ログを表示します：

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

一般的な問題には、データベースマイグレーションの失敗、設定ファイルの構文エラー、依存関係の欠落などが含まれます。データベースマイグレーションの問題の場合は、手動で修正する必要があるかもしれません：

```bash
sudo gitlab-rake db:migrate
```

アップグレード前にデータのバックアップを取り、バージョン間の重大な変更を理解するために公式のアップグレードドキュメントを注意深く読むことを推奨しています。

### 異なるバージョン間で安全にアップグレードする方法は？

GitLabのアップグレードは、バージョンの跳躍制限に従う必要があり、一度に1つのメジャーバージョンを超えてアップグレードしないでください。例えば、15.xから16.xに升级し、15.xから17.xに直接升级しないでください。まず中間のバージョンに升级し、正常に動作することを確認してから升级を続けてください。

アップグレード前に以下のステップを実行します：データのバックアップ、互換性問題のチェック、リリースノートの阅读。升级中は電力とネットワークを安定させ、升级後にすべての機能が正常に動作することを確認してください。
