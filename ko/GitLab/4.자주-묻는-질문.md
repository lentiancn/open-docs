# GitLab 자주 묻는 질문

## 설치 문제

### GitLab 설치가 실패하면 어떻게 해야 하나요?

설치 실패는 일반적으로 몇 가지 일반적인 원인으로 인해 발생합니다. 먼저 시스템이 운영 체제 버전, 메모리 및 디스크 공간을 포함한 최소 요구 사항을 충족하는지 확인하세요. 자체 호스팅 스크립트를 사용한 설치가 실패하면 종속성 패키지를 수동으로 설치한 다음 단계별로 설치해 보세요.

자세한 오류 정보를 얻으려면 설치 로그를 확인하세요:

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# 로그 보기
sudo tail -f /var/log/gitlab/install.log
```

일반적인 오류에는 포트 충돌(포트 80, 443, 22 사용 중), 메모리 부족, SSL 인증서 문제가 포함됩니다. 포트 충돌이 있는 경우 포트를 사용하는 서비스를 중지하거나 GitLab 구성을 변경하여 다른 포트를 사용하세요.

### GitLab 액세스 주소를 수정하려면 어떻게 해야 하나요?

설치 후 GitLab의 액세스 주소를 수정해야 하는 경우 /etc/gitlab/gitlab.rb 파일에서 external_url 매개변수를 편집한 다음 다시 구성하세요:

```bash
sudo nano /etc/gitlab/gitlab.rb
# external_url 수정
sudo gitlab-ctl reconfigure
```

수정 후 새 주소에 정상적으로 접근할 수 있도록 DNS 확인 또는 로컬 hosts 파일이 올바르게 구성되어 있는지 확인하세요. HTTP에서 HTTPS로 변경하는 경우 SSL 인증서도 구성해야 합니다.

## 사용 문제

### SSH 키 연결이 실패하면 어떻게 해야 하나요?

SSH 키 연결 실패는 여러 가지 원인으로 인해 발생할 수 있습니다. 먼저 공개 키가 GitLab 계정의 SSH 키 설정에 올바르게 추가되었는지 확인하세요:

```bash
# 로컬 공개 키 보기
cat ~/.ssh/id_ed25519.pub

# 연결 테스트
ssh -T git@gitlab.example.com
```

연결 테스트가 실패하면 SSH 클라이언트 구성을 확인하세요:

```bash
# SSH 구성 편집
nano ~/.ssh/config

# 다음 추가
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

원격 저장소 URL이 SSH 형식(git@gitlab.example.com:group/project.git)을 사용하고 있는지 HTTPS 형식이 아닌지 확인하세요.

### 병합 요청을 병합할 수 없으면 어떻게 해야 하나요?

병합 요청은 다음과 같은 이유로 병합하지 못할 수 있습니다: 충돌 존재, CI/CD 테스트 실패, 보호된 브랜치 규칙 미충족, 승인 부족 등. 해결 방법은 하나씩 확인하는 것입니다:

- 병합 충돌: 병합 요청 페이지에서 "브랜치 업데이트"를 클릭하고 대상 브랜치를 소스 브랜치에 병합한 다음 충돌을 해결하고 다시 푸시
- CI 실패: 실패한 작업의 로그를 보고 문제를 해결한 다음 파이프라인 다시 실행
- 보호된 브랜치: 프로젝트 관리자에게 보호된 브랜치 설정 수정을 요청하거나 권한이 있는 멤버에게 병합 요청

### CI/CD 파이프라인 실행 실패를 해결하려면 어떻게 해야 하나요?

파이프라인이 실패하면 CI/CD → 파이프라인 페이지에서 실패한 작업의 로그를 먼저 확인하세요. 로그는 특정 오류 정보와 실패 이유를 보여줍니다. 일반적인 문제에는 종속성 다운로드 실패, 스크립트 명령 오류, 테스트 케이스 실패 등이 포함됩니다.

문제를 파악하기 어려운 경우 환경 변수 출력, 설치 전후 파일 상태 확인 등 파이프라인 구성에 디버깅 단계를 추가할 수 있습니다. 또한 대화형 디버깅 기능을 사용하여 파이프라인 작업에서 대화형 셸을 시작하여 문제를 해결할 수 있습니다.

## 성능 문제

### GitLab 액세스가 매우 느립니다. 어떻게 해야 하나요?

느린 액세스는 서버 리소스 부족, 네트워크 지연 또는 부적절한 구성으로 인해 발생할 수 있습니다. 먼저 서버 부하를 확인하세요:

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

CPU 또는 메모리 사용률이 매우 높으면 서버 구성을 업그레이드하거나 GitLab 설정을 최적화해야 할 수 있습니다. 데이터베이스 성능도 확인할 수 있습니다:

```bash
sudo gitlab-rake gitlab:git:fsck  # Git 저장소 무결성 확인
sudo gitlab-rake db:index_cache_daily  # 데이터베이스 캐시 최적화
```

네트워크 문제의 경우 정적 리소스 제공을 가속화하기 위해 CDN을 사용하거나 사용자에게 더 가까운 네트워크 위치에 GitLab을 배포하는 것을 고려하세요.

### GitLab 성능을 최적화하려면 어떻게 해야 하나요?

GitLab 성능 최적화는 여러 가지 접근 방식이 있습니다. 데이터베이스 최적화는 중요합니다. 정기적인 유지보수는 성능을 크게 향상시킬 수 있습니다:

```bash
# 데이터베이스 정리
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# 미사용 파일 정리
sudo gitlab-rake gitlab:cleanup:builds
```

대형 저장소의 경우 Git 객체 풀을 활성화하여 디스크 사용량을 줄이거나 Git LFS를 구성하여 큰 파일을 저장할 수 있습니다. CI/CD 캐시를 합리적으로 구성하면 빌드 속도를 높일 수 있습니다.

## 보안 문제

### SSL 인증서를 구성하려면 어떻게 해야 하나요?

GitLab은 Let's Encrypt를 사용한 자동 SSL 구성 또는 사용자 정의 인증서 사용을 지원합니다. 먼저 구성 파일을 편집하세요:

```ruby
# Let's Encrypt 자동 구성
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# 사용자 정의 인증서
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

구성을 적용하려면 sudo gitlab-ctl reconfigure를 실행하세요. 자체 서명 인증서를 사용하는 경우 클라이언트가 HTTPS를 통해 접근하려면 해당 인증서를 신뢰해야 합니다.

### GitLab을 백업하고 복원하려면 어떻게 해야 하나요?

정기적인 백업은 데이터 손실을 방지하는 데 중요합니다. GitLab은 내장 백업 도구를 제공합니다:

```bash
# 백업 만들기
sudo gitlab-backup create

# 백업 파일 위치
ls -la /var/opt/gitlab/backups/
```

백업 파일에는 데이터베이스, 업로드된 파일 및 Git 저장소가 포함됩니다. 복원하려면: GitLab 서비스를 중지하고 백업 파일을 기본 위치에 복사한 다음 복원 명령을 실행하세요:

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

백업 파일을 객체 스토리지 서비스 또는 다른 서버와 같은 원격 스토리지로 자동 동기화하는 것이 권장됩니다.

## 업그레이드 문제

### 업그레이드 후 오류가 발생하면 어떻게 해야 하나요?

업그레이드 후 호환성 오류 또는 기능 이상 현상이 발생할 수 있습니다. 먼저 GitLab 상태를 확인하세요:

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

특정 오류를 이해하려면 업그레이드 로그를 확인하세요:

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

일반적인 문제에는 데이터베이스 마이그레이션 실패, 구성 파일 구문 오류, 종속성 누락 등이 포함됩니다. 데이터베이스 마이그레이드 문제의 경우 수동으로 수정해야 할 수 있습니다:

```bash
sudo gitlab-rake db:migrate
```

업그레이드 전에 데이터 백업을 만들고 버전 간 주요 변경 사항을 이해하기 위해 공식 업그레이드 문서를 주의 깊게 읽는 것이 권장됩니다.

### 다양한 버전 간에 안전하게 업그레이드하려면 어떻게 해야 하나요?

GitLab 업그레이드는 한 번에 한 메이저 버전을 초과하여 건너뛰지 않는 버전 점프 제한을 따라야 합니다. 예: 15.x에서 16.x로 업그레이드, 15.x에서 17.x로 직접 업그레이드하지 마세요. 먼저 중간 버전으로 업그레이드하고 정상 작동을 확인한 다음 업그레이드를 계속하세요.

업그레이드 전에 다음 단계를 수행하세요: 데이터 백업, 호환성 문제 확인, 릴리스 노트 읽기. 업그레이드プロセス中전원과 네트워크를 안정적으로 유지하고 업그레이드 후 모든 기능이 정상적으로 작동하는지 확인하세요.
