# Preguntas Frecuentes sobre Node.js (FAQ)

## Preguntas Generales

### ¿Qué es Node.js?
Node.js es un entorno de ejecución de JavaScript de código abierto y multiplataforma que ejecuta código JavaScript fuera de un navegador web. Permite a los desarrolladores usar JavaScript para escribir herramientas de línea de comandos y para scripting del lado del servidor, generando contenido web dinámico antes de que se envíe al navegador del usuario.

### ¿Por qué se creó Node.js?
Node.js fue creado por Ryan Dahl en 2009 para abordar las limitaciones de los servidores web tradicionales que utilizaban un modelo basado en hilos para manejar conexiones concurrentes. Node.js introdujo un modelo de E/S orientado a eventos y no bloqueante que lo hace ligero y eficiente para construir aplicaciones de red escalables.

### ¿Es Node.js de un solo hilo?
Las aplicaciones Node.js se ejecutan en un solo hilo para la ejecución de JavaScript, pero pueden manejar múltiples operaciones concurrentes a través de E/S asíncrona y el bucle de eventos. Además, Node.js proporciona APIs como `worker_threads` y `child_process` para utilizar múltiples núcleos de CPU cuando sea necesario.

### ¿Qué es npm?
npm (Node Package Manager) es el gestor de paquetes predeterminado para Node.js y el registro de software más grande del mundo. Permite a los desarrolladores compartir y reutilizar paquetes de código, gestionar dependencias y ejecutar scripts definidos en archivos package.json.

## Instalación y Configuración

### ¿Qué versión debo instalar: LTS o Current?
- **LTS (Soporte a Largo Plazo)**: Recomendado para aplicaciones de producción. Proporciona estabilidad, actualizaciones de seguridad y soporte a largo plazo (30 meses en total).
- **Current**: Contiene las últimas características pero tiene un ciclo de soporte más corto. Mejor para desarrollo, pruebas de nuevas características o aprendizaje.

### ¿Cómo cambio entre versiones de Node.js?
Usa un gestor de versiones de Node:
- **nvm** (macOS/Linux): `nvm install 18 && nvm use 18`
- **nvm-windows** (Windows): Comandos similares a nvm basado en Unix
- **fnm** (Fast Node Manager): Alternativa multiplataforma con mejor rendimiento

### ¿Por qué obtengo errores de permisos al instalar paquetes globales?
Esto típicamente ocurre cuando npm intenta escribir en directorios del sistema que requieren permisos elevados. Soluciones:
1. Configura npm para usar un directorio diferente: `npm config set prefix ~/.npm-global`
2. Agrega `~/.npm-global/bin` a tu PATH
3. Usa un gestor de versiones de Node que maneje correctamente los permisos

### ¿Cómo verifico mi instalación de Node.js?
Ejecuta estos comandos en tu terminal:
```bash
node --version    # Muestra la versión de Node.js
npm --version     # Muestra la versión de npm
node -e "console.log('¡Funcionando!')"  # Prueba funcionalidad básica
```

## Preguntas de Desarrollo

### ¿Cuál es la diferencia entre CommonJS y ES Modules?
- **CommonJS** (`require()`/`module.exports`): El sistema de módulos original de Node.js, síncrono, ampliamente soportado
- **ES Modules** (`import`/`export`): Estándar moderno de JavaScript, asíncrono, requiere extensión `.mjs` o `"type": "module"` en package.json

Ejemplo CommonJS:
```javascript
// math.js
exports.add = (a, b) => a + b;

// app.js
const { add } = require('./math');
```

Ejemplo ES Modules:
```javascript
// math.mjs
export const add = (a, b) => a + b;

// app.mjs
import { add } from './math.mjs';
```

### ¿Cómo manejo operaciones asíncronas?
Tres enfoques principales:

1. **Callbacks** (tradicional pero puede llevar al "infierno de callbacks"):
```javascript
fs.readFile('file.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log(data);
});
```

2. **Promesas** (mejor manejo de errores y encadenamiento):
```javascript
fs.promises.readFile('file.txt', 'utf8')
  .then(data => console.log(data))
  .catch(err => console.error(err));
```

3. **Async/Await** (más legible, parece síncrono):
```javascript
async function readFile() {
  try {
    const data = await fs.promises.readFile('file.txt', 'utf8');
    console.log(data);
  } catch (err) {
    console.error(err);
  }
}
```

### ¿Qué es el bucle de eventos y cómo funciona?
El bucle de eventos es el mecanismo de Node.js para manejar operaciones asíncronas sin bloquear el hilo principal. Continuamente verifica callbacks pendientes y los ejecuta cuando sus operaciones asociadas se completan.

Fases del bucle de eventos:
1. **Temporizadores**: Ejecuta callbacks de setTimeout() y setInterval()
2. **Callbacks pendientes**: Ejecuta callbacks de E/S diferidos a la siguiente iteración del bucle
3. **Inactivo, preparar**: Solo para uso interno
4. **Encuesta**: Recupera nuevos eventos de E/S; ejecuta callbacks relacionados con E/S
5. **Verificar**: Ejecuta callbacks de setImmediate()
6. **Callbacks de cierre**: Ejecuta callbacks de eventos de cierre (ej. socket.on('close'))

### ¿Cómo depuro aplicaciones Node.js?
Múltiples opciones de depuración:

1. **Inspector integrado**: `node --inspect app.js`
2. **DevTools de Chrome**: Conéctate a `chrome://inspect`
3. **Depurador de VS Code**: Soporte integrado para depuración de Node.js
4. **Registro en consola**: Simple pero efectivo para depuración básica
5. **Herramientas de depuración dedicadas**: Como Node.js Inspector Manager (NiM)

## Rendimiento y Escalabilidad

### ¿Cómo escalo aplicaciones Node.js?
Varias estrategias:

1. **Clustering**: Usa el módulo integrado `cluster` para crear procesos secundarios que compartan puertos del servidor
2. **Balanceo de carga**: Usa proxies inversos como NGINX para distribuir tráfico entre múltiples instancias de Node.js
3. **Microservicios**: Divide aplicaciones grandes en servicios más pequeños e independientes
4. **Caching**: Implementa capas de caching (Redis, Memcached) para reducir la carga de la base de datos
5. **Optimización de base de datos**: Usa pooling de conexiones y optimización de consultas

### ¿Qué causa fugas de memoria en Node.js?
Causas comunes:
- **Variables globales**: Acumulación de datos en el ámbito global
- **Listeners de eventos no cerrados**: No eliminar listeners de eventos cuando los objetos se destruyen
- **Closures**: Mantener referencias a objetos grandes más tiempo del necesario
- **Temporizadores**: No limpiar intervalos y timeouts
- **Buffers**: No gestionar adecuadamente datos binarios grandes

Herramientas de detección:
- `process.memoryUsage()` para monitoreo básico
- Instantáneas de heap con DevTools de Chrome
- Herramientas de perfilado de memoria como Clinic.js

### ¿Cómo manejo tareas intensivas en CPU?
Dado que Node.js es de un solo hilo para la ejecución de JavaScript, las tareas intensivas en CPU pueden bloquear el bucle de eventos. Soluciones:

1. **Hilos de trabajo**: Usa el módulo `worker_threads` para procesamiento paralelo
```javascript
const { Worker } = require('worker_threads');
const worker = new Worker('./cpu-intensive-task.js');
```

2. **Procesos secundarios**: Genera procesos separados usando `child_process`
```javascript
const { spawn } = require('child_process');
const child = spawn('python', ['script.py']);
```

3. **Descarga a servicios externos**: Usa colas de mensajes (RabbitMQ, Kafka) para procesar tareas pesadas de forma asíncrona
4. **Optimiza algoritmos**: Perfila y optimiza el código real que causa problemas de rendimiento

## Seguridad

### ¿Cómo aseguro mi aplicación Node.js?
Prácticas clave de seguridad:

1. **Validación de entrada**: Siempre valida y sanitiza la entrada del usuario
2. **Gestión de dependencias**: Actualiza regularmente las dependencias y ejecuta `npm audit`
3. **Variables de entorno**: Almacena secretos en variables de entorno, nunca en el código
4. **Limitación de tasa**: Prevén ataques de fuerza bruta con middleware de limitación de tasa
5. **Helmet.js**: Usa middleware de cabeceras de seguridad
6. **CORS**: Configura adecuadamente el Intercambio de Recursos de Origen Cruzado
7. **HTTPS**: Usa siempre HTTPS en producción
8. **Autenticación**: Implementa autenticación y autorización adecuadas

### ¿Cómo prevengo vulnerabilidades comunes?
- **Inyección SQL**: Usa consultas parametrizadas u ORMs
- **XSS (Cross-Site Scripting)**: Escapa la entrada del usuario antes de renderizar
- **CSRF (Cross-Site Request Forgery)**: Usa tokens CSRF
- **Contaminación de prototipos**: Evita funciones de fusión de objetos inseguras
- **Travesía de rutas**: Valida rutas de archivos y usa `path.resolve()` con precaución

## Despliegue y Producción

### ¿Qué gestor de procesos debo usar?
Opciones populares:

1. **PM2**: Más popular, rico en características, soporta clustering y monitoreo
2. **Forever**: Gestor de procesos simple, bueno para casos de uso básicos
3. **Systemd**: Gestor de servicios nativo de Linux, se integra bien con el SO
4. **Docker**: Enfoque de contenerización, excelente para microservicios

### ¿Cómo manejo la configuración específica del entorno?
Usa variables de entorno y archivos de configuración:

```javascript
// config.js
const config = {
  development: {
    port: 3000,
    dbUrl: 'mongodb://localhost:27017/myapp'
  },
  production: {
    port: process.env.PORT || 8080,
    dbUrl: process.env.DATABASE_URL
  }
};

module.exports = config[process.env.NODE_ENV || 'development'];
```

### ¿Qué herramientas de monitoreo debo usar?
- **Monitoreo de Rendimiento de Aplicaciones (APM)**: New Relic, Datadog, AppSignal
- **Registro**: Winston, Bunyan, con registro centralizado (pila ELK, Splunk)
- **Verificaciones de salud**: Endpoints personalizados que verifican la salud de la aplicación
- **Métricas**: Prometheus con Grafana para visualización
- **Seguimiento de errores**: Sentry, Rollbar para monitoreo de excepciones

### ¿Cómo implemento un apagado elegante?
Maneja señales de proceso para limpiar recursos:

```javascript
const server = http.createServer(app);

const shutdown = () => {
  server.close(() => {
    console.log('Servidor cerrado');
    process.exit(0);
  });
  
  // Apagado forzado después de 10 segundos
  setTimeout(() => {
    console.error('No se pudieron cerrar las conexiones a tiempo, apagando forzosamente');
    process.exit(1);
  }, 10000);
};

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## Ecosistema y Comunidad

### ¿Cuáles son los frameworks más populares de Node.js?
- **Express.js**: Minimalista, flexible, más ampliamente utilizado
