# Hadoop 簡介

## 什麼是 Hadoop

Apache Hadoop 是由 Apache 軟體基金會開發的開源分散式計算框架，專為處理和儲存大規模數據集而設計。Hadoop 提供了可靠的、可擴展的分散式數據儲存和計算解決方案，能夠在由普通硬體組成的叢集上處理海量數據。Hadoop 的核心設計理念是將計算移動到數據所在的節點，而不是將大量數據移動到計算節點，從而大幅減少網路傳輸開銷，提升處理效率。

Hadoop 專案的起源可以追溯到 2003 年，當時 Google 發表了關於 Google File System (GFS) 的論文，隨後在 2004 年又發表了關於 MapReduce 的論文。受這些論文的啟發，Yahoo! 的工程師 Doug Cutting 和 Mike Cafarella 開始開發一個開源實現，即 Hadoop。Hadoop 於 2006 年正式成為 Apache 軟體基金會的項目，其名稱來自 Doug Cutting 兒子的玩具大象。

Hadoop 採用 Java 語言開發，具有良好的跨平台特性。其核心元件包括 HDFS（Hadoop Distributed File System，分散式檔案系統）和 YARN（Yet Another Resource Negotiator，資源調度器），以及 MapReduce 編程模型。Hadoop 的設計哲學是「一切皆可擴展」，即系統可以通過簡單地添加更多節點來線性擴展儲存容量和計算能力，這種簡單而強大的設計使 Hadoop 成為大數據處理領域的基礎設施。

## 核心元件

### HDFS - 分斥式檔案系統

HDFS 是 Hadoop 的分散式檔案系統，專為儲存超大型檔案而設計。HDFS 採用主從架構，包含一個 NameNode（名稱節點）和多個 DataNode（資料節點）。NameNode 負責管理檔案系統的命名空間和中繼資料，維護檔案系統的目錄結構，記錄每個檔案的塊列表以及每個塊所在的 DataNode 位置。DataNode 負責實際的資料儲存，定期向 NameNode 報告其儲存的塊資訊。

HDFS 具有高容錯性，通過資料複製機制確保資料的可靠性。預設情況下，每個資料塊會被複製到三個不同的 DataNode 上，即使某個節點發生故障，系統也能從其他副本中恢復資料。HDFS 優化了大型檔案順序讀取的效能，適合批次處理場景，但不太適合低延遲的隨機讀寫場景。HDFS 還支援機架感知功能，能夠智慧地將資料塊的副本分配到不同機架的節點上，進一步提高資料的可用性和系統的容錯能力。

HDFS 的架構設計遵循「寫一次讀多次」（Write Once Read Many）的原則，這意味著檔案一旦寫入後就不應該被修改。這種設計簡化了一致性模型，提高了讀取效能，也使得資料的複製和分發更加簡單。HDFS 還提供了快照功能，可以建立檔案系統的唯讀時間點副本，用於資料備份和恢復。

### YARN - 資源調度器

YARN 是 Hadoop 2.x 引入的資源管理平台，負責叢集資源的統一管理和調度。YARN 將資源管理和應用程式執行分離，使得 Hadoop 能夠支援更多類型的工作負載，而不僅僅是 MapReduce 批次處理作業。YARN 的核心元件包括 ResourceManager（資源管理器）、NodeManager（節點管理器）和 ApplicationMaster（應用程式主節點）。

ResourceManager 是 YARN 的核心元件，負責整個叢集的資源分配和管理。它接收來自應用程式的資源請求，並根據叢集的可用資源進行調度。ResourceManager 由調度器（Scheduler）和應用程式管理器（ApplicationsManager）組成。調度器負責按照預定的策略分配資源，應用程式管理器負責接收應用程式提交請求，並為應用程式啟動 ApplicationMaster。

NodeManager 是每個工作節點上的代理，負責管理該節點上的資源，包括 CPU、記憶體、磁碟和網路資源。NodeManager 定期向 ResourceManager 報告其資源使用情況和容器狀態，並執行來自 ResourceManager 的指令。ApplicationMaster 是每個應用程式的管理者，負責與 ResourceManager 協商資源，並與 NodeManager 合作執行任務。

### MapReduce - 編程模型

MapReduce 是 Hadoop 的核心計算模型，用於大規模資料的並行處理。MapReduce 編程模型將資料處理過程分為兩個主要階段：Map 階段和 Reduce 階段。在 Map 階段，系統將輸入資料分割成獨立的資料塊，然後並行處理每個資料塊，輸出鍵值對形式的中間結果。在 Reduce 階段，系統將具有相同鍵的中間結果進行聚合和處理，最終得到計算結果。

MapReduce 框架自動處理資料分發、任務調度、負載平衡、容錯處理等複雜細節，使開發者可以專注於業務邏輯的實現。這種抽象大大簡化了分散式程式的開發，開發者只需要編寫 Map 函數和 Reduce 函數，分散式執框架會自動處理行的全部複雜性。MapReduce 還支援自定義分區器（Partitioner）和組合器（Combiner），用於優化資料流動和減少網路傳輸。

MapReduce 作業的執行過程包括輸入分片、Map 任務執行、中間結果 shuffle、Reduce 任務執行和結果輸出等階段。shuffle 階段是 MapReduce 的核心過程，負責將 Map 任務的輸出按照鍵進行分區、排序和合併，然後傳輸給相應的 Reduce 任務。這個過程是計算密集型和網路密集型的，對作業的整體效能有重要影響。

## 技術架構

### NameNode 與 Secondary NameNode

NameNode 是 HDFS 的核心元件，維護整個檔案系統的中繼資料。它在記憶體中保存檔案系統的完整中繼資料，包括檔案名稱、檔案屬性、檔案與資料塊的映射關係以及資料塊與 DataNode 的映射關係。為了保證中繼資料的持久性，NameNode 還會將中繼資料寫入本地檔案系統的事務日誌（EditLog）中。當 NameNode 啟動時，它會讀取映像檔（FsImage）並重放事務日誌來恢復檔案系統狀態。

Secondary NameNode 並不是 NameNode 的備份，它的主要作用是定期合併 FsImage 和 EditLog，幫助 NameNode 減少啟動時間和 EditLog 檔案大小。Secondary NameNode 通常運行在獨立的機器上，它定期從 NameNode 下載 FsImage 和 EditLog，在本地進行合併操作，然後將新的 FsImage 上傳回 NameNode。這個過程稱為檢查點（Checkpoint），可以有效減少 NameNode 的記憶體壓力和啟動時間。

### ResourceManager 架構

ResourceManager 採用雙層調度器架構。第一層是容量調度器（Capacity Scheduler）和公平調度器（Fair Scheduler），它們負責在不同的佇列和應用程式之間分配叢集資源。第二層是 ApplicationMaster，每個應用程式都有自己的 ApplicationMaster，負責與 ResourceManager 協商資源並管理應用程式的執行。

容量調度器允許多個組織共享叢集資源，每個組織擁有獨立的佇列，保證每個組織都能獲得一定的資源配額。公平調度器則強調公平性，會自動在所有運行的應用程式之間平衡資源分配，使得每個應用程式都能獲得相對公平的資源。兩種調度器都可以通過設定檔進行靈活定制，滿足不同場景的需求。

### 資料本地性

資料本地性是 Hadoop 的重要優化策略，旨在減少網路資料傳輸。Hadoop 盡可能將計算任務分配到存儲有相應資料的節點上執行，這就是「計算移動到資料」的理念。資料本地性分為三個級別：節點本地（計算和資料在同一節點上）、機架本地（計算和資料在同一機架上）和跨機架（計算和資料在不同機架上）。

Hadoop 在任務調度時會優先考慮節點本地的資料，如果該節點不可用，則退而求其次選擇機架本地的資料。為了實現這一優化，NameNode 會向 ResourceManager 報告每個資料塊的位置資訊，ResourceManager 在調度任務時會參考這些資訊做出最優決策。資料本地性對於大規模資料處理非常重要，可以顯著提升叢集的整體效能。

## 應用場景

Hadoop 廣泛應用於大數據處理的各個領域。在資料分析領域，Hadoop 可以用於處理和分析海量的日誌資料、使用者行為資料、業務交易資料等，幫助企業從海量資料中提取有價值的資訊。在資料倉庫領域，Hadoop 可以作為傳統資料倉庫的補充，處理結構化和非結構化資料，提供更低成本的資料儲存和分析能力。

在機器學習領域，Hadoop 可以作為大規模特徵工程和模型訓練的基礎設施，支援分散式機器學習演算法的運行。在資料湖架構中，Hadoop 的 HDFS 提供了廉價而可靠的資料儲存層，可以存儲來自各種來源的原始資料。在即時處理方面，雖然 Hadoop 本身不擅長即時計算，但可以與 Apache Spark、Apache Flink 等流處理框架結合，實現近即時的資料處理和分析。

Hadoop 也是許多其他大數據技術的基礎。許多流行的分散式計算框架，如 Apache Spark、Apache Hive、Apache Pig 等，都是在 Hadoop 基礎上構建的，或者與 Hadoop 深度整合。掌握 Hadoop 的原理和實踐，對於理解和使用其他大數據技術具有重要意義。

## 發展歷程

Hadoop 自 2006 年誕生以來，經歷了多個重要版本的迭代。Hadoop 1.0 時代，MapReduce 同時負責資源調度和任務執行，架構相對簡單但擴展性受限。Hadoop 2.0 引入了 YARN，將資源管理和任務執行分離，大大提高了系統的擴展性和靈活性，支援更多類型的工作負載。

Hadoop 3.0 帶來了多項重要改進，包括支援更多的 NameNode（HA Federation）、容器化執行環境、GPU 加速等。Hadoop 3.x 還優化了儲存效率，提供了基於糾錯碼的資料保護機制，可以在保證資料可靠性的同時大幅減少儲存開銷。目前，Hadoop 生態系統仍在持續發展和完善，新的特性和優化不斷被引入。

Hadoop 的發展也催生了龐大而活躍的生態系統。Apache Hive 提供了類似 SQL 的查詢語言，使得熟悉 SQL 的使用者可以輕鬆處理大數據。Apache Pig 提供了高階資料流語言，簡化了複雜資料處理任務的開發。Apache HBase 提供了面向列的分散式資料庫，支援隨機讀寫。Apache Sqoop 簡化了關係資料庫與 HDFS 之間的資料遷移。這些項目共同構成了完整的大數據處理解決方案。
