# Mockito Benutzerhandbuch

## Erstellen von Mock-Objekten

Das Erstellen von Mock-Objekten ist der erste Schritt bei der Verwendung von Mockito fuer Tests. Es gibt verschiedene Moeglichkeiten, Mock-Objekte zu erstellen, und Entwickler koennen die geeignete Methode根据ihren Beduerfnissen waehlen.

### Verwenden der mock()-Methode

Der einfachste Weg ist die Verwendung der statischen Methode mock() der Mockito-Klasse, um Mock-Objekte zu erstellen. Diese Methode akzeptiert einen Class-Parameter und gibt eine Mock-Instanz dieser Klasse zurueck.

```java
import static org.mockito.Mockito.*;

List<String> mockedList = mock(List.class);
Map<String, Integer> mockedMap = mock(Map.class);
```

Ab Mockito 4.10.0 koennen Sie den Class-Parameter weglassen und der Compiler leitet den Typ automatisch ab.

```java
List<String> mockedList = mock();
```

### Verwenden der @Mock-Annotation

Die @Mock-Annotation bietet einen saubereren Deklarationsstil, besonders nuetzlich wenn mehrere Mock-Objekte in einer Testklasse erstellt werden muessen. Die Verwendung der Annotation erfordert zuerst die Initialisierung des Mockito-Frameworks.

```java
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @Before
    public void setup() {
        MockitoAnnotations.initMocks(this);
    }
}
```

Die Verwendung von MockitoJUnitRunner kann die Initialisierung automatisch uebernehmen.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

Die Verwendung von JUnit 5's @ExtendWith-Annotation erreicht ebenfalls die automatische Initialisierung.

```java
@ExtendWith(MockitoExtension.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

## Definieren des Methodenverhaltens

Das Definieren des Methodenverhaltens wird auch als Stubbing bezeichnet. Es wird verwendet, um zu definieren, welche Werte zurueckgegeben oder welche Exceptions geworfen werden sollen, wenn Methoden auf Mock-Objekten aufgerufen werden.

### Grundlegende Rueckgabewert-Konfiguration

Verwenden Sie die when().thenReturn()-Kettensyntax, um Rueckgabewerte zu definieren.

```java
when(mockedList.size()).thenReturn(100);
when(mockedList.get(0)).thenReturn("first");
when(mockedList.get(1)).thenReturn("second");
```

Sie koennen mehrere aufeinanderfolgende Rueckgabewerte definieren, und nachfolgende Aufrufe geben unterschiedliche Werte zurueck.

```java
when(mockedList.get(anyInt()))
    .thenReturn("first")
    .thenReturn("second")
    .thenThrow(new RuntimeException("Error"));

System.out.println(mockedList.get(0)); // gibt aus: first
System.out.println(mockedList.get(0)); // gibt aus: second
System.out.println(mockedList.get(0)); // wirft Exception
```

### Definieren des Exception-Wurfs

Verwenden Sie thenThrow(), um zu definieren, dass Exceptions geworfen werden sollen, wenn Methoden aufgerufen werden.

```java
when(mockedList.get(0)).thenThrow(new IndexOutOfBoundsException("Index 0"));
```

Fuer Methoden mit void-Rueckgabetyp verwenden Sie doThrow().

```java
doThrow(new RuntimeException("Error")).when(mockedList).clear();
```

### Verwenden von Answer fuer benutzerdefinierte Antworten

Wenn vordefinierte Rueckgabewerte nicht ausreichen, koennen Sie die Answer-Schnittstelle implementieren, um Rueckgabewerte dynamisch zu berechnen.

```java
when(mockedList.get(anyInt())).thenAnswer(invocation -> {
    int index = invocation.getArgument(0);
    return "Value at index " + index;
});
```

## Verifizieren von Interaktionen

Verifizierung wird verwendet, um zu bestaetigen, dass der zu testende Code wie erwartet korrekt mit Mock-Objekten interagiert.

### Grundlegende Verifizierung

Verwenden Sie verify(), um zu ueberpruefen, ob eine Methode aufgerufen wurde.

```java
mockedList.add("one");
verify(mockedList).add("one");
```

### Ueberpruefen der Aufrufanzahl

Sie koennen ueberpruefen, wie oft eine Methode aufgerufen wurde.

```java
mockedList.add("one");
mockedList.add("one");
mockedList.add("one");

verify(mockedList, times(3)).add("one");       // genau 3 mal
verify(mockedList, atLeast(2)).add("one");     // mindestens 2 mal
verify(mockedList, atMost(3)).add("one");      // hoechstens 3 mal
verify(mockedList, never()).add("two");        // nie aufgerufen
verify(mockedList, atLeastOnce()).add("one");  // mindestens einmal
```

### Ueberpruefen der Aufrufreihenfolge

Der InOrder-Verifizierer kann ueberpruefen, ob Methodenaufrufe in einer bestimmten Reihenfolge ausgefuehrt wurden.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
inOrder.verify(mockedList).add("third");
```

### Ueberpruefen der Aufrufargumente

Argument-Matcher koennen verwendet werden, um Aufrufargumente flexibel zu verifizieren.

```java
verify(mockedList).add(anyString());
verify(mockedList).get(anyInt());
verify(mockedList).contains(argThat(s -> s.length() > 5));
```

## Argument-Matcher

Mockito bietet reichhaltige Argument-Matcher fuer flexibles Matching von Methodenparametern.

### Integrierte Matcher

```java
anyInt()          // beliebige Ganzzahl
anyString()       // beliebige Zeichenkette
any()             // beliebiges Objekt
anyList()         // beliebige Liste
anyMap()          // beliebige Map
isNull()          // Nullwert
isNotNull()       // Nicht-Nullwert
eq(value)         // exakte Uebereinstimmung
startsWith()     // beginnt mit angegebener Zeichenkette
endsWith()       // endet mit angegebener Zeichenkette
contains()       // enthaelt angegebene Zeichenkette
```

### Benutzerdefinierte Matcher

Benutzerdefinierte Argument-Matcher koennen mit der argThat()-Methode erstellt werden.

```java
when(mockedList.add(argThat(s -> s != null && s.length() > 3)))
    .thenReturn(true);
```

## Verwenden von Spy

Im Gegensatz zu Mocks behalten Spies das meiste Verhalten des echten Objekts bei, waehrend spezifische Methoden gestubbt werden koennen.

### Erstellen eines Spy

```java
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);
```

Oder verwenden Sie die @Spy-Annotation.

```java
@Spy
private List<String> list = new ArrayList<>();
```

### Definieren des Spy-Verhaltens

Im Gegensatz zu Mocks muessen Sie vorsichtig sein, wenn Sie when() mit Spies verwenden, da dies echte Methodenaufrufe beeinflussen kann. Es wird empfohlen, die doReturn().when()-Syntax zu verwenden.

```java
doReturn(100).when(spyList).size();
```

Spies rufen echte Methoden auf, es sei denn, sie sind explizit gestubbt.

```java
spyList.add("element");  // echte Methode wird aufgerufen
verify(spyList).add("element");  // Verifizierung erfolgreich
```

## Verwenden von @InjectMocks

Die @Mock-Annotation wird verwendet, um das zu testende Objekt automatisch zu erstellen und mit @Mock oder @Spy annotierte Felder darin zu injizieren.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @InjectMocks
    private UserService userService;
    
    @Test
    public void testCreateUser() {
        User user = new User("testUser");
        when(userRepository.save(any(User.class))).thenReturn(user);
        
        User result = userService.createUser("testUser");
        
        assertEquals("testUser", result.getUsername());
        verify(userRepository).save(any(User.class));
    }
}
```

## Verwenden von ArgumentCaptor

ArgumentCaptor wird verwendet, um waehrend Methodenaufrufen uebergebene Argumente zu erfassen, um detailliertere Assertions zu ermoeglichen.

```java
@Mock
private List<String> mockedList;

@Test
public void testCaptureArgument() {
    mockedList.add("test");
    
    ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
    verify(mockedList).add(captor.capture());
    
    assertEquals("test", captor.getValue());
}
```

Wenn eine Methode mehrmals aufgerufen wird, koennen Sie die Argumente aller Aufrufe erfassen.

```java
mockedList.add("first");
mockedList.add("second");
mockedList.add("third");

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList, times(3)).add(captor.capture());

assertEquals(3, captor.getAllValues().size());
assertEquals("second", captor.getAllValues().get(1));
```

## BDD-Style-Tests

Mockito unterstuetzt Behavior-Driven Development (BDD)-Style-Tests unter Verwendung der given().willReturn()-Syntax.

```java
import static org.mockito.BDDMockito.*;

@Test
public void testBddStyle() {
    // Given
    given(mockedList.get(0)).willReturn("first");
    
    // When
    String result = mockedList.get(0);
    
    // Then
    assertEquals("first", result);
}
```

## Haeufig verwendete Muster

### Testen von Void-Methoden

```java
@Test
public void testVoidMethod() {
    doNothing().when(mockedList).clear();
    mockedList.clear();
    verify(mockedList).clear();
}
```

### Mocken von Methodenketten

```java
@Test
public void testMethodChaining() {
    when(mockedList.iterator()).thenReturn(Collections.emptyIterator());
    
    Iterator<String> iterator = mockedList.iterator();
    assertFalse(iterator.hasNext());
}
```

### Mocken der Exception-Behandlung

```java
@Test(expected = RuntimeException.class)
public void testException() {
    when(mockedList.get(0)).thenThrow(new RuntimeException("Error"));
    mockedList.get(0);
}
```

## Best Practices

Beim Verwenden von Mockito befolgen Sie Best Practices, um die Qualitaet und Wartbarkeit des Testcodes sicherzustellen. Mocken Sie keine Typen, die Sie besitzen, oder Value-Objekte, da dies die Authentizitaet der Tests bricht. Mocken Sie nur notwendige Abhaengigkeiten, um die Authentizitaet des Tests zu wahren. Jede Testmethode sollte nur eine Funktion testen. Organisieren Sie den Testcode angemessen mit @Before fuer gemeinsame Einstellungen. Verifizierung sollte moderat sein, konzentrieren Sie sich auf wichtige Interaktionen statt jeden Methodenaufruf zu verifizieren.
