# Node.js 자주 묻는 질문(FAQ)

## 일반적인 질문

### Node.js란 무엇인가요?
Node.js는 웹 브라우저 외부에서 JavaScript 코드를 실행하는 오픈 소스 크로스 플랫폼 JavaScript 런타임 환경입니다. 개발자가 명령줄 도구를 작성하고 서버 측 스크립팅을 수행하여 사용자의 웹 브라우저에 전송되기 전에 동적 웹 페이지 콘텐츠를 생성할 수 있도록 합니다.

### Node.js는 왜 만들어졌나요?
Node.js는 동시 연결을 처리하기 위해 스레드 기반 모델을 사용하는 기존 웹 서버의 한계를 해결하기 위해 Ryan Dahl이 2009년에 만들었습니다. Node.js는 확장 가능한 네트워크 애플리케이션을 구축하기 위해 가볍고 효율적인 비동기 I/O 모델을 도입했습니다.

### Node.js는 싱글 스레드인가요?
Node.js 애플리케이션은 JavaScript 실행을 위해 단일 스레드에서 실행되지만, 비동기 I/O 및 이벤트 루프를 통해 여러 동시 작업을 처리할 수 있습니다. 또한 필요할 때 여러 CPU 코어를 활용할 수 있는 `worker_threads` 및 `child_process`와 같은 API를 제공합니다.

### npm이란 무엇인가요?
npm(Node Package Manager)은 Node.js의 기본 패키지 관리자이며 세계 최대의 소프트웨어 레지스트리입니다. 개발자가 코드 패키지를 공유하고 재사용하며, 종속성을 관리하고, package.json 파일에 정의된 스크립트를 실행할 수 있도록 합니다.

## 설치 및 설정

### LTS와 Current 중 어떤 버전을 설치해야 하나요?
- **LTS(장기 지원)**: 프로덕션 애플리케이션에 권장됩니다. 안정성, 보안 업데이트 및 장기 지원(총 30개월)을 제공합니다.
- **Current**: 최신 기능을 포함하지만 지원 주기가 짧습니다. 개발, 새로운 기능 테스트 또는 학습에 적합합니다.

### Node.js 버전 간에 어떻게 전환하나요?
Node 버전 관리자를 사용합니다:
- **nvm**(macOS/Linux): `nvm install 18 && nvm use 18`
- **nvm-windows**(Windows): Unix 기반 nvm과 유사한 명령어
- **fnm**(Fast Node Manager): 더 나은 성능을 제공하는 크로스 플랫폼 대안

### 전역 패키지 설치 시 권한 오류가 발생하는 이유는 무엇인가요?
이는 일반적으로 npm이 상승된 권한이 필요한 시스템 디렉토리에 쓰려고 할 때 발생합니다. 해결 방법:
1. npm을 다른 디렉토리를 사용하도록 구성: `npm config set prefix ~/.npm-global`
2. `~/.npm-global/bin`을 PATH에 추가
3. 권한을 적절히 처리하는 Node 버전 관리자 사용

### Node.js 설치를 어떻게 확인하나요?
터미널에서 다음 명령어를 실행합니다:
```bash
node --version    # Node.js 버전 표시
npm --version     # npm 버전 표시
node -e "console.log('Working!')"  # 기본 기능 테스트
```

## 개발 관련 질문

### CommonJS와 ES Modules의 차이점은 무엇인가요?
- **CommonJS**(`require()`/`module.exports`): 원래 Node.js 모듈 시스템, 동기식, 광범위하게 지원됨
- **ES Modules**(`import`/`export`): 현대 JavaScript 표준, 비동기식, `.mjs` 확장자 또는 package.json에서 `"type": "module"` 필요

CommonJS 예제:
```javascript
// math.js
exports.add = (a, b) => a + b;

// app.js
const { add } = require('./math');
```

ES Modules 예제:
```javascript
// math.mjs
export const add = (a, b) => a + b;

// app.mjs
import { add } from './math.mjs';
```

### 비동기 작업을 어떻게 처리하나요?
세 가지 주요 접근 방식:

1. **콜백**(전통적이지만 "콜백 지옥"으로 이어질 수 있음):
```javascript
fs.readFile('file.txt', 'utf8', (err, data) => {
  if (err) throw err;
  console.log(data);
});
```

2. **프로미스**(더 나은 오류 처리 및 체이닝):
```javascript
fs.promises.readFile('file.txt', 'utf8')
  .then(data => console.log(data))
  .catch(err => console.error(err));
```

3. **Async/Await**(가장 읽기 쉬움, 동기식처럼 보임):
```javascript
async function readFile() {
  try {
    const data = await fs.promises.readFile('file.txt', 'utf8');
    console.log(data);
  } catch (err) {
    console.error(err);
  }
}
```

### 이벤트 루프란 무엇이며 어떻게 작동하나요?
이벤트 루프는 메인 스레드를 차단하지 않고 비동기 작업을 처리하는 Node.js의 메커니즘입니다. 관련 작업이 완료되면 보류 중인 콜백을 지속적으로 확인하고 실행합니다.

이벤트 루프 단계:
1. **타이머**: setTimeout() 및 setInterval() 콜백 실행
2. **보류 중인 콜백**: 다음 루프 반복으로 연기된 I/O 콜백 실행
3. **유휴, 준비**: 내부 용도 전용
4. **폴**: 새 I/O 이벤트 검색; I/O 관련 콜백 실행
5. **체크**: setImmediate() 콜백 실행
6. **닫기 콜백**: 닫기 이벤트 콜백 실행(예: socket.on('close'))

### Node.js 애플리케이션을 어떻게 디버깅하나요?
여러 디버깅 옵션:

1. **내장 인스펙터**: `node --inspect app.js`
2. **Chrome DevTools**: `chrome://inspect`에 연결
3. **VS Code 디버거**: Node.js 디버깅 지원 내장
4. **콘솔 로깅**: 기본 디버깅에 간단하지만 효과적
5. **전용 디버깅 도구**: Node.js Inspector Manager(NiM)와 같은 도구

## 성능 및 확장성

### Node.js 애플리케이션을 어떻게 확장하나요?
여러 전략:

1. **클러스터링**: 내장된 `cluster` 모듈을 사용하여 서버 포트를 공유하는 자식 프로세스 생성
2. **로드 밸런싱**: NGINX와 같은 리버스 프록시를 사용하여 여러 Node.js 인스턴스 간에 트래픽 분산
3. **마이크로서비스**: 큰 애플리케이션을 더 작고 독립적인 서비스로 분할
4. **キャッシング**: Redis, Memcached와 같은 캐싱 계층을 구현하여 데이터베이스 부하 감소
5. **데이터베이스 최적화**: 연결 풀링 및 쿼리 최적화 사용

### Node.js에서 메모리 누수가 발생하는 원인은 무엇인가요?
일반적인 원인:
- **전역 변수**: 전역 범위에 데이터 축적
- **닫히지 않은 이벤트 리스너**: 객체가 파괴될 때 이벤트 리스너 제거하지 않음
- **클로저**: 큰 객체에 대한 참조를 필요 이상으로 오래 유지
- **타이머**: 간격 및 타임아웃 지우지 않음
- **버퍼**: 큰 바이너리 데이터를 적절히 관리하지 않음

감지 도구:
- 기본 모니터링을 위한 `process.memoryUsage()`
- Chrome DevTools를 통한 힙 스냅샷
- Clinic.js와 같은 메모리 프로파일링 도구

### CPU 집약적 작업을 어떻게 처리하나요?
Node.js는 JavaScript 실행을 위해 싱글 스레드이므로 CPU 집약적 작업은 이벤트 루프를 차단할 수 있습니다. 해결 방법:

1. **워커 스레드**: 병렬 처리를 위해 `worker_threads` 모듈 사용
```javascript
const { Worker } = require('worker_threads');
const worker = new Worker('./cpu-intensive-task.js');
```

2. **자식 프로세스**: `child_process`를 사용하여 별도의 프로세스 시작
```javascript
const { spawn } = require('child_process');
const child = spawn('python', ['script.py']);
```

3. **외부 서비스에 오프로드**: RabbitMQ, Kafka와 같은 메시지 큐를 사용하여 무거운 작업을 비동기적으로 처리
4. **알고리즘 최적화**: 성능 문제를 일으키는 실제 코드를 프로파일링하고 최적화

## 보안

### Node.js 애플리케이션을 어떻게 보호하나요?
핵심 보안 모범 사례:

1. **입력 검증**: 항상 사용자 입력을 검증하고 정리
2. **종속성 관리**: 종속성을 정기적으로 업데이트하고 `npm audit` 실행
3. **환경 변수**: 비밀 정보를 코드가 아닌 환경 변수에 저장
4. **속도 제한**: 속도 제한 미들웨어로 무차별 대입 공격 방지
5. **Helmet.js**: 보안 헤더 미들웨어 사용
6. **CORS**: 교차 출처 리소스 공유를 적절히 구성
7. **HTTPS**: 프로덕션에서는 항상 HTTPS 사용
8. **인증**: 적절한 인증 및 권한 부여 구현

### 일반적인 취약점을 어떻게 방지하나요?
- **SQL 인젝션**: 매개변수화된 쿼리 또는 ORM 사용
- **XSS(교차 사이트 스크립팅)**: 렌더링 전에 사용자 입력 이스케이프
- **CSRF(교차 사이트 요청 위조)**: CSRF 토큰 사용
- **프로토타입 오염**: 안전하지 않은 객체 병합 함수 피하기
- **경로 탐색**: 파일 경로를 검증하고 `path.resolve()`를 주의해서 사용

## 배포 및 프로덕션

### 어떤 프로세스 관리자를 사용해야 하나요?
인기 있는 옵션:

1. **PM2**: 가장 인기 있고 기능이 풍부하며 클러스터링 및 모니터링 지원
2. **Forever**: 간단한 프로세스 관리자, 기본적인 사용 사례에 적합
3. **Systemd**: 네이티브 Linux 서비스 관리자, OS와 잘 통합됨
4. **Docker**: 컨테이너화 접근 방식, 마이크로서비스에 탁월

### 환경별 구성을 어떻게 처리하나요?
환경 변수 및 구성 파일 사용:

```javascript
// config.js
const config = {
  development: {
    port: 3000,
    dbUrl: 'mongodb://localhost:27017/myapp'
  },
  production: {
    port: process.env.PORT || 8080,
    dbUrl: process.env.DATABASE_URL
  }
};

module.exports = config[process.env.NODE_ENV || 'development'];
```

### 어떤 모니터링 도구를 사용해야 하나요?
- **애플리케이션 성능 모니터링(APM)**: New Relic, Datadog, AppSignal
- **로깅**: Winston, Bunyan, 중앙 집중식 로깅(ELK 스택, Splunk)
- **상태 확인**: 애플리케이션 상태를 확인하는 사용자 정의 엔드포인트
- **메트릭**: 시각화를 위한 Prometheus 및 Grafana
- **오류 추적**: 예외 모니터링을 위한 Sentry, Rollbar

### 우아한 종료를 어떻게 구현하나요?
리소스를 정리하기 위해 프로세스 신호를 처리:

```javascript
const server = http.createServer(app);

const shutdown = () => {
  server.close(() => {
    console.log('Server closed');
    process.exit(0);
  });
  
  // 10초 후 강제 종료
  setTimeout(() => {
    console.error('Could not close connections in time, forcefully shutting down');
    process.exit(1);
  }, 10000);
};

process.on('SIGTERM', shutdown);
process.on('SIGINT', shutdown);
```

## 생태계 및 커뮤니티

### 가장 인기 있는 Node.js 프레임워크는 무엇인가요?
- **Express.js**: 미니멀리스트, 유연, 가장 널리 사용됨
- **Fastify**: 고성능, 낮은 오버헤드
- **Koa**: 모던, async/await 사용, Express 제작자
- **NestJS**: TypeScript 우선, 엔터프라이즈급 아키텍처
- **Hapi**: 구성 중심, 엔터프라이즈 지향

기여 방법:

4. **커뮤니티 지원**: 포럼, Stack Overflow에서 다른 사람 도움
5. **에반젤리즘**: 미트업에서 발표, 블로그 게시물 작성
6. **워킹 그룹**: 공식 Node.js 워킹 그룹 참여
