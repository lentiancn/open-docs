# Введение в Hadoop

## Что такое Hadoop

Apache Hadoop — это фреймворк с открытым исходным кодом для распределённых вычислений, разработанный Apache Software Foundation и предназначенный для обработки и хранения больших наборов данных. Hadoop предоставляет надёжные и масштабируемые решения для распределённого хранения и вычисления данных, способные обрабатывать огромные объёмы данных на кластерах из обычного оборудования. Основная философия дизайна Hadoop заключается в перемещении вычислений к данным, а не перемещении больших объёмов данных к вычислительным узлам, что значительно сокращает накладные расходы на сетевую передачу и повышает эффективность обработки.

История проекта Hadoop восходит к 2003 году, когда Google опубликовал статью о Google File System (GFS), а в 2004 году — статью о MapReduce. Вдохновлённые этими статьями, инженеры Yahoo! Даг Каттинг и Майк Кафарелла начали разработку реализации с открытым исходным кодом, которая стала Hadoop. В 2006 году Hadoop официально стал проектом Apache Software Foundation, а его название происходит от игрушечного слона сына Дага Каттинга.

Hadoop разработан на Java и обладает отличными возможностями кроссплатформенности. Его основные компоненты включают HDFS (Hadoop Distributed File System) и YARN (Yet Another Resource Negotiator), а также модель программирования MapReduce. Философия дизайна Hadoop — «всё масштабируемо»: система может линейно масштабировать ёмкость хранилища и вычислительную мощность, просто добавляя больше узлов. Эта простая, но мощная концепция сделала Hadoop предпочтительной инфраструктурой в области обработки больших данных.

## Основные компоненты

### HDFS — Распределённая файловая система

HDFS — это распределённая файловая система Hadoop, предназначенная для хранения чрезвычайно больших файлов. HDFS использует архитектуру «ведущий-ведомый» с одним NameNode и несколькими DataNode. NameNode управляет пространством имён файловой системы и метаданными, поддерживая структуру каталогов файловой системы, записывая списки блоков каждого файла и места расположения каждого блока на DataNode. DataNode отвечают за фактическое хранение данных, периодически сообщая NameNode информацию о хранимых блоках.

HDFS обладает высокой отказоустойчивостью благодаря механизму репликации данных. По умолчанию каждый блок данных реплицируется на три разных DataNode, что обеспечивает возможность восстановления данных из других реплик даже при отказе узла. HDFS оптимизирован для последовательного чтения больших файлов, что делает его подходящим для сценариев пакетной обработки, но не идеальным для операций случайного чтения/записи с низкой задержкой. HDFS также поддерживает распознавание стоек, интеллектуально распределяя реплики блоков на узлы в разных стойках для дальнейшего повышения доступности данных и отказоустойчивости системы.

Архитектура HDFS следует принципу «запись один раз, чтение много раз» (WORM), что означает, что файлы не должны изменяться после записи. Этот дизайн упрощает модель согласованности, повышает производительность чтения и упрощает репликацию и распространение данных. HDFS также предоставляет функцию создания моментальных снимков, позволяющую создавать резервные копии файловой системы только для чтения для резервного копирования и восстановления данных.

### YARN — Менеджер ресурсов

YARN — это платформа управления ресурсами, представленная в Hadoop 2.x, отвечающая за унифицированное управление ресурсами и планирование кластера. YARN разделяет управление ресурсами и выполнение приложений, позволяя Hadoop поддерживать больше типов рабочих нагрузок, помимо пакетных заданий MapReduce. Основные компоненты YARN включают ResourceManager, NodeManager и ApplicationMaster.

ResourceManager — это основной компонент YARN, отвечающий за распределение и управление ресурсами во всём кластере. Он получает запросы ресурсов от приложений и планирует их на основе доступных ресурсов кластера. ResourceManager состоит из планировщика (Scheduler) и менеджера приложений (ApplicationsManager). Планировщик отвечает за распределение ресурсов согласно заранее определённым политикам, а менеджер приложений принимает запросы на отправку приложений и запускает ApplicationMaster для приложений.

NodeManager — это агент на каждом рабочем узле, отвечающий за управление ресурсами на этом узле, включая ЦП, память, диск и сетевые ресурсы. NodeManager периодически сообщает ResourceManager об использовании ресурсов и состоянии контейнеров, а также выполняет инструкции от ResourceManager. ApplicationMaster — это менеджер каждого приложения, отвечающий за согласование ресурсов с ResourceManager и координацию с NodeManager для выполнения задач.

### MapReduce — Модель программирования

MapReduce — это основная вычислительная модель Hadoop для параллельной обработки больших данных. Модель программирования MapReduce разделяет обработку данных на две основные фазы: Map и Reduce. На фазе Map система разделяет входные данные на независимые фрагменты и обрабатывает каждый фрагмент параллельно, выводя промежуточные результаты в виде пар «ключ-значение». На фазе Reduce система агрегирует и обрабатывает промежуточные результаты с одинаковыми ключами для получения окончательных результатов вычислений.

Платформа MapReduce автоматически обрабатывает сложные детали, такие как распределение данных, планирование задач, балансировка нагрузки и отказоустойчивость, позволяя разработчикам сосредоточиться на реализации бизнес-логики. Эта абстракция значительно упрощает разработку распределённых программ — разработчикам нужно только написать функции Map и Reduce, а платформа обрабатывает всю сложность распределённого выполнения. MapReduce также поддерживает пользовательские разделители (Partitioner) и комбайнеры (Combiner) для оптимизации потока данных и сокращения сетевой передачи.

Процесс выполнения задания MapReduce включает этапы: разбиение ввода, выполнение задач Map, перемешивание промежуточных результатов, выполнение задач Reduce и вывод результатов. Фаза перемешивания — это основной процесс MapReduce, отвечающий за разделение, сортировку и слияние выходных данных задач Map по ключам и их передачу соответствующим задачам Reduce. Этот процесс требует интенсивных вычислений и сетевых ресурсов и оказывает значительное влияние на общую производительность задания.

## Техническая архитектура

### NameNode и Secondary NameNode

NameNode — это основной компонент HDFS, хранящий метаданные всей файловой системы. Он хранит полные метаданные файловой системы в памяти, включая имена файлов, атрибуты файлов, сопоставления между файлами и блоками данных, а также сопоставления между блоками и DataNode. Для обеспечения постоянства метаданных NameNode также записывает метаданные в журнал транзакций (EditLog) в локальной файловой системе. При запуске NameNode считывает файл образа (FsImage) и воспроизводит журнал транзакций для восстановления состояния файловой системы.

Secondary NameNode не является резервной копией NameNode; его основная цель — периодически объединять FsImage и EditLog, помогая NameNode сократить время запуска и размер файла EditLog. Secondary NameNode обычно работает на отдельной машине, периодически загружая FsImage и EditLog от NameNode, выполняя слияние локально и загружая новый FsImage обратно на NameNode. Этот процесс, называемый контрольной точкой, может эффективно снизить нагрузку на память NameNode и время запуска.

### Архитектура ResourceManager

ResourceManager использует двухуровневую архитектуру планировщика. Первый уровень включает планировщик ёмкости (Capacity Scheduler) и справедливый планировщик (Fair Scheduler), которые отвечают за распределение ресурсов кластера между различными очередями и приложениями. Второй уровень — это ApplicationMaster: каждое приложение имеет собственный ApplicationMaster, ответственный за согласование ресурсов с ResourceManager и управление выполнением приложения.

Планировщик ёмкости позволяет нескольким организациям совместно использовать ресурсы кластера, при этом каждая организация имеет собственную очередь, гарантируя, что каждая организация получает определённую квоту ресурсов. Справедливый планировщик делает акцент на справедливости, автоматически балансируя распределение ресурсов между всеми запущенными приложениями, чтобы каждое приложение получало относительно справедливые ресурсы. Оба планировщика можно гибко настраивать через конфигурационные файлы для удовлетворения различных требований сценариев.

### Локальность данных

Локальность данных — это важная стратегия оптимизации в Hadoop, направленная на сокращение сетевой передачи данных. Hadoop пытается назначить вычислительные задачи узлам, на которых хранятся соответствующие данные — это философия «перемещения вычислений к данным». Локальность данных имеет три уровня: узловая (вычисления и данные на одном узле), стоечная (вычисления и данные в одной стойке) и межстоечная (вычисления и данные в разных стойках).

Hadoop при планировании задач сначала учитывает данные на уровне узла; если этот узел недоступен, он переходит к данным на уровне стойки. Для реализации этой оптимизации NameNode сообщает ResourceManager информацию о местоположении каждого блока, который ResourceManager использует для принятия оптимальных решений при планировании. Локальность данных очень важна для обработки больших данных и может значительно повысить общую производительность кластера.

## Области применения

Hadoop широко используется в различных областях обработки больших данных. В области анализа данных Hadoop может обрабатывать и анализировать огромные объёмы данных журналов, данные о поведении пользователей и бизнес-транзакции, помогая компаниям извлекать ценную информацию из огромных наборов данных. В области хранилищ данных Hadoop может дополнить традиционные хранилища данных, обрабатывая структурированные и неструктурированные данные с более низкой стоимостью хранения и аналитических возможностей.

В области машинного обучения Hadoop может служить инфраструктурой для крупномасштабной разработки признаков и обучения моделей, поддерживая распределённые алгоритмы машинного обучения. В архитектурах озёр данных HDFS предоставляет недорогой и надёжный слой хранения данных для хранения необработанных данных из различных источников. Для обработки в реальном времени, хотя сам Hadoop не подходит для вычислений в реальном времени, его можно комбинировать с потоковыми платформами, такими как Apache Spark и Apache Flink, для достижения почти обработки и анализа данных в реальном времени.

Hadoop также является основой для многих других технологий больших данных. Многие популярные фреймворки распределённых вычислений, такие как Apache Spark, Apache Hive и Apache Pig, построены на Hadoop или глубоко интегрированы с ним. Освоение принципов и практики Hadoop имеет большое значение для понимания и использования других технологий больших данных.

## История развития

С момента своего создания в 2006 году Hadoop прошёл несколько важных версий. В эпоху Hadoop 1.0 MapReduce отвечал как за планирование ресурсов, так и за выполнение задач — простая архитектура, но ограниченная масштабируемость. Hadoop 2.0 представил YARN, разделяющий управление ресурсами и выполнение задач, что значительно повысило масштабируемость и гибкость системы, поддерживая больше типов рабочих нагрузок.

Hadoop 3.0 принёс значительные улучшения, включая поддержку нескольких NameNode (HA Federation), контейнеризированные среды выполнения и ускорение на GPU. Hadoop 3.x также оптимизировал эффективность хранения, обеспечивая защиту данных на основе кодирования стирания, которое может значительно сократить накладные расходы на хранение, обеспечивая при этом надёжность данных. В настоящее время экосистема Hadoop продолжает развиваться и совершенствоваться, постоянно внедряются новые функции и оптимизации.

Развитие Hadoop также породило большую и активную экосистему. Apache Hive предоставляет SQL-подобный язык запросов, позволяющий пользователям, знакомым с SQL, легко обрабатывать большие данные. Apache Pig предоставляет высокоуровневый язык потоков данных, упрощающий разработку сложных задач обработки данных. Apache HBase предоставляет распределённую колоночную базу данных с поддержкой случайного чтения/записи. Apache Sqoop упрощает миграцию данных между реляционными базами данных и HDFS. Вместе эти проекты образуют полное решение для обработки больших данных.
