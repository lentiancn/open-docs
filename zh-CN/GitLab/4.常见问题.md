# GitLab 常见问题

## 安装问题

### GitLab 安装失败怎么办？

安装失败通常由几个常见原因导致。首先检查系统是否满足最低要求，包括操作系统版本、内存和磁盘空间。如果使用自托管脚本安装失败，可以尝试手动安装依赖包，然后分步骤安装。

查看安装日志可以获取详细错误信息：

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# 查看日志
sudo tail -f /var/log/gitlab/install.log
```

常见错误包括：端口冲突（80、443、22 端口被占用）、内存不足、SSL 证书问题等。如果是端口冲突，需要先停止占用端口的服务，或者修改 GitLab 配置使用其他端口。

### 如何修改 GitLab 访问地址？

安装后如果需要修改 GitLab 的访问地址，编辑 /etc/gitlab/gitlab.rb 文件中的 external_url 参数，然后重新配置：

```bash
sudo nano /etc/gitlab/gitlab.rb
# 修改 external_url
sudo gitlab-ctl reconfigure
```

修改后需要确保 DNS 解析或本地 hosts 文件正确配置，以便新地址可以正常访问。如果从 HTTP 改为 HTTPS，还需要配置 SSL 证书。

## 使用问题

### SSH 密钥无法连接怎么办？

SSH 密钥连接失败可能由多种原因导致。首先确认公钥已正确添加到 GitLab 账户的 SSH 密钥设置中：

```bash
# 查看本地公钥
cat ~/.ssh/id_ed25519.pub

# 测试连接
ssh -T git@gitlab.example.com
```

如果测试连接失败，检查 SSH 客户端配置：

```bash
# 编辑 SSH 配置
nano ~/.ssh/config

# 添加以下内容
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

确保远程仓库 URL 使用 SSH 格式（git@gitlab.example.com:group/project.git），而不是 HTTPS 格式。

### Merge Request 无法合并怎么办？

合并请求可能因为以下原因无法合并：存在冲突、CI/CD 测试失败、未满足保护分支规则、缺少审批等。解决方法是逐一排查：

- 合并冲突：在合并请求页面点击“更新分支”，将目标分支合并到源分支，解决冲突后重新推送
- CI 失败：查看失败的作业日志，修复问题后重新运行流水线
- 分支保护：联系项目管理员修改保护分支设置，或由有权限的成员合并

### CI/CD 流水线运行失败如何排查？

流水线失败时，首先在 CI/CD → 流水线页面查看失败作业的日志。日志会显示具体的错误信息和失败原因。常见问题包括：依赖下载失败、脚本命令错误、测试用例失败等。

如果问题难以定位，可以在流水线配置中添加调试步骤，例如输出环境变量、安装前后检查文件状态等。还可以使用 interactive debugging 功能，在流水线作业中启动交互式 shell 进行排查。

## 性能问题

### GitLab 访问速度很慢怎么办？

访问速度慢可能由服务器资源不足、网络延迟或配置不当导致。首先检查服务器负载：

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

如果 CPU 或内存使用率很高，可能需要升级服务器配置或优化 GitLab 设置。还可以检查数据库性能：

```bash
sudo gitlab-rake gitlab:git:fsck  # 检查 Git 仓库完整性
sudo gitlab-rake db:index_cache_daily  # 优化数据库缓存
```

对于网络问题，可以考虑使用 CDN 加速静态资源，或在更接近用户的网络位置部署 GitLab。

### 如何优化 GitLab 性能？

优化 GitLab 性能可以从多个方面入手。数据库优化是重要的一环，定期维护可以显著提升性能：

```bash
# 清理数据库
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# 清理未使用的文件
sudo gitlab-rake gitlab:cleanup:builds
```

对于大型仓库，可以启用 Git 对象池减少磁盘占用，或者配置 Git LFS 存储大型文件。合理设置 CI/CD 缓存可以加快构建速度。

## 安全问题

### 如何配置 SSL 证书？

GitLab 支持 Let's Encrypt 自动配置 SSL，也可以使用自定义证书。首先编辑配置文件：

```ruby
# Let's Encrypt 自动配置
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# 自定义证书
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

配置后运行 sudo gitlab-ctl reconfigure 使配置生效。如果使用自签名证书，客户端需要信任该证书才能通过 HTTPS 访问。

### 如何备份和恢复 GitLab？

定期备份是防止数据丢失的重要措施。GitLab 提供内置备份工具：

```bash
# 创建备份
sudo gitlab-backup create

# 备份文件位置
ls -la /var/opt/gitlab/backups/
```

备份文件包含数据库、上传文件和 Git 仓库。恢复时，停止 GitLab 服务，复制备份文件到默认位置，然后运行恢复命令：

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

建议将备份文件自动同步到远程存储，如对象存储服务或另一台服务器。

## 升级问题

### 升级后出现错误怎么办？

升级后可能出现兼容性错误或功能异常。首先检查 GitLab 状态：

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

查看升级日志了解具体错误：

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

常见问题包括：数据库迁移失败、配置文件语法错误、依赖缺失等。如果是数据库迁移问题，可能需要手动修复：

```bash
sudo gitlab-rake db:migrate
```

建议在升级前备份数据，并仔细阅读官方升级文档，了解版本间的重大变更。

### 如何在不同版本间安全升级？

升级 GitLab 应遵循版本跳跃限制，每次升级跳跃不超过一个主版本。例如从 15.x 升级到 16.x，而不是直接从 15.x 升级到 17.x。先升级到中间版本，确认正常运行后再继续升级。

升级前执行以下步骤：备份数据、检查兼容性问题、阅读发布说明。升级过程中保持电源和网络稳定，升级完成后验证所有功能正常。
