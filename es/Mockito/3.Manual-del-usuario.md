# Manual del usuario de Mockito

## Creacion de objetos Mock

Crear objetos mock es el primer paso para usar Mockito en pruebas. Hay multiples formas de crear objetos mock, y los desarrolladores pueden elegir el metodo apropiado segun sus necesidades.

### Usando el metodo mock()

La forma mas basica es usar el metodo estatico mock() de la clase Mockito para crear objetos mock. Este metodo acepta un parametro Class y devuelve una instancia mock de esa clase.

```java
import static org.mockito.Mockito.*;

List<String> mockedList = mock(List.class);
Map<String, Integer> mockedMap = mock(Map.class);
```

A partir de Mockito 4.10.0, puede omitir el parametro Class y el compilador inferira automaticamente el tipo.

```java
List<String> mockedList = mock();
```

### Usando la anotacion @Mock

La anotacion @Mock proporciona un estilo de declaracion mas limpio, especialmente util cuando se necesitan crear multiples objetos mock en una clase de prueba. Usar la anotacion requiere inicializar el framework Mockito primero.

```java
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @Before
    public void setup() {
        MockitoAnnotations.initMocks(this);
    }
}
```

Usar MockitoJUnitRunner puede manejar la inicializacion automaticamente.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

Usar la anotacion @ExtendWith de JUnit 5 tambien logra la inicializacion automatica.

```java
@ExtendWith(MockitoExtension.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
}
```

## Definicion del comportamiento del metodo

Definir el comportamiento del metodo aussi connu como stubbing. Se usa para definir que valores deben devolverse o que excepciones deben lanzarse cuando se llaman metodos en objetos mock.

### Configuracion basica del valor de retorno

Use la sintaxis de cadena when().thenReturn() para definir valores de retorno.

```java
when(mockedList.size()).thenReturn(100);
when(mockedList.get(0)).thenReturn("first");
when(mockedList.get(1)).thenReturn("second");
```

Puede definir multiples valores de retorno consecutivos, y cada llamada subsiguiente devolvera valores diferentes.

```java
when(mockedList.get(anyInt()))
    .thenReturn("first")
    .thenReturn("second")
    .thenThrow(new RuntimeException("Error"));

System.out.println(mockedList.get(0)); // salida: first
System.out.println(mockedList.get(0)); // salida: second
System.out.println(mockedList.get(0)); // lanza excepcion
```

### Definicion de lanzamiento de excepciones

Use thenThrow() para definir excepciones que deben lanzarse cuando se llaman metodos.

```java
when(mockedList.get(0)).thenThrow(new IndexOutOfBoundsException("Index 0"));
```

Para metodos con tipo de retorno void, use doThrow().

```java
doThrow(new RuntimeException("Error")).when(mockedList).clear();
```

### Usando Answer para respuestas personalizadas

Si los valores de retorno predefinidos son insuficientes, puede implementar la interfaz Answer para calcular dinamicamente valores de retorno.

```java
when(mockedList.get(anyInt())).thenAnswer(invocation -> {
    int index = invocation.getArgument(0);
    return "Value at index " + index;
});
```

## Verificacion de interacciones

La verificacion se usa para confirmar que el codigo bajo prueba interactua correctamente con los objetos mock segun lo esperado.

### Verificacion basica

Use verify() para verificar si un metodo fue llamado.

```java
mockedList.add("one");
verify(mockedList).add("one");
```

### Verificacion del numero de llamadas

Puede verificar cuantas veces se llamo un metodo.

```java
mockedList.add("one");
mockedList.add("one");
mockedList.add("one");

verify(mockedList, times(3)).add("one");       // exactamente 3 veces
verify(mockedList, atLeast(2)).add("one");    // al menos 2 veces
verify(mockedList, atMost(3)).add("one");     // como maximo 3 veces
verify(mockedList, never()).add("two");       // nunca llamado
verify(mockedList, atLeastOnce()).add("one"); // al menos una vez
```

### Verificacion del orden de llamadas

El verificador InOrder puede verificar si las llamadas a metodos se ejecutaron en un orden especifico.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
inOrder.verify(mockedList).add("third");
```

### Verificacion de argumentos de llamada

Se pueden usar emparejadores de argumentos para verificar flexiblemente los argumentos de llamada.

```java
verify(mockedList).add(anyString());
verify(mockedList).get(anyInt());
verify(mockedList).contains(argThat(s -> s.length() > 5));
```

## Emparejadores de argumentos

Mockito proporciona ricos emparejadores de argumentos para la coincidencia flexible de parametros de metodos.

### Emparejadores incorporados

```java
anyInt()          // cualquier entero
anyString()       // cualquier cadena
any()             // cualquier objeto
anyList()         // cualquier lista
anyMap()          // cualquier mapa
isNull()          // valor null
isNotNull()       // valor no null
eq(value)         // coincidencia exacta
startsWith()      // comienza con cadena especificada
endsWith()        // termina con cadena especificada
contains()        // contiene cadena especificada
```

### Emparejadores personalizados

Se pueden crear emparejadores de argumentos personalizados usando el metodo argThat().

```java
when(mockedList.add(argThat(s -> s != null && s.length() > 3)))
    .thenReturn(true);
```

## Usando Spy

A diferencia de los mocks, los spies retienen la mayor parte del comportamiento del objeto real mientras permiten que metodos especificos sean stubbed.

### Creando un Spy

```java
List<String> realList = new ArrayList<>();
List<String> spyList = spy(realList);
```

O use la anotacion @Spy.

```java
@Spy
private List<String> list = new ArrayList<>();
```

### Definiendo el comportamiento del Spy

A diferencia de los mocks, debe tener cuidado al usar when() con spies porque puede afectar las llamadas a metodos reales. Se recomienda usar la sintaxis doReturn().when().

```java
doReturn(100).when(spyList).size();
```

Los Spy llaman a metodos reales a menos que se stubbeen explicitamente.

```java
spyList.add("element");  // el metodo real sera llamado
verify(spyList).add("element");  // la verificacion pasa
```

## Usando @InjectMocks

La anotacion @InjectMocks se usa para crear automaticamente el objeto bajo prueba e inyectar campos anotados con @Mock o @Spy en el.

```java
@RunWith(MockitoJUnitRunner.class)
public class UserServiceTest {
    @Mock
    private UserRepository userRepository;
    
    @Mock
    private EmailService emailService;
    
    @InjectMocks
    private UserService userService;
    
    @Test
    public void testCreateUser() {
        User user = new User("testUser");
        when(userRepository.save(any(User.class))).thenReturn(user);
        
        User result = userService.createUser("testUser");
        
        assertEquals("testUser", result.getUsername());
        verify(userRepository).save(any(User.class));
    }
}
```

## Usando ArgumentCaptor

ArgumentCaptor se usa para capturar argumentos pasados durante llamadas a metodos para aserciones mas detalladas.

```java
@Mock
private List<String> mockedList;

@Test
public void testCaptureArgument() {
    mockedList.add("test");
    
    ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
    verify(mockedList).add(captor.capture());
    
    assertEquals("test", captor.getValue());
}
```

Cuando un metodo se llama multiples veces, puede capturar los argumentos de todas las llamadas.

```java
mockedList.add("first");
mockedList.add("second");
mockedList.add("third");

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList, times(3)).add(captor.capture());

assertEquals(3, captor.getAllValues().size());
assertEquals("second", captor.getAllValues().get(1));
```

## Pruebas estilo BDD

Mockito soporta pruebas estilo Desarrollo Guiado por Comportamiento (BDD) usando la sintaxis given().willReturn().

```java
import static org.mockito.BDDMockito.*;

@Test
public void testBddStyle() {
    // Given
    given(mockedList.get(0)).willReturn("first");
    
    // When
    String result = mockedList.get(0);
    
    // Then
    assertEquals("first", result);
}
```

## Patrones de uso comunes

### Probando metodos void

```java
@Test
public void testVoidMethod() {
    doNothing().when(mockedList).clear();
    mockedList.clear();
    verify(mockedList).clear();
}
```

### Simulando cadenas de metodos

```java
@Test
public void testMethodChaining() {
    when(mockedList.iterator()).thenReturn(Collections.emptyIterator());
    
    Iterator<String> iterator = mockedList.iterator();
    assertFalse(iterator.hasNext());
}
```

### Simulando manejo de excepciones

```java
@Test(expected = RuntimeException.class)
public void testException() {
    when(mockedList.get(0)).thenThrow(new RuntimeException("Error"));
    mockedList.get(0);
}
```

## Mejores practicas

Al usar Mockito, siga las mejores practicas para asegurar la calidad y mantenibilidad del codigo de prueba. No simule tipos que posee o objetos de valor, ya que esto rompe la autenticidad de las pruebas. Solo simule las dependencias necesarias para mantener la autenticidad de la prueba. Cada metodo de prueba debe enfocarse en probar una caracteristica. Organice el codigo de prueba adecuadamente usando @Before para configuracion comun. La verificacion debe ser moderada, enfocandose en interacciones importantes en lugar de verificar cada llamada a metodo.
