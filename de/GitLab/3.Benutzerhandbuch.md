# GitLab Benutzerhandbuch

> Umfassende Anleitung zur Verwendung von GitLab

---

## Schnellstart

### Erstes Projekt erstellen

1. Klicken Sie nach dem Anmelden auf "Neues Projekt"
2. Wählen Sie eine Projektvorlage oder ein leeres Projekt
3. Füllen Sie Projektname, Beschreibung und Sichtbarkeitsstufe aus
4. Klicken Sie auf "Projekt erstellen"

### Projekt lokal klonen

```bash
# HTTPS klonen
git clone https://gitlab.example.com/benutzername/projekt.git

# SSH klonen
git clone git@gitlab.example.com:benutzername/projekt.git
```

## Code-Management

### Branch-Operationen

```bash
# Neuen Branch erstellen
git checkout -b feature/neue-funktion

# Branch zum Remote pushen
git push -u origin feature/neue-funktion

# Alle Branches anzeigen
git branch -a

# Lokalen Branch löschen
git branch -d feature/neue-funktion

# Remote-Branch löschen
git push origin --delete feature/neue-funktion
```

### Merge Requests

1. Nach dem Entwickeln lokalen Branch zum Remote pushen
2. Projektseite öffnen, auf "Merge Request erstellen" klicken
3. Titel und Beschreibung ausfüllen, Quell- und Zielbranch wählen
4. Reviewer festlegen, relevante Labels und Meilensteine hinzufügen
5. Merge Request erstellen und auf Code-Review warten

### Code-Review

- **Inline-Kommentare**: Reviewer können direkt auf Codezeilen kommentieren
- **Genehmigungsregeln**: Anforderbare Genehmigungsanzahl für Merges konfigurierbar
- **Pipeline-Prüfungen**: Merges können an erfolgreiche Pipelines gebunden werden
- **Konflikte lösen**: Vor dem Merge müssen mögliche Konflikte behoben werden

## Issue-Tracking

### Issue erstellen

1. Gehen Sie zu Projekt → Issues → Neues Issue
2. Titel und Beschreibung ausfüllen (Markdown unterstützt)
3. Labels, Meilensteine und Assignees festlegen
4. Gewichtung und Fälligkeitsdatum setzen

### Boards verwenden

1. Gehen Sie zu Projekt → Issues → Board
2. Spalten erstellen (z.B. Offen, In Bearbeitung, Erledigt)
3. Issue-Karten per Drag & Drop in entsprechende Spalten ziehen
4. Nach Mitgliedern oder Labels filtern

### Meilensteine

- Meilensteine mit Start- und Enddatum erstellen
- Relevante Issues mit Meilensteinen verknüpfen
- Meilenstein-Fortschritt verfolgen

## CI/CD-Pipelines

### .gitlab-ci.yml Grundstruktur

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - echo "Building..."
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/
    expire_in: 1 week

test:
  stage: test
  script:
    - echo "Testing..."
    - npm test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - echo "Deploying..."
  only:
    - main
  environment:
    name: production
```

### Runner-Konfiguration

#### Docker Executor

```yaml
job:
  image: node:18
  services:
    - postgres:14
  variables:
    POSTGRES_DB: test_db
    POSTGRES_USER: test_user
    POSTGRES_PASSWORD: test_password
```

#### Cache-Konfiguration

```yaml
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/
    - .npm/
```

### Deployment-Umgebungen

- **Produktion**: Für offizielle Releases
- **Staging**: Für Tests vor der Veröffentlichung
- **Entwicklung**: Für tägliche Entwicklung

## Container-Registry

### Image pushen

```bash
# Bei GitLab Registry anmelden
docker login registry.gitlab.example.com

# Image taggen
docker tag my-image:latest registry.gitlab.example.com/benutzer/projekt/my-image:latest

# Image pushen
docker push registry.gitlab.example.com/benutzer/projekt/my-image:latest
```

### Image pullen

```bash
docker pull registry.gitlab.example.com/benutzer/projekt/my-image:latest
```

## Sicherheits-Scanning

### Automatische Scanning-Funktionen

- **Abhängigkeits-Scanning**: Sicherheitslücken in Projektabhängigkeiten prüfen
- **Container-Scanning**: Sicherheitsprobleme in Docker-Images prüfen
- **SAST**: Statische Anwendungssicherheitstests
- **DAST**: Dynamische Anwendungssicherheitstests

### Sicherheits-Dashboard anzeigen

1. Gehen Sie zu Projekt → Sicherheit & Compliance → Sicherheits-Dashboard
2. Alle Schwachstellen anzeigen
3. Nach Schweregrad filtern
4. Issue zur Verfolgung erstellen

## Projektmanagement-Tipps

### Vorlagen verwenden

- Standard-Branch-Schutzregeln für neue Projekte festlegen
- Projektvorlagen konfigurieren
- Projektvererbung nutzen

### Mitglieder verwalten

- Projektmitglieder hinzufügen und Zugriffsebenen festlegen
- Gruppen zur Verwaltung mehrerer Projekte nutzen
- Zugriffstokens konfigurieren

### Webhooks

- Webhook-Integrationen mit externen Diensten konfigurieren
- Unterstützte Trigger: push, merge request, tag usw.

## Best Practices

1. **Feature-Branches verwenden**: Jede Funktion oder Korrektur in separaten Branches entwickeln
2. **Regelmäßige Commits**: Häufig committen, klare Commit-Nachrichten verwenden
3. **Code-Review**: Jeder Code muss vor dem Merge reviewed werden
4. **Automatisierte Tests**: CI/CD-Pipelines müssen Testschritte enthalten
5. **Dokumentation pflegen**: Wiki zur Projekt dokumentation nutzen
6. **Sicherheit**: Regelmäßig Sicherheits-Scan-Berichte prüfen
