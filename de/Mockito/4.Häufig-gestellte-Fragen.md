# Haeufig gestellte Fragen zu Mockito

## Grundlegende Fragen

### Was ist der Unterschied zwischen Mockito und PowerMock

Mockito wird hauptsaechlich zum Mocken regulaerer Objekte verwendet, waehrend PowerMock statische Methoden, Konstruktoren, private Methoden und mehr mocken kann. PowerMock verwendet jedoch Bytecode-Manipulationstechniken, die die Testisolierung aufheben, was Tests naeher an Integrationstests als an Unit-Tests macht. Die offizielle Empfehlung ist, zuerst Mockito zu bevorzugen und PowerMock nur in Betracht zu ziehen, wenn Mockito bestimmte Szenarien wirklich nicht handhaben kann, wie das Mocken statischer Methoden. Seit Mockito 2.x wird das Mocken statischer Methoden auch über mockito-inline unterstützt.

### Wann sollte ich Spy statt Mock verwenden

Verwenden Sie Spy, wenn Sie den Grossteil des echten Verhaltens benoetigen, aber nur wenige Methoden mocken moechten. Spy behieltiebt den Grossteil der Funktionalitaet des echten Objekts bei, waehrend spezifische Methoden gestubbt werden koennen. Mock ersetzt das echte Objekt vollstaendig, sodass keine echte Methodenlogik ausgefuehrt wird. Verwenden Sie Mock, wenn Sie die volle Kontrolle über das Objektverhalten benoetigen. In der Praxis wird Spy relativ selten verwendet, da in den meisten Faellen die Verwendung echter Testobjekte oder vollständiger Mocks die Anforderungen erfuellt.

### Kann Mockito fuer finale Klassen und Methoden verwendet werden

Seit Mockito 2.1.0 koennen Sie Mockito konfigurieren, um finale Klassen und Methoden zu mocken. Sie muessen die mockito-inline-Abhaigkeit anstelle von mockito-core zu Ihrem Projekt hinzufuegen. mockito-inline aktiviert den dynamischen Unterklassen-Mechanismus, um finale Klassen und Methoden zu behandeln. Diese Funktion erweitert die Anwendbarkeit von Mockito erheblich und macht fast jede Java-Klasse mockbar.

### Warum geben Mock-Objekt-Methoden null zurueck

Standardmaessig geben alle Methoden von Mock-Objekten null oder Standardwerte fuer ihre Typen zurueck (int gibt 0 zurueck, boolean gibt false zurueck, Sammlungstypen geben leere Sammlungen zurueck, usw.). Dies ist das Standard-Antwortverhalten von Mockito. Wenn Sie bestimmte Rückgabewerte benoetigen, muessen Sie diese explizit mit when().thenReturn() festlegen.

## Verwendungsfragen

### Die @Mock-Annotation funktioniert nicht, was soll ich tun

Die @Mock-Annotation erfordert die Initialisierung des Mockito-Frameworks, damit sie wirksam wird. Stellen Sie sicher, dass Sie eines von MockitoAnnotations.initMocks(this), @RunWith(MockitoJUnitRunner.class) oder @ExtendWith(MockitoExtension.class) ( fuer JUnit 5) in Ihren Tests verwenden. Bei korrekter Initialisierung erstellen mit @Mock annotierte Felder automatisch Mock-Objekte, bevor Testmethoden ausgefuehrt werden.

### Wie man void-Methoden mockt

Fuer Methoden mit void-Rueckgabetyp verwenden Sie doThrow(), doAnswer() oder doNothing(), um Verhalten zu definieren.

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### Wie man Methodenaufrufargumente verifiziert

Sie koennen Argument-Matcher zur Verifizierung verwenden oder ArgumentCaptor verwenden, um Argumente fuer detaillierte Assertions zu erfassen.

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### Wie man Iteratoren mockt

Das Mocken von Iteratoren erfordert Aufmerksamkeit auf die Koordination zwischen hasNext()- und next()-Methoden.

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### Wie man Methodenüberladung handhabt

Wenn eine Klasse überladene Methoden hat, unterscheidet Mockito verschiedene Methoden basierend auf Parametertypen und -anzahl. Stellen Sie sicher, dass die Parametertypen in when() genau mit der Signatur der zu mockenden Methode übereinstimmen.

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // überladene Methode
```

## Fortgeschrittene Fragen

### Warum funktioniert when().thenReturn() nicht

Haeufige Gruende sind: Festlegen von Verhalten fuer dieselbe Methode mehrmals (nur das letzte gilt); Verwenden von Spy und Aufrufen der echten Methode; Methode gibt void zurueck (sollte doReturn().when() verwenden); Argument-Matcher falsch verwendet fuehrt zu Matchfehlern. Ueberpruefen Sie Ihren Code, um sicherzustellen, dass die Syntax fuer die Verhaltensfestlegung korrekt ist.

### Wie man abstrakte Klassen testet

Beim Testen abstrakter Klassen koennen Sie Mockito verwenden, um konkrete Methoden der abstrakten Klasse zu mocken, oder Spy verwenden, um einige echte Verhaltensweisen beizubehalten. Ein anderer Ansatz ist das Erstellen einer konkreten Testunterklasse zum Ausfuehren von Tests.

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### Wie man generische Methoden mockt

Beim Mocken generischer Methoden seien Sie sich der durch Typloeschung verursachten Einschraenkungen bewusst. Verwenden Sie any() oder anyObject(), um generische Parameter zu matchen.

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### Wie man die Aufrufreihenfolge verifiziert

Verwenden Sie InOrder, um zu verifizieren, dass Methodenaufrufe in einer bestimmten Reihenfolge ausgefuehrt wurden.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### Wie man Objekte teilweise mockt

Verwenden Sie Spy fuer teilweises Mocking oder mitSettings(), um das Mock-Verhalten anzupassen.

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## Leistungsfragen

### Tests laufen langsam, was soll ich tun

Stellen Sie sicher, dass Sie keine unnötigen echten Methodenaufrufe verwenden. Ueberpruefen Sie, ob Sie in jedem Test neue Mock-Objekte erstellen. Fuers Projekte, die viel Mockito verwenden, sollten Sie die Verwendung von @RunWith(MockitoJUnitRunner.Silent.class) in Betracht ziehen, um die Protokollausgabe zu reduzieren.

### Wie man den Speicherverbrauch von Mock-Objekten reduziert

Mockito-Mock-Objekte selbst verbrauchen nicht viel Speicher. Wenn das Erstellen einer grossen Anzahl von Mock-Objekten jedoch die Leistung beeintraechtigt, sollten Sie die Verwendung von @Mock-Annotation mit @Scope in Betracht ziehen, um Mock-Objekte wiederzuverwenden, oder statische Mock-Fabrikmethoden verwenden.

## Debugging-Fragen

### Wie man Mock-Objekt-Aufrufprotokolle anzeigt

Sie koennen die Protokollierungsfunktion von Mockito verwenden oder Listener hinzufuegen, um Aufrufe zu verfolgen.

```java
Mockito.framework().addListener(new VerificationListener());
```

Oder verwenden Sie die Maven/Gradle-Testausgabe, um detaillierte Mock-Aufrufinformationen anzuzeigen.

### Wie man debuggt, wenn Tests fehlschlagen

Mockito bietet detaillierte Fehlermeldungen. Ueberpruefen Sie die Unterschiede zwischen erwarteten und tatsaechlichen Aufrufen, die bei verify-Fehlern angezeigt werden. Verwenden Sie die Funktion Mockito-Protokolle drucken, um das Verhalten des Mock-Objekts zu verstehen. Vereinfachen Sie Testfaelle schrittweise, um Probleme zu lokalisieren.
