# 쿠버네티스 자주 묻는 질문

## 기본 개념

### 쿠버네티스란?

쿠버네티스(K8s)는 컨테이너화된 애플리케이션을 자동으로 배포, 확장, 관리하는 오픈소스 시스템이다. 애플리케이션을 구성하는 컨테이너를 논리적 단위로 그룹화하여 관리와 발견을 쉽게 만든다.

### 파드와 컨테이너의 차이점은?

파드는 쿠버네티스에서 배포 가능한 가장 작은 단위이다. 파드는 하나 이상의 컨테이너를 포함할 수 있다. 파드당 하나의 컨테이너만 실행하는 것이 권장된다.同一 파드 내의 컨테이너는 네트워크와 스토리지 리소스를 공유한다.

### 디플로이먼트와 스테이트풀셋의 차이점은?

디플로이먼트는 상태 없는 애플리케이션 관리에 사용되며, 선언적 업데이트, 스케일링, 롤백을 지원한다. 스테이트풀셋은 상태 있는 애플리케이션 관리에 사용되며, 안정적인 네트워크 ID와 영구 스토리지를 제공하여 영구가 필요한 데이터베이스 등에 적합하다.

### 네임스페이스란?

네임스페이스는 클러스터 리소스를 여러 논리적 파티션으로 나누어 다양한 팀이나 프로젝트 간 리소스 격리에 도움이 된다. 일반적인 기본 네임스페이스에는 default, kube-system, kube-public이 있다.

## 설치 및 구성

### Minikube와 Kind의 차이점은?

Minikube는 완전한 단일 노드 Kubernetes 클러스터를 만들어 학습 및 개발에 적합하다. Kind (Kubernetes in Docker)는 Docker 컨테이너를 사용하여 Kubernetes 노드를 시뮬레이션하며, 가볍고 빠른 테스트에 적합하다.

### 컨테이너 런타임은 어떻게 선택하나?

일반적인 컨테이너 런타임에는 Docker, containerd, CRI-O가 있다. containerd는 Docker의 핵심 컨테이너 런타임이며, 현재 널리 권장된다.

### kubeadm init이 실패하면 어떻게 하나?

일반적인 원인:
- 포트가 이미 사용 중: 6443, 10250, 10255 포트 확인
- 스왑이 비활성화되지 않음: `swapoff -a` 실행
- 방화벽 규칙: 노드 간 네트워크 통신 확인
- SELinux/AppArmor: 허용 모드로 설정하거나 비활성화 가능

### 다중 컨트롤 플레인 클러스터는 어떻게 구성하나?

kubeadm을 사용하여 고가용성 클러스터 생성:
1. 여러 서버 노드 준비
2. kubeadm init으로 첫 번째 컨트롤 플레인 노드 생성
3. 인증서 생성 및 다른 노드에 복사
4. kubeadm join으로 추가 컨트롤 플레인 추가

## 네트워크 및 스토리지

### 파드가 서로 통신하지 못하면 어떻게 하나?

일반적인 원인:
- 네트워크 플러그인이 올바르게 설치되지 않음
- 방화벽이 파드 네트워크 CIDR 통신을 차단
- CNI 플러그인 구성 오류
- kube-proxy가 올바르게 실행되고 있는지 확인

### 파드에 영구 스토리지를 어떻게 구성하나?

퍼시스턴트볼륨(PV)과 퍼시스턴트볼륨클레임(PVC)을 생성해야 한다. PV는 로컬 스토리지, 네트워크 스토리지(NFS, iSCSI), 또는 클라우드 스토리지(AWS EBS, GCE PD, Azure Disk)일 수 있다.

### 서비스 유형은 무엇이 있나요?

- ClusterIP: 클러스터 내에서 접근 가능한 가상 IP
- NodePort: 각 노드에서 포트 열기
- LoadBalancer: 클라우드 공급자의 로드 밸런서 사용
- ExternalName: 서비스를 외부 도메인 이름에 매핑

## 리소스 관리

### 컨테이너 리소스 제한은 어떻게 설정하나?

파드 또는 디플로이먼트 spec에 resources 필드 추가:
```yaml
resources:
  limits:
    memory: "512Mi"
    cpu: "500m"
  requests:
    memory: "256Mi"
    cpu: "250m"
```

### 리소스쿼터란?

리소스쿼터는 네임스페이스의 총 리소스를 제한하여 단일 팀이나 프로젝트가 너무 많은 리소스를 소비하는 것을 방지한다.

### 리밋레인지란?

리밋레인지는 네임스페이스의 파드와 컨테이너에 기본 리소스 제한 및 요청 값을 설정하는 데 사용된다.

## 배포 및 운영

### 애플리케이션의 롤링 업데이트는 어떻게 구현하나?

디플로이먼트를使用时, 쿠버네티스는 자동으로 롤링 업데이트를 수행한다:
```bash
kubectl set image deployment/my-app my-app=my-app:v2
kubectl rollout status deployment/my-app
```

### 배포를 롤백하려면 어떻게 하나?

```bash
# 이전 버전으로 롤백
kubectl rollout undo deployment/my-app

# 특정 리비전으로 롤백
kubectl rollout undo deployment/my-app --to-revision=2
```

### HPA란?

Horizontal Pod Autoscaler(HPA)는 CPU, 메모리 등의 메트릭스를 기반으로 파드 복제본 수를 자동으로 조정한다. Metrics Server가 필요하다.

### 헬스 체크는 어떻게 구성하나?

생성/readiness 프로브 구성:
```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

## 보안

### RBAC이란?

역할 기반 접근 제어(RBAC)는 Kubernetes 클러스터에서 권한을 관리하는 데 사용된다. Role과 ClusterRole을 생성하여 사용자 또는 그룹에 바인딩할 수 있다.

### 민감 정보는 어떻게 관리하나?

시크릿을 사용하여 비밀번호, 키, 인증서 등의 민감한 정보를 저장한다. kubectl 명령줄로 생성하거나 YAML 파일로 정의할 수 있다.

### 클러스터는 어떻게 보호하나?

- RBAC 인증 활성화
- 네트워크 정책을 사용하여 파드 간 통신 제한
- Kubernetes 버전 정기적으로 업데이트
- etcd 데이터 암호화
- 통신에 TLS 사용
- kubelet 접근 권한 제한

## 문제 해결

### 파드가 Pending 상태로 유지되면?

가능한 원인:
- 리소스 부족 (CPU 또는 메모리)
- 노드 셀렉터 또는 테인트 불일치
- PVC가 올바르게 바인되지 않음
- `kubectl describe pod <pod-name>`로 세부 정보 확인

### 파드가 ImagePullBackOff 상태로 유지되면?

일반적인 원인:
- 이미지 이름 잘못됨
- 이미지 태그가 존재하지 않음
- 프라이빗 레지스트리 인증 문제
- 네트워크 문제로 이미지 다운로드 불가

### 파드가 CrashLoopBackOff 상태로 유지되면?

- 컨테이너 로그 확인: kubectl logs --previous <pod-name>
- 애플리케이션이 올바르게 시작하는지 확인
- 리소스 구성이 올바른지 확인
- 종속 서비스가 사용 가능한지 확인

### 네트워크 문제는 어떻게 디버깅하나?

- kube-proxy가 올바르게 실행되고 있는지 확인
- CNI 플러그인 상태 확인
- kubectl exec을 사용하여 컨테이너 내에서 네트워크 연결 테스트
- iptables 규칙 보기
- 네트워크 진단 도구 사용 (curl, ping, traceroute)

## 성능 최적화

### 클러스터 성능은 어떻게 향상시키나요?

- 적절한 리소스 요청 및 제한 구성
- 안티 어피니티를 사용하여 파드 분산
- API 캐싱 활성화하여 요청 감소
- 로컬 스토리 사용하여 네트워크 지연 감소
- 이미지 크기 최적화하여 다운로드 시간 단축

### 대규모 클러스터는 어떻게 처리하나?

- 페더레이션을 사용하여 여러 클러스터 관리
- 적절한 etcd 하드웨어 구성
- API 캐싱 활성화
- 로드 밸런서를 사용하여 요청 분산
