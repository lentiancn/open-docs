# Manuel Utilisateur GitLab

## Gestion des Comptes

### Inscription et Connexion

Pour utiliser GitLab pour la première fois, cliquez sur "S'inscrire" sur la page de connexion pour créer un compte. L'inscription nécessite de fournir un nom d'utilisateur, une adresse e-mail et un mot de passe. Le choix du nom d'utilisateur doit être fait avec soin, car il ne peut pas être modifié après la création. Après l'inscription, vous recevrez un e-mail de vérification. Cliquez sur le lien dans l'e-mail pour compléter la vérification.

Après la connexion, vous pouvez améliorer votre profil dans les paramètres personnels, y compris l'avatar, le nom, le nom complet et la bio. Il est recommandé de compléter les informations, car elles seront affichées dans les listes de membres du projet et les historiques de commits. Vous pouvez également lier des clés SSH pour pousser du code via la ligne de commande.

### Configuration de Clé SSH

Les clés SSH permettent une authentification sécurisée avec les dépôts GitLab sans saisir le nom d'utilisateur et le mot de passe à chaque fois. Tout d'abord, générez une paire de clés SSH :

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

La clé publique générée se trouve dans ~/.ssh/id_ed25519.pub. Connectez-vous à GitLab, allez dans les paramètres utilisateur, cliquez sur l'onglet "Clés SSH", collez le contenu de la clé publique dans la zone de texte et enregistrez. Après avoir ajouté la clé, vous pouvez cloner des dépôts en utilisant SSH :

```bash
git clone git@gitlab.example.com:username/project.git
```

## Opérations de Projet

### Création d'un Projet

Après la connexion, cliquez sur le bouton "Nouveau Projet" dans la barre de navigation supérieure pour créer un nouveau projet. Vous pouvez choisir de créer un projet vide, créer à partir d'un modèle ou importer un projet existant. Lors de la création, vous devez définir le nom du projet, la description, le niveau de visibilité (Public, Interne, Privé) et le nom de branche par défaut.

Les noms de projet doivent être concis et clairs, en utilisant l'anglais et des tirets pour le nommage. Les projets privés ne sont accessibles qu'aux membres du projet. Les projets internes sont visibles pour tous les utilisateurs connectés au sein de l'entreprise. Les projets publics sont visibles par tout le monde. Après la création, vous pouvez modifier ces options dans la page des paramètres du projet.

### Gestion des Branches

GitLab utilise main comme branche principale par défaut (les versions précédentes utilisaient master). Cliquez sur "Branches" sur la page du projet pour voir et gérer toutes les branches. Créez une nouvelle branche en : cliquant sur "Nouvelle Branche" sur la page des branches, ou en utilisant les commandes Git :

```bash
git checkout -b feature/new-feature
```

Les branches protégées sont une fonctionnalité importante pour la collaboration d'équipe. Allez dans Paramètres du Projet → Dépôt → Branches Protégées pour définir quels utilisateurs peuvent pousser ou fusionner vers des branches spécifiques. Il est recommandé de protéger la branche main, en permettant uniquement à des rôles spécifiques ou aux mainteneurs de fusionner le code.

### Merge Requests

Les Merge Requests sont la fonctionnalité centrale pour la collaboration de code dans GitLab. Après avoir terminé le développement d'une fonctionnalité et poussé la branche, cliquez sur "Merge Requests" → "Nouvelle Merge Request" sur la page du projet. Sélectionnez la branche source et la branche cible, remplissez le titre et la description, puis créez.

La page Merge Request montre les différences entre les branches source et cible, y compris les modifications de fichiers et l'historique des commits. Les membres de l'équipe peuvent effectuer des révisions de code ici, en ajoutant des commentaires ligne par ligne. Une fois la révision terminée, si les conditions de fusion sont remplies (comme passer les tests CI et obtenir des approbations), la branche peut être fusionnée.

## Pipelines CI/CD

### Configuration .gitlab-ci.yml

GitLab CI/CD utilise le fichier .gitlab-ci.yml à la racine du projet pour définir les pipelines. C'est un fichier de configuration YAML décrivant chaque étape de construction et travail. Voici un exemple simple :

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

Cette configuration définit trois étapes : build pour construire le projet, test pour exécuter les tests, et deploy pour déployer l'application. Les artifacts sont utilisés pour passer les artifacts de construction entre les travaux, et only spécifie exécuter les travaux de déploiement uniquement sur la branche main.

### GitLab Runner

GitLab Runner est l'agent qui exécute les tâches CI/CD. GitLab fournit des Runners partagés, ou vous pouvez enregistrer vos propres Runners privés. Installez GitLab Runner sur votre serveur :

```bash
# Installer
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Enregistrer Runner
sudo gitlab-runner register
```

L'enregistrement nécessite de fournir l'URL de l'instance GitLab et le jeton d'enregistrement, qui peuvent être trouvés dans Paramètres du Projet → CI/CD → Runners. Les Runners peuvent être configurés pour exécuter les travaux en parallèle pour améliorer la vitesse de construction.

### Surveillance des Pipelines

Après chaque push de code, le pipeline est déclenché automatiquement. Vous pouvez voir tous les statuts d'exécution des pipelines sur la page du Projet → CI/CD → Pipelines. Cliquez sur un pipeline spécifique pour voir le statut d'exécution de chaque travail, y compris la sortie des journaux, le temps de construction et les raisons de l'échec.

GitLab fournit également un Éditeur de Pipeline qui crée et modifie visuellement les configurations CI/CD, éliminant le besoin d'écrire manuellement des fichiers YAML. L'éditeur fournit la vérification de syntaxe et l'aperçu en temps réel, réduisant les erreurs de configuration.

## Suivi des Problèmes

### Création et Gestion des Problèmes

Le système de problèmes est la base de la gestion de projet GitLab. Sur la page du projet, cliquez sur "Problèmes" → "Nouveau Problème" pour créer un nouveau problème. Vous devez remplir le titre et la description, et pouvez choisir d'assigner à des membres spécifiques, définir des étiquettes, des jalons et des dates d'échéance.

Le statut des problèmes est géré via des étiquettes ou des jalons. Les étiquettes courantes incluent bug, feature, enhancement, etc. Vous pouvez utiliser la vue tableau pour visualiser le statut des problèmes, en regroupant les problèmes par colonnes (ex: À Faire, En Cours, Terminé).

### Liaison des Problèmes au Code

L'utilisation de mots-clés plus des numéros de problème dans les messages de commit ou les descriptions de Merge Request lie automatiquement le code aux problèmes. Par exemple :

- Fixes #123 - Corrige le problème 123, fermera automatiquement le problème après la fusion
- Closes #456 - Ferme le problème 456
- See #789 - Référence le problème 789

Ce mécanisme de liaison aide les équipes à suivre les progrès du travail et à comprendre le contexte et l'objectif du problème lors de la révision de code.

## Wikis et Documentation

Chaque projet GitLab comprend une fonctionnalité wiki intégrée pour créer la documentation du projet. Les pages wiki sont écrites en format Markdown, prenant en charge la coloration syntaxique, les tableaux et les images. Le contenu wiki prend en charge le contrôle de version comme les dépôts de code, vous permettant de voir les versions historiques et de revenir en arrière.

Il est recommandé de maintenir la documentation suivante pour les projets : README pour l'introduction du projet, CONTRIBUTING pour le guide de contributions, les guides de déploiement et la documentation API. Ces documents peuvent réduire considérablement la difficulté d'intégration et favoriser les contributions de la communauté.
