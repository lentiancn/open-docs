# Preguntas frecuentes sobre Mockito

## Preguntas basicas

### Cual es la diferencia entre Mockito y PowerMock

Mockito se usa principalmente para simular objetos regulares, mientras que Power puede simular métodos estaticos, constructores, métodos privados y mas. Sin embargo, PowerMock usa tecnicas de manipulacion de bytecode que rompen el aislamiento de las pruebas, haciendo que las pruebas sean mas cercanas a pruebas de integracion que a pruebas unitarias. La recomendacion oficial es preferir Mockito primero, y solo considerar PowerMock cuando Mockito realmente no puede manejar ciertos escenarios como simular métodos estaticos. Desde Mockito 2.x, la simulacion de métodos estaticos tambien es compatible a traves de mockito-inline.

### Cuando debo usar Spy en lugar de Mock

Use Spy cuando necesita la mayor parte del comportamiento real pero desea simular solo algunos métodos. Spy retiene la mayor parte de la funcionalidad del objeto real mientras permite que métodos especificos sean stubbed. Mock reemplaza completamente el objeto real, por lo que no se ejecuta ninguna lógica de metodo real. Use Mock cuando necesita control total sobre el comportamiento del objeto. En la practica, Spy se usa relativamente poco porque en la mayoria de los casos usar objetos de prueba reales o mocks completos satisface los requisitos.

### Se puede usar Mockito para clases y métodos finales

Desde Mockito 2.1, puede configurar Mockito para simular clases y métodos finales. Necesita agregar la dependencia mockito-inline en lugar de mockito-core. mockito-inline habilita el mecanismo de subclases dinamicas para manejar clases y métodos finales. Esta caracteristica mejora enormemente la aplicabilidad de Mockito, haciendo que casi cualquier clase Java sea simulable.

### Por qué los métodos del objeto mock devuelven null

De manera predeterminada, todos los métodos de los objetos mock devuelven null o valores predeterminados para sus tipos (int devuelve 0, boolean devuelve false, los tipos de coleccion devuelven colecciones vacias, etc.). Este es el comportamiento predeterminado de Answer de Mockito. Si necesita valores de retorno especificos, debe establecerlos explicitamente usando when().thenReturn().

## Preguntas de uso

### La anotacion @Mock no funciona, que debo hacer

La anotacion @Mock requiere inicializar el framework Mockito para que surta efecto. Asegurese de usar uno de MockitoAnnotations.initMocks(this), @RunWith(MockitoJUnitRunner.class), o @ExtendWith(MockitoExtension.class) (para JUnit 5) en sus pruebas. Si se inicializa correctamente, los campos anotados con @Mock crearan automaticamente objetos mock antes de que se ejecuten los métodos de prueba.

### Como simular métodos void

Para métodos con tipo de retorno void, use doThrow(), doAnswer(), o doNothing() para definir el comportamiento.

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### Como verificar los argumentos de llamada del metodo

Puede usar emparejadores de argumentos para verificación, o usar ArgumentCaptor para capturar argumentos para aserciones detalladas.

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### Como simular iteradores

La simulacion de iteradores requiere atencion a la coordinacion entre los métodos hasNext() y next().

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### Como manejar sobrecarga de métodos

Cuando una clase tiene métodos sobrecargados, Mockito distingue diferentes métodos basandose en los tipos y cantidades de parametros. Asegurese de que los tipos de parametros en when() coincidan exactamente con la firma del metodo que desea simular.

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // metodo sobrecargado
```

## Preguntas avanzadas

### Por qué when().thenReturn() no funciona

Las razones comunes incluyen: establecer comportamiento en el mismo metodo multiples veces (solo el ultimo tiene efecto); usar Spy y llamar al metodo real; el metodo devuelve void (deberia usar doReturn().when()); el emparejador de argumentos usado incorrectamente causando falla de coincidencia. Verifique que la sintaxis para establecer el comportamiento sea correcta.

### Como probar clases abstractas

Al probar clases abstractas, puede usar Mockito para simular métodos concretos de la clase abstracta, o usar Spy para retener algo de comportamiento real. Otro enfoque es crear una subclase de prueba concreta para ejecutar pruebas.

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### Como simular métodos genericos

Al simular métodos genericos, sea consciente de las limitaciones causadas por el borrado de tipos. Use any() o anyObject() para coincidir con parametros genericos.

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### Como verificar el orden de llamadas

Use InOrder para verificar que las llamadas a métodos se ejecuten en un orden especifico.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### Como simular objetos parcialmente

Use Spy para simulacion parcial, o use withSettings() para personalizar el comportamiento de Mock.

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## Preguntas de rendimiento

### Las pruebas se ejecutan lentamente, que debo hacer

Asegurese de no estar usando llamadas a métodos reales innecesarias. Verifique si esta creando nuevos objetos Mock en cada prueba. Para proyectos que usan mucho Mockito, considere usar @RunWith(MockitoJUnitRunner.Silent.class) para reducir la salida de registros.

### Como reducir el uso de memoria de los objetos mock

Los objetos mock de Mockito por si mismos no consumen mucha memoria. Sin embargo, si crear una gran cantidad de objetos mock todavia afecta el rendimiento, considere usar la anotacion @Mock con @Scope para reutilizar objetos mock, o usar métodos de fabrica de mock estaticos.

## Preguntas de depuración

### Como ver los registros de llamadas del objeto mock

Puede usar la funcion de registro de Mockito o agregar oyentes para rastrear llamadas.

```java
Mockito.framework().addListener(new VerificationListener());
```

O use la salida de prueba de Maven/Gradle para ver informacion detallada de llamadas mock.

### Como depurar cuando las pruebas fallan

Mockito proporciona mensajes de error detallados. Verifique las diferencias entre las llamadas esperadas y reales que se muestran en los fallos de verify. Use la funcion Imprimir Registros de Mockito para comprender el comportamiento del objeto mock. Simplifique los casos de prueba paso a paso para localizar problemas.
