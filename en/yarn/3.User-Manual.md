# Yarn User Manual

## Basic Commands

### General Commands

- `yarn add` - Add dependencies to the project
- `yarn install` - Install the project dependencies
- `yarn remove` - Remove dependencies from the project
- `yarn run` - Run a script defined in the package.json
- `yarn init` - Create a new package
- `yarn info` - See information related to packages
- `yarn why` - Display the reason why a package is needed

### Advanced Commands

- `yarn dlx` - Run a package in a temporary environment
- `yarn dedupe` - Deduplicate dependencies with overlapping ranges
- `yarn up` - Upgrade dependencies across the project
- `yarn pack` - Generate a tarball from the active workspace
- `yarn rebuild` - Rebuild the project's native packages

### NPM Related Commands

- `yarn npm audit` - Perform a vulnerability audit against the installed packages
- `yarn npm publish` - Publish the active workspace to the npm registry
- `yarn npm login` - Store new login info to access the npm registry
- `yarn npm logout` - Logout of the npm registry
- `yarn npm whoami` - Display the name of the authenticated user

### Plugin Management

- `yarn plugin list` - List the available official plugins
- `yarn plugin import` - Download a plugin
- `yarn plugin remove` - Remove a plugin
- `yarn plugin runtime` - List the active plugins

## Workspaces

Yarn was the first package manager built specifically around workspaces, allowing you to split your project into sub-components.

### Workspace Configuration

Configure workspaces in your root `package.json`:

```json
{
  "workspaces": [
    "packages/*"
  ]
}
```

### Workspace Commands

- `yarn workspaces list` - List all workspaces
- `yarn workspace <name> <command>` - Run a command in a specific workspace

### Cross-references

Packages from monorepos often need to depend on each other. Yarn makes it very easy thanks to the special `workspace:` protocol:

```json
{
  "dependencies": {
    "@my-org/utils": "workspace:^"
  }
}
```

The `workspace:` protocol accepts either a regular semver range, or the special `^` / `~` / `*` tokens.

### Focused Installs

A common concern when discovering workspaces is how you need to install all of their dependencies whenever you wish to work on a single one of them. Yarn provides a solution via `yarn workspaces focus`.

This command takes a list of workspaces, extends the list to include transitive dependencies, and excludes everything else from the install:

```bash
yarn workspaces focus @my-org/app
```

### Parallel Execution

Scripts from multiple workspaces can be run in parallel if they share the same name, by using `yarn workspaces foreach`:

```bash
yarn workspaces foreach --all -pt npm publish
```

## Configuration Files

### .yarnrc.yml

Yarn v2+ uses `.yarnrc.yml` file for configuration:

```yaml
nodeLinker: pnp
yarnPath: .yarn/releases/yarn-4.0.0.cjs
```

### .yarnrc (Yarn v1)

Yarn v1 uses `.yarnrc` file:

```
--registry https://registry.npmjs.org
--strict-semver true
```

## Version Differences

### Yarn v4+ vs Yarn v1

| Feature | Yarn v4+ | Yarn v1 |
|---------|----------|---------|
| Architecture | Plugin-based (Berry) | Traditional |
| Installation Speed | Faster (parallel installs) | Standard speed |
| Disk Space | More efficient (PnP) | Standard node_modules |
| Workspaces | Native support | Requires additional setup |
| Plugin System | Built-in support | Not supported |
| Configuration | .yarnrc.yml | .yarnrc |

### Migration Considerations

- Yarn v4+ uses PnP by default, which may be incompatible with some tools
- CI/CD configurations need to be updated to accommodate the new installation flow
- Plugin system provides more customization options
- Workspace functionality is more powerful and flexible
