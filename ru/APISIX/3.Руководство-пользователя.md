# Руководство пользователя

В этом руководстве подробно описаны основные возможности и способы использования Apache APISIX: настройка маршрутов, управление upstream-серверами, балансировка нагрузки, ограничение скорости, аутентификация и другие важные функции.

## Базовые понятия

### Route (маршрут)

Маршрут — это путь для доступа к upstream-сервису. В APISIX маршрут сначала сопоставляется с запросом клиента по заданным правилам, затем загружаются и выполняются нужные плагины, и наконец запрос отправляется на соответствующий upstream-сервер.

Самый простой маршрут содержит только путь и адрес upstream-сервера.

### Upstream ( upstream)

Upstream — это группа серверов с одинаковой функциональностью, своего рода абстракция над виртуальным хостом. APISIX может автоматически распределять нагрузку между несколькими сервисными узлами через настроенные правила.

### Consumer (потребитель)

Потребитель — это приложение или разработчик, использующий API. В APISIX потребитель должен иметь уникальное имя и быть привязан к плагину аутентификации.

### Plugin (плагин)

APISIX предоставляет богатую систему плагинов для расширения функциональности шлюза. Плагины можно настраивать на уровне маршрута, потребителя или глобально.

## Настройка маршрутов

Создать маршрут можно через управляющий API. Вот пример маршрута, который перенаправляет запросы на httpbin.org:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

При успешном создании вернётся `HTTP/1.1 201 Created`.

### Проверка маршрута

```bash
curl "http://127.0.0.1:9080/ip"
```

Должен вернуться примерно такой ответ:

```json
{
  "origin": "183.94.122.205"
}
```

### Правила сопоставления маршрутов

APISIX поддерживает разные способы сопоставления:

**По URI:**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**По хосту:**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**По методу запроса:**

```json
{
  "methods": ["GET", "POST"]
}
```

**По заголовкам:**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## Балансировка нагрузки

APISIX поддерживает разные алгоритмы балансировки: round-robin (простой перебор), weighted round-robin (с весами), least connections (наименьшее количество соединений) и другие.

### Настройка взвешенной балансировки

Создаём маршрут с несколькими upstream-серверами:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**Параметры:**

- `type`: тип алгоритма балансировки — `roundrobin` (перебор), `weighted_roundrobin` (с весами), `least_conn` (наименьшее число соединений)
- `nodes`: узлы upstream-сервиса в формате `{"host:port": вес}`
- `pass_host`: при значении `node` заголовки запроса передаются на upstream
- `scheme`: протокол upstream — `http` или `https`

### Проверка балансировки

Генерируем 100 запросов для проверки:

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

Запросы должны распределиться примерно поровну между двумя серверами.

## Ограничение скорости

В APISIX гибкая система ограничения скорости через плагин `limit-count`.

### Включение ограничения скорости

Добавляем плагин к существующему маршруту:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**Параметры:**

- `count`: максимальное число запросов в окне
- `time_window`: размер окна в секундах
- `rejected_code`: HTTP-статус при превышении лимита

В этом примере лимит — 2 запроса каждые 10 секунд.

### Проверка ограничения

Генерируем 100 запросов:

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

Результат: 2 запроса успешны (200), 98 отклонены (503).

### Отключение ограничения

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### Продвинутые настройки

**С использованием переменных:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**С Redis-кластером:**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## Аутентификация

APISIX поддерживает разные способы аутентификации: Key Auth, Basic Auth, JWT, OAuth2 и другие.

### Key Authentication (аутентификация по ключу)

#### Создание потребителя

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "secret-key"
    }
  }
}'
```

> ⚠️ В продакшене используйте сложные ключи.

#### Включение аутентификации на маршруте

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### Проверка аутентификации

**Запрос без ключа:**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

Ответ: `HTTP/1.1 401 Unauthorized`

**Запрос с неправильным ключом:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: wrong-key'
```

Ответ: `HTTP/1.1 401 Unauthorized`

**Запрос с правильным ключом:**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: secret-key'
```

Ответ: `HTTP/1.1 200 OK`

### Другие способы аутентификации

**JWT:**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "user-key",
      "secret": "my-secret-key"
    }
  }
}
```

**Basic Auth:**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**HMAC:**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "my-secret-key"
    }
  }
}
```

## Управление сервисами

### Canary-релизы (канареечный выпуск)

APISIX поддерживает canary-релизы на основе весов:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### Переписывание запросов

Используем плагин `proxy-rewrite` для изменения запроса:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### Кэширование ответов

Плагин `proxy-cache` кэширует ответы upstream-серверов:

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## Настройка плагинов

### Порядок выполнения плагинов

В разделе `plugins` можно указать фазу выполнения через `phase`:

- `rewrite` — фаза переписывания запроса
- `access` — фаза контроля доступа
- `before_proxy` — перед проксированием
- `header_filter` — фильтрация заголовков ответа
- `body_filter` — фильтрация тела ответа
- `log` — логирование

### Отключение плагина

Установите `_meta.disable` в `true`:

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/route-id" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## Мониторинг и наблюдаемость

### Интеграция с Prometheus

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

Доступ к метрикам:

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### Логирование

APISIX поддерживает разные способы вывода логов:

**HTTP-логи:**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

**Ошибки:**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

## Рекомендации

1. **Используйте осмысленные ID**: для маршрутов, upstream-серверов и других ресурсов
2. **Настраивайте разумные таймауты**: учитывайте время ответа upstream-сервисов
3. **Включайте проверки здоровья**: для критичных сервисов настройте active и passive health checks
4. **Настраивайте ограничение скорости**: исходя из бизнес-требований и возможностей сервера
5. **Включайте аутентификацию**: в продакшене это обязательно
6. **Используйте TLS**: для внешних интерфейсов нужен HTTPS
7. **Мониторьте ключевые метрики**: настройте Prometheus и алерты
