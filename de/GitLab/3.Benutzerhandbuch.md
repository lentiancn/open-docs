# GitLab Benutzerhandbuch

## Kontoverwaltung

### Registrierung und Anmeldung

Um GitLab zum ersten Mal zu verwenden, klicken Sie auf der Anmeldeseite auf "Registrieren", um ein Konto zu erstellen. Für die Registrierung müssen Sie einen Benutzernamen, eine E-Mail-Adresse und ein Passwort angeben. Die Wahl des Benutzernamens sollte sorgfältig getroffen werden, da er nach der Erstellung nicht mehr geändert werden kann. Nach der Registrierung erhalten Sie eine Bestätigungs-E-Mail. Klicken Sie auf den Link in der E-Mail, um die E-Mail-Verifizierung abzuschließen.

Nach der Anmeldung können Sie Ihr Profil in den persönlichen Einstellungen verbessern, einschließlich Avatar, Name, vollständiger Name und Bio. Es wird empfohlen, vollständige Informationen auszufüllen, da diese in Projektmitgliederlisten und Commit-Verläufen angezeigt werden. Sie können auch SSH-Schlüssel binden, um Code über die Befehlszeile zu pushen.

### SSH-Schlüssel-Konfiguration

SSH-Schlüssel ermöglichen die sichere Authentifizierung mit GitLab-Repositorys, ohne jedes Mal Benutzernamen und Passwort einzugeben. Generieren Sie zunächst ein SSH-Schlüsselpaar:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Der generierte öffentliche Schlüssel befindet sich in ~/.ssh/id_ed25519.pub. Melden Sie sich bei GitLab an, gehen Sie zu den Benutzereinstellungen, klicken Sie auf die Registerkarte "SSH-Schlüssel", fügen Sie den Inhalt des öffentlichen Schlüssels in das Textfeld ein und speichern Sie. Nach dem Hinzufügen des Schlüssels können Sie Repositorys über SSH klonen:

```bash
git clone git@gitlab.example.com:username/project.git
```

## Projektoperationen

### Ein Projekt erstellen

Nach der Anmeldung klicken Sie auf die Schaltfläche "Neues Projekt" in der oberen Navigationsleiste, um ein neues Projekt zu erstellen. Sie können wählen, ob Sie ein leeres Projekt erstellen, aus einer Vorlage erstellen oder ein bestehendes Projekt importieren möchten. Beim Erstellen müssen Sie den Projektnamen, die Beschreibung, die Sichtbarkeitsstufe (Öffentlich, Intern, Privat) und den Standard-Branch-Namen festlegen.

Projektnamen sollten kurz und klar sein, wobei Sie Englisch und Bindestriche für die Benennung verwenden. Private Projekte sind nur für Projektmitglieder zugänglich. Interne Projekte sind für alle angemeldeten Benutzer im Unternehmen sichtbar. Öffentliche Projekte sind für alle sichtbar. Nach der Erstellung können Sie diese Optionen auf der Projekteinstellungsseite ändern.

### Branch-Verwaltung

GitLab verwendet standardmäßig main als Hauptbranch (frühere Versionen verwendeten master). Klicken Sie auf "Branches" auf der Projektseite, um alle Branches anzuzeigen und zu verwalten. Erstellen Sie einen neuen Branch durch: Klicken Sie auf "Neuer Branch" auf der Branch-Seite oder verwenden Sie Git-Befehle:

```bash
git checkout -b feature/new-feature
```

Geschützte Branches sind eine wichtige Funktion für die Teamzusammenarbeit. Gehen Sie zu Projekteinstellungen → Repository → Geschützte Branches, um festzulegen, welche Benutzer auf bestimmte Branches pushen oder mergen können. Es wird empfohlen, den main-Branch zu schützen und nur bestimmten Rollen oder Maintainern das Mergen zu erlauben.

### Merge Requests

Merge Requests sind die Kernfunktion für die Code-Zusammenarbeit in GitLab. Nach dem Abschluss der Funktionsentwicklung und dem Pushen des Branches klicken Sie auf "Merge Requests" → "Neuer Merge Request" auf der Projektseite. Wählen Sie den Quellbranch und den Zielbranch, füllen Sie Titel und Beschreibung aus und erstellen Sie ihn.

Die Merge Request-Seite zeigt die Unterschiede zwischen Quell- und Zielbranch, einschließlich Dateiänderungen und Commit-Verlauf. Teammitglieder können hier Code-Reviews durchführen und Kommentare Zeile für Zeile hinzufügen. Nachdem der Review abgeschlossen ist und Merge-Bedingungen erfüllt sind (wie Bestehen der CI-Tests und Erhalten von Genehmigungen), kann der Branch gemergt werden.

## CI/CD-Pipelines

### .gitlab-ci.yml-Konfiguration

GitLab CI/CD verwendet die Datei .gitlab-ci.yml im Projektstammverzeichnis, um Pipelines zu definieren. Dies ist eine YAML-Konfigurationsdatei, die jede Build-Stufe und Aufgabe beschreibt. Hier ist ein einfaches Beispiel:

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

Diese Konfiguration definiert drei Stufen: build für das Erstellen des Projekts, test für das Ausführen von Tests und deploy für das Bereitstellen der Anwendung. Artefakte werden verwendet, um Build-Artefakte zwischen Aufgaben zu übergeben, und only gibt an, dass Deploy-Aufgaben nur auf dem main-Branch ausgeführt werden sollen.

### GitLab Runner

GitLab Runner ist der Agent, der CI/CD-Aufgaben ausführt. GitLab bietet gemeinsame Runner, oder Sie können Ihre eigenen privaten Runner registrieren. Installieren Sie GitLab Runner auf Ihrem Server:

```bash
# Installieren
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Runner registrieren
sudo gitlab-runner register
```

Die Registrierung erfordert die Angabe der GitLab-Instanz-URL und des Registrierungstokens, die Sie unter Projekteinstellungen → CI/CD → Runner finden. Runner können für parallele Aufgabenausführung konfiguriert werden, um die Build-Geschwindigkeit zu verbessern.

### Pipeline-Überwachung

Nach jedem Code-Push wird die Pipeline automatisch ausgelöst. Sie können alle Pipeline-Ausführungsstatus auf der Seite Projekt → CI/CD → Pipelines anzeigen. Klicken Sie auf eine bestimmte Pipeline, um den Ausführungsstatus jeder Aufgabe anzuzeigen, einschließlich Protokollausgabe, Build-Zeit und Fehlerursachen.

GitLab bietet auch einen Pipeline-Editor, der CI/CD-Konfigurationen visuell erstellt und bearbeitet, ohne YAML-Dateien manuell schreiben zu müssen. Der Editor bietet Syntaxprüfung und Echtzeitvorschau, um Konfigurationsfehler zu reduzieren.

## Issue-Tracking

### Erstellen und Verwalten von Issues

Das Issue-System ist die Grundlage des GitLab-Projektmanagements. Klicken Sie auf der Projektseite auf "Issues" → "Neues Issue", um ein neues Issue zu erstellen. Sie müssen Titel und Beschreibung ausfüllen und können Mitglieder zuweisen, Labels, Meilensteine und Fälligkeitsdaten festlegen.

Der Issue-Status wird durch Labels oder Meilensteine verwaltet. Übliche Labels sind bug, feature, enhancement usw. Sie können die Board-Ansicht verwenden, um den Issue-Status zu visualisieren und Issues nach Spalten zu gruppieren (z.B. Zu erledigen, In Bearbeitung, Erledigt).

### Issues mit Code verknüpfen

Die Verwendung von Schlüsselwörtern plus Issue-Nummern in Commit-Nachrichten oder Merge Request-Beschreibungen verknüpft automatisch Code mit Issues. Zum Beispiel:

- Fixes #123 - Behebt Issue 123, wird das Issue nach dem Mergen automatisch schließen
- Closes #456 - Schließt Issue 456
- See #789 - Verweist auf Issue 789

Dieser Verknüpfungsmechanismus hilft Teams, den Arbeitsfortschritt zu verfolgen und den Hintergrund und Zweck des Issues während des Code-Reviews zu verstehen.

## Wikis und Dokumentation

Jedes GitLab-Projekt enthält integrierte Wiki-Funktionen zum Erstellen von Projektdokumentation. Wiki-Seiten werden im Markdown-Format geschrieben und unterstützen Syntaxhervorhebung, Tabellen und Bilder. Wiki-Inhalte unterstützen wie Code-Repositorys die Versionskontrolle, sodass Sie historische Versionen anzeigen und zurücksetzen können.

Es wird empfohlen, folgende Dokumentation für Projekte zu pflegen: README für Projektvorstellung, CONTRIBUTING für Beitragshandbücher, Bereitstellungsanleitungen und API-Dokumentation. Diese Dokumente können die Einarbeitung erheblich erleichtern und Community-Beiträge fördern.
