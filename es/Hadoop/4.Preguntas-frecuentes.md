# Preguntas Frecuentes sobre Hadoop

## Problemas Relacionados con HDFS

### Qué Hacer Cuando el Espacio de Almacenamiento de HDFS Es Insuficiente

El espacio de almacenamiento insuficiente de HDFS es un problema común que puede tener múltiples causas. Primero verifique el uso del disco de DataNode:

```bash
hdfs dfsadmin -report
```

Si el espacio es efectivamente insuficiente, tome las siguientes medidas: elimine datos históricos que ya no necesite; use hdfs balancer para balancear la distribución de datos y liberar algunos nodos; agregue nuevos nodos DataNode para expandir la capacidad de almacenamiento del clúster; reduzca el factor de replicación para disminuir la sobrecarga de almacenamiento (debe sopesar la seguridad de los datos).

```bash
# Eliminar archivos
hdfs dfs -rm -r /user/olddata

# Balancear datos
hdfs balancer -threshold 10
```

### Cómo Manejar la Memoria Insuficiente de NameNode

NameNode almacena todos los metadatos en memoria; si hay demasiados archivos, la memoria puede volverse insuficiente. Mitigue aumentando la configuración de memoria de NameNode en hadoop-env.sh configurando el parámetro -Xmx en HADOOP_NAMENODE_OPTS; habilitando la funcionalidad de instantáneas de NameNode para crear regularmente copias de solo lectura de datos importantes; actualizando a Hadoop 2.x o superior, que tiene mejor optimización de gestión de metadatos.

```bash
# Establecer en hadoop-env.sh
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### Qué Hacer Cuando DataNode No Inicia

El fallo de inicio de DataNode puede tener múltiples causas. Primero revise los registros de DataNode:

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

Causas comunes y soluciones: espacio en disco insuficiente—limpie discos o agregue nuevos discos; problemas de permisos—asegúrese de que el usuario de Hadoop tenga acceso de lectura/escritura a los directorios de datos; ID de clúster inconsistente—si agrega un nuevo DataNode, ejecute hdfs namenode -upgrade en el DataNode o elimine el directorio de almacenamiento y reformatee.

```bash
# Ver permisos del directorio de DataNode
ls -la /usr/local/hadoop/data/

# Eliminar directorio de almacenamiento de DataNode y reiniciar
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### Cómo Recuperar Archivos Eliminados Accidentalmente

HDFS proporciona funcionalidad de papelera para recuperar archivos eliminados. Primero confirme que la papelera está habilitada:

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

Si descubre eliminación accidental, restaure desde la papelera:

```bash
# Restaurar archivo
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

Si la papelera ha sido vaciada o el período de retención ha pasado, los datos no se pueden recuperar. Por lo tanto, en operaciones diarias: mantenga la papelera habilitada (habilitada por defecto); haga respaldo de datos importantes; confirme cuidadosamente antes de eliminar.

### Cómo Manejar Bloques HDFS Dañados

Los bloques de datos pueden dañarse debido a fallas de disco o problemas de red. HDFS tiene mecanismos integrados de detección de integridad de datos: cuando se detectan bloques dañados, automáticamente replica datos de otras réplicas para recuperación. Verificación y reparación manual:

```bash
# Verificar integridad del sistema de archivos
hdfs fsck / -files -blocks -locations

# Eliminar archivos de bloque dañados
hdfs fsck / -delete
```

## Problemas Relacionados con MapReduce

### Qué Hacer Cuando los Trabajos Se Ejecutan Lentamente

Los trabajos MapReduce lentos pueden ser causados por múltiples factores. Primero, revise los registros de historial del trabajo y analice el tiempo empleado en cada fase:

```bash
# Ver detalles del trabajo
yarn logs -applicationId application_xxx -logFiles *

# Ver en JobHistory Server
mapred job -history job_history_file
```

Direcciones comunes de optimización: aumentar el número de tareas Reduce para mejorar el paralelismo; ajustar el tamaño del heap de JVM para evitar OOM; habilitar Combiner para reducir el volumen de transmisión de red; optimizar la asimetría de datos usando particionadores personalizados para balancear la carga; elegir formatos de archivo apropiados (Parquet, ORC, etc.).

### Cómo Manejar la Asimetría de Datos

La asimetría de datos significa que los datos se distribuyen de manera desigual durante la fase Reduce, causando que algunas tareas Reduce se ejecuten mucho más tiempo que otras. Las soluciones incluyen: usar particionadores personalizados para redistribuir datos según la lógica de negocio; agregar prefijo aleatorio para dejar que los datos se dispersen primero a múltiples Reduces, luego agregar secundariamente; usar salting de clave—eliminar prefijo después de la agregación.

```java
// Ejemplo de particionador personalizado
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // Redistribuir según lógica de negocio
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### Cómo Solucionar Trabajos MapReduce que Fállan

Cuando los trabajos fallan, primero revise los registros de error para determinar la causa del fallo:

```bash
# Ver estado del trabajo
yarn application -status application_xxx

# Ver razones del fallo de tareas
mapred job -status job_id
```

Causas comunes de fallo: Java OutOfMemory (OOM)—aumente mapreduce.map.memory.mb y mapreduce.reduce.memory.mb; error de formato de datos de entrada—verifique si los datos coinciden con el formato esperado; excepción de código personalizado—revise la lógica del código y mensajes de exception; timeout de red—aumente el parámetro mapreduce.task.timeout.

## Problemas Relacionados con YARN

### Qué Hacer Cuando ResourceManager No Inicia

El fallo de inicio de ResourceManager generalmente es causado por errores de configuración o conflictos de puertos. Revise los registros:

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

Problemas comunes y soluciones: puerto en uso—modifique el puerto de ResourceManager en yarn-site.xml; error de configuración de ZooKeeper (si usa HA)—revise la configuración relacionada de zkfc; memoria insuficiente—asegúrese de que la máquina tenga suficiente memoria física; error de archivo de configuración—revise cuidadosamente yarn-site.xml y capacity-scheduler.xml.

```bash
# Ver uso de puerto
netstat -tlnp | grep 8032
```

### Cómo Manejar la Baja Utilización de Recursos del Clúster

La baja utilización de recursos del clüster puede ser causada por: patrones de envío de trabajos irrazonables—trabajos demasiado pequeños hacen que la sobrecarga de programación exceda el tiempo de computación; configuración inapropiada del programador—asignación de recursos de cola irrazonable; mala localidad de datos—datos de entrada y nodos de computación no en la misma ubicación.

Sugerencias de optimización: envíe trabajos pequeños en lotes; configure razonablemente la capacidad y prioridades de cola; optimice el diseño de datos para mejorar la localidad de datos; use el programador apropiado (Capacity Scheduler o Fair Scheduler).

### Cómo Limitar el Uso de Recursos de Usuarios

Use colas YARN y control de acceso para limitar el uso de recursos de usuarios:

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- Máximo múltiplo de recursos de cola para un solo usuario -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- Máximo de trabajos en ejecución en cola -->
</property>
```

## Problemas de Operaciones del Clúster

### Cómo Monitorear el Clúster de Hadoop

Hadoop proporciona múltiples métodos de monitoreo: Interfaz Web (NameNode puerto 9870, ResourceManager puerto 8088); Monitoreo JMX (Java Management Extensions); Sistemas de monitoreo externos como Ganglia, Prometheus; Herramientas de línea de comandos (hdfs dfsadmin, yarn node, etc.).

Se recomienda usar Prometheus + Grafana para construir una plataforma de monitoreo completa—recolectar métricas JMX de Hadoop y visualizarlas.

### Cómo Realizar Actualización del Clüster

La actualización del clúster de Hadoop requiere operación cuidadosa. Pasos recomendados: primero verifique la compatibilidad de la nueva versión en un entorno de prueba; respalde metadatos de NameNode y datos de HDFS; actualice nodos DataNode uno por uno, luego actualice NameNode; use actualización gradual para reducir el tiempo de inactividad.

```bash
# Actualización gradual de DataNode
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Actualizar Hadoop
tar -xzf hadoop-new.tar.gz -C /usr/local/
# Iniciar DataNode
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### Cómo Configurar Alta Disponibilidad de NameNode

NameNode HA evita el punto único de fallo de NameNode. Pasos de configuración: configure dos NameNodes en hdfs-site.xml; configure el directorio de ediciones compartidas (usando ZooKeeper o NFS); inicie ZooKeeper Failover Controller (ZKFC).

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

Después de la configuración, inicialice ZKFC:

```bash
hdfs zkfc -formatZK
```

### Cómo Manejar la Desactivación de Nodos

La desactivación de nodos requiere manejo cuidadoso para asegurar que no se pierdan datos. Use la funcionalidad de decommission para desconectar nodos de manera segura:

```bash
# Agregar nodo a excluir en archivo workers
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# Actualizar lista de nodos
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

Espere a que se complete la replicación de datos—el nodo entrará automáticamente en estado decommissioned. Confirme que los datos se han replicado a otros nodos antes de apagar el nodo.

## Problemas de Desarrollo

### Cómo Depurar Programas MapReduce

La depuración de programas MapReduce es difícil. Métodos recomendados: primero pruebe la lógica en modo local; use Contadores para registrar métricas clave; agregue salida de registro—imprima valores de variables clave en métodos Map o Reduce; use MRUnit para pruebas unitarias.

```java
// Usando Contador
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### Cómo Elegir InputFormat y OutputFormat

InputFormat determina cómo leer los datos de entrada: TextInputFormat (predeterminado, lee por línea); KeyValueTextInputFormat (divide por tab); SequenceFileInputFormat (formato binario); DBInputFormat (lee de base de datos).

OutputFormat determina cómo emitir resultados: TextOutputFormat (predeterminado, formato de texto); SequenceFileOutputFormat (formato binario); NullOutputFormat (sin salida de archivo); DBOutputFormat (emite a base de datos).

### Cómo Manejar el Problema de Archivos Pequeños

Grandes cantidades de archivos pequeños aumentan la presión de memoria de NameNode y reducen la eficiencia del procesamiento. Soluciones: usar Hadoop Archives (HAR) para empaquetar archivos pequeños; usar SequenceFile para fusionar archivos pequeños; usar HBase para almacenar grandes cantidades de archivos pequeños; ajustar el tamaño de bloque para reducir el número de archivos pequeños.
