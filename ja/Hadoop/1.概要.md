# Hadoop 入門

## Hadoopとは

Apache Hadoopは、Apache Software Foundationが開発するオープンソースの分散コンピューティングフレームワークで、大規模データセットの処理と保存のために設計されています。Hadoopは、信頼性が高く、スケーラブルな分散データストレージとコンピューティングソリューションを提供し、一般的なハードウェアで構成されたクラスター上で大量データを処理できます。Hadoopの中核的な設計理念は、計算をデータのあるノードに移動させることであり、大量のデータを計算ノードに移動させることではないため、ネットワーク転送オーバーヘッドを大幅に削減し、処理効率を向上させます。

Hadoopプロジェクトの起源は2003年に遡ります。当時、GoogleはGoogle File System（GFS）に関する論文を発表し、その後2004年にはMapReduceに関する論文を発表しました。これらの論文に触発され、Yahoo!のエンジニアであるDoug CuttingとMike Cafarellaは、オープンソース実装の開発を開始し、これがHadoopとなりました。2006年、Hadoopは正式にApache Software Foundationのプロジェクトとなり、その名称はDoug Cuttingの息子の玩具のゾウに由来しています。

HadoopはJavaで開発されており、優れたクロスプラットフォーム機能を備えています。中核的なコンポーネントには、HDFS（Hadoop分散ファイルシステム）とYARN（Yet Another Resource Negotiator）、およびMapReduceプログラミングモデルが含まれます。Hadoopの設計哲学は「すべて拡張可能」であり、システムは単にノードを追加することでストレージ容量と計算能力を線形に拡張できます。このシンプルで強力な設計により、Hadoopはビッグデータ処理分野のインフラストラクチャとなっています。

## 中核コンポーネント

### HDFS - 分散ファイルシステム

HDFSは、Hadoopの分散ファイルシステムで、超大型ファイルの保存のために設計されています。HDFSは、1つのNameNodeと複数のDataNodeで構成されるマスタースレーブアーキテクチャを採用しています。NameNodeはファイルシステムの名前空間とメタデータを管理し、ファイルシステムのディレクトリ構造を維持し、各ファイルのブロックリストと各ブロックのDataNode上の場所を記録します。DataNodeは実際のデータ保存を担当し、保存されたブロック情報を定期的にNameNodeに報告します。

HDFSはデータ複製メカニズムを通じて高い耐障害性を備えています。デフォルトでは、各データブロックは3つの異なるDataNodeに複製され、あるノードが故障してもシステムは他のレプリカからデータを回復できます。HDFSは大きなファイルのシーケンシャル読み取りのパフォーマンスに最適化されており、バッチ処理シナリオに適していますが、低レイテンシなランダム読み取り/書き込みには適していません。HDFSはまた、ラック認識をサポートしており、異なるラックのノードにブロックレプリ카をインテリジェントに分配して、データの可用性とシステムの耐障害性をさらに向上させます。

HDFSのアーキテクチャ設計は「Write Once Read Many」（WORM）原則に従っています。つまり、ファイルは書き込まれた後に変更すべきではありません。この設計は一貫性モデルを調整し、読み取りパフォーマンスを向上させ、データの複製と分散を簡素化します。HDFSはまた、スナップショット機能を提供しており、バックアップと回復のためにファイルシステムの読み取り専用ポイントインタイムコピーを作成できます。

### YARN - リソースマネージャ

YARNは、Hadoop 2.xで導入されたリソース管理プラットフォームで、クラスターリソースの統一管理与管理与スケジューリングを担当します。YARNはリソース管理与アプリケーション実行を分離し、HadoopがMapReduceバッチジョブだけでなく、より多くの種類のワークロードをサポートできるようになりました。YARNの中核的なコンポーネントには、ResourceManager、NodeManager、ApplicationMasterがあります。

ResourceManagerはYARNの中核的なコンポーネントで、クラ全体のリソース管理与スケジューリングを担当します。リソース要求を受け取り、利用可能なクラスターリソースに基づいてスケジューリングを行います。ResourceManagerは、SchedulerとApplicationsManagerで構成されます。Schedulerはプリセットのポリシーに従ってリソースを分配し、ApplicationsManagerはアプリケーション提出要求を受け取り、アプリケーションのApplicationMasterを起動します。

NodeManagerは各ワーカーノード上のエージェントで、そのノード上のリソース（CPU、メモリ、ディスク、ネットワークリソース）の管理を担当します。NodeManagerはResourceManagerにリソース使用状況とコンテナ状態を定期的に報告し、ResourceManagerからの命令を実行します。ApplicationMasterは各アプリケーションの管理자로、ResourceManagerとネゴシエーションしてリソースを取得し、NodeManagerと協力してタスクを実行します。

### MapReduce - プログラミングモデル

MapReduceは、大規模データの並列処理のためのHadoopの中核的なコンピューティングモデルです。MapReduceプログラミングモデルは、データ処理プロセスをMapフェーズとReduceフェーズの2つの主要フェーズに分割します。Mapフェーズでは、システムが入力データを独立したチャンクに分割し、各チャンクを並列処理し、中間結果をキーと値のペアとして出力します。Reduceフェーズでは、同じキーを持つ中間結果をを集約して処理し、最終的な計算結果を生成します。

MapReduceフレームワークは、データ分散、タスクスケジューリング、負荷分散、障害処理などの複雑な詳細を自動的に処理し、開発者はビジネスロジックの実装に集中できます。この抽象化により分散プログラムの开发が大幅に簡素化されます。開発者はMap関数とReduce関数を記述するだけで、フレームワークが分散実行の全複雑性を処理します。MapReduceはまた、カスタムパーティショナーとコシナーをサポートしており、データフローの最適化とネットワーク転送の削減に使用できます。

MapReduceジョブの実行プロセスには、入力スプリット、Mapタスク実行、中間結果シャッフル、Reduceタスク実行、結果出力などの段階が含まれます。シャッフルフェーズはMapReduceの中核的なプロセスで、Mapタスクの出力をキーでパーティショニング、、ソート、結合し、対応するReduceタスクに転送します。このプロセスは計算集約型和ネットワーク集約型であり、ジョブ全体のパフォーマンスに大きな影響を与えます。

## 技術アーキテクチャ

### NameNodeとSecondary NameNode

NameNodeはHDFSの中核的なコンポーネントで、ファイルシステム全体のメタデータを維持します。ファイル名、ファイル属性、ファイルとデータブロックのマッピング、ブロックとDataNodeのマッピングなど、ファイルシステムの完全なメタデータをメモリに保存します。メタデータの永続性を確保するため、NameNodeはメタデータをローカルファイルシステムのトランザクションログ（EditLog）にも書き込みます。NameNode起動時に、イメージファイル（FsImage）を読み取り、トランザクションログをリプレイしてファイルシステム状態を回復します。

Secondary NameNodeはNameNodeのバックアップではありません。その主な役割はFsImageとEditLogを定期的にマージし、NameNodeの起動時間とEditLogファイルサイズを削減することです。Secondary NameNodeは通常別のマシンで実行され、NameNodeからFsImageとEditLogを定期的にダウンロードし、ローカルでマージ操作を実行して、新しいFsImageをNameNodeにアップロードします。このプロセスはチェックポイント」と呼ばれ、NameNodeのメモリ压力と起動時間を効果的に削減できます。

### ResourceManagerアーキテクチャ

ResourceManagerは2層スケジューラーアーキテクチャを採用しています。第1層はCapacity SchedulerとFair Schedulerで、異なるキューとアプリケーション間でクラスターリソースを分配します。第2層はApplicationMasterで、各アプリケーションにはResourceManagerとネゴシエーションしてリソースを取得し、アプリケーションの実行を管理する独自のApplicationMasterがあります。

Capacity Schedulerは複数の組織がクラスターリソースを共有することを許可し、各組織が独自のキューを持ち、各組織が特定のリソースクォータを取得することを保証します。Fair Schedulerは公平性を強調し、実行中のすべてのアプリケーション間でリソース分配を自動的にバランスさせ、各アプリケーションが比較的公平なリソースを取得できるようにします。両方のスケジューラーは設定ファイルで柔軟に構成でき、さまざまなシナリオの要件を満たします。

### データローカリティ

データローカリティは、ネットワークデータ転送を減らすことを目指すHadoopの重要な最適化戦略です。Hadoopは、対応するデータが保存されているノードに計算タスクを割り当てようとします，这就是「計算をデータに移動させる」という理念です。データローカリティには3つのレベルがあります：ノードローカル（計算とデータが同じノード上）、ラックローカル（計算とデータが同じラック上）、クラック間（計算とデータが異なるラック上）。

Hadoopはタスクスケジューリング時にノードローカルのデータを優先し、そのノードが利用できない場合はラックローカルのデータにフォールバックします。この最適化を実現するために、NameNodeは各ブロックの場所情報をResourceManagerに報告し、ResourceManagerはタスクスケジューリング時にこの情報を参照して最適な決定を行います。データローカリティは大規模データ処理に非常に重要であり、クラスター全体のパフォーマンスを大幅に向上させることができます。

## ユースケース

Hadoopは、ビッグデータ処理のさまざまな分野に広く使用されています。データ分析分野では、Hadoopを使用して大量のログデータ、ユーザ行動データ、ビジネストランザクションデータを処理および分析し、企業が大量データから価値のある情報を抽出するのに役立ちます。データウェアハウス分野では、Hadoopは従来のデータウェアハウスの補足として機能し、低コストのストレージと分析機能で構造化データと非構造化データを処理できます。

機械学習分野では、Hadoopは大規模な特徴エンジニアリングとモデルトレーニングのインフラストラクチャとして機能し、分散機械学習アルゴリズムをサポートします。データレイクアーキテクチャでは、HDFSはさまざまなソースからの生データを保存するための安価で信頼性の高いデータストレージ層を提供します。リアルタイム処理については、Hadoop自体はリアルタイムコンピューティングには向いていませんが、Apache SparkやApache Flinkなどのストリーミングフレームワークと組み合わせて、ほぼリアルタイムのデータ処理和分析を実現できます。

Hadoopは他の多くのビッグデータ技術の基盤でもあります。Apache Spark、Apache Hive、Apache Pigなどの一般的な分散コンピューティングフレームワークは、Hadoopの上に構築されているか、Hadoopと深く統合されています。Hadoopの原則と実践をマスターすることは、他のビッグデータ技術を理解し、使用するために非常に重要です。

## 発展の歴史

Hadoopは2006年の誕生以来、複数のメジャーソングバーションの反復を経験しました。Hadoop 1.0時代、MapReduceはリソーススケジューリングとタスク実行の両方を担当し、アーキテクチャは比較的シンプルでしたが拡張性に限界がありました。Hadoop 2.0ではYARNが導入され、リソース管理与タスク実行が分離され、システムの拡張性と柔軟性が大幅に向上し、より多くの種類のワークロードをサポートするようになりました。

Hadoop 3.0には、複数のNameNodeサポート（HA Federation）、コンテナ化された実行環境、GPUアクセラレーションなど、多くの重要な改善がもたらされました。Hadoop 3.xはまた、ストレージ効率を最適化し、データの信頼性を確保しながらストレージオーバーヘッドを大幅に削減できるイージャーコードベースのデータ保護メカニズムを提供しています。現在、Hadoopエコシステムは引き続き発展と改善を続けており、新しい機能と最適化が常に導入されています。

Hadoopの発展は、巨大で活発なエコシステムも生み出しました。Apache HiveはSQLに似たクエリ言語を提供し、SQLに精通したユーザーがビッグデータを簡単に処理できます。Apache Pigは高水準のデータフロー言語を提供し、複雑なデータ処理タスクの開発を簡素化します。Apache HBaseはランダム読み取り/書き込みをサポートする列指向の分散データベースを提供します。Apache SqoopはリレーショナルデータベースとHDFS間のデータ移行を簡素化します。これらのプロジェクト togetherは、完全なビッグデータ処理ソリューションを形成しています。
