# Hadoop 常见问题

## HDFS 相关问题

### HDFS 存储空间不足怎么办

HDFS 存储空间不足是一个常见问题，可能由多种原因导致。首先检查 DataNode 的磁盘使用情况：

```bash
hdfs dfsadmin -report
```

如果确实空间不足，可以采取以下措施：删除不再需要的历史数据；使用 hdfs balancer 均衡数据分布，释放部分节点；增加新的 DataNode 节点扩展集群存储容量；调整副本数，减少存储开销（需权衡数据安全性）。

```bash
# 删除文件
hdfs dfs -rm -r /user/olddata

# 均衡数据
hdfs balancer -threshold 10
```

### NameNode 内存不足如何处理

NameNode 将所有元数据存储在内存中，如果文件数量过多，可能导致内存不足。可以通过以下方式缓解：增加 NameNode 的内存配置，在 hadoop-env.sh 中设置 HADOOP_NAMENODE_OPTS 的 -Xmx 参数；启用 NameNode 快照功能，定期创建只读副本以备份重要数据；升级到 Hadoop 2.x 或更高版本，它们对元数据管理有更好的优化。

```bash
# 在 hadoop-env.sh 中设置
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### DataNode 启动失败怎么办

DataNode 启动失败可能由多种原因导致。首先检查 DataNode 日志：

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

常见原因和解决方案包括：磁盘空间不足，需要清理磁盘或添加新磁盘；权限问题，确保 Hadoop 用户对数据目录有读写权限；集群 ID 不一致，如果是新加入的 DataNode，需要在 DataNode 上执行 hdfs namenode -upgrade 或删除 storage 目录后重新格式化。

```bash
# 检查 DataNode 目录权限
ls -la /usr/local/hadoop/data/

# 删除 DataNode 的 storage 目录并重新启动
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### 如何恢复误删的文件

HDFS 提供了回收站功能，可以在删除文件后恢复。首先确认回收站是否启用：

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

如果发现误删，可以从回收站恢复：

```bash
# 恢复文件
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

如果回收站已被清空或过了保留时间，数据将无法恢复。因此，日常操作中建议：启用回收站功能（默认启用）；重要数据做好备份；删除前仔细确认。

### HDFS 块损坏如何处理

数据块可能因磁盘故障或网络问题而损坏。HDFS 有内置的数据完整性检测机制，发现损坏块时会自动从其他副本复制数据进行恢复。手动检查和修复：

```bash
# 检查文件系统完整性
hdfs fsck / -files -blocks -locations

# 删除损坏的块文件
hdfs fsck / -delete
```

## MapReduce 相关问题

### 作业运行缓慢怎么办

MapReduce 作业运行缓慢可能由多种因素导致。首先查看作业历史日志，分析各个阶段的耗时：

```bash
# 查看作业详情
yarn logs -applicationId application_xxx -logFiles *

# 在 JobHistory Server 查看
mapred job -history job_history_file
```

常见优化方向包括：增加 Reduce 任务数量以提高并行度；调整 JVM 堆内存大小避免 OOM；启用 Combiner 减少网络传输数据量；优化数据倾斜，使用自定义分区器平衡负载；选择合适的文件格式（Parquet、ORC 等）。

### 如何处理数据倾斜

数据倾斜是指数据在 Reduce 阶段分布不均，导致部分 Reduce 任务运行时间远超其他任务。解决方案包括：使用自定义分区器，根据业务特点重新分配数据；添加随机前缀使数据先分散到多个 Reduce，再二次聚合；对 Key 进行加盐处理，聚合后再去除前缀。

```java
// 自定义分区器示例
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // 根据业务逻辑重新分配
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### MapReduce 作业失败如何排查

作业失败时，首先查看错误日志确定失败原因：

```bash
# 查看作业状态
yarn application -status application_xxx

# 查看任务失败原因
mapred job -status job_id
```

常见失败原因包括：Java 内存溢出（OOM），增加 mapreduce.map.memory.mb 和 mapreduce.reduce.memory.mb；输入数据格式错误，检查数据是否符合预期格式；自定义代码异常，检查代码逻辑和异常信息；网络超时，增加 mapreduce.task.timeout 参数。

## YARN 相关问题

### ResourceManager 无法启动怎么办

ResourceManager 启动失败通常由配置错误或端口冲突导致。检查日志：

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

常见问题和解决方案：端口被占用，修改 yarn-site.xml 中的 ResourceManager 端口； ZooKeeper 配置错误（如果使用 HA），检查 zkfc 相关配置；内存不足，确保机器有足够的物理内存；配置文件错误，仔细检查 yarn-site.xml 和 capacity-scheduler.xml。

```bash
# 查看端口占用
netstat -tlnp | grep 8032
```

### 集群资源利用率低怎么办

集群资源利用率低可能由以下原因导致：任务提交模式不合理，作业过小导致调度开销大于计算时间；调度器配置不当，队列资源分配不合理；数据本地性差，输入数据与计算节点不在同一位置。

优化建议：批量提交小作业；合理配置队列容量和优先级；优化数据布局，提高数据本地性；使用合适的调度器（Capacity Scheduler 或 Fair Scheduler）。

### 如何限制用户资源使用

可以通过 YARN 队列和访问控制限制用户资源使用：

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- 单用户最大可用队列资源的倍数 -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- 队列最大运行作业数 -->
</property>
```

## 集群运维问题

### 如何监控 Hadoop 集群

Hadoop 提供了多种监控方式：Web UI（NameNode 9870 端口、ResourceManager 8088 端口）；JMX（Java Management Extensions）监控；Ganglia、Prometheus 等外部监控系统；命令行工具（hdfs dfsadmin、yarn node 等）。

推荐使用 Prometheus + Grafana 搭建完整的监控平台，收集 Hadoop 的 JMX 指标并可视化展示。

### 如何进行集群升级

Hadoop 集群升级需要谨慎操作，建议按以下步骤进行：在测试环境先验证新版本的兼容性；备份 NameNode 的元数据和 HDFS 数据；逐一升级 DataNode 节点，最后升级 NameNode；使用 rolling upgrade 进行滚动升级，减少停机时间。

```bash
# 滚动升级 DataNode
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# 升级 Hadoop
tar -xzf hadoop-new.tar.gz -C /usr/local/
# 启动 DataNode
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### NameNode 高可用如何配置

NameNode HA 可以避免 NameNode 单点故障。配置步骤包括：在 hdfs-site.xml 中配置两个 NameNode；配置共享编辑目录（通常使用 ZooKeeper 或 NFS）；启动 ZooKeeper Failover Controller (ZKFC)。

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

配置完成后，使用以下命令初始化 ZKFC：

```bash
hdfs zkfc -formatZK
```

### 如何处理节点下线

节点下线需要谨慎处理，确保数据不丢失。使用 decommission 功能让节点安全下线：

```bash
# 在 workers 文件中添加要下线的节点到 exclude 文件
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# 刷新节点列表
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

等待数据复制完成后，节点会自动进入 decommissioned 状态，确认数据已复制到其他节点后再关闭节点。

## 开发相关问题

### 如何调试 MapReduce 程序

MapReduce 程序调试比较困难，推荐以下方法：在本地模式下先测试逻辑；使用 Counter 记录关键指标；添加日志输出，在 Map 或 Reduce 方法中打印关键变量值；使用 MRUnit 进行单元测试。

```java
// 使用 Counter 计数
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### 如何选择 InputFormat 和 OutputFormat

InputFormat 决定了如何读取输入数据：TextInputFormat（默认，按行读取）；KeyValueTextInputFormat（按制表符分割）；SequenceFileInputFormat（二进制格式）；DBInputFormat（从数据库读取）。

OutputFormat 决定了如何输出结果：TextOutputFormat（默认，文本格式）；SequenceFileOutputFormat（二进制格式）；NullOutputFormat（不输出文件）；DBOutputFormat（输出到数据库）。

### 如何处理小文件问题

大量小文件会导致 NameNode 内存压力增大和处理效率降低。解决方案包括：使用 Hadoop Archives (HAR) 将小文件打包；使用 SequenceFile 合并小文件；使用 HBase 存储大量小文件；调整块大小，减少小文件数量。
