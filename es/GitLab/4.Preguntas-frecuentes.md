# GitLab Preguntas Frecuentes

## Problemas de Instalación

### ¿Qué Hacer Si la Instalación de GitLab Falla?

El fallo de instalación generalmente es causado por algunas razones comunes. Primero, verifique si el sistema cumple con los requisitos mínimos, incluyendo versión del sistema operativo, memoria y espacio en disco. Si la instalación usando el script autohospedado falla, intente instalar manualmente los paquetes de dependencia y luego instalar paso a paso.

Vea los registros de instalación para obtener información detallada de errores:

```bash
# Ubuntu/Debian
sudo apt-get install -y gitlab-ce 2>&1 | tee /tmp/gitlab-install.log

# Ver registros
sudo tail -f /var/log/gitlab/install.log
```

Errores comunes incluyen: conflictos de puertos (puertos 80, 443, 22 están ocupados), memoria insuficiente, problemas de certificados SSL, etc. Si hay un conflicto de puertos, detenga el servicio que ocupa el puerto o modifique la configuración de GitLab para usar otros puertos.

### ¿Cómo Modificar la Dirección de Acceso de GitLab?

Si necesita modificar la dirección de acceso de GitLab después de la instalación, edite el parámetro external_url en /etc/gitlab/gitlab.rb, luego reconfigurar:

```bash
sudo nano /etc/gitlab/gitlab.rb
# Modificar external_url
sudo gitlab-ctl reconfigure
```

Después de la modificación, asegúrese de que la resolución DNS o el archivo hosts local estén correctamente configurados para que la nueva dirección sea accesible normalmente. Si cambia de HTTP a HTTPS, también necesita configurar certificados SSL.

## Problemas de Uso

### ¿Qué Hacer Si la Conexión de Clave SSH Falla?

El fallo de conexión de clave SSH puede ser causado por múltiples razones. Primero, confirme que la clave pública se haya agregado correctamente en las claves SSH de la cuenta de GitLab:

```bash
# Ver clave pública local
cat ~/.ssh/id_ed25519.pub

# Probar conexión
ssh -T git@gitlab.example.com
```

Si la prueba de conexión falla, verifique la configuración del cliente SSH:

```bash
# Editar config SSH
nano ~/.ssh/config

# Agregar lo siguiente
Host gitlab.example.com
  HostName gitlab.example.com
  User git
  IdentityFile ~/.ssh/id_ed25519
  IdentitiesOnly yes
```

Asegúrese de que la URL del repositorio remoto use formato SSH (git@gitlab.example.com:group/project.git), no formato HTTPS.

### ¿Qué Hacer Si un Merge Request No Se Puede Fusionar?

Los Merge Requests pueden no fusionarse debido a las siguientes razones: existen conflictos, pruebas CI/CD fallaron, no se cumplen las reglas de rama protegida, faltan aprobaciones, etc. La solución es investigar uno por uno:

- Conflictos de fusión: Haga clic en "Actualizar Rama" en la página de Merge Request, fusione la rama objetivo en la rama fuente, resuelva los conflictos, luego envíe de nuevo
- Falla de CI: Vea el registro del trabajo fallido, solucione el problema, luego ejecute el pipeline de nuevo
- Rama protegida: Contacte al administrador del proyecto para modificar la configuración de ramas protegidas, o tenga un miembro con permisos fusionar

### ¿Cómo Solucionar Fallos de Ejecución de Pipelines CI/CD?

Cuando un pipeline falla, primero vea el registro del trabajo fallido en la página de CI/CD → Pipelines. El registro mostrará información específica de errores y razones de falla. Problemas comunes incluyen: fallo de descarga de dependencias, errores de comandos de script, fallos de casos de prueba, etc.

Si el problema es difícil de localizar, puede agregar pasos de depuración en la configuración del pipeline, como imprimir variables de estado, verificar estado de archivos antes y después de la instalación, etc. También puede usar la función de depuración interactiva para iniciar un shell interactivo en trabajos de pipeline para investigación.

## Problemas de Rendimiento

### GitLab Accede Muy Lento, ¿Qué Hacer?

El acceso lento puede ser causado por recursos insuficientes del servidor, latencia de red o configuración inadecuada. Primero verifique la carga del servidor:

```bash
sudo gitlab-ctl status
top -bn1 | head -20
```

Si el uso de CPU o memoria es muy alto, puede necesitar actualizar la configuración del servidor u optimizar la configuración de GitLab. También puede verificar el rendimiento de la base de datos:

```bash
sudo gitlab-rake gitlab:git:fsck  # Verificar integridad del repositorio Git
sudo gitlab-rake db:index_cache_daily  # Optimizar caché de base de datos
```

Para problemas de red, considere usar CDN para acelerar la entrega de recursos estáticos, o implementar GitLab en ubicaciones de red más cercanas a los usuarios.

### ¿Cómo Optimizar el Rendimiento de GitLab?

Optimizar el rendimiento de GitLab puede abordarse desde múltiples aspectos. La optimización de la base de datos es importante. El mantenimiento regular puede mejorar significativamente el rendimiento:

```bash
# Limpiar base de datos
sudo gitlab-rake gitlab:db:analyze
sudo gitlab-rake gitlab:db:optimize

# Limpiar archivos no usados
sudo gitlab-rake gitlab:cleanup:builds
```

Para repositorios grandes, puede habilitar grupos de objetos Git para reducir el uso de disco, o configurar Git LFS para almacenar archivos grandes. Configurar razonablemente el almacenamiento en caché de CI/CD puede acelerar las construcciones.

## Problemas de Seguridad

### ¿Cómo Configurar Certificados SSL?

GitLab soporta configuración automática de SSL con Let's Encrypt, o puede usar certificados personalizados. Primero edite el archivo de configuración:

```ruby
# Configuración automática de Let's Encrypt
letsencrypt['enable'] = true
external_url 'https://gitlab.example.com'

# Certificado personalizado
nginx['ssl_certificate'] = "/etc/gitlab/ssl/server.crt"
nginx['ssl_certificate_key'] = "/etc/gitlab/ssl/server.key"
```

Ejecute sudo gitlab-ctl reconfigure para aplicar la configuración. Si usa certificados autofirmados, los clientes necesitan confiar en el certificado para acceder a través de HTTPS.

### ¿Cómo Respaldar y Restaurar GitLab?

Los respaldos regulares son importantes para prevenir pérdida de datos. GitLab proporciona una herramienta de respaldo incorporada:

```bash
# Crear respaldo
sudo gitlab-backup create

# Ubicación de archivos de respaldo
ls -la /var/opt/gitlab/backups/
```

Los archivos de respaldo incluyen bases de datos, archivos subidos y repositorios Git. Para restaurar: detenga el servicio de GitLab, copie el archivo de respaldo a la ubicación predeterminada, luego ejecute el comando de restauración:

```bash
sudo gitlab-ctl stop
sudo gitlab-backup restore BACKUP=timestamp
sudo gitlab-ctl reconfigure
sudo gitlab-ctl start
```

Se recomienda sincronizar automáticamente archivos de respaldo a almacenamiento remoto, como servicios de almacenamiento de objetos u otro servidor.

## Problemas de Actualización

### ¿Qué Hacer Si Appearen Errores Después de Actualizar?

Después de actualizar, pueden ocurrir errores de compatibilidad o anomalías funcionales. Primero verifique el estado de GitLab:

```bash
sudo gitlab-ctl status
sudo gitlab-rake gitlab:check
```

Vea los registros de actualización para entender errores específicos:

```bash
sudo tail -100 /var/log/gitlab/gitlab-rails/production.log
```

Problemas comunes incluyen: fallo de migración de base de datos, errores de sintaxis de archivo de configuración, dependencias faltantes, etc. Si es un problema de migración de base de datos, puede necesitar corregir manualmente:

```bash
sudo gitlab-rake db:migrate
```

Se recomienda respaldar datos antes de actualizar y leer cuidadosamente la documentación oficial de actualización para entender cambios importantes entre versiones.

### ¿Cómo Actualizar Segur Between Diferentes Versiones?

Actualizar GitLab debe seguir límites de salto de versión, saltando no más de una versión mayor a la vez. Por ejemplo, actualice de 15.x a 16.x, no directamente de 15.x a 17.x. Primero actualice a la versión intermedia, confirme operación normal, luego continúe actualizando.

Ejecute los siguientes pasos antes de actualizar: respaldar datos, verificar problemas de compatibility, leer notas de lanzamiento. Mantenga energía y red estables durante el proceso de actualización, y verifique que todas las funciones sean normales después de actualizar.
