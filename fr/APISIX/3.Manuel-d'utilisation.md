# Manuel d'utilisation

Ce manuel détaille les fonctionnalités principales et les méthodes d'utilisation d'Apache APISIX, notamment la configuration du routage, la gestion des upstream, l'équilibrage de charge, la limitation de débit et l'authentification.

## Concepts de base

### Route (Routage)

Une Route est le chemin d'accès aux services upstream. Dans APISIX, une route fait d'abord correspondre les requêtes clientes selon des règles prédéfinies, puis charge et exécute les plugins appropriés, et enfin transmet la requête à un service upstream spécifique.

La route la plus simple ne contient que le chemin de correspondance et l'adresse upstream.

### Upstream

Un Upstream est un ensemble de nœuds partageant les mêmes fonctionnalités, représentant une abstraction des hôtes virtuels. Un Upstream peut effectuer un équilibrage de charge sur plusieurs nœuds de service selon des règles préconfigurées.

### Consumer (Consommateur)

Un Consumer désigne l'application ou le développeur qui utilise l'API. Dans APISIX, un consommateur nécessite un nom unique au niveau global et doit être associé à un plugin d'authentification.

### Plugin

APISIX propose un système de plugins riche pour étendre les fonctionnalités de la passerelle. Les plugins peuvent être configurés au niveau de la route, du consommateur ou de manière globale.

## Configurer le routage

Vous pouvez facilement créer des routes via l'API d'administration. L'exemple suivant crée une route qui transfère les requêtes vers httpbin.org :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "getting-started-ip",
  "uri": "/ip",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:80": 1
    }
  }
}'
```

En cas de succès, la réponse sera `HTTP/1.1 201 Created`.

### Vérifier la route

```bash
curl "http://127.0.0.1:9080/ip"
```

En temps normal, vous recevrez quelque chose comme :

```json
{
  "origin": "183.94.122.205"
}
```

### Règles de correspondance des routes

APISIX prend en charge plusieurs types de correspondance :

**Correspondance URI :**

```json
{
  "uri": "/api/v1/*",
  "uri": ["/api/v1/*", "/api/v2/*"]
}
```

**Correspondance Host :**

```json
{
  "host": "example.com",
  "host": ["example.com", "*.example.com"]
}
```

**Correspondance par méthode HTTP :**

```json
{
  "methods": ["GET", "POST"]
}
```

**Correspondance par en-tête :**

```json
{
  "headers": {
    "Content-Type": "application/json",
    "X-API-Key": "*"
  }
}
```

## Équilibrage de charge

APISIX prend en charge plusieurs algorithmes d'équilibrage de charge : round robin (tourniquet), round robin pondéré, nombre minimal de connexions, etc.

### Configurer l'équilibrage pondéré

Créez une route avec plusieurs services upstream :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "load-balance-route",
  "uri": "/headers",
  "upstream": {
    "type": "roundrobin",
    "nodes": {
      "httpbin.org:443": 1,
      "mock.api7.ai:443": 1
    },
    "pass_host": "node",
    "scheme": "https"
  }
}'
```

**Explication des paramètres :**

- `type` : Type d'algorithme d'équilibrage, supporte `roundrobin` (tourniquet), `weighted_roundrobin` (pondéré), `least_conn` (minimal connexions)
- `nodes` : Nœuds du service upstream, au format `{"host:port": poids}`
- `pass_host` : Défini à `node` pour transmettre les en-têtes de requête à l'upstream
- `scheme` : Protocole upstream, supporte `http` et `https`

### Vérifier l'équilibrage de charge

Générez 100 requêtes pour tester la répartition :

```bash
hc=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/headers" -sL | grep "httpbin" | wc -l); echo httpbin.org: $hc, mock.api7.ai: $((100 - $hc))
```

Les requêtes seront réparties de manière quasi-égale entre les deux services upstream.

## Limitation de débit

APISIX offre une fonctionnalité flexible de limitation de débit via le plugin `limit-count`.

### Activer la limitation

Ajoutez le plugin de limitation à une route existante :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "count": 2,
      "time_window": 10,
      "rejected_code": 503
    }
  }
}'
```

**Explication des paramètres :**

- `count` : Nombre maximum de requêtes autorisées dans la fenêtre de temps
- `time_window` : Durée de la fenêtre de temps (en secondes)
- `rejected_code` : Code de statut HTTP renvoyé en cas de dépassement

Cette configuration limite le trafic entrant à 2 requêtes maximum toutes les 10 secondes.

### Vérifier la limitation

Générez 100 requêtes pour tester :

```bash
count=$(seq 100 | xargs -I {} curl "http://127.0.0.1:9080/ip" -I -sL | grep "503" | wc -l); echo "200": $((100 - count)), "503": $count
```

Résultat : 2 requêtes réussies (200), 98 requêtes rejetées (503).

### Désactiver la limitation

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

### Configuration avancée

**Utiliser des variables :**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr"
    }
  }
}
```

**Utiliser un cluster Redis :**

```json
{
  "plugins": {
    "limit-count": {
      "count": 100,
      "time_window": 60,
      "key": "remote_addr",
      "policy": "redis",
      "redis_host": "127.0.0.1",
      "redis_port": 6379,
      "redis_password": "",
      "redis_database": 1
    }
  }
}
```

## Authentification

APISIX prend en charge plusieurs méthodes d'authentification : Key Auth, Basic Auth, JWT, OAuth2, etc.

### Authentification par clé (Key Auth)

#### Créer un consommateur

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/consumers" -X PUT -d '
{
  "username": "tom",
  "plugins": {
    "key-auth": {
      "key": "secret-key"
    }
  }
}'
```

> ⚠️ En production, utilisez une clé complexe.

#### Activer l'authentification par clé sur une route

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/getting-started-ip" -X PATCH -d '
{
  "plugins": {
    "key-auth": {}
  }
}'
```

#### Vérifier l'authentification

**Requête sans clé :**

```bash
curl -i "http://127.0.0.1:9080/ip"
```

Réponse : `HTTP/1.1 401 Unauthorized`

**Avec une clé erronée :**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: wrong-key'
```

Réponse : `HTTP/1.1 401 Unauthorized`

**Avec la bonne clé :**

```bash
curl -i "http://127.0.0.1:9080/ip" -H 'apikey: secret-key'
```

Réponse : `HTTP/1.1 200 OK`

### Autres méthodes d'authentification

**Authentification JWT :**

```json
{
  "plugins": {
    "jwt-auth": {
      "key": "user-key",
      "secret": "my-secret-key"
    }
  }
}
```

**Authentification Basic :**

```json
{
  "plugins": {
    "basic-auth": {
      "hide_credentials": false
    }
  }
}
```

**Authentification HMAC :**

```json
{
  "plugins": {
    "hmac-auth": {
      "secret_key": "my-secret-key"
    }
  }
}
```

## Gouvernance des services

### Déploiement progressif (Canary)

APISIX supporte les déploiements progressifs et canary basés sur le poids :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "canary-route",
  "uri": "/api/*",
  "upstream": {
    "type": "weighted_roundrobin",
    "nodes": [
      {"host": "v1.example.com", "port": 80, "weight": 90},
      {"host": "v2.example.com", "port": 80, "weight": 10}
    ]
  }
}'
```

### Réécriture de requêtes

Utilisez le plugin `proxy-rewrite` pour réécrire les requêtes :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes" -X PUT -d '
{
  "id": "rewrite-route",
  "uri": "/old-path/*",
  "plugins": {
    "proxy-rewrite": {
      "uri": "/new-path/$uri",
      "headers": {
        "X-New-Header": "value"
      }
    }
  },
  "upstream": {
    "nodes": {
      "backend.example.com:80": 1
    }
  }
}'
```

### Cache des réponses

Utilisez le plugin `proxy-cache` pour mettre en cache les réponses upstream :

```json
{
  "plugins": {
    "proxy-cache": {
      "cache_key": ["uri", "query_string"],
      "cache_http_status": [200, 301, 404],
      "cache_ttl": 60
    }
  }
}
```

## Configuration des plugins

### Ordre d'exécution des plugins

Vous pouvez spécifier la phase d'exécution des plugins dans la configuration `plugins` :

- `rewrite` : Phase de réécriture de requête
- `access` : Phase de contrôle d'accès
- `before_proxy` : Phase avant代理
- `header_filter` : Phase de filtrage des en-têtes de réponse
- `body_filter` : Phase de filtrage du corps de réponse
- `log` : Phase de journalisation

### Désactiver un plugin

Définissez `_meta.disable` à `true` :

```bash
curl -i "http://127.0.0.1:9180/apisix/admin/routes/route-id" -X PATCH -d '
{
  "plugins": {
    "limit-count": {
      "_meta": {
        "disable": true
      }
    }
  }
}'
```

## Surveillance et observabilité

### Intégration Prometheus

```json
{
  "plugins": {
    "prometheus": {
      "prefer_name": true
    }
  }
}
```

Accéder aux métriques :

```bash
curl "http://127.0.0.1:9091/prometheus"
```

### Journalisation

APISIX supporte plusieurs types de sortie de logs :

**Logs HTTP :**

```json
{
  "plugins": {
    "http-logger": {
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

**Logs d'erreurs :**

```json
{
  "plugins": {
    "error-log-logger": {
      "level": "ERROR",
      "uri": "http://log-server:8080/logs"
    }
  }
}
```

## Bonnes pratiques

1. **Utilisez des ID explicites** : Nommez vos routes et upstream de manière descriptive
2. **Configurez des timeout appropriés** : Adaptez les paramètres de timeout au temps de réponse de vos services upstream
3. **Activez les contrôles de santé** : Configurez des health checks actifs et passifs pour les services critiques
4. **Configurez une limitation adaptée** : Définissez des règles de limitation selon les besoins métier et les capacités serveur
5. **Activez l'authentification** : L'authentification est obligatoire en production
6. **Utilisez TLS** : Les interfaces exposées publiquement doivent utiliser HTTPS
7. **Surveillez les indicateurs clés** : Configurez Prometheus avec des alertes
