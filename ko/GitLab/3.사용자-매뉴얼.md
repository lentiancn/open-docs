# GitLab 사용자 매뉴얼

## 계정 관리

### 등록 및 로그인

처음으로 GitLab을 사용하려면 로그인 페이지에서 "등록"을 클릭하여 계정을 만드세요. 등록에는 사용자 이름, 이메일 주소 및 비밀번호를 제공해야 합니다. 사용자 이름 선택은 신중하게 해야 합니다. 생성 후에는 변경할 수 없습니다. 등록 후 확인 이메일을 받게 됩니다. 이메일의 링크를 클릭하여 이메일 확인을 완료하세요.

로그인 후 개인 설정에서 아바타, 이름, 전체 이름 및 자기 소개를 포함한 프로필을 개선할 수 있습니다. 프로젝트 멤버 목록 및 커밋 기록에 표시되므로 완전한 정보를하는 것을 권장합니다. 또한 명령줄로 코드를 푸시하기 위해 SSH 키를 바인딩할 수 있습니다.

### SSH 키 구성

SSH 키를 사용하면 매번 사용자 이름과 비밀번호를 입력하지 않고 GitLab 저장소로 안전하게 인증할 수 있습니다. 먼저 SSH 키 쌍을 생성하세요:

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

생성된 공개 키는 ~/.ssh/id_ed25519.pub 파일에 있습니다. GitLab에 로그인하고 사용자 설정으로 이동하여 "SSH 키" 탭을 클릭하고 공개 키 내용을 텍스트 상자에 붙여넣고 저장하세요. 키를 추가하면 SSH를 사용하여 저장소를 클론할 수 있습니다:

```bash
git clone git@gitlab.example.com:username/project.git
```

## 프로젝트 작업

### 프로젝트 만들기

로그인 후 상단 탐색 모음에서 "새 프로젝트" 버튼을 클릭하여 새 프로젝트를 만드세요. 빈 프로젝트를 만들거나 템플릿에서 만들거나 기존 프로젝트를 가져올 수 있습니다. 생성할 때 프로젝트 이름, 설명, 가시성 수준(공개, 내부, 비공개) 및 기본 브랜치 이름을 설정해야 합니다.

프로젝트 이름은 영어와 하이픈을 사용하여하고 명확해야 합니다. 비공개 프로젝트는 프로젝트 멤버만 액세스할 수 있습니다. 내부 프로젝트는 기업 내 모든 로그인 사용자에게 표시됩니다. 공개 프로젝트는 모든 사람에게 표시됩니다. 생성 후 프로젝트 설정 페이지에서 이러한 옵션을 수정할 수 있습니다.

### 브랜치 관리

GitLab은 기본으로 main을 주요 브랜치로 사용합니다(이전 버전에서는 master 사용). 프로젝트 페이지에서 "브랜치"를 클릭하여 모든 브랜치를 보고 관리하세요. 새 브랜치를 생성하는 방법: 브랜치 페이지에서 "새 브랜치"를 클릭하거나 Git 명령을 사용:

```bash
git checkout -b feature/new-feature
```

보호된 브랜치는 팀 협업을 위한 중요한 기능입니다. 프로젝트 설정 → 저장소 → 보호된 브랜치로 이동하여 특정 브랜치로 푸시하거나 병합할 수 있는 사용자를 설정하세요. main 브랜치를 보호하고 특정 역할 또는 메인테이너만 코드를 병합할 수 있도록 허용하는 것이 권장됩니다.

### 병합 요청

병합 요청은 GitLab에서 코드 협업의 핵심 기능입니다. 기능 개발을 완료하고 브랜치를 푸시한 후 프로젝트 페이지에서 "병합 요청" → "새 병합 요청"을 클릭하세요. 소스 브랜치와 대상 브랜치를 선택하고 제목과 설명을 작성한 후 만드세요.

병합 요청 페이지는 파일 변경 및 커밋 히스토리를 포함한 소스 브랜치와 대상 브랜치의 차이점을 보여줍니다. 팀 멤버는 여기에서 코드 리뷰를 수행하고 줄 단위로 댓글을 추가할 수 있습니다. 리뷰가 완료되고 병합 조건(예: CI 테스트 통과, 승인 획득)이 충족되면 브랜치를 병합할 수 있습니다.

## CI/CD 파이프라인

### .gitlab-ci.yml 구성

GitLab CI/CD는 프로젝트 루트의 .gitlab-ci.yml 파일을 사용하여 파이프라인을 정의합니다. 이는 각 빌드 단계와 작업을 설명하는 YAML 구성 파일입니다. 간단한 예:

```yaml
stages:
  - build
  - test
  - deploy

build:
  stage: build
  script:
    - npm install
    - npm run build
  artifacts:
    paths:
      - dist/

test:
  stage: test
  script:
    - npm run test
  coverage: '/Coverage: \d+\.\d+%/'

deploy:
  stage: deploy
  script:
    - ./deploy.sh
  only:
    - main
```

이 구성은 세 가지 단계를 정의합니다: build는 프로젝트 빌드, test는 테스트 실행, deploy는 애플리케이션 배포입니다. 아티팩트는 작업 간에 빌드 아티팩트를 전달하는 데 사용되며, only는 배포 작업을 main 브랜치에서만 실행하도록 지정합니다.

### GitLab Runner

GitLab Runner는 CI/CD 작업을 실행하는 에이전트입니다. GitLab은 공유 Runner를 제공하거나 자체 개인 Runner를 등록할 수 있습니다. 서버에 GitLab Runner를 설치하세요:

```bash
# 설치
curl -L https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh | sudo bash
sudo apt-get install gitlab-runner

# Runner 등록
sudo gitlab-runner register
```

등록에는 프로젝트 설정 → CI/CD → Runner에서 찾을 수 있는 GitLab 인스턴스 URL과 등록 토큰을 제공해야 합니다. Runner를 병렬로 실행하도록 구성하여 빌드 속도를 향상시킬 수 있습니다.

### 파이프라인 모니터링

코드를 푸시할 때마다 파이프라인이 자동으로 트리거됩니다. 프로젝트 → CI/CD → 파이프라인 페이지에서 모든 파이프라인 실행 상태를 볼 수 있습니다. 특정 파이프라인을 클릭하면 로그 출력, 빌드 시간 및 실패 이유를 포함한 각 작업의 실행 상태를 볼 수 있습니다.

GitLab은 CI/CD 구성을 시각적으로 생성하고 편집할 수 있는 파이프라인 편집기도 제공하여 YAML 파일을 수동으로 작성할 필요가 없습니다. 편집기는 구문 검사 및 실시간 미리보기를 제공하여 구성 오류를 줄입니다.

## 문제 추적

### 만들기 및 관리

문제 시스템은 GitLab 프로젝트 관리의 기초입니다. 프로젝트 페이지에서 "문제" → "새 문제"를 클릭하여 새 문제를 만드세요. 제목과 설명을 작성하고 특정 멤버에게 할당하거나 레이블, 마일스톤 및 기한을 설정할 수 있습니다.

문제는 레이블 또는 마일스톤을 통해 관리됩니다. 일반적인 레이블에는 bug, feature, enhancement 등이 포함됩니다. 보드 뷰를 사용하여 문제 상태를 시각화하고 문제를 열(예: 할 일, 진행 중, 완료)로 그룹화할 수 있습니다.

### 코드에 문제 연결

커밋 메시지 또는 병합 요청 설명에서 키워드 번호를 사용하면 코드가 문제에 자동으로 연결됩니다. 예:

- Fixes #123 - 문제 123을 수정하고 병합 후 자동으로 문제를 닫음
- Closes #456 - 문제 456을 닫음
- See #789 - 문제 789를 참조

이 연결 메커니즘은 팀이 작업 진행 상황을 추적하고 코드 리뷰 동안 문제의 배경과 목적을 이해하는 데 도움이 됩니다.

## 위키 및 문서

모든 GitLab 프로젝트에는 프로젝트 문서를 만들기 위한 내장 위키 기능이 있습니다. 위키 페이지는 구문 강조, 테이블 및 이미지를 지원하는 마크다운 형식으로 작성됩니다. 위키 콘텐츠는 코드 저장소와 마찬가지로 버전 제어를 지원하여 이전 버전을보고 롤백할 수 있습니다.

프로젝트에 다음 문서를 유지할 것을 권장합니다: README는 프로젝트 소개, CONTRIBUTING은 기여 가이드, 배포 가이드 및 API 문서입니다. 이 문서는 온보딩을 크게 줄이고 커뮤니티 기여를 촉진할 수 있습니다.
