# Часто задаваемые вопросы о Hadoop

## Проблемы, связанные с HDFS

### Что делать при недостаточном пространстве хранения HDFS

Недостаточное пространство хранения HDFS — распространённая проблема, которая может иметь несколько причин. Сначала проверьте использование диска DataNode:

```bash
hdfs dfsadmin -report
```

Если пространство действительно недостаточно, примите следующие меры: удалите ненужные исторические данные; используйте hdfs balancer для балансировки распределения данных и освобождения некоторых узлов; добавьте новые узлы DataNode для расширения ёмкости хранилища кластера; уменьшите коэффициент репликации для сокращения накладных расходов на хранение (необходимо учитывать безопасность данных).

```bash
# Удалить файлы
hdfs dfs -rm -r /user/olddata

# Балансировать данные
hdfs balancer -threshold 10
```

### Как обработать недостаточную память NameNode

NameNode хранит все метаданные в памяти; при слишком большом количестве файлов память может закончиться. Смягчите: увеличьте конфигурацию памяти NameNode в hadoop-env.sh, установив параметр -Xmx в HADOOP_NAMENODE_OPTS; включите функцию создания моментальных снимков NameNode для регулярного создания резервных копий важных данных; перейдите на Hadoop 2.x или более позднюю версию, которая имеет лучшую оптимизацию управления метаданными.

```bash
# Установить в hadoop-env.sh
export HADOOP_NAMENODE_OPTS="-Xmx4g -Xms4g"
```

### Что делать, если DataNode не запускается

Сбой запуска DataNode может иметь несколько причин. Сначала проверьте журналы DataNode:

```bash
cat $HADOOP_HOME/logs/hadoop-datanode-*.log
```

Распространённые причины и решения: недостаточно дискового пространства — очистите диски или добавьте новые; проблемы с разрешениями — убедитесь, что пользователь Hadoop имеет доступ к каталогам данных для чтения и записи; несовместимый ID кластера — при добавлении нового DataNode запустите hdfs namenode -upgrade на DataNode или удалите каталог хранилища и переформатируйте.

```bash
# Проверить разрешения каталога DataNode
ls -la /usr/local/hadoop/data/

# Удалить каталог хранилища DataNode и перезапустить
rm -rf /usr/local/hadoop/data/datanode/*
systemctl restart hadoop-datanode
```

### Как восстановить случайно удалённые файлы

HDFS предоставляет функцию корзины для восстановления удалённых файлов. Сначала подтвердите, что корзина включена:

```bash
hdfs dfs -ls /user/username/.Trash/Current/
```

При обнаружении случайного удаления восстановите из корзины:

```bash
# Восстановить файл
hdfs dfs -mv /user/username/.Trash/Current/filepath /user/username/
```

Если корзина была очищена или срок хранения истёк, данные невозможно восстановить. Поэтому в повседневных операциях: держите корзину включенной (включена по умолчанию); создавайте резервные копии важных данных; внимательно проверяйте перед удалением.

### Как обработать повреждённые блоки HDFS

Блоки данных могут быть повреждены из-за сбоев диска или проблем с сетью. HDFS имеет встроенные механизмы обнаружения целостности данных: при обнаружении повреждённых блоков он автоматически реплицирует данные из других реплик для восстановления. Ручная проверка и восстановление:

```bash
# Проверить целостность файловой системы
hdfs fsck / -files -blocks -locations

# Удалить повреждённые файлы блоков
hdfs fsck / -delete
```

## Проблемы, связанные с MapReduce

### Что делать, если задания выполняются медленно

Медленные задания MapReduce могут быть вызваны несколькими факторами. Сначала проверьте журналы истории заданий и проанализируйте время, затраченное на каждый этап:

```bash
# Просмотреть детали задания
yarn logs -applicationId application_xxx -logFiles *

# Просмотреть в JobHistory Server
mapred job -history job_history_file
```

Распространённые направления оптимизации: увеличьте количество задач Reduce для повышения параллелизма; настройте размер кучи JVM, чтобы избежать OOM; включите Combiner для сокращения объёма сетевой передачи; оптимизируйте перекос данных, используя пользовательские разделители для балансировки нагрузки; выберите подходящие форматы файлов (Parquet, ORC и т.д.).

### Как обработать перекос данных

Перекос данных означает, что данные неравномерно распределены на этапе Reduce, из-за чего некоторые задачи Reduce выполняются гораздо дольше других. Решения включают: используйте пользовательские разделители для перераспределения данных согласно бизнес-логике; добавьте случайный префикс, чтобы данные сначала распределились по нескольким Reduce, затем агрегируйте повторно; используйте соление ключей — удалите префикс после агрегации.

```java
// Пример пользовательского разделителя
public class SaltedPartitioner extends Partitioner<Text, Text> {
    @Override
    public int getPartition(Text key, Text value, int numPartitions) {
        // Перераспределить согласно бизнес-логике
        int hash = key.hashCode() % numPartitions;
        return Math.abs(hash);
    }
}
```

### Как устранить сбои заданий MapReduce

При сбое заданий сначала проверьте журналы ошибок, чтобы определить причину сбоя:

```bash
# Просмотреть состояние задания
yarn application -status application_xxx

# Просмотреть причины сбоя задач
mapred job -status job_id
```

Распространённые причины сбоев: Java OutOfMemory (OOM) — увеличьте mapreduce.map.memory.mb и mapreduce.reduce.memory.mb; ошибка формата входных данных — проверьте, соответствуют ли данные ожидаемому формату; исключение пользовательского кода — проверьте логику кода и сообщения об исключениях; тайм-аут сети — увеличьте параметр mapreduce.task.timeout.

## Проблемы, связанные с YARN

### Что делать, если ResourceManager не запускается

Сбой запуска ResourceManager обычно вызван ошибками конфигурации или конфликтами портов. Проверьте журналы:

```bash
cat $HADOOP_HOME/logs/yarn-resourcemanager-*.log
```

Распространённые проблемы и решения: порт уже используется — измените порт ResourceManager в yarn-site.xml; ошибка конфигурации ZooKeeper (при использовании HA) — проверьте связанную конфигурацию zkfc; недостаточно памяти — убедитесь, что машина имеет достаточно физической памяти; ошибка конфигурационного файла — внимательно проверьте yarn-site.xml и capacity-scheduler.xml.

```bash
# Проверить использование порта
netstat -tlnp | grep 8032
```

### Как обработать низкое использование ресурсов кластера

Низкое использование ресурсов кластера может быть вызвано: неразумными шаблонами отправки заданий — слишком маленькие задания приводят к тому, что накладные расходы на планирование превышают время вычисления; неправильная конфигурация планировщика — неразумное распределение ресурсов очереди; плохая локальность данных — входные данные и вычислительные узлы не в одном месте.

Предложения по оптимизации: отправляйте небольшие задания партиями; разумно настройте ёмкость и приоритеты очереди; оптимизируйте расположение данных для улучшения локальности данных; используйте подходящий планировщик (Capacity Scheduler или Fair Scheduler).

### Как ограничить использование ресурсов пользователями

Используйте очереди YARN и контроль доступа для ограничения использования ресурсов пользователями:

```xml
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.user-limit-factor</name>
    <value>1</value>  <!-- Максимальный кратный ресурсов очереди для одного пользователя -->
</property>
<property>
    <name>yarn.scheduler.capacity.root.<queue-name>.maximum-applications</name>
    <value>100</value>  <!-- Максимальное количество выполняемых заданий в очереди -->
</property>
```

## Проблемы эксплуатации кластера

### Как отслеживать кластер Hadoop

Hadoop предоставляет несколько методов мониторинга: веб-интерфейс (NameNode порт 9870, ResourceManager порт 8088); мониторинг JMX (Java Management Extensions); внешние системы мониторинга, такие как Ganglia, Prometheus; инструменты командной строки (hdfs dfsadmin, yarn node и т.д.).

Рекомендуется использовать Prometheus + Grafana для создания полной платформы мониторинга — собирайте метрики JMX Hadoop и визуализируйте их.

### Как выполнить обновление кластера

Обновление кластера Hadoop требует осторожных действий. Рекомендуемые шаги: сначала проверьте совместимость новой версии в тестовой среде; создайте резервную копию метаданных NameNode и данных HDFS; обновляйте узлы DataNode по одному, затем обновляйте NameNode; используйте постепенное обновление для сокращения времени простоя.

```bash
# Постепенное обновление DataNode
$HADOOP_HOME/bin/hdfs --daemon stop datanode
# Обновить Hadoop
tar -xzf hadoop-new.tar.gz -C /usr/local/
# Запустить DataNode
$HADOOP_HOME/bin/hdfs --daemon start datanode
```

### Как настроить высокую доступность NameNode

Высокая доступность NameNode позволяет избежать единой точки отказа NameNode. Шаги настройки: настройте два NameNode в hdfs-site.xml; настройте общий каталог редактирования (обычно с использованием ZooKeeper или NFS); запустите контроллер отказов ZooKeeper (ZKFC).

```xml
<property>
    <name>dfs.ha.automatic-failover.enabled</name>
    <value>true</value>
</property>
<property>
    <name>ha.zookeeper.quorum</name>
    <value>zk1:2181,zk2:2181,zk3:2181</value>
</property>
```

После настройки инициализируйте ZKFC:

```bash
hdfs zkfc -formatZK
```

### Как обработать вывод узла из эксплуатации

Вывод узла из эксплуатации требует осторожного обращения для обеспечения отсутствия потери данных. Используйте функцию вывода из эксплуатации для безопасного отключения узлов:

```bash
# Добавить узел в файл исключения в workers
echo "slave3" >> $HADOOP_HOME/etc/hadoop/workers.exclude

# Обновить список узлов
hdfs dfsadmin -refreshNodes
yarn rmadmin -refreshNodes
```

Дождитесь завершения репликации данных — узел автоматически перейдёт в состояние decommissioned. Подтвердите, что данные были реплицированы на другие узлы, перед выключением узла.

## Проблемы разработки

### Как отлаживать программы MapReduce

Отладка программ MapReduce затруднена. Рекомендуемые методы: сначала протестируйте логику в локальном режиме; используйте счётчики для записи важных метрик; добавьте вывод журналов — печатайте значения важных переменных в методах Map или Reduce; используйте MRUnit для модульного тестирования.

```java
// Использование счётчика
context.getCounter("MyCounters", "InvalidRecord").increment(1);
```

### Как выбрать InputFormat и OutputFormat

InputFormat определяет, как читать входные данные: TextInputFormat (по умолчанию, читает построчно); KeyValueTextInputFormat (разделяет по табуляции); SequenceFileInputFormat (двоичный формат); DBInputFormat (читает из базы данных).

OutputFormat определяет, как выводить результаты: TextOutputFormat (по умолчанию, текстовый формат); SequenceFileOutputFormat (двоичный формат); NullOutputFormat (без вывода файла); DBOutputFormat (выводит в базу данных).

### Как обработать проблему с маленькими файлами

Большое количество маленьких файлов увеличивает давление на память NameNode и снижает эффективность обработки. Решения: используйте Hadoop Archives (HAR) для упаковки маленьких файлов; используйте SequenceFile для объединения маленьких файлов; используйте HBase для хранения большого количества маленьких файлов; настройте размер блока, чтобы уменьшить количество маленьких файлов.
