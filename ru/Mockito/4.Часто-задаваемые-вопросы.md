# Часто задаваемые вопросы о Mockito

## Базовые вопросы

### В чем разница между Mockito и PowerMock

Mockito в основном используется для mockирования обычных объектов, в то время как PowerMock может создавать mock-объекты для статических методов, конструкторов, приватных методов и т.д. Однако PowerMock использует технологии манипуляции байт-кодом, которые нарушают изоляцию тестов, делая тесты ближе к интеграционным, чем к юнит-тестам. Официальная рекомендация - сначала использовать Mockito и рассматривать PowerMock только когда Mockito действительно не может обработать определенные сценарии, такие как mockирование статических методов. Начиная с Mockito 2.x, mockирование статических методов также поддерживается через mockito-inline.

### Когда следует использовать Spy вместо Mock

Используйте Spy, когда вам нужно большая часть реального поведения, но вы хотите создать mock только для нескольких методов. Spy сохраняет большую часть функциональности реального объекта, позволяя при этом создавать заглушки для конкретных методов. Mock полностью заменяет реальный объект, поэтому никакая логика реального метода не выполняется. Используйте Mock, когда вам нужен полный контроль над поведением объекта. На практике Spy используется относительно редко, поскольку в большинстве случаев использование реальных тестовых объектов или полных mock-объектов удовлетворяет требованиям.

### Можно ли использовать Mockito для финальных классов и методов

Начиная с Mockito 2.1.0, можно настроить Mockito для mockирования финальных классов и методов. Вам нужно добавить зависимость mockito-inline вместо mockito-core в ваш проект. mockito-inline включает механизм динамических подклассов для обработки финальных классов и методов. Эта функция значительно расширяет применимость Mockito, делая почти любой Java-класс mockируемым.

### Почему методы mock-объекта возвращают null

По умолчанию все методы mock-объектов возвращают null или значения по умолчанию для своих типов (int возвращает 0, boolean возвращает false, типы коллекций возвращают пустые коллекции и т.д.). Это поведение Mockito по умолчанию. Если вам нужны конкретные возвращаемые значения, вы должны явно указать их с помощью when().thenReturn().

## Вопросы по использованию

### Аннотация @Mock не работает, что делать

Аннотация @Mock требует инициализации фреймворка Mockito, чтобы вступить в силу. Убедитесь, что вы используете один из способов инициализации: MockitoAnnotations.initMocks(this), @RunWith(MockitoJUnitRunner.class) или @ExtendWith(MockitoExtension.class) (для JUnit 5) в ваших тестах. При правильной инициализации поля с аннотацией @Mock автоматически создают mock-объекты перед выполнением тестовых методов.

### Как создавать mock для void-методов

Для методов с типом возврата void используйте doThrow(), doAnswer() или doNothing() для определения поведения.

```java
doThrow(new RuntimeException("Error")).when(mockedObject).voidMethod();
doAnswer(invocation -> null).when(mockedObject).voidMethod();
doNothing().when(mockedObject).voidMethod();
```

### Как проверять аргументы вызова метода

Можно использовать сопоставители аргументов для проверки или использовать ArgumentCaptor для захвата аргументов для детальных проверок.

```java
verify(mockedList).add(anyString());

ArgumentCaptor<String> captor = ArgumentCaptor.forClass(String.class);
verify(mockedList).add(captor.capture());
assertTrue(captor.getValue().length() > 0);
```

### Как создавать mock для итераторов

Mockирование итераторов требует внимания к координации между методами hasNext() и next().

```java
Iterator<String> mockIterator = mock(Iterator.class);
when(mockIterator.hasNext()).thenReturn(true, true, false);
when(mockIterator.next()).thenReturn("first", "second");

assertTrue(mockIterator.hasNext());
assertEquals("first", mockIterator.next());
assertTrue(mockIterator.hasNext());
assertEquals("second", mockIterator.next());
assertFalse(mockIterator.hasNext());
```

### Как обрабатывать перегрузку методов

Когда класс имеет перегруженные методы, Mockito различает разные методы на основе типов и количества параметров. Убедитесь, что типы параметров в when() точно соответствуют сигнатуре метода, для которого вы хотите создать mock.

```java
when(mockedList.get(0)).thenReturn("string");      // get(int index)
when(mockedList.get(anyInt())).thenReturn("any"); // перегруженный метод
```

## Продвинутые вопросы

### Почему when().thenReturn() не работает

Распространенные причины включают: установка поведения для одного и того же метода несколько раз (действует только последнее); использование Spy и вызов реального метода; метод возвращает void (следует использовать doReturn().when()); сопоставитель аргументов используется неправильно, что приводит к несоответствию. Проверьте ваш код, чтобы убедиться, что синтаксис для установки поведения правильный.

### Как тестировать абстрактные классы

При тестировании абстрактных классов можно использовать Mockito для mockирования конкретных методов абстрактного класса или использовать Spy для сохранения некоторого реального поведения. Другой подход заключается в создании конкретного тестового подкласса для выполнения тестов.

```java
@Test
public void testAbstractClass() {
    AbstractClass mock = mock(AbstractClass.class, Mockito.CALLS_REAL_METHODS);
    when(mock.concreteMethod()).thenReturn("mocked");
}
```

### Как создавать mock для generic-методов

При mockировании generic-методов учитывайте ограничения, вызванные стиранием типов. Используйте any() или anyObject() для сопоставления generic-параметров.

```java
when(mockedObject.genericMethod(any())).thenReturn("result");
```

### Как проверять порядок вызовов

Используйте InOrder для проверки того, что вызовы методов были выполнены в определенном порядке.

```java
InOrder inOrder = inOrder(mockedList);
inOrder.verify(mockedList).add("first");
inOrder.verify(mockedList).add("second");
```

### Как частично создавать mock объектов

Используйте Spy для частичного mockирования или withSettings() для настройки поведения Mock.

```java
MyClass spy = spy(new MyClass());
doReturn("mocked").when(spy).method();
```

## Вопросы производительности

### Тесты выполняются медленно, что делать

Убедитесь, что вы не используете ненужные вызовы реальных методов. Проверьте, создаете ли вы новые Mock-объекты в каждом тесте. Для проектов, использующих много Mockito, рассмотрите возможность использования @RunWith(MockitoJUnitRunner.Silent.class) для уменьшения вывода журналов.

### Как уменьшить использование памяти Mock-объектами

Mock-объекты Mockito сами по себе не потребляют много памяти. Однако если создание большого количества mock-объектов все еще влияет на производительность, рассмотрите возможность использования аннотации @Mock с @Scope для повторного использования mock-объектов или используйте статические фабричные методы Mock.

## Вопросы отладки

### Как просмотреть журналы вызовов Mock-объекта

Можно использовать функцию ведения журнала Mockito или добавить прослушиватели для отслеживания вызовов.

```java
Mockito.framework().addListener(new VerificationListener());
```

Или используйте вывод тестов Maven/Gradle для просмотра подробной информации о вызовах mock.

### Как отлаживать, когда тесты падают

Mockito предоставляет подробные сообщения об ошибках. Проверьте различия между ожидаемыми и фактическими вызовами, показанными при ошибках проверки. Используйте функцию печати журналов Mockito для понимания поведения mock-объекта. Упрощайте тестовые случаи шаг за шагом, чтобы найти проблемы.
